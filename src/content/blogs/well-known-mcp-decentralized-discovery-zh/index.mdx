An agent 的工具列表通常是一个 JSON 文件，固定在 agent 的配置中——一组固定的 MCP 服务器 URL，每个 URL 对应一个 agent 允许调用的工具表面。新增服务器 → 编辑配置 → 重启 agent。设计很简单，这基本上是当前野外所有生产级 MCP 部署的工作方式。但这恰恰是 MCP 生态必须迈出下一步所面临的障碍：让 agent 发现某台主机上存在哪些工具，而无需任何人预先配置。

正在兴起的约定是 `.well-known/mcp` 端点——托管在任何提供 MCP 服务的主机上的一个可预测 URL，返回一个 JSON 文档，描述这些服务是什么、如何认证、以及它们暴露了哪些能力。这个想法直接移植自 OAuth 2.0（`.well-known/oauth-authorization-server`）和 OpenID Connect（`.well-known/openid-configuration`），两者都解决了同样的引导问题：客户端如何在不把知识手工写入客户端的情况下，发现服务器支持什么？

这篇文章讨论的是：为什么 agent 生态正在向 `.well-known/mcp` 汇聚；它实际上必须包含什么才能有用；它跨越了什么样的信任边界；以及设计在哪里会泄露。我会假设读者熟悉 MCP 的请求-响应形态（传输层请参见我们的 [MCP 无状态传输文章](/zh/blog/mcp-stateless-transport/)）。

## 引导问题

考虑一个应该能在 GitHub 上执行操作的 agent。目前 agent 运营者的工作流程是：

1. 找到 GitHub MCP 服务器（某个 URL，散落在某处）。
2. 把 URL 复制到 agent 的配置中。
3. 把认证凭据复制到 agent 的密钥存储中。
4. 重启 agent。

现在把这个过程乘以生产 agent 触及的几十个工具表面——代码托管、问题跟踪器、云 API、内部服务、监控仪表盘。每一个都需要手动接线。当一个服务迁移到新 URL 时，每个 agent 的配置都必须更新。当一个服务新增能力时，每个 agent 都必须被告知。这正是 web 在 DNS 和 `robots.txt` 出现之前的模式：每个客户端都持有自己手工编写的列表，记录哪里有什么。

`.well-known/mcp` 约定问了一个不同的问题。如果一个 agent 知道一个服务的主机名——比如 `github.com` 或 `linear.app`——它能否在没有任何带外配置的情况下，发现该主机提供的 MCP 表面？答案应该是：agent 获取 `https://<host>/.well-known/mcp`，得到一个描述服务器（们）的 JSON 文档，然后决定是否连接。**没有注册中心拥有这个映射。没有配置文件需要更新。服务本身是其自身 MCP 表面的事实来源。**

这不是一个注册中心。注册中心仍然存在，仍然重要——比如"哪些 MCP 服务器是可信的、经过验证的、已编入索引的"——约定不会取代那个功能。`.well-known/mcp` 取代的是**从配置文件到端点的手工接线 URL**。

## 发现文档实际上包含什么

一个最小的 `.well-known/mcp` 响应大致如下：

```json
{
  "mcp_version": "2025-06-18",
  "servers": [
    {
      "name": "github-code",
      "url": "https://mcp.github.com/v1",
      "transport": "streamable-http",
      "capabilities": ["tools", "resources"],
      "scope_hint": "read:code,write:pr",
      "auth": {
        "type": "oauth2",
        "authorization_server": "https://github.com/.well-known/oauth-authorization-server"
      },
      "fingerprint": "sha256:3a1b...e9f0"
    },
    {
      "name": "github-issues",
      "url": "https://mcp.github.com/v1/issues",
      "transport": "streamable-http",
      "capabilities": ["tools"],
      "scope_hint": "read:issues,write:issues",
      "auth": {
        "type": "oauth2",
        "authorization_server": "https://github.com/.well-known/oauth-authorization-server"
      },
      "fingerprint": "sha256:7c2d...a41b"
    }
  ],
  "server_card_format": "https://modelcontextprotocol.io/spec/server-card/v1"
}
```

有五样东西在发挥作用：

1. **`mcp_version`** — 服务器使用的协议版本。落后于此版本的 agent 可以优雅地拒绝，而不是抛出不透明的错误。
2. **`servers[].url`** 和 **`transport`** — 实际的端点和 wire 格式。主机级别的 `.well-known` 文档可以广告位于不同子路径的多个服务器，这一点很重要，因为真实服务没有一个单一的整块 MCP 表面——GitHub 有代码操作和问题操作，Linear 有项目和搜索，等等。
3. **`capabilities`** — 服务器实现了哪些 MCP 原语。只暴露 `resources` 而不暴露 `tools` 的服务器是只读表面；agent 可以根据自己需要决定是否费力去连接。
4. **`auth`** — 指向认证机制的指针。关键是这个指针又是另一个 `.well-known` 端点，所以发现是级联进行的——agent 可以沿着这条链从主机名到 MCP 服务器到 OAuth 元数据到 token 端点，无需任何预配置的知识。
5. **`fingerprint`** — 服务器能力清单的内容哈希，这样 agent 可以检测漂移。如果服务器的工具集在连接之间意外变化了，指纹会变化，agent 可以拒绝继续并等待明确的重授权。**这是防供应链攻击的部分；我稍后会展开。**

这个文档**不包含**的是详细工具模式本身。那些是在 agent 决定连接后，通过 MCP 连接获取的。发现文档刻意保持最小化——足以决定是否连接，不足以实际调用工具。这与 OAuth 元数据文档的模式匹配：它们告诉你**如何**认证，而不是**资源是什么**。

## 为什么是主机级别而不是子域级别

一个微妙的设计选择：`.well-known/mcp` 端点是按主机作用域的，而不是按子域或路径的。每个源（origin）一个文档。OAuth 2.0 和 OpenID Connect 技术上允许在路径限定的 URL（RFC 8414 §3.1 允许在 issuer 路径下有 `.well-known`）进行 issuer 作用域的元数据，但在实践中真正传播的是简单的主机根变体，同样的简洁原则在这里也适用：**应该只有一个明显的查找位置**，否则发现就不是真正的发现，只是"这里有个建议，试几个地方直到某个能用"。

主机作用域有一个后果：在同一主机上有多个独立 MCP 产品的大型组织，需要在一个文档中列出所有这些。GitHub 的问题跟踪器、Actions 和代码托管都是同一 `github.com` 上 `.well-known/mcp` 下的子服务器。**这是功能，不是 bug**——这意味着 agent 一次查找就能看到某个厂商的完整 MCP 表面，而不必探测十几个子域。

它还排除了"不想被发现的服务器端" MCP 表面。如果你的 MCP 服务器不在你主机的 `.well-known/mcp` 中列出，遵循发现约定的 agent 不会找到它，你将不得不回退到客户端侧的手工接线配置。对于大多数生产服务，这正是你想要的——`.well-known` 文档是公共 MCP 表面，私有或未列出的服务器是另一个问题，由私有注册中心处理。

## 指纹与供应链问题

这个设计中最有趣的部分，也是大多数手工替代方案遗漏的部分，是 `fingerprint` 字段。Agent 工具表面是供应链攻击的目标：如果攻击者能在 agent 不注意的情况下修改 agent 将要调用的工具集，他们就可以将任意行为注入 agent 的推理循环。一个"小的、隐蔽的"改变——重命名一个工具、添加一个新参数、改变描述——可能产生巨大的下游影响。

指纹是能力清单（工具名称、描述、参数模式、资源 URI）的哈希值，由服务器端计算并包含在发现文档中。**Agent 的契约是：**

1. **首次连接时**，获取发现文档，读取指纹，并存储它。
2. **每次后续连接时**，重新获取发现文档。如果指纹与存储的匹配，静默继续。
3. **如果指纹变了**，停止连接，并将变化呈现给用户（或策略引擎）。

这给运营者提供了一种可行的方式来限定他们对第三方 MCP 服务器的信任。**你批准的是一个特定的指纹；你批准的不是一个"服务器今天声称的任何东西"。** 语义类似于 SSH 主机密钥，这是一个刻意的类比——我们有三十年的操作者心智模型来处理"密钥变了，我应该信任这个吗？"，这比 MCP 配置文件目前隐含的开放式信任是一个更好的基础。

较弱的版本，目前可用的，是在 agent 配置中固定工具模式。问题是配置中的固定迫使操作者在每次合法更新时都要介入。指纹方法将那个循环移到了变化实际发生的时刻，而不是每次 agent 想连接的时刻，让其余时间都是自动的。

## 现有实现的样子

发现循环，大致如下，作为 agent 运行时的一部分：

```python
import httpx

async def discover_mcp(host: str) -> list[dict]:
    url = f"https://{host}/.well-known/mcp"
    async with httpx.AsyncClient(timeout=5.0) as client:
        resp = await client.get(url, headers={"Accept": "application/json"})
    if resp.status_code != 200:
        return []
    doc = resp.json()
    # Minimal schema validation
    if "mcp_version" not in doc or "servers" not in doc:
        raise ValueError(f"Invalid .well-known/mcp at {host}")
    return doc["servers"]

async def connect_with_pinning(
    host: str,
    pinned: dict[tuple[str, str], str],  # (host, server_name) -> fingerprint
) -> list["MCPClient"]:
    servers = await discover_mcp(host)
    clients = []
    for s in servers:
        key = (host, s["name"])
        fp = s.get("fingerprint")
        if key in pinned and pinned[key] != fp:
            raise FingerprintMismatch(key, pinned[key], fp)
        pinned[key] = fp
        clients.append(await mcp_client_connect(s["url"], s["transport"], s["auth"]))
    return clients
```

关于这段代码，有三点值得注意：

1. **发现调用非常便宜**。一次 HTTP GET 到 well-known 路径，JSON 解析，完成。可以按主机缓存，TTL 很短（几分钟到几小时）。
2. **指纹固定是本地状态**。Agent 运营者的配置现在是 `(host, server_name) -> fingerprint` 的映射，而不是 host 到完整工具模式的映射。映射很小，语义很清晰："我批准了这个主机上这个特定服务器的特定版本。"**按服务器固定而非按主机固定在实践中很重要**——像 `github.com` 这样的主机可以独立发布代码托管 MCP 服务器和问题 MCP 服务器。
3. **发现机制优雅降级**。不提供 `.well-known/mcp` 的主机 просто 无法被发现——agent 回退到它已有的任何预配置 URL 列表。采用发现模式不需要每个服务器同时采用它。

## 设计泄露的地方

有两个问题值得明确说出来，因为它们对任何在生产环境中实现这个的人来说很重要。

**内网发现比公共发现更难。** `github.com` 上的 `.well-known/mcp` 是一回事；`mcp.internal.acme.corp` 上的 `.well-known/mcp` 取决于 agent 能否解析并访问那个内部主机名。对于使用公共服务的公共互联网 agent，这个模式立竿见影。对于与企业内部服务通信的企业 agent，发现需要与基于 DNS 的服务发现（SRV 记录）、私有 MCP 注册中心或两者配对。**`.well-known` 文档不能取代服务发现**——它补充了服务发现，在"这个服务在哪里"的问题通过其他方式回答之后，回答"这个服务提供什么"的问题。

**指纹的信任模型比看起来弱。** 指纹是服务器计算的。控制服务器的攻擊者可以为篡改后的清单计算一个指纹，让 agent 在首次连接时接受它。**指纹检测的是"相对于先前已知状态的漂移"，而不是被入侵的初始状态。** 这与 SSH 密钥固定的信任模型相同，有相同的失败模式（首次使用时信任）。这是一个真实的约束，但不是新约束，SSH 采用的缓解措施也是如此：通过带有签名证明的注册中心进行第一次连接到任何重要事物的带外验证，或者通过人类操作员显式检查工具。

**这两个都不是致命的。它们是设计有意识地将复杂性推向上层而不是试图在一个协议中解决一切的地方。**

## 为什么这现在很重要

当前 agent 生态被困在两个坏选项之间：不可扩展的手工接线工具配置，和集中信任、拖慢迭代的集中式注册中心。`.well-known/mcp` 是中间选项——**服务器是其自身 MCP 表面的事实来源，agent 自动发现该表面，信任锚定在运营者可以审计的指纹上，没有人必须运行权威注册中心。**

我不断回想到的类比是 DNS 加上 `robots.txt`。DNS 给了我们没有人拥有的主机级发现；`robots.txt` 给了我们没有人必须手动聚合的主机级声明。Agent 生态需要同样的两个原语，而 `.well-known/mcp` 是最接近两者的东西。这个规范正在从 OAuth 2.0 元数据、OpenID Connect 发现和二十年的 web 标准实践中拼凑出来，这不是弱点。**这是这个模式将真正起作用的原因——它不需要任何人发明任何新东西，它只需要 agent 生态注意到它面临的问题是 web 二十年前解决的问题。**
