Mechanistic interpretability — the effort to reverse-engineer what neural networks actually compute — has historically been resource-constrained in a particular way. Training a sparse autoencoder on a large model's activations requires running millions of forward passes, storing massive activation datasets, and doing so for every layer you want to understand. For researchers outside the major labs, this meant working with small open models or accepting that findings might not transfer to the frontier systems where alignment actually matters.

Gemma Scope 2, released by Google DeepMind in December 2025, changes this equation for the Gemma 3 model family. It is not a paper or a benchmark — it is a pre-computed infrastructure: SAEs and Transcoders for every layer of every Gemma 3 model size (270M through 27B parameters), trained on approximately 110 PB of data with over a trillion total parameters across all the tools. The entire toolkit is released under Apache 2.0.

This matters because it makes serious mechanistic interpretability accessible without a compute budget measured in hundreds of thousands of dollars.

## What Sparse Autoencoders Do

Before getting into what's new in Gemma Scope 2, a brief recap of the core tool: Sparse Autoencoders (SAEs).

A transformer's residual stream at any layer is a high-dimensional vector (768 or 1024 or more dimensions for typical models). But representational geometry research suggests these vectors are superpositions of many more "features" — directions in activation space that correspond to human-interpretable concepts like "this is Python code," "the subject is performing an action," or "the context involves deception."

Standard PCA or probing can find some of these directions, but the superposition means features are entangled. SAEs solve this by learning a sparse, overcomplete dictionary:

```
residual_stream = Σᵢ fᵢ(x) · eᵢ
```

where `fᵢ(x)` is a scalar activation (usually ≥ 0 due to ReLU) and `eᵢ` is a learned feature direction. The sparsity constraint — most `fᵢ(x)` should be zero for any given input — forces the dictionary to find directions that only activate for specific, coherent semantic patterns.

```python
import torch
import torch.nn as nn

class SparseAutoencoder(nn.Module):
    """
    Minimal SAE for decomposing transformer residual streams.
    
    The real Gemma Scope SAEs use JumpReLU activations and Matryoshka training;
    this shows the core structure.
    """
    def __init__(self, d_model: int, n_features: int, l1_coeff: float = 1e-3):
        super().__init__()
        self.d_model = d_model
        self.n_features = n_features
        self.l1_coeff = l1_coeff
        
        # Encoder: residual stream → sparse feature activations
        self.W_enc = nn.Parameter(torch.randn(d_model, n_features) * 0.01)
        self.b_enc = nn.Parameter(torch.zeros(n_features))
        
        # Decoder: sparse features → residual stream reconstruction
        self.W_dec = nn.Parameter(torch.randn(n_features, d_model) * 0.01)
        self.b_dec = nn.Parameter(torch.zeros(d_model))
        
        # Normalize decoder columns to unit sphere (important for interpretability)
        with torch.no_grad():
            self.W_dec.data = nn.functional.normalize(self.W_dec.data, dim=1)
    
    def encode(self, x: torch.Tensor) -> torch.Tensor:
        """x: (..., d_model) → feature activations: (..., n_features)"""
        pre_acts = x @ self.W_enc + self.b_enc
        return torch.relu(pre_acts)  # Sparsity via ReLU
    
    def decode(self, f: torch.Tensor) -> torch.Tensor:
        """f: (..., n_features) → reconstructed residual: (..., d_model)"""
        return f @ self.W_dec + self.b_dec
    
    def forward(self, x: torch.Tensor) -> dict:
        f = self.encode(x)
        x_hat = self.decode(f)
        
        # Reconstruction loss + sparsity penalty
        recon_loss = (x - x_hat).pow(2).mean()
        sparsity_loss = f.abs().mean()
        total_loss = recon_loss + self.l1_coeff * sparsity_loss
        
        return {
            "loss": total_loss,
            "features": f,
            "reconstructed": x_hat,
            "l0": (f > 0).float().sum(-1).mean(),  # Average features active
        }


# Using Gemma Scope 2 SAEs via the HuggingFace hub
# pip install transformer_lens sae_lens

from sae_lens import SAE

# Load pre-trained SAE for Gemma 3 layer 12 (residual stream)
sae, cfg_dict, _ = SAE.from_pretrained(
    release="gemma-scope-2-27b-pt-res",
    sae_id="layer_12/width_16k/average_l0_71",
)

# Get activations from the model
import transformer_lens
model = transformer_lens.HookedTransformer.from_pretrained("google/gemma-3-27b")

tokens = model.to_tokens("The proof proceeds by contradiction. Assume")
logits, cache = model.run_with_cache(tokens)

# Extract residual stream at layer 12
residual = cache["resid_post", 12]  # (1, seq_len, d_model)

# Decompose into sparse features
feature_acts = sae.encode(residual)  # (1, seq_len, n_features)
print(f"Average features active per token: {(feature_acts > 0).float().mean(-1).mean():.1f}")

# Find the most active features on the final token
final_token_acts = feature_acts[0, -1, :]  # (n_features,)
top_features = final_token_acts.topk(10)
print("Top 10 active features:")
for idx, val in zip(top_features.indices, top_features.values):
    print(f"  Feature {idx.item()}: {val.item():.3f}")
```

## Transcoders: Mapping MLP Computations

SAEs decompose what the residual stream *contains* at each layer. Transcoders answer a different question: what transformation does the MLP at each layer perform on those features?

An MLP block computes `output = W_out · ReLU(W_in · x + b_in) + b_out`. This is a nonlinear function from the residual stream to a delta that gets added back. Transcoders learn a sparse approximation of this function as a sum of "feature → feature" transitions:

```
MLP(x) ≈ Σᵢ g_i(x) · v_i
```

where `g_i(x)` activates when input feature `i` is present, and `v_i` is the direction this causes in the output. This gives you a human-readable circuit description: "when the model detects feature X in the context, the MLP at layer L adds feature Y to the residual stream."

This kind of circuit analysis is what allowed Anthropic to identify the "Indirect Object Identification" circuit in GPT-2 (Wang et al., 2022) — a specific sequence of attention heads and MLP neurons that implement the algorithm for figuring out which pronoun refers to which named entity. Gemma Scope 2's Transcoders make this type of analysis tractable at the scale of Gemma 3's full 27B model.

## What's New: Skip-Transcoders

The most technically interesting addition in Gemma Scope 2 is the skip-transcoder. Standard Transcoders capture how the MLP at layer L transforms features within that layer. But what about features that appear at layer 5 and influence computations at layer 15, skipping everything in between?

This "skip connection" phenomenon in information flow isn't about the model's literal residual connections — it's about semantic features that persist across many layers without much transformation, then suddenly become active input to a computation far downstream.

Skip-transcoders are trained to approximate `f(x_L1) ≈ x_L2` — learning the transformation between activations at non-adjacent layers L1 and L2. When the learned transformation is sparse and interpretable, it reveals which features are causally propagating across the layers between L1 and L2.

This has implications for understanding multi-step reasoning:

```python
# Conceptual code illustrating skip-transcoder analysis
from gemma_scope import SkipTranscoder, load_model

model = load_model("gemma-3-27b")
skip_tc = SkipTranscoder.from_pretrained("gemma-scope-2-27b", source_layer=8, target_layer=20)

problem = "Let n be a prime number greater than 3. Show that n² - 1 is divisible by 24."

tokens = model.tokenize(problem)
activations_l8 = model.get_activations(tokens, layer=8)
activations_l20 = model.get_activations(tokens, layer=20)

# Find which features at layer 8 causally contribute to layer 20
skip_features = skip_tc.encode(activations_l8)  # (seq_len, n_features)
reconstructed_l20 = skip_tc.decode(skip_features)

# Features with high activation in the skip-transcoder
# represent information being "carried" from layer 8 to 20
skip_residual = reconstructed_l20 - activations_l8.mean(dim=0)
dominant_skip_features = skip_features.mean(0).topk(20)

# These features represent semantic content being preserved 
# across layers 8-20 — likely mathematical/logical structure
# in this case that helps at the reasoning layers
```

## Cross-Layer Transcoders and Causal Flow

Cross-layer Transcoders generalize this to any pair of layers, learning the transformation `(layer_a, layer_b) → sparse approximation`. This makes it possible to systematically map causal information flow through the model for specific computations.

For CoT faithfulness research — the question of whether a model's visible reasoning steps actually caused its conclusion — this is particularly valuable. By comparing the skip-transcoder-derived information flow for "model produces correct CoT then correct answer" vs. "model produces plausible-sounding CoT then wrong answer," you can ask: at what layer does the information determining the final answer appear, and does it correlate with which CoT steps were generated?

Early results from Anthropic's line of work on CoT faithfulness (Lanham et al., 2023) used activation patching to show that models sometimes reach conclusions through reasoning paths that bypass their stated chain-of-thought. Cross-layer Transcoders make this type of analysis more systematic and fine-grained.

## The Matryoshka Training Innovation

Gemma Scope 2 SAEs are trained with Matryoshka representation learning. The idea: a set of 16K learned features should contain a meaningful 4K-feature subset that captures the most important information, which itself contains a 1K-feature subset that captures the most important of that, and so on.

Formally, during training, you add auxiliary losses that require smaller subsets of the feature dictionary to also achieve good reconstruction:

```python
class MatryoshkaSAE(nn.Module):
    """
    SAE with nested representation learning.
    Smaller feature subsets must also reconstruct the residual stream.
    """
    def __init__(self, d_model: int, n_features: int, 
                 nest_sizes: list[int] = [1024, 4096, 16384]):
        super().__init__()
        assert n_features == max(nest_sizes)
        self.nest_sizes = sorted(nest_sizes)
        
        self.W_enc = nn.Parameter(torch.randn(d_model, n_features) * 0.01)
        self.b_enc = nn.Parameter(torch.zeros(n_features))
        self.W_dec = nn.Parameter(torch.randn(n_features, d_model) * 0.01)
        self.b_dec = nn.Parameter(torch.zeros(d_model))
    
    def encode(self, x: torch.Tensor) -> torch.Tensor:
        return torch.relu(x @ self.W_enc + self.b_enc)
    
    def forward(self, x: torch.Tensor) -> dict:
        f = self.encode(x)
        
        total_loss = 0.0
        # Loss for each nested subset
        for k in self.nest_sizes:
            # Use only the first k features (Matryoshka nesting)
            f_k = f[..., :k]
            x_hat = f_k @ self.W_dec[:k] + self.b_dec
            
            recon_loss = (x - x_hat).pow(2).mean()
            sparsity_loss = f_k.abs().mean()
            
            # Higher weight for smaller nests (they need to work harder)
            weight = self.nest_sizes[-1] / k
            total_loss = total_loss + weight * (recon_loss + 1e-3 * sparsity_loss)
        
        return {"loss": total_loss, "features": f}
```

The practical benefit: you can choose different granularity levels at analysis time. The top 1K features give you a coarse map of what the model is "thinking about" — high-level semantic categories. The full 16K features give fine-grained distinctions within those categories. This makes exploratory analysis faster (use coarse features to identify interesting layers/tokens) and detailed analysis tractable (zoom into specific features of interest).

## Why This Infrastructure Matters

Gemma Scope 2 is useful as a research tool, but its broader significance is as a proof of concept for a model-interpretability pipeline that could become standard practice.

Currently, training and evaluating SAEs is a research project in itself. Most academic groups cannot afford the compute to train high-quality SAEs on frontier models. Gemma Scope 2 eliminates this barrier for Gemma 3, and the 40+ papers citing the original Gemma Scope (covering Gemma 2) suggest the community will use it.

More importantly, it demonstrates a workable answer to the question: how does a lab pursuing safety research maintain meaningful visibility into increasingly capable models? The answer isn't necessarily to give everyone access to model weights — it's to provide pre-computed interpretability tools that are useful for identifying concerning behaviors without requiring white-box access to the training process.

The limitation is that Gemma Scope 2 is optimized for post-hoc analysis, not for real-time monitoring during deployment. The SAE forward pass adds significant overhead. For production monitoring — the scenario where you want to flag concerning activations as they happen — you'd need either much faster SAEs or different tooling entirely. This gap between analysis-time interpretability and deployment-time monitoring is an open engineering problem.

What Gemma Scope 2 solves, concretely, is making it cheap and fast for the research community to study what a state-of-the-art open model actually computes, at every layer, for any input. That's not everything, but it's a genuine and substantial step.
