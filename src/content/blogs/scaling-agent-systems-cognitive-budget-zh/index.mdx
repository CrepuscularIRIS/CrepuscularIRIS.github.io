多智能体编排框架在 2025 年讲着同一个故事：单个模型是瓶颈，所以多生成几个。LangGraph、CrewAI、AutoGen、MetaGPT，以及每一个 MCP 编排 Demo，都遵循相同的范式——一个规划器，几个专业 Agent，一个批评者，一个综合者。这套逻辑表面上说得通：一个 Agent 不够好，几个专业化 Agent 合力应该更好。

这个逻辑错得足够频繁，值得认真对待。[Google Research 的《迈向 Agent 系统扩展的科学》（2026 年 4 月）](https://research.google/blog/towards-a-science-of-scaling-agent-systems/)在两个基准测试上运行了 180 种多智能体配置，结论应该改变从业者的设计思路。在一个基准上，集中式多智能体协调比单 Agent 提升了 80.9%。在另一个上，所有多智能体配置的性能下滑了 39%—70%。相同框架、相同模型、相同推理步骤数——结果截然相反。

决定性变量不是大多数编排框架暴露的那个。而是：底层任务是*可并行*的还是*串行*的，以及 Agent 团队的协调开销是否小于单 Agent 处理同一问题所花的认知预算。

本文梳理这一发现，解释其背后的机制，并提炼出一套决策流程：多 Agent 何时有用，何时有害，以及真的无法判断时该怎么办。

## 两个基准测试

Google 的研究选了两类任务，正是因为它们的结构属性截然不同。

**类 FinanceBench 分析。** 给定一份 10-K 或 10-Q，回答关于它的特定事实性或推断性问题（分部营收、一次性费用、衍生品敞口等）。这类任务分解干净：不同专家可以并行阅读文档的不同部分，提取结构化信息，再交给综合器合成最终答案。子任务间的依赖图浅——大多数子任务只依赖原始文档，而不依赖彼此的输出。

**PlanCraft。** 一个规划基准，每一步都紧密依赖前一步（构建合成链、导航依赖图、执行中修复损坏的计划）。任务从根本上是串行的——第 N 步的输出是第 N+1 步的输入——每个分支点都需要*同一个*推理者在脑子里装着所有前序状态。

数字很直白：

| 任务类型 | 单 Agent | 集中式多 Agent | 去中心化多 Agent |
|---------|:---:|:---:|:---:|
| 金融（可并行） | 基准 | **+80.9%** | +45% |
| PlanCraft（串行） | 基准 | **−39%** | **−70%** |

可并行任务：集中式 > 去中心化 > 单 Agent。串行任务：单 Agent > 集中式 > 去中心化。两端配置之间 150 个百分点的差距——做实际工作的模型相同——完全来自协调结构的不同。

## 认知预算模型

Google 论文提出的机制（也是我在生产部署中觉得最有预测力的）把每次 Agent 交互视为消耗*认知预算*：花在协调、上下文传递和结果整合上的总 token 和注意力，而非花在任务本身上的。

对于内在难度为 $D$（以推理 token 计）的任务，单 Agent 大约花 $D$ 个 token 解决问题。一个 $k$ Agent 的集中式团队大约花 $D / k + C(k)$，其中 $C(k)$ 是协调开销：

- 每个专家需要上下文导入（我的子任务是什么，共享状态是什么）
- 每个专家输出一份报告供协调器读取
- 协调器需要调和相互冲突或重叠的报告
- 任何共享状态（计划、草稿板）都要序列化、修改、再序列化

$C(k)$ 在可并行任务上至少线性增长，在串行任务上大约*二次方*增长——因为每增加一个 Agent，就在依赖图上新增一条边，状态每次变化时都得遍历一遍。

多 Agent 有益的条件正是 $D/k + C(k) < D$，即"$C(k) < D(k-1)/k$"——协调成本必须小于拆分任务带来的加速。满足条件需要：

1. 任务分解干净（子任务耦合低，$D/k$ 真的能代表专家工作量）。
2. 协调开销低（结构化交接，而非自由对话）。
3. 每个子任务的工作量远大于每个子任务的交接成本。

任何一条不满足——尤其是条件 1——协调开销就会主导，多 Agent 系统输给将所有状态保留在本地的单 Agent。

## 串行任务为何专门摧毁多 Agent

PlanCraft 的结果比金融结果更有启发性。−70% 的下滑不是噪声，是可预测的失败模式。三个具体机制叠加：

**序列化损耗。** 每次一个 Agent 把状态交接给另一个，状态都要经过自然语言（或结构化 JSON，本质一样）。序列化有损耗——接收方 Agent 必须从显式文本重建发送方持有的隐式理解。对于串行任务，这种损耗在每一步叠加。到第十个 Agent 行动时，对原始计划的累积误读已经大到足以产生一个完全不同的计划。

**分布式故障定位。** 串行计划失败时，有人得找出*哪一步*出了问题。单 Agent 系统里，Agent 在自己的上下文中保存了完整轨迹，可以推断哪一步错了。多 Agent 系统里，没有任何一个 Agent 拥有完整轨迹——它们只有自己的子任务和交接记录。调试和修复需要元 Agent 把轨迹重新拼起来，而这个元 Agent 本身也受序列化损耗影响。

**冗余重新计算。** 串行任务中，后期步骤往往需要重新推导早期步骤的推理中已有的事实。单 Agent 系统里，这些事实就在上下文窗口中。多 Agent 系统里，它们必须被重新计算或重新检索，往往由不知道早期推导存在的不同 Agent 来做。这是认知层面的缓存缺失，随计划深度扩展。

可并行任务完全绕开了这三个问题。分析同一份 10-K 不同部分的专家之间没有序列化（只对综合器序列化一次）。不存在分布式故障定位，因为每个专家的输出可以独立检验。也不存在冗余重新计算，因为子任务在构造上就是独立的。

## 集中式 vs. 去中心化：为何集中式在多 Agent 有效时更好

Google 研究的次要发现几乎和主要发现同样实用。在各种多 Agent 配置中，*集中式*协调（一个编排器路由子任务并聚合结果）在几乎所有多 Agent 有帮助的基准上都优于*去中心化*协调（Agent 彼此自由发消息）。

原因同样在协调项上。去中心化系统的通信边数是 $O(k^2)$——任何 Agent 都可以和任何其他 Agent 对话，它们确实会这样做，因为被提示"与团队协作"的 LLM Agent 会生成比必要多得多的消息。集中式系统的边数是 $O(k)$：每个 Agent 只和编排器对话。编排器把协调复杂性吸收进一个可以被连贯推理的单一上下文窗口中。

这和生产中有效的 Agent 系统高度吻合。Claude Code 的子 Agent 模型是集中式的：父级 Claude 生成子 Agent，读取它们的报告，从不让它们彼此对话。Cursor 的 Agent 模式是集中式的。那些让 Agent 自由发消息的"蜂群"框架，往往会产生三个 Agent 用七轮对话争论谁先行动这种失控会话。

实际含义：如果要用多 Agent，从星形拓扑开始。给一个编排器读取所有子 Agent 输出的权限，不给子 Agent 任何彼此通信的渠道。只有在有证据表明集中式版本因特定原因失败时，才添加对等通信。

## 决策流程

Google 的数据加上生产 Agent 部署中的有效经验，给出了相当具体的决策流程。在引入多 Agent 框架前，按顺序走这几步：

**第一步：任务是否可并行？** 画出子任务的依赖图。如果大体上是星形（一份文档，多个独立问题）或基本不相交（不同文件，独立分析），多 Agent 可能有帮助。如果是链状或深度 DAG，保持单 Agent。PlanCraft 的失败说明串行任务即使有最优编排也会下滑。

**第二步：总工作量是否足以覆盖协调开销？** 协调成本在 token 层面大约是每次 Agent 调用固定的。如果任务能舒适地放进单个上下文窗口，且单 Agent 版本在一分钟内跑完，多 Agent 只增加延迟，不增加能力。在 Google 的具体实验中，收支平衡点大约在单 Agent 需要 5 万+ 推理 token 的任务——把这个阈值当作一个量级参考而非通用常量，它会随模型、任务族和协调协议而变化。

**第三步：能否清晰表达分解方式？** 如果专家子任务可以描述为"把这个固定提示词应用到这个固定输入上"，多 Agent 可行。如果子任务要求专家推理其他专家在做什么，就退回到了串行问题。

**第四步：如果前三步说可以，用集中式而非去中心化。** 一个编排器，多个叶子专家，星形拓扑。没有对等通信。

**第五步：限制团队规模。** Google 的数据显示大多数任务在超过 4 个专家后回报递减，超过 8 个后出现负回报。更大的团队是警示信号：通常说明任务实际上不可并行，框架在用 Agent 掩盖协调失败。

## 与当前趋势的关系

2026 年的三个趋势改变了这套决策流程的读法。

**更大的单模型上下文窗口改变了收支平衡点。** 当前前沿模型（本文写作时的 Claude Opus 和 GPT-5 系列）声称 20 万+ token 上下文，在我的实际使用中能可靠处理前 10 万 token 的真实、密集内容。很多*看起来*需要并行处理的工作——"从五个角度分析这份 10-K"——现在放进一个窗口就够了。多 Agent 的收支平衡点在上移，而非下移。2024 年需要拆分的任务，2026 年往往不需要了。

**模型内部的自我验证（Opus 4.7 的"我会检查自己的输出"行为）吃掉了集中式多 Agent 部分的价值主张。** 集中式多 Agent 在可并行任务上之所以有帮助，原因之一是编排器能发现各专家之间的不一致。如果单 Agent 能可靠地发现自己的不一致，外部编排器做的独特工作就少了。这不会消除并行处理的加速效益，但缩小了*质量*差距。

**递归语言模型（RLM）对很多可并行任务来说是比多 Agent 更好的替代方案。** 带 `llm_batch` 的 RLM 根节点就是一个集中式多 Agent 系统，其中"Agent"是无状态的子 LLM 调用，使用同质提示词，编排器是根节点的 Python 代码。这比框架管理的多 Agent 团队简单得多，扩展性也更好：`llm_batch` 里 100 个子 LLM 调用轻而易举；LangGraph DAG 里 100 个专业 Agent 是调试噩梦。

综合来看，趋势方向是：*多 Agent 作为生产模式在收窄，而非扩大*。大上下文吸收了多 Agent 过去处理的场景。自我验证吸收了编排器过去处理的场景。RLM 吸收了扇入扇出工作流过去处理的场景。剩下的合理用例是确实可分解、专家边界清晰的任务——这比框架生态系统暗示的更小的细分市场。

## 我已经不再用多 Agent 做的事

作为对上述框架的验证，我自己 Agent 工作中的具体例子。

- **PR 的代码审查。** 以前：每种语言/文件类型一个专家，加一个综合器。现在：一次 Opus 调用，传入完整 diff 和结构化评审标准。综合器做的工作比专家还多，而且大部分是在调和对同一行代码意见不同的专家输出。单 Agent 更一致，成本约低 3 倍。
- **多跳研究问题。** 以前：研究 Agent、验证 Agent、写作 Agent。现在：一个带网络搜索的 Agent，提示它在写作前验证自己的说法。验证 Agent 唯一的真实工作是捕捉幻觉——模型被提示后会原生做到这一点。
- **长任务的计划生成。** 以前：规划器、估算器、执行器。现在：一个既规划又执行的 Agent，加上显式的检查点提示。估算 Agent 从来没有对规划 Agent 提出过有用的不同意见。

我*仍然*保留多 Agent 的场景：

- **对放不进上下文的文档语料库做并行文档分析。** 经典 map-reduce 形态；集中式编排器加同质叶子提示词。这是 RLM 形式化的场景，也是 Google 金融结果实证验证的。
- **红队/蓝队评估**，对抗结构本身就是任务定义的一部分（一个 Agent 生成，另一个攻击，裁判打分）。这里"多 Agent"不是优化手段，而是任务定义本身。

## 仔细读这个结果

对于 Google 论文，有一点要注意：它使用的基准是合成的，任务结构已知。生产中结构通常是混合的——一个任务可能前半段可并行（分析这些文档），后半段是串行的（从分析中生成连贯的计划）。上述框架适用于*每个阶段*，而非整个任务。并行阶段用多 Agent，串行阶段用单 Agent，接受阶段切换的协调成本。"我们是多 Agent 架构"或"我们是单 Agent 架构"这种一刀切答案，输给按阶段分别选择的系统。

Google 论文讲述的更大故事——也是它值得在这个研究之后持续关注的原因——是**Agent 系统设计开始产生有实证依据的工程原则**，而非基于直觉的最佳实践。180 种配置的扫描，正是 2025 年多 Agent 讨论中缺失的方法论。更多这样的研究，以这种配置覆盖度，会缩小框架声称和实际交付之间的差距。

我对这个方向的预测：到 2026 年底，主流 Agent 模式将是"一个大上下文模型做自我验证，偶尔并行扇出到持久运行时中返回到变量的子调用"，而 2025 年代的多 Agent 框架将大约像微服务框架遭遇"模块化单体"反应后的样子——过度营销、过度应用、在特定场景有用、退出默认技术栈。
