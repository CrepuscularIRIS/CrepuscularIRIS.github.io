过去一年里，我审视过的每一个自主编码智能体的复盘报告，在漏斗顶端讲述的都是同一个故事：模型有能力写出修复方案，脚手架有能力运行测试，但失败出在智能体打开了错误的文件。在我所采样的 SWE-bench 轨迹日志中，大约 **30%** 的失败最终归因于文献中委婉称呼的"定位错误"——智能体找到了一个看起来靠谱的函数，但那个函数并不是真正的故障点，它做了一个干净、有测试覆盖的修改，生成的补丁对问题毫无作用。另外约 **25%** 的失败属于我所说的"扩展定位错误"：智能体找到了正确的函数，但遗漏了调用方、配置项或副作用——修复本应也涉及这些部分。

这不是新发现。SWE-agent 论文在 2024 年就标注了这个问题，Aider 自 2023 年起就在发布故障模式拆解。2026 年发生变化的是：几个生产级智能体团队已经不再把这个问题当作"迟早会被更好的嵌入模型解决的检索问题"，而是开始围绕定位器（localizer）作为一等公民来重新设计智能体架构。本文讨论的是这种重新设计在实践中的样子，以及为什么"定位优先"这个框架会改变你构建智能体其余部分的方式。

## 故障分类

在修复架构之前，有必要精确地了解实际失败的是什么。上个月，我从三个公开的智能体系统收集了 SWE-bench Verified 任务样本的轨迹，对故障进行了分类，最终得到了大致这样的分布：

| 故障类别 | 占比 | 实际发生了什么 |
|---|---|---|
| 错误的文件编辑 | ~30% | 智能体编辑了一个与真正 bug 位置同名或同功能的函数，但并不是它 |
| 遗漏依赖 | ~25% | 智能体修复了核心函数，但忽略了调用方、配置项或同级模块 |
| 测试预言机 | ~20% | 智能体生成的补丁通过了自身测试，但未通过隐藏测试 |
| 重试空转 | ~15% | 智能体在同一个位置反复编辑，换了略微不同的提示词 |
| 环境不匹配 | ~10% | 智能体的沙盒构建方式与评估器不同 |

前两行都是定位失败。它们合计占所有故障的 **一半以上**。一个更强的编辑模型修不好它们。更完善的沙盒修不好它们。修复必须在编辑器接触任何文件之前发生。

有意思的细节在第二行。"遗漏依赖"看起来应该可以通过运行测试来解决——如果你破坏了调用方，测试难道不应该catch到吗？经验证，并非如此。大量真实世界的 bug 测试覆盖率稀疏，而本应暴露这个 bug 的那些调用方在 CI 中根本没有被测试。智能体的测试运行通过了，因为测试所需的证据根本没有被采集。这就是"测试预言机"失败类别渗透到定位环节的表现：智能体无法判断自己遗漏了依赖，因为它的可观察环境中没有任何信息告诉它。

## 为什么仅靠检索不等于定位

对于"智能体老是打开错误的文件"这个问题，本能反应是投入一个更强的检索器。嵌入代码库，嵌入问题，找到 top-k 最相似的文件，把这些交给智能体。我看过的每个编码智能体都有某个版本的这个机制，它有帮助，但不够充分。从"检索"到"定位"的差距，是**相似性**与**因果性**之间的差距。

检索返回的是*看起来*与查询相关的文件。定位返回的是*导致*查询中描述的症状的文件。这是两个不同的问题。一条 issue 说"当邮件包含大写字符时登录页崩溃并返回 500 错误"，用相似性搜索会浮现出很多文件：登录处理器、邮件验证器、认证测试、用户模型、表单组件。真正的 bug 可能藏在三层以下的一个辅助函数里，它在两个代码路径中对邮件小写化处理不一致。检索器会把登录处理器排在第一位，因为它匹配查询的文本。但定位器需要追踪邮件通过调用图的路径，比较两条路径，并找出分歧点。

在消融实验中这个区别表现得很尖锐。在 Agentless 论文和我自己做过的内部实验中，把纯检索定位器替换为检索+执行定位器——一个真正运行代码、观察失败路径、沿栈追踪的定位器——将 SWE-bench 解决率提升了数个百分点。增量不在编辑质量，而在定位器将症状转换为调用图坐标的能力。

## 定位优先架构

我所说的"定位优先"是一种架构选择：智能体管道中的第一个专家既不是规划器，也不是检索器，而是定位器，它的工作是在任何其他组件考虑改什么之前，输出一个调用图坐标——一个具体的函数、一个具体的调用方集合、一个具体的配置面。其余管道将这个坐标视为真值。

```
Issue text
    │
    ▼
┌─────────────────────┐
│     Localizer        │ ← search + execute + trace
│                      │
│  Outputs:            │
│   - target function  │
│   - caller set       │
│   - config surface   │
│   - confidence score │
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│      Planner         │ ← plans edits within the coordinate
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│      Editor          │ ← writes patches
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│     Validator        │ ← runs tests, regresses on callers
└──────────┬──────────┘
           │ fail
           ▼
┌─────────────────────┐
│   Localizer Redo     │ ← NOT Editor Redo
│                      │
│ If validator fails,  │
│ re-run the localizer │
│ with failure as new  │
│ evidence.            │
└─────────────────────┘
```

关键的设计决策是底部的反馈循环。当验证器失败时，默认的本能是把失败信息发送回编辑器，附带"你的补丁破坏了这些测试，再试一次"的消息。这就是重试空转失败模式。编辑器会继续在同一个坐标上修补，而测试会继续失败，因为坐标本身就是错的。

定位优先系统将验证失败路由回*定位器*，而不是编辑器。定位器将测试失败视为新的证据，可能用失败栈作为额外查询重新运行搜索，要么确认要么修正坐标。编辑器只有在线索器产生了新的或确认的目标后才会再次运行。这么做每次迭代变慢了，但整体收敛快得多，因为你不会在错误的位置白白消耗重试次数。

## 定位器实际在做什么

生产级定位器本身就是一个小型智能体。接口很简单——输入 issue，输出坐标——但实现包含四个不能压缩成单次 LLM 调用的组件。

**证据采集。** 定位器运行匹配 issue 的测试，捕获栈跟踪，grep 错误字符串，追踪 import。这是机械性的操作，不应该让 LLM 介入，除非是作为规划运行哪些命令的调度者。输出是一堆原始证据。

**调用图投影。** 从证据出发，定位器构建代码库的子图：出现在栈跟踪中的函数、它们的直接调用方和被调用方、以及它们读取的配置值。这是静态分析的地盘，应该使用 `tree-sitter`、`pyright` 或语言特定的调用图提取器等工具。一个好的定位器不要求 LLM 计算调用图；它要求 LLM 阅读调用图。至关重要的是，喂给 LLM 的是子图的*摘要*——函数签名、边列表和几条相关片段——而不是完整图的 dump。把原始 200KB 的图灌进 context 会可靠地触发长上下文模型被记录在案的"中间丢失"退化问题。投影步骤是保持 context 聚焦的关键。

**假设生成。** 有了证据和调用图子图，LLM 最终被用于提出候选目标函数并按可能性排序。这部分才是一个大模型真正值回成本的地方。Prompt 包含 issue 文本、证据、子图，并明确要求一个带解释的候选排名列表。

**确认。** 对于排名靠前的候选，定位器尝试确认因果性。便宜且可靠的方式是运行项目自己的失败测试（或从 issue 正文中提取的重现场景），加上插桩——print 语句、Python `sys.settrace` hook 或调试器断点——验证候选函数确实出现在失败执行路径上。为任意内部函数合成新的隔离单元测试在一般情况下要难得多（你需要正确地 mock 状态和依赖），文献也没有声称解决了这个问题。在实践中真正有回报的重现步骤是基于现有失败进行 trace 确认，它将候选集从"看似合理"缩小到"明确在失败路径上"。

```python
# 定位器管道的草图
async def localize(issue: Issue, repo: Repo) -> Coordinate:
    evidence = await collect_evidence(issue, repo)  # grep, traces, test runs
    subgraph = await build_callgraph_subgraph(evidence, repo)

    candidates = await llm_rank_candidates(
        issue=issue,
        evidence=evidence,
        subgraph=subgraph,
        n=5,
    )

    for candidate in candidates:
        reproduction = await attempt_reproduction(candidate, issue, repo)
        if reproduction.confirmed:
            return Coordinate(
                target=candidate.function,
                callers=subgraph.callers_of(candidate.function),
                config=subgraph.config_reads_by(candidate.function),
                confidence=reproduction.score,
            )

    # No confirmed candidate — escalate rather than guess
    raise LocalizationEscalation(candidates=candidates, evidence=evidence)
```

底部的升级（escalation）不是 bug。一个无法确认的定位器应该说清楚，而不是把最佳猜测发给编辑器。这是架构获得可靠性的地方——编辑只有在定位器有足够信心承诺一个坐标时才发生。一个有原则的"定位失败"比一个在错误地方的自信编辑要好得多。

## 这如何改变了编辑器的工作

坐标驱动的编辑器与提示词驱动的编辑器是两种不同的动物。传统编辑器接收"这是 issue，这是一些文件，生成一个补丁"。坐标驱动的编辑器接收"这是函数，这是它的调用方，这是它读取的配置面——生成一个修复这个特定函数并处理这些特定调用方的补丁"。范围更窄，失败模式也不同。

具体来说，编辑器不再需要猜测哪些调用方可能受影响。定位器已经告诉它了。编辑器现在可以更小、更机械。对于很多 bug，坐标驱动的编辑器可以是模板化的——"修改返回类型并更新这三个调用方"——而不是一次开放式的 LLM 调用。你在编辑器上省下的 token 预算可以重新投入到定位器，而那才是智能体真正在失败的地方。

这也是经济学改变的地方。坐标驱动的编辑更容易缓存。如果定位器的输出是一个结构化的坐标，两个不同的 issue 如果落到同一个坐标上，就可以共享一个编辑计划。如果你的智能体处理一类 bug 的重复变体，规划器的缓存命中率会上升。定位器成本被摊薄，编辑器成本每次调用都很小，整体系统每个已解决 issue 的成本降低。

## 开放问题

定位优先架构在我看过的智能体基准测试上确实优于定位靠后的架构，但优势幅度比上面的叙述可能暗示的要小，而且它们也带来了自己的一堆麻烦。

**单体仓库（monorepo）定位仍然很难。** 调用图投影步骤假设你可以从证据构建一个有用的子图。在一个有 200 万行代码、依赖注入密集的单体仓库里，子图就是整个仓库。你需要语言特定的启发式方法（框架感知的解析器、运行时绑定的处理），这些不能泛化。

**重现场景成本高。** 对每个 top-5 候选运行候选重现场景是计算密集的。在实践中你重现场景 top-1，只有 top-1 失败时才回退到 top-2。这是一个简单的启发式方法，在 top-1 是红鲱鱼的 bug 上表现不佳。

**定位器需要自己的评测。** 这个领域有很好的端到端智能体性能基准测试（SWE-bench、SWE-Bench Pro、Terminal-Bench），但在隔离环境下的定位评测基本没有。你无法判断你的定位器是否在改进，除非有一个独立于编辑质量来衡量定位精度的评测。我一直在手工构建一个小型的内部评测，它已经帮我们避免了智能体栈中的好几次回归。

**升级 UX 尚未成熟。** 当定位器说"我无法确认一个坐标"时，人类操作员看到的东西很重要。今天大多数系统会列出一串候选，让人类选择。这比输出一个错误答案要好，但不如让人类看到定位器采集的证据让他们自己指向正确的函数。定位升级的 UX 是一个开放的设计问题。

## 心理翻转

一旦你内化了定位优先，最能改变一切的观念转变是：**应该用坐标来衡量智能体的不确定性，而不是用 token**。传统编码智能体在文本生成层面追踪不确定性——logits、采样温度、若干合理延续的可能性。定位优先智能体在调用图层面追踪不确定性——哪个函数、哪些调用方、哪条配置路径。第一种不确定性并不能预测正确性。第二种可以。

一旦你完成了这个翻转，几个工程决策会重新定位。一个更大的编辑器模型变得不如一个更严格的定位器管道有吸引力。采样温度调优变得不如重现场景阈值调优重要。当测试失败时，第一个问题变成"定位器把坐标做对了吗？"而不是"编辑器需要再试一次吗？"——条件于坐标，编辑器通常是正确的。智能体变成了先理解代码、后写代码的东西，这就是高级工程师所做的，大概也是为什么这个框架感觉对了。
