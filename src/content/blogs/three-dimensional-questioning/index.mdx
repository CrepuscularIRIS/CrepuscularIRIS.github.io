Most deep learning practitioners learn architectures as black boxes. ResNet works, so use ResNet. Transformer works, so use Transformer. This is fine for engineering, but it's the wrong stance for research. Understanding *why* an architecture works — and what it implicitly assumes — is necessary for knowing when to abandon it.

The three axes below form a systematic framework for evaluating any architectural proposal from first principles. They apply equally to evaluating existing architectures (why does ResNet fail on task X?) and to designing new ones (what constraint is my task imposing?). The framework emerged from analyzing residual connection variants, but the axes generalize.

## Axis 1: Interaction Order

**The question**: What is the highest order of interaction this architecture can represent?

Neural networks process information through interactions between elements — tokens, channels, layers, modalities. The order of interaction determines expressiveness: first-order processes each element independently, second-order processes pairs, and higher-order processes arbitrary subsets.

This matters because tasks have an intrinsic interaction order. Scene understanding requires understanding that "tablecloth + plates + silverware + food" jointly define a dining scene — a high-order relationship that cannot be decomposed into pairwise interactions without information loss. Sequence modeling requires that "not ... but ..." is a dependency between non-adjacent tokens — a relationship that fixed-window architectures can fail to capture.

The interaction order of common architectures:

| Architecture | Interaction Order | Mechanism |
|---|---|---|
| MLP | First-order | Each token processed independently |
| CNN | Second-order | Local pairwise within fixed kernel |
| Standard ResNet | Second-order | Adjacent layer interactions only |
| Transformer (self-attention) | Second-order | All-pairs dot products per head |
| Hypergraph NN | Higher-order | Hyperedges connect arbitrary subsets |

The practical implication: architectures operating at order $k$ require $O(\text{depth})$ to synthesize order $> k$ information. A transformer synthesizing a 4-way interaction must chain three second-order attention operations, which introduces a gradient path length constraint and makes the synthesis implicit rather than direct.

```python
import torch
import torch.nn as nn
from typing import Optional

class SecondOrderAttention(nn.Module):
    """Standard self-attention: all-pairs second-order interactions."""
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: (B, T, D)
        # Attention(Q, K, V): token i attends to token j — pairwise
        # A 4-way interaction requires at least 3 stacked attention layers
        q, k, v = x, x, x
        scores = torch.bmm(q, k.transpose(1, 2)) / (x.shape[-1] ** 0.5)
        return torch.bmm(torch.softmax(scores, dim=-1), v)


class HigherOrderHyperedgeAttention(nn.Module):
    """
    Hyperedge-based attention: explicit high-order interactions.
    
    A hyperedge groups k tokens and produces one output that 
    represents their joint interaction — not decomposed into pairs.
    This is the difference between "scene" and "object relationship".
    """
    def __init__(self, d_model: int, n_hyperedges: int = 16, edge_size: int = 4):
        super().__init__()
        self.edge_size = edge_size
        # Learnable hyperedge membership weights
        self.edge_membership = nn.Parameter(torch.randn(n_hyperedges, d_model))
        # For each hyperedge, aggregate the member tokens into one representation
        self.edge_proj = nn.Linear(d_model, d_model)
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """
        x: (B, T, D)
        For each hyperedge e, compute:
          membership_score[e, t] = softmax over tokens of token t's affinity for e
          edge_repr[e] = sum of member token representations weighted by membership
        Then each token aggregates information from all hyperedges it belongs to.
        """
        B, T, D = x.shape
        
        # (B, T, n_edges): how strongly does each token belong to each hyperedge?
        membership = torch.softmax(
            torch.einsum("btd,ed->bte", x, self.edge_membership) / (D ** 0.5),
            dim=1  # softmax over tokens → sparse edge membership
        )
        
        # (B, n_edges, D): edge representation = weighted sum of member tokens
        edge_repr = torch.einsum("bte,btd->bed", membership, x)
        edge_repr = self.edge_proj(edge_repr)  # learned transformation
        
        # (B, T, D): each token aggregates from all edges it belongs to
        out = torch.einsum("bte,bed->btd", membership, edge_repr)
        return out + x  # residual
```

**The diagnostic question**: If your task requires your model to decide based on a 4-way or 5-way joint interaction between elements, is your architecture capable of representing that without impractical depth?

## Axis 2: Uniformity Assumptions

**The question**: Which dimensions does this architecture treat uniformly, and is that treatment correct for your task?

Uniformity is the assumption that all elements along a dimension are equivalent and should receive equal computational treatment. It's a strong inductive bias that's often wrong. Six dimensions where uniformity is commonly and incorrectly assumed:

**Depth uniformity**: Standard residual connections assume each layer contributes equally to the final representation:

$$h_{l+1} = h_l + f(h_l) \quad \Rightarrow \quad h_L = h_0 + \sum_{l=0}^{L-1} f(h_l)$$

The implicit weight on each layer's contribution is 1. In practice, different depths encode qualitatively different information (low-level features at shallow layers, high-level semantics at deep layers), and some layers contribute much more than others. This is empirically demonstrated by layer-dropping experiments: large models often remain functional after removing 30-40% of layers, which wouldn't be true if all layers contributed equally.

**Width uniformity**: Standard convolutions treat all channels identically — each output channel is a learned linear combination of all input channels with equal computational priority. But channels specialize during training; some encode task-relevant features and some become near-redundant. Channel attention (SE-Networks, ECA-Net) addresses this, but the underlying architecture starts uniform.

**Spatial uniformity**: A 3×3 convolution samples a fixed grid regardless of image content. Regions with high information density (object boundaries, fine-grained texture) receive the same number of sampling points as background. Deformable convolutions address this; standard convolutions don't.

**Temporal uniformity**: Standard transformer attention has no built-in recency bias — absent content-dependent cues, all positions receive equivalent weighting. There is no architectural prior that recent context is more relevant than distant context; that must be learned entirely from data.

```python
class NonUniformDepthAggregation(nn.Module):
    """
    Learnable depth aggregation: each layer gets a learned weight 
    rather than the fixed weight=1 of standard residuals.
    
    This is the core idea behind Attention Residuals (Moonshot AI, 2026):
    instead of h_{L} = h_0 + sum(f(h_l)), compute
    h_{L} = sum(alpha_l * h_l) where alpha_l are learned or content-dependent.
    """
    def __init__(self, n_layers: int, d_model: int):
        super().__init__()
        # Content-dependent aggregation weights (attention over layer history)
        self.query_proj = nn.Linear(d_model, n_layers)
    
    def forward(
        self,
        current: torch.Tensor,
        history: list[torch.Tensor],
    ) -> torch.Tensor:
        """
        current: (B, D) — current layer representation
        history: list of (B, D) — all previous layer representations
        
        Compute attention-weighted sum over history.
        """
        if not history:
            return current
        
        # Compute content-dependent weights
        raw_weights = self.query_proj(current)[:, :len(history)]  # (B, n_prev)
        alpha = torch.softmax(raw_weights, dim=-1)                  # (B, n_prev)
        
        # Aggregate with learned weights
        history_stack = torch.stack(history, dim=1)   # (B, n_prev, D)
        aggregated = (alpha.unsqueeze(-1) * history_stack).sum(dim=1)  # (B, D)
        
        return aggregated + current  # residual connection preserved

    def forward_static_weights(
        self,
        current: torch.Tensor,
        history: list[torch.Tensor],
    ) -> torch.Tensor:
        """
        Alternative: uniform weights (not content-dependent).
        Simpler than learned attention; still breaks depth uniformity vs. standard residuals
        by pooling over all prior layers rather than only the immediately preceding one.
        """
        if not history:
            return current
        n_prev = len(history)
        # Uniform average over history — no learning needed for this variant
        alpha = torch.full((n_prev,), 1.0 / n_prev, device=current.device)
        history_stack = torch.stack(history, dim=1)   # (B, n_prev, D)
        return (alpha.unsqueeze(0).unsqueeze(-1) * history_stack).sum(dim=1) + current
```

**The diagnostic question**: For each dimension of your architecture (depth, width, spatial, temporal, batch, modality), ask: is treating all elements equally along this dimension a correct assumption for your task? If not, you've identified an architectural constraint worth removing.

## Axis 3: Separability Constraints

**The question**: Which dimensions does this architecture process independently (separately), and does your task require them to be coupled?

Separability is the architectural choice to process dimensions in sequence (first spatial, then channel; first temporal, then modality) rather than jointly. It's a computational shortcut that works when the dimensions are genuinely independent — and fails when they're not.

The most common separability assumption in CNNs is **spatial-channel separability**: convolutions aggregate spatial information first (within the kernel), then mix channels (across the output dimension). This works for natural image recognition because texture and shape information can often be extracted independently. It fails for tasks requiring spatial-semantic entanglement: knowing that the same spatial pattern means "wing" in one context and "fin" in another requires joint spatial-semantic processing.

```python
class DepthwiseSeparableConv(nn.Module):
    """
    Standard separable convolution: spatial and channel dimensions processed independently.
    Efficient but assumes spatial and channel information are separable.
    """
    def __init__(self, in_channels: int, out_channels: int, kernel_size: int = 3):
        super().__init__()
        # Spatial aggregation: each channel independently
        self.depthwise = nn.Conv2d(in_channels, in_channels, kernel_size, 
                                    padding=kernel_size//2, groups=in_channels)
        # Channel mixing: each spatial position independently
        self.pointwise = nn.Conv2d(in_channels, out_channels, 1)
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # Processes spatial THEN channel — never jointly
        x = self.depthwise(x)  # spatial: each channel sees only itself
        x = self.pointwise(x)  # channel: each position sees only itself
        return x


class JointSpatialChannelAttention(nn.Module):
    """
    Joint spatial-channel processing: what a token at position (h, w, c)
    can attend to is determined by both spatial proximity AND channel identity.
    
    More expensive but breaks the separability assumption — the model can
    learn that "pattern at (h, w)" means different things in different channels.
    """
    def __init__(self, d_model: int, n_heads: int = 8):
        super().__init__()
        self.spatial_channel_attn = nn.MultiheadAttention(
            embed_dim=d_model, num_heads=n_heads, batch_first=True
        )
        self.norm = nn.LayerNorm(d_model)
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """
        x: (B, H*W*C, D) — flattened spatial+channel tokens
        Each token is a (h, w, c) triple rather than a spatial location alone.
        Attention is over the joint space — position and channel simultaneously.
        """
        attn_out, _ = self.spatial_channel_attn(x, x, x)
        return self.norm(attn_out + x)
```

**The separability diagnostic**: For dimensions A and B, ask: can you compute the optimal function of A given B's value, without knowing B at the time you compute A? If yes, A and B are separable for this task. If no — if the right spatial aggregation depends on which channel you're in — the separability assumption is introducing an error.

**Depth-time separability** is the constraint most relevant to sequence modeling with long-range dependencies. Standard transformers process the depth dimension (layer number) completely separately from the temporal dimension (sequence position). There's no mechanism for "what the model did at layer 3 for token 47" to directly influence "what it does at layer 7 for token 12." Whether this matters depends on whether your task requires this kind of cross-depth temporal interaction.

## Using the Framework: A Diagnostic Procedure

Given an architecture or a task, apply the three axes in sequence:

**Step 1: Identify the required interaction order.** What is the minimum interaction order that the optimal solution requires? If you can compute the output from pairwise interactions, second-order suffices. If not, you need either higher-order mechanisms or deep stacking (with the attendant gradient path cost).

**Step 2: Scan for uniformity violations.** For each dimension (depth, width, spatial, temporal, batch, modality), ask: does this task require non-uniform treatment? Mark where uniformity is wrong. These are the locations of compulsory architectural debt — if you don't fix them, you're using model scale to compensate for structural errors.

**Step 3: Identify necessary couplings.** For each pair of dimensions, ask: does the optimal processing of dimension A require knowing dimension B? If yes and the architecture separates them, that's a separability error — information that the architecture cannot represent.

```python
from dataclasses import dataclass, field

@dataclass
class ArchitecturalAudit:
    """Output of the three-axis diagnostic applied to an architecture."""
    
    # Axis 1: Interaction Order
    max_representable_order: int
    task_required_order: int
    order_gap: int = 0
    
    # Axis 2: Uniformity
    uniformity_violations: list[str] = field(default_factory=list)
    # e.g., ["depth: task requires dynamic layer selection",
    #         "spatial: task requires non-uniform sampling density"]
    
    # Axis 3: Separability
    required_couplings: list[tuple[str, str]] = field(default_factory=list)
    # e.g., [("spatial", "channel"), ("depth", "time")]
    
    def __post_init__(self):
        self.order_gap = max(0, self.task_required_order - self.max_representable_order)
    
    def severity(self) -> str:
        """Rough severity classification."""
        issues = len(self.uniformity_violations) + len(self.required_couplings)
        if self.order_gap > 0:
            return "CRITICAL (interaction order mismatch)"
        if issues > 3:
            return "HIGH (multiple uniformity/separability constraints)"
        if issues > 1:
            return "MEDIUM"
        return "LOW"

# Example: auditing a standard Transformer for scene-understanding
transformer_audit = ArchitecturalAudit(
    max_representable_order=2,       # attention is pairwise
    task_required_order=4,           # scene = 4+ jointly defined objects
    uniformity_violations=[
        "depth: all layers weighted equally in residual sum",
        "temporal: equidistant context weighting regardless of recency",
    ],
    required_couplings=[
        ("spatial", "semantic"),     # scene regions have context-dependent meaning
        ("modality", "temporal"),    # in video, visual and audio vary jointly
    ],
)

print(f"Severity: {transformer_audit.severity()}")
print(f"Order gap: {transformer_audit.order_gap}")
print(f"Uniformity violations: {len(transformer_audit.uniformity_violations)}")
```

## The Purpose of the Framework

The three-axis framework doesn't tell you how to build architectures. It tells you what constraints your architecture is imposing, so you can check whether those constraints are correct for your task.

Most architectural papers don't state their implicit assumptions. They show benchmark results that are consistent with the proposed mechanism, but the benchmark may not distinguish between "the mechanism works" and "the mechanism adds parameters" or "the benchmark is aligned with the architecture's uniformity assumptions."

Using these three axes forces the question: is this architecture correct for this task, or is it simply the best available approximation of the correct architecture? If the latter, the performance ceiling is set by the approximation error, and more compute will hit that ceiling eventually. If the former, you're working with the right inductive biases and scaling becomes productive rather than compensatory.
