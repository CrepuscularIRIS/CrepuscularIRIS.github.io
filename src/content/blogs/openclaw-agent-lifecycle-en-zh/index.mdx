单一通用智能体在长期任务中会遇到两道结构性壁垒。第一道是**上下文污染**：实现、审查、研究塞进同一个对话，注意力分散，糟糕的决策层层叠加。第二道是**没有停止函数**：没有外部仲裁者，智能体倾向于无限礼貌地重试，并给出越来越复杂的解释，说明为什么"这次一定能成"。

这些问题不是假设出来的。一个同时写代码又审查代码的单一智能体，容易滑向自我辩护——它的上下文里已经塞满了为每个决定辩护的理由，"审查"会退化为确认偏见的练习。这正是 2025–2026 年多个多智能体分析所指出观点的工程化实现：一个没有独立信息源的多智能体系统，其预期效用被系统内单个最佳智能体的效用所限制。五个看到相同事物、以相同方式推理的智能体，并不比一个好的智能体更好——成本却高了五倍。

那么真正的收益来自哪里？不是并行处理，而是**认知隔离**：每个智能体掌握其他智能体不具备的信息或能力，它们之间的耦合是狭窄且结构性的，而非对话式的。

OpenClaw 是我一直在个人 AI 操作系统中运行的五智能体分解方案。这篇文章是架构的完整说明：为何是五个智能体、为何需要心跳机制、为何选择 JSONL 邮箱，以及——唯一真正重要的设计决策——为何停止机制是硬编码的，而非提示词驱动的。

## 五个角色

| 智能体 | 角色 | 核心职责 | 不做什么 |
|-------|------|---------|---------|
| **Lacia** | 编排 | 任务分解、优先级排序、路由分发 | 不实现、不研究 |
| **Methode** | 构建 | 代码编写、测试、构建验证 | 不做架构决策 |
| **Satonus** | 审查 | 代码审查、质量评估、通过/失败判定 | 不修复发现的问题 |
| **Snowdrop** | 研究 | 文献查阅、API 文档、选项对比 | 不做最终决策 |
| **Kouka** | 交付 | 打包、停止决策、"已读"追踪 | 不研究、不实现 |

这个分工不是随意的。两条规则驱动它：*写代码的人不审查自己的代码*，*收集证据的人不决定如何使用证据*。Satonus 永远不修复它发现的问题——因为修复者兼任审查者的循环，正是确认偏见的温床。Snowdrop 收集选项但不选择——因为证据提供者不应该成为证据的消费者。

### 为什么是五个而非三个或十个

五个是软件工程生命周期的最小完整覆盖：计划、构建、验证、研究、交付。少于五个意味着某个智能体需要身兼两职，这会把上下文污染带回来。多于五个会增加通信复杂度而不带来明显收益。

一个组合数学的快速验证：五个智能体有 $\binom{5}{2} = 10$ 条成对通道。邮箱协议采用*主要星型拓扑*：Lacia 是任务分发、审查结果、交付的默认路由器，因此绝大多数有意义的协调流量走四条以 Lacia 为中心的通道。直接非 Lacia 路径也存在（例如 Methode 直接向 Satonus 请求审查，一旦任务进入执行阶段，或者 Kouka 向 Snowdrop 分发研究任务），但它们是狭窄的、契约性的，且同样被记录。这种区分很关键。超过 N=3 的去中心化 N 对 N 通信很快会变得认知上无法管理。以 Lacia 为中心的骨干保持了路由的可追踪性，同时让一个智能体拥有全局视图，其他智能体保持本地视图。

### 独立信息源

为什么五个 OpenClaw 智能体可以超越一个调优良好的单一智能体？因为每个智能体都持有其他智能体不具备的信号：

- **Methode** 看到仓库的实时状态和真实的构建/测试反馈——这些硬信号并非来自语言模型的推理分布。
- **Satonus** 运行在独立的上下文中。它从不看到 Methode 实现时的推理过程，只看到产物。那是结构性的距离，不是提示词技巧。
- **Snowdrop** 持有外部世界的认知基础——论文、API 文档、选项矩阵——这些不在其他智能体的上下文中。
- **Kouka** 持有交付历史和停止指标数据。结构化数据，而非 LLM 推理出的叙事。

这种非对称性是刻意设计的。如果两个智能体最终持有重叠的信息，分裂它们带来的收益就会消失——你只是在为同一个决策支付 2 倍的推理费用。

## 心跳机制

系统以**30 分钟心跳周期**运行。这个数字不是美学的考量。它大致是一只 Claude Code 会话在上下文窗口开始以丢失关键细节的方式压缩之前，能够进行有效工作的真实时钟时间。短于 30 分钟则无法完成一个连贯的工作单元。长于 30 分钟则丢失之前构建的上下文。

每个心跳的形状是相同的：

```
1. 智能体读取自己的邮箱（~/inbox/<agent>.jsonl）
2. 按优先级排序待处理消息
3. 执行最高优先级任务
4. 执行计划的定时任务（如 Blog-Maintenance-Kouka）
5. 如果没有工作剩余 → 向 Lacia 发出 idle_report
6. Lacia 汇总空闲信号，决定是否升级给人类
```

关键属性：**智能体之间不互相轮询。** 它们只读取自己的邮箱。行为是邮箱内容的纯函数。重放邮箱历史，就能重放决策过程。这正是让系统可调试的原因——没有这一点，多智能体系统是随机性的，事后分析几乎不可能。

### 心跳错拍

五个心跳是故意**错开五分钟**，而非同步：

```
T=0min   Lacia  心跳 — 分发任务
T=5min   Methode 心跳 — 接收工作
T=10min  Snowdrop 心跳 — 接收研究请求
T=15min  Satonus 心跳 — 接收审查请求
T=20min  Kouka   心跳 — 接收交付请求
T=25min  （空闲缓冲区）
T=30min  Lacia  心跳 — 下一周期
```

错拍保证：Lacia 在 T=0 写入的消息，在 Methode 于 T=5 苏醒时已在其收件箱中。没有竞态条件，没有"从空收件箱读取"的坑。代价是端到端延迟：完整的分发→构建→审查→交付循环需要约 30 分钟，而非理想状态下的五分钟。这种权衡对于无人值守系统是正确的——当没有人类在看管时，可靠性比吞吐量更重要。

### 粒度假设

30 分钟心跳有一个隐藏的前提：**工作单元必须能塞进一个心跳的上下文中。** "在 auth/service.py 中添加 JWT 中间件"可以。"重构整个认证系统"不行。那个重构必须由 Lacia 分解为多个心跳大小的子任务，每个都有明确的验收标准。

这并非巧合——这正是 Lacia 存在的理由。如果每个任务从一开始就是一个心跳大小，就不需要编排器了。Lacia 的工作是把人类规模的目标（"发布这个功能"）翻译为智能体规模的任务（"在 auth/middleware.py 中添加 JWT 验证，验收标准：三个特定测试通过"）。这种翻译是协调层的全部价值所在。

## 邮箱协议

邮箱是 JSONL。每条消息是一个独立的 JSON 对象，追加到接收者的收件箱文件。示例行（截断）：

```jsonl
{"id":"msg-001","from":"lacia","to":"methode","type":"task_assignment","priority":1,"subject":"Implement control-plane logging module","body":{"spec":"...","acceptance_criteria":["JSONL output","timestamp and agent ID"],"deadline":"2026-04-10T12:00:00Z"},"timestamp":"2026-04-10T10:00:00Z"}
{"id":"msg-002","from":"satonus","to":"lacia","type":"review_verdict","priority":1,"subject":"Control-plane logging PASS","body":{"verdict":"PASS","findings":[],"coverage":"87%"},"timestamp":"2026-04-10T11:30:00Z"}
```

### 消息类型契约

| type | Sender | Receiver | Meaning | Priority |
|------|--------|----------|---------|----------|
| `task_assignment` | Lacia | Methode/Snowdrop | Assign work | 1 (high) |
| `review_request` | Lacia/Methode | Satonus | Request review | 1 (high) |
| `review_verdict` | Satonus | Lacia | PASS/FAIL verdict | 1 (high) |
| `delivery_request` | Lacia | Kouka | Ship a reviewed artifact | 1 (high) |
| `delivery_note` | Kouka | Lacia | Delivery confirmation | 2 (med) |
| `idle_report` | any | Lacia | Idle heartbeat report | 3 (low) |
| `research_dispatch` | Kouka/Lacia | Snowdrop | Research task | 2 (med) |
| `escalation` | any | Lacia | Anomaly escalation | 0 (urgent) |

一个典型的端到端流程形成因果链：

```
Lacia → task_assignment("implement JWT middleware") → Methode
  Methode 实现，生成 commit
Methode → review_request(commit_sha="abc123") → Satonus
  Satonus 审查
Satonus → review_verdict(FAIL, findings=["missing token expiry check"]) → Lacia
  Lacia 将 FAIL 转换为新任务
Lacia → task_assignment("fix token expiry check") → Methode
  Methode 修复，再次提交
Methode → review_request(commit_sha="def456") → Satonus
Satonus → review_verdict(PASS) → Lacia
Lacia → delivery_request → Kouka
  Kouka 验证构建，更新 seen 记录
Kouka → delivery_note(status="shipped") → Lacia
```

Lacia 参与每一个跳跃——它持有完整的因果链。其他人只看到他们的本地切片。这种非对称是刻意设计的。局部视图的智能体，在构造上就无法在其范围之外做决策。

### 为何选 JSONL 而非 SQLite 或 Redis

四个理由，按重要性排序：

1. **Git 友好。** 每行一条消息。diff 可读。对 `~/inbox/` 做 `git log` 就是系统的通信历史。
2. **无依赖。** 没有数据库进程。不再有一样东西在凌晨三点静默故障。
3. **原子追加。** POSIX `O_APPEND` 使单行追加是原子的，解决了每条消息的完整性问题。以 Lacia 为中心的拓扑已经避免了大多数对同一文件的并发写入，实际上很少看到争用。多 writer 到一个收件箱的全局*排序*是另一个问题——见下面的未解决问题。
4. **人类可读。** 当出了问题，`cat` 文件然后读它。这句话在调试存储在 Redis 中的多智能体系统之前听起来微不足道。

代价是查询效率——线性扫描一万条消息会变慢。变通方案是每月归档到 `~/inbox/archive/YYYY-MM/`。更深的理念是**可观测性优先于性能**。在智能体系统中，最坏的故障不是慢——是难以理解。JSONL 加上 Git 给你免费的版本化通信历史。SQLite 理论上可以匹配；实践中它只是多了一样需要 grep 的东西。

## 停止机制：最重要的设计决策

在无人值守的智能体系统中，危险的失败模式不是崩溃。崩溃是安全的——它们立即停止，响亮地停止。危险的模式是**无限礼貌的努力**：智能体不断重试一个从当前状态无法达成的任务，白白燃烧 token 和时间，没有任何产出。

Kouka 不只是打包器。Kouka 是**停止决策者**。停止规则是*硬编码的*，不是 LLM 判断的。

**触发条件（满足任一即触发）：**

| Condition | Threshold | Reason |
|-----------|-----------|--------|
| 任务时长无进展 | >24h | 超过单个任务的合理跨度 |
| 连续心跳状态无变化 | 2 | 智能体可能在空转 |
| Token 燃烧但无 commit | >50k tokens | 输入/输出比率失衡 |
| 连续构建失败 | 3 | 可能是系统性问题而非偶发 |
| Satonus 对同一任务 FAIL | 3 | 方向可能在设计层面就是错的 |

**停止动作序列：**

```
1. 将任务标记为 wontfix 或 deferred
2. 记录停止理由及上下文快照
3. 通知 Lacia 重新排序
4. 如果有 Git 分支在飞，创建 WIP commit 保存状态
5. 向 ~/logs/stoppage/YYYY-MM-DD.jsonl 写入停止记录供审计
```

关键细节是规则 0：**停止是硬编码的，不是 LLM 判断的。** 如果你让智能体决定是否停止，它几乎总是决定继续。被乐观框架训练的语言模型，会乐观地评估自己的进展。停止必须是外部的、机械的、不可谈判的。

这是对使用持续信任评分和渐进降级的治理框架的刻意对比——本质上是一个环模型，在"自主"和"终止"之间有多个中间状态。连续模型表达力更强。OpenClaw 选择了二值——继续或停止——因为二值更容易推理，更难被玩花样。代价是缺乏中间状态。有时候你想要"减速但不要停止"，OpenClaw 无法优雅地表达这一点。未来工作。

## 五智能体架构何时真正带来收益

OpenClaw 并非总是值得。做一个对比：

| Property | Single Agent | OpenClaw 5-Agent |
|----------|--------------|------------------|
| 上下文清洁度 | 低（职责混杂） | 高（角色隔离） |
| 错误隔离 | 无 | Satonus 门控 |
| 停止机制 | 依赖提示词 | 结构化（硬编码） |
| 可观测性 | 低（黑盒） | 高（邮箱审计日志） |
| 可扩展性 | 线性复杂度增长 | 按需添加角色类型 |
| 部署复杂度 | 低（单进程） | 中（心跳 + cron） |
| 单任务延迟 | 低 | 高（约每周期 30 分钟） |
| 成本 | 低 | 约 5 倍基础 + 通信开销 |

经验法则：**如果任务从头到尾能塞进一个心跳，使用单一智能体。如果需要多个心跳，或者沉默失败的代价高于编排成本，使用五智能体形态。** 每周博客维护、多天功能开发、研究-实现-审查循环——这些合适。修改一个拼写错误不合适。

## 未解决问题

已知限制：

- **延迟很高。** 完整的分发→构建→审查→交付循环需要约 30 分钟。紧急修复需要一个快速通道逃生口——Lacia 可以给任务打上 `priority: 0` 标签，强制 Methode 和 Satonus 在当前心跳内行动，绕过五分钟错拍。
- **邮箱写入没有全局排序保证。** 在病理情况下（多个智能体写入同一 JSONL），消息可能交错。星型拓扑几乎所有时候都能避免这个问题，但这不是根本性修复。
- **心跳内状态是易失的。** 如果智能体会话在心跳中途死亡，中间状态随之消失。频繁的 checkpoint commit 缓解了问题；持久执行模式（为每个非确定性事件做日志）才能根本解决问题，这也是 Cloudflare 风格的 "fibers" 正在推动的方向。
- **空闲成本累积。** 即使是空闲的智能体，每个心跳也必须加载上下文并发出报告。跨五个智能体那是真实的基线支出。空闲睡眠策略（连续三次空闲后跳过心跳）已在路线图上。

进行中：

- 控制平面决策日志——每个 Lacia 路由决策及理由，用于事后分析
- 邮箱并发压力测试——五个智能体高频写入一个 JSONL 文件的持续测试
- Satonus 审查质量指标——历史 PASS/FAIL 数据用于校准审查标准是否过严或过松

计划中：

- 自适应心跳频率——紧急任务用 5 分钟周期，常规任务用 30 分钟周期
- 置信度评分审查裁决（替代二元 PASS/FAIL）以表达"条件通过"
- 心跳边界上下文压缩——保留决策点，丢弃中间推理
- 空闲智能体睡眠——连续空闲报告后跳过心跳
