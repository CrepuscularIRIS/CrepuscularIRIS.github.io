每一个 Agent 框架都把上下文窗口当作固定大小的缓冲区。满了就截断、摘要或检索。这些只是应对机制，而非真正的记忆管理。它们丢弃信息时不了解什么是有价值的，检索信息时也不了解已经存在什么。

最近在语义 KV 压缩方面的工作——基于 DeepMind、Meta 等机构的缓存高效注意力研究——采用了一种不同的方法：将 KV 缓存视为一种有分层存储和明确生命周期策略的被管理资源。Agent 不再面对一个溢出就完事的扁平缓冲区，而是获得了一个记忆层级体系——热、温、冷存储——信息根据语义相关性和时间新旧在层级间流动。

这不是一个小的优化。它改变了 Agent 能做的事情。拥有 128K 上下文且简单截断的编码 Agent，在对话变长时会丢失早期的文件读取记录。而具有分层记忆的同一 Agent，会保留它读取过的每个文件的压缩表示，在需要时可以完全恢复。这种差异在需要跨小时推理的任务中显现——恰恰是当前 Agent 失败的那类任务。

## KV 缓存作为被管理的资源

在标准 transformer 中，KV 缓存为上下文中的每个 token 存储键值对。对于 128K 上下文窗口，使用 64 个注意力头和 128 维键/值（bf16 精度）的模型，KV 缓存每个层存储约 128K × 64 × 128 × 2（键+值）× 2 字节 = **约 4.3GB** 状态。这个状态是扁平的——每个 token 的 KV 对具有相同的存储成本和相同的访问优先级。

语义压缩方法引入了一个次级 Agent——记忆管理器——来监控主 Agent 的 KV 缓存并做出存储决策。当缓存接近容量时，记忆管理器识别可以压缩而不造成重大信息丢失的 token 块：

```python
from dataclasses import dataclass, field
import torch
from typing import Literal

@dataclass(frozen=True)
class KVBlock:
    keys: torch.Tensor        # [num_tokens, num_heads, head_dim]
    values: torch.Tensor       # [num_tokens, num_heads, head_dim]
    token_ids: tuple[int, ...]
    timestamp: int
    semantic_hash: str

    @property
    def num_tokens(self) -> int:
        return self.keys.shape[0]

Tier = Literal["hot", "warm", "cold"]

@dataclass
class TieredKVCache:
    hot: list[KVBlock] = field(default_factory=list)
    warm: list[torch.Tensor] = field(default_factory=list)
    cold_index: list[str] = field(default_factory=list)

    hot_capacity: int = 32768
    warm_capacity: int = 65536

    @property
    def hot_tokens(self) -> int:
        return sum(b.num_tokens for b in self.hot)

    def needs_eviction(self) -> bool:
        return self.hot_tokens > self.hot_capacity
```

三层结构的工作原理类似于 CPU 缓存层级：

**热层**：最近上下文的完整 KV 对。每个 token 以完全保真度表示。这就是标准 KV 缓存——快速访问，高存储成本。保存最后 N 个 token，其中 N 通常是总上下文预算的 25-50%。

**温层**：较旧上下文的压缩语义嵌入。压缩模型将例如 512 个 KV 对的块缩减为捕获块语义内容的单个密集向量。访问需要解压缩，这增加了延迟但能恢复大部分信息。存储成本比完整 KV 对降低 **50-100 倍**。

**冷层**：最旧上下文的元数据索引。只保留语义哈希和文本摘要。完全恢复需要从外部存储（文件系统、数据库、对话日志）重新读取原始内容。存储成本最低，访问延迟最高。

## 压缩作为语义分诊

记忆管理器的核心决策是将哪些 KV 块压缩（热→温）或归档（温→冷）。随机驱逐或简单 LRU（最近最少使用）会不加区分地丢弃信息。语义方法在做出存储决策前评估每个块对**未来可能查询的相关性**。

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class BlockRelevance:
    recency_score: float       # decays with time
    reference_count: int       # how often attention heads attend to this block
    semantic_stability: float  # how much the block's meaning has been superseded
    task_alignment: float      # relevance to the current task description

    @property
    def composite_score(self) -> float:
        return (
            0.3 * self.recency_score
            + 0.25 * min(self.reference_count / 10.0, 1.0)
            + 0.25 * (1.0 - self.semantic_stability)
            + 0.2 * self.task_alignment
        )


def select_eviction_candidates(
    blocks: list[KVBlock],
    relevance_scores: list[BlockRelevance],
    target_reduction: int,
) -> list[int]:
    scored = sorted(
        enumerate(relevance_scores),
        key=lambda x: x[1].composite_score,
    )
    candidates = []
    freed = 0
    for idx, _ in scored:
        candidates.append(idx)
        freed += blocks[idx].num_tokens
        if freed >= target_reduction:
            break
    return candidates
```

`semantic_stability` 指标是最有趣的部分。它衡量一个块的信息是否已被后续上下文**取代**。如果 Agent 在第 5 步读取了一个文件，然后在第 20 步读取了同一文件的更新版本，那么第 5 步的块具有高语义稳定性（其含义现在是冗余的——新版本包含相同信息加更新）。高稳定性意味着该块可以被积极压缩，因为其独特信息含量很低。

这就是该方法与标准缓存驱逐策略的分歧点。LRU 驱逐最旧的数据。LFU（最不频繁使用）驱逐访问最少的。语义驱逐问的是："如果我压缩这个块，会丢失多少**独特**信息？" 一个内容已被新数据取代的频繁访问块会被压缩。一个包含尚未出现在其他地方的独特信息的低频访问块会被保留。

## 压缩模型

实际的压缩步骤使用一个小型快速模型（类似于蒸馏 BERT 或轻量级编码器），将一系列 KV 对映射到固定大小的密集向量：

```python
import torch
import torch.nn as nn

class KVCompressor(nn.Module):
    def __init__(
        self,
        kv_dim: int = 128,
        num_heads: int = 64,
        compressed_dim: int = 256,
    ):
        super().__init__()
        input_dim = kv_dim * num_heads * 2  # keys + values
        self.encoder = nn.Sequential(
            nn.Linear(input_dim, compressed_dim * 4),
            nn.GELU(),
            nn.Linear(compressed_dim * 4, compressed_dim),
        )
        self.decoder = nn.Sequential(
            nn.Linear(compressed_dim, compressed_dim * 4),
            nn.GELU(),
            nn.Linear(compressed_dim * 4, input_dim),
        )

    def compress(self, kv_block: torch.Tensor) -> torch.Tensor:
        # kv_block: [num_tokens, num_heads * head_dim * 2]
        pooled = kv_block.mean(dim=0)  # [num_heads * head_dim * 2]
        return self.encoder(pooled)     # [compressed_dim]

    def decompress(
        self, compressed: torch.Tensor, num_tokens: int
    ) -> torch.Tensor:
        reconstructed = self.decoder(compressed)
        return reconstructed.unsqueeze(0).expand(num_tokens, -1)
```

压缩是有损的。在 token 维度上进行平均池化会丢弃块内的位置信息。编码器-解码器对会引入重构误差。问题是丢失的信息对下游推理是否重要。

答案取决于内容类型。对于代码块（高度依赖位置，每个字符都重要），压缩会显著降低质量。对于自然语言描述、摘要和对话上下文（高度冗余，含义在转述后仍存活），语义内容可以更大幅度压缩——即使积极池化，大意仍能保留，尽管精确的 token 级重构会丢失。

这提示了一种**内容感知压缩策略**：代码块和结构化数据在热层停留更长时间。自然语言上下文优先压缩。记忆管理器可以从 token 分布推断内容类型（代码具有独特的括号、缩进和标识符 token 模式）。

## 对 Agent 架构的影响

分层记忆从三个方面改变了 Agent 系统的设计空间。

**会话持久性变得实用。** 一个在多小时任务中工作的编码 Agent，目前在对话超过窗口时会丢失所有上下文。有了分层记忆，早期上下文会被压缩而非消失。Agent 保持对数小时前所做决策的意识——不是 token 级保真度，而是语义级，这通常足以维持连贯性。

**多文件推理得到改善。** 为了理解代码库而读取 20 个文件的 Agent，目前在读取后续文件时会将早期文件推出上下文。分层记忆同时保留所有文件的压缩表示，支持朴素截断所阻止的跨文件推理。Agent 可以问"哪个文件定义了这个文件实现的接口？"并从温层记忆回答，无需重新读取文件。

**成本-性能权衡变得明确。** 分层容量（热/温/冷分配）是可调参数，权衡成本与能力。廉价部署使用 16K 热 / 32K 温 / 无限冷。高级部署使用 64K 热 / 128K 温。记忆管理器的压缩策略是控制面，而非上下文窗口本身。

## 生物学的类比

分层架构以惊人的保真度映射到生物记忆系统。工作记忆（热层）以完全保真度保存 4-7 个项目用于即时操作。短时记忆（温层）保留压缩的大意表征，在几分钟到几小时的时间尺度上消退。长期记忆（冷层）存储可通过联想线索检索的巩固模式。

对于 BCI（脑机接口）应用，这种类比不仅仅是隐喻性的。来自脑电图（EEG）的神经信号产生连续的高维数据流。处理实时神经输入的 Agent 面临同样的记忆管理问题：输入流超过任何固定缓冲区，且信号同时包含瞬态噪声（可以安全丢弃）和有意义的神经模式（必须保留）。应用于神经特征向量的语义压缩可以实现维持数小时前观察到的脑状态模式意识的 BCI Agent——这是当前滑动窗口方法无法提供的能力。

与海马记忆巩固的联系是直接的。在慢波睡眠期间，海马体以压缩形式重放近期经历，强化重要记忆，并允许新皮层将它们整合到长期存储中。记忆管理器的压缩步骤发挥着类似的作用：定期审查 KV 缓存，识别哪些块包含持久信息与瞬态噪声，并相应地进行压缩。

## 缺失的部分

当前方法有三个悬而未决的问题。

**压缩质量难以在线评估。** 记忆管理器不知道压缩块以后是否需要完全保真度。它基于启发式相关性分数做出压缩决策，但一段上下文的真正相关性由尚未发生的未来查询决定。这是缓存中的基本预测问题——没有完美解决方案，只有更好的启发式方法。

**跨层注意力尚未解决。** 标准 transformer 注意力在 KV 对上操作。温层数据以压缩向量形式存在，而非 KV 对。关注温层内容需要么即时解压缩（增加延迟），要么修改注意力机制以处理混合保真度输入（增加架构复杂性）。两种方法都未在规模上得到验证。

**训练记忆管理器成本高昂。** 压缩模型和相关性评分函数需要在反映真实 Agent 使用模式的数据上进行训练。这种数据不存在于标准预训练语料库中——需要 Agent 在真实任务上运行，记录其 KV 缓存访问模式，并使用结果作为训练信号。引导问题是循环的：你需要一个好的记忆管理器来收集好的数据，以训练记忆管理器。

尽管存在这些悬而未决的问题，概念上的转变才是贡献所在。上下文窗口不是溢出的固定资源——它是一种具有明确生命周期策略的被管理资源。一旦你这样框架它，解决方案空间就打开了：分层存储、内容感知驱逐、压缩表示、联想检索。这些技术在系统 engineering 中是众所周知的。将它们应用于 LLM 推理需要解决上述领域特定挑战，但方向是明确的，工具是可用的。

2027 年表现最好的 Agent，不会是拥有最大上下文窗口的那些。它们会是拥有最好记忆管理器的那些。
