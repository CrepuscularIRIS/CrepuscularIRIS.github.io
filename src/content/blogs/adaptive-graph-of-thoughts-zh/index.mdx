过去两年，关于语言模型推理的主流叙事一直是思维链（Chain-of-Thought, CoT）：提示模型逐步推理，它在复杂任务上的表现就会显著提升。这项技术是真实的，带来的增益也是真实的，而且它现在已经如此深入地嵌入模型的训练方式中，与其说是一种提示技巧，不如说是一种架构特性。

但 CoT 中隐含的线性假设，正在悄然成为天花板。

当一个问题存在多个相互独立的维度，每个维度都需要专门的分析——而且某个分支的结论会约束或否定另一个分支的结论时——线性思维链无法在不人为压缩的情况下表示这种结构。你要么强行赋予一个串行顺序，引入虚假的依赖关系；要么把所有内容塞进单个上下文窗口，让交叉检查变得隐晦且容易出错。

2026 年初，尤其是来自 Google DeepMind 和 Qwen 团队的研究中，一个研究方向正在获得关注：**自适应思维图（Adaptive Graph of Thoughts, AGoT）**——将内部推理结构视为有向无环图（DAG），而非序列。

---

## 线性链错在哪里

以一个中等复杂的工程问题为例：「我们应该将这项服务从单体架构迁移到微服务吗？」要回答好这个问题，需要分析性能特征、部署复杂度、团队技能分布、数据一致性需求和运维成本——大致是并行的，因为这些维度之间并没有严格的先后依赖关系。

标准的 CoT 链将其串行化：

```
1. 性能分析 → 结论_A
2. 基于结论_A，分析部署复杂度 → 结论_B
3. 基于结论_B，分析团队技能 → 结论_C
...
```

问题在于，`结论_A` 现在正在污染每一个后续步骤的框架。如果模型在第 1 步得出了稍有偏差的结论——比如它锚定在了某个特定的延迟顾虑上——这种框架偏见就会沿着整个链条传播。到第 5 步时，你是在一个已经被早期错误逐步扭曲的上下文中进行推理，而你无法回溯。

这一点在经验上已有充分记录。在多步推理任务上，长 CoT 链随着链长增加表现出持续的性能退化，尤其是在早期步骤涉及模糊性时。模型无法「回去」——自回归生成已经固定了那些 token。

## DAG 模型

AGoT 将问题分解为一个由推理节点构成的有向无环图。每个节点是一个聚焦的子问题。边表示逻辑依赖关系——并非所有节点都需要等待其他所有节点完成后才能开始。

结构如下：

```
问题：「我们应该迁移到微服务吗？」
         │
    ┌────┴────┐
    │    按独立维度分支
    ▼         ▼         ▼
[性能]     [运维成本]   [团队技能]
    │         │         │
    └────┬────┘         │
         ▼              │
   [数据一致性]          │
         │              │
         └──────┬───────┘
                ▼
      [综合与交叉验证]
                │
                ▼
         [最终建议]
```

性能、运维成本和团队技能可以并行分析——它们之间没有逻辑依赖。数据一致性依赖于性能分析（因为一致性需求与延迟预算相互影响）。综合节点从所有三个分支拉取结果，检查矛盾，然后生成最终建议。

这不仅仅是一种结构上的优雅。并行分支防止了早期错误的传播。汇合节点处的交叉验证步骤在综合之前，显式检查独立分支的输出是否相互一致。汇合处的矛盾是一个可恢复的信号——模型可以重新分析一个或两个分支——而不是被埋藏的偏见。

## 实际实现方式

AGoT  capable 模型的实际实现通过研究者所称的 **思维块并行（Thought Block Parallelism）** 来工作。在推理过程中，模型生成并行的「思维块」——概念上相互独立的推理上下文，彼此看不到对方的中间 token。汇合节点接收其上游块的压缩输出作为显式输入，而非作为尾随上下文。

```python
from dataclasses import dataclass, field
from typing import Optional
import asyncio

@dataclass
class ThoughtNode:
    node_id: str
    prompt: str
    dependencies: list[str] = field(default_factory=list)
    result: Optional[str] = None
    confidence: float = 0.0

class AdaptiveGraphOfThoughts:
    def __init__(self, model_client):
        self.model = model_client
        self.nodes: dict[str, ThoughtNode] = {}
        self.results: dict[str, str] = {}
    
    def add_node(self, node_id: str, prompt: str, depends_on: list[str] = None) -> None:
        self.nodes[node_id] = ThoughtNode(
            node_id=node_id,
            prompt=prompt,
            dependencies=depends_on or []
        )
    
    async def execute_node(self, node: ThoughtNode) -> str:
        """执行单个推理节点，附带其依赖上下文。"""
        context_parts = [node.prompt]
        
        if node.dependencies:
            dep_context = "\n".join([
                f"[来自 {dep}]: {self.results[dep]}"
                for dep in node.dependencies
                if dep in self.results
            ])
            context_parts.append(f"\n来自上游推理的上下文：\n{dep_context}")
        
        full_prompt = "\n".join(context_parts)
        result = await self.model.generate(full_prompt)
        return result
    
    async def execute_graph(self) -> str:
        """按拓扑顺序执行所有节点，可能时并行执行。"""
        completed = set()
        
        while len(completed) < len(self.nodes):
            # 找出所有依赖都已完成的节点
            ready = [
                node for node_id, node in self.nodes.items()
                if node_id not in completed
                and all(dep in completed for dep in node.dependencies)
            ]
            
            if not ready:
                raise ValueError("推理图中检测到环")
            
            # 并行执行就绪节点
            tasks = [self.execute_node(node) for node in ready]
            results = await asyncio.gather(*tasks)
            
            for node, result in zip(ready, results):
                self.results[node.node_id] = result
                completed.add(node.node_id)
        
        return self.results.get("synthesis", "")

# 使用示例
async def analyze_architecture_decision(question: str, model) -> str:
    aot = AdaptiveGraphOfThoughts(model)
    
    # 并行分支——彼此之间没有依赖
    aot.add_node("performance", f"分析性能影响：{question}")
    aot.add_node("ops_cost", f"分析运维成本影响：{question}")
    aot.add_node("team_skills", f"分析团队能力需求：{question}")
    
    # 数据一致性依赖于性能分析
    aot.add_node(
        "data_consistency",
        f"分析数据一致性需求：{question}",
        depends_on=["performance"]
    )
    
    # 综合节点检查所有分支是否存在矛盾
    aot.add_node(
        "synthesis",
        f"综合分析并检查矛盾：{question}\n"
        f"如果任意两个分支相互矛盾，请明确指出冲突。",
        depends_on=["performance", "ops_cost", "team_skills", "data_consistency"]
    )
    
    return await aot.execute_graph()
```

这是实际 AGoT 实现内部机制的简化版本。真实实现会为每个节点添加置信度评分，允许综合层在检测到矛盾时请求重新执行上游节点，并剪除已收敛到几乎相同结论的分支。

## 汇合验证步骤

AGoT 中技术上最重要的创新不是并行分支——而是汇合节点处发生的事情。一个朴素的实现只会把上游结果拼接起来。AGoT 则将汇合视为显式的 **一致性验证** 步骤。

汇合节点接收来自上游节点的结构化输出，并执行三步流程：

1. **一致性检测**：上游结论是否指向兼容的方向？
2. **矛盾解决**：如果不是，哪个分支更可能是错的？汇合能否在不重新计算的情况下解决这一矛盾？
3. **综合置信度**：给定解决方案后，汇合输出的置信度如何？

当矛盾解决失败时——当汇合无法调和来自上游的不兼容结论时——系统将其标记为信号，以一个或多个上游节点用从冲突中推导出的额外约束重新执行。这就是「自适应」的含义：图结构不是固定的。它可以在执行过程中生长出新的验证边。

```python
@dataclass
class JunctionVerificationResult:
    conclusions: list[str]
    contradictions: list[tuple[str, str, str]]  # (分支_a, 分支_b, 冲突描述)
    resolution: str
    confidence: float
    requires_recompute: list[str]  # 需要重新执行的节点 ID

async def verify_junction(
    model,
    node_id: str,
    upstream_results: dict[str, str]
) -> JunctionVerificationResult:
    """在 DAG 汇合节点进行显式交叉验证。"""
    
    verification_prompt = f"""你是一个多分支推理的一致性验证器。

上游结论：
{chr(10).join(f"[{node}: {result}]" for node, result in upstream_results.items())}

任务：
1. 列出每个分支的关键结论。
2. 识别分支之间的直接矛盾。
3. 对于每个矛盾：哪个分支更可能是正确的，为什么？
4. 提供一个带有置信度评分（0-1）的综合结论。
5. 如果矛盾无法解决，列出应重新执行的分支 ID。

以结构化 JSON 格式回复。"""

    raw = await model.generate(verification_prompt)
    # parse_verification_json 将模型的 JSON 响应解析为
    # JunctionVerificationResult —— 具体实现取决于你的模型客户端的
    # 输出格式（例如 json.loads(raw) + 数据类构造）。
    return parse_verification_json(raw)  # type: ignore[name-defined]
```

## 实践中带来的改变

对于当前模型的使用者来说，AGoT 大体上是不可见的——它是一种内部推理机制，而非 API 特性。但它有可见的影响。

**多步推理的基准性能**：来自 Google DeepMind 的早期报告显示，基于 AGoT 的推理在 MATH-500、GPQA 和 MMLU-Pro 基准的任务上持续优于线性 CoT——尤其是在涉及多个独立变量的问题上。在天然适合线性顺序的任务上（每一步逻辑上紧随前一步的逐步证明），增益较小。

**长链上的「推理漂移」减少**：使用 AGoT 的模型在复杂任务上表现出更低的方差。线性 CoT 链随着长度增加变得不那么可靠，因为小错误会复合累积。AGoT 的汇合验证在矛盾传播之前将其捕获，这似乎也稳定了输出质量。

**对未知情况的更好校准**：当 AGoT 分支真正无法解决某个问题时——当信息根本不存在时——置信度评分会将这种不确定性清晰地传播到综合层。线性 CoT 倾向于要么在不确定性中编造内容，要么突然停止，关于发生了哪种情况的信号更少。

---

## 局限性

AGoT 并非没有代价。分支的并行执行需要比等效总长度的线性链更多的总计算量。汇合验证是额外的推理步骤。对于真正具有自然线性结构的简单问题，AGoT 增加了开销却没有带来收益。

分解问题也仍然困难。如何将一个复杂问题划分为真正独立（而非只是看起来独立）的分支，这本身就是一个推理挑战。当前实现通过一个规划步骤来处理这个问题，该步骤在主图之前运行，增加了另一次推理调用。

而且图结构需要是无环的——这对大多数问题来说没问题，但排除了结论 C 直接反馈给前提 A 的迭代精化循环。这些情况仍然需要不同的机制。

## 与推理时扩展的关联

AGoT 最好被理解为一种结构化的推理时计算扩展形式。T²（Train-to-Test）扩展范式的核心洞见——允许更长时间推理的较小模型可以胜过较大的静态模型——直接适用于此。AGoT 不仅让模型思考得更久；它让模型思考得**更宽**，在收敛到结论之前同时探索多个分支。

该架构使计算分配变得显式。简单的子问题获得小的分支。复杂的、模糊的子问题获得更深的分支和更多的验证。综合层按分支复杂度成比例地聚合。这是一种比简单增加线性链中推理 token 总数更有原则的计算分配方式。

对于任何使用 LLM 进行复杂分析的系统构建者——架构决策、多变量优化、研究综合——理解 AGoT 很重要，因为它将改变模型在相同模型规模和推理预算下实际能做的事情。
