Anthropic 发布了 Claude Opus 4.7，上下文窗口扩展至 100 万 token。公告读起来像是例行版本更新——在 Opus 4.6 的"Mythos"架构上做了一次提升，没有新命名——但旗舰推理模型上下文扩展 5 倍可不是小改动，因为 Agent Harness 的设计是基于所配套模型的上下文限制的。每一条压缩启发式规则、每一次摘要注入、每一份记忆文件，都对应着一个具体的数值，当这个数值发生变化，Harness 的某些部分就成了冗余。

真正有意思的问题不是"你现在能往 100 万 token 里塞什么？"——答案显然是"更多东西"。真正有意思的问题是，**你围绕 20 万 token 限制构建的 Harness 组件，哪些刚丢了饭碗，哪些反而更重要了**。

## 1M 阈值不新——但推理模型在 1M 是新的

Gemini 自 2024 年起就在推理能力模型上提供 100 万 token 上下文，200 万变体也已存在。Opus 4.7 的新意在于，能在 1M 上下文中运行的模型，同时也是你在自主编码循环中最艰难架构决策时会选的那个——认知路由模式中的编排器、跨供应商审查流中的综合器、生成器-验证器管道的最终验证者。廉价快速模型的长上下文早已可用；但能力前沿模型的长上下文一直是稀缺资源。

这种稀缺性很重要，因为 Agent Harness 围绕它形成了分工。在几乎所有生产级 Claude Code 配置中，这个模式都很清晰：Opus（或其他等价 Anthropic 层级）做规划，Sonnet 或 Haiku 执行，外部检索层——向量数据库、ripgrep、结构化记忆文件——将聚焦后的上下文反馈给规划器。规划器的上下文是稀缺资源，整个 Harness 围绕节约它来构建。

Opus 4.7 的 1M 上下文颠覆了这个约束。稀缺资源从"规划器可用的 token 数"变成了**这些 token 内部的信号密度**。这重新定义了 Harness 应该做什么。

## 三个刚丢饭碗的 Harness 组件

当规划器有 100 万 token 可用时，以下三样东西明显变得不那么有用了：

**对已完成工作的大胆摘要。** 在 20 万上下文的世界里，足够长的 Agent 会话会迫使你压缩早期的对话轮次——通过模型生成的摘要、结构化的"目前学到了什么"笔记，或有损截断。每一次压缩都会引入信息丢失，而这种丢失在最无法预测后续步骤需要什么的时候最让人痛。在 100 万 token 下，Claude Code 会话可以运行数小时的工具调用而不触顶。"每 N 轮生成一次摘要"的旧循环现在成了净负面——它为了解决模型已经不再有的问题而丢弃了状态。

**对自身代码仓库做 RAG 来解决架构决策。** 在项目本身上做检索增强生成，是一种 workaround，因为 LLM 无法持有整个代码库。在 20 万上下文下，你需要检索器挑出正确的五个文件。在 100 万上下文下，你可以把大多数项目的完整 `src/` 树直接扔进上下文，让模型通过注意力自己做检索。这不会杀死 RAG——它杀死的是**专门用来 workaround 代码上下文限制的 RAG**。对外部知识（文档、法律、科学文献）的 RAG 仍然是唯一选项。

**多阶段"规划器 → 摘要 → 执行器"管道，其中摘要是在阶段间传递的唯一产物。** 如果你的 Harness 有一个中间阶段，唯一的职责是把计划压缩成执行器能装下的东西，这个阶段就是在信息保真度上的纯税。在 100 万上下文下，执行器可以直接读取完整计划加上制定该计划的完整上下文。管道缩短了一跳。

这些都曾在 2024-2025 年解决了真实问题。一旦问题消失，它们没有一个是无害的。Agent Harness 中的死亡组件是活跃风险——它们仍在执行，仍在消耗 token，仍以意想不到的方式失败，同时什么都不贡献。

## 三个变得更重要的 Harness 组件

硬币的另一面是，**信号密度**成了约束，一组不同的 Harness 组件现在承担了重量。

**通过结构化上下文进行注意力控制。** 有 100 万 token 可用的模型，如果你让它愉快地遍历 100 万 token，信号就会被稀释。Harness 现在需要像一个优秀的人类研究助理那样行事：优先呈现最相关的材料，按相关性递减排序其余内容，清晰划定活跃内容和档案内容之间的边界。在 100 万 token 下比在 20 万 token 下更重要的，是 Claude Code 中显式 `<system-reminder>` 块用于高优先级信息的模式，加上 `CLAUDE.md` 文件中的结构化章节——因为风险已经翻转：从"信息装不下"变成"重要信息被埋没了"。

**KV-cache 管理作为性能基元。** 在 100 万 token 下，每一轮的预填充成本主导了成本和延迟。把长前缀（代码库转储、项目约定、技能清单）固定在上下文开头、从不修改——这样的 Harness 可以利用 prompt caching 为那部分节省 80-95%。在 20 万 token 下这是 nice-to-have 的优化；在 100 万 token 下，这是一个 Harness 成本每轮几美分还是几美元的区别。你的上下文布局的 cache 友好性现在是头等工程问题。（经济学分析见 [KV-Cache Compression and Long-Context Economics](/blog/kv-cache-compression-long-context-economics)。）

**长上下文中断言的溯源追踪。** 在 5 万 token 的加载代码中，幻觉的行号只是个偶发小麻烦。在 80 万 token 的混合源码、日志、提交历史中，模型可以自信地编造一个看起来可信的引用，因为它可能来自那一大堆中的任何地方。Harness 需要给每个断言盖上可检索的来源戳："`src/auth.py` 第 142 行"或"`a7bf92c` 的提交信息"。没有这个，你会得到置信度-能力不匹配的灾难：规划器基于不存在的来源自信地推理。这是 100 万上下文版的 RAG 系统多年来依赖的"在检索段落中为答案提供依据"模式。

## 管道 vs 交换机框架在此的应用

从短上下文前沿模型到长上下文前沿模型的转变，是一个更普遍移动的具体实例——即从[管道到交换机](/blog/from-pipes-to-switches)的移动。管道假设信息流经有序阶段，每个阶段只能持有少量状态。交换机假设状态中的任意点可以在任意时刻从任意其他点查询。20 万上下文通过使串行压缩不可避免来强制执行管道纪律。100 万上下文让你把系统作为交换机来运行，用检索式注意力在完整工作状态上操作，而非分阶段交接。

交换机框架对你如何在"模型"和"Harness"之间划定边界有影响。在管道时代，Harness 做了很多模型做不到的结构性工作——排序、压缩、检索。在交换机时代，Harness 做了很多模型仍然做不到的**策展**工作——决定什么进入上下文、附加什么溯源、什么被缓存而什么被刷新。策展人的工作比排序人的工作更难，因为失败模式不再是"我们触到了限制"，而是"我们往大注意力场前塞了垃圾，得到了自信的错误输出"。

## 这则发布对 Agent 框架设计的含义

[2026 年 Agent 框架收敛](/blog/agent-framework-convergence-2026) 中记录的五模式收敛——顺序、并发、交接、群聊、分层——是从围绕短上下文规划器设计的框架中推导出来的。五个模式中，三个（顺序、交接、分层）明确依赖阶段边界的信息压缩。如果规划器有 100 万 token，阶段边界会失去部分存在理由。

这不会压平编排栈。真正改变的是哪一层产生差异化。当上下文是瓶颈时，框架在压缩和委托的巧妙性上竞争。当信号密度是瓶颈时，框架在**上下文组合**上竞争——加载哪些文件、以什么顺序、附加什么标注、缓存还是实时、附有什么显式溯源指针。竞争优势表面从"我们怎么塞更多进去？"移到"我们怎么判断什么重要？"

这就是为什么协议层（MCP、A2A、AP2）可能变得比框架层更重要，而非更不重要：协议标准化了上下文如何从工具和其他 Agent 那里组装，这正是策展问题。当规划器能看到一切时，框架内部路由逻辑的重要性下降；**什么被交给规划器**的互操作标准重要性上升。

## 经济学警告

这一切都不是免费的。Opus 4.7 的 100 万上下文不是 Opus 4.6 的 20 万上下文相同的价格点，而且完整 100 万上下文的预填充计算成本不小。实际的部署模式不是"总是跑在 1M"，而是"跑在信号密度所证明的长度"。自动根据任务缩放上下文窗口的 Agent Harness——短任务用短上下文，长任务解锁完整窗口——会胜过默认每个调用都最大上下文的那个。这和 `temperature` 和 `top_p` 一直面对的校准问题相同：有用范围是任务的函数，不是单一全局设置。

成本规划参考：预计中位 Claude Code 会话不会触及 100 万 token。预计特定长时域会话（代码库迁移、数小时自主调试、大型研究综合）会触及，并通过消除压缩损失失败模式来证明费用合理。Agent 流量中位成本与峰值成本的比值是需要关注的指标。

## 结语：Harness 每 18 个月必须重写一次

更广泛的模式是，前沿模型的每一次重大能力跳跃都会使为前代构建的 Harness 部分失效。从 8K 到 32K 让结构化摘要变得多余。从 32K 到 20 万让大多数滑动窗口方法过时。从 20 万到 100 万让对自有上下文的大胆预压缩适得其反。

跨越这些代际存续的 Harness 组件是做**策展**的——溯源、布局、缓存、注意力控制——而非做**压缩**的。持久的工程问题是，模型前面应该有什么信息，而非如何让它装进去。Opus 4.7 的 100 万上下文是一个契机，让你重新审视 Harness 代码的每一行并追问：这是解决"装不下"问题，还是解决"不重要"问题？如果是前者，删掉它。如果是后者，它刚刚变得更重要了。

能力曲线没有停止移动。Harness 层必须跟上——在 2026 年，这意味着重写那些为一个上下文窗口只有现在 20% 的模型而写的部分。
