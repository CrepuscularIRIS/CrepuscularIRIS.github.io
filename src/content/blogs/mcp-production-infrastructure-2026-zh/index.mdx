Model Context Protocol 最初的定位是 LLM 调用外部工具的标准化方案。JSON-RPC、一份服务端清单、stdio 传输。Anthropic 开源它的 18 个月以来，MCP 积累了超过 10,000 个公开服务端，被所有主流 Agent CLI 采用。但当你把一个为本地工具调用设计的协议部署为服务数千个并发 Agent 的生产基础设施时，它会遇到硬性限制。

2026 年 MCP 路线图——现由 Linux 基金会旗下的 Agentic AI Foundation 治理——针对三个具体缺口做出回应：传输层的可扩展性、服务端的可发现性、以及长时间运行任务的管理。每个缺口产生了一项具体的规范变更。

## Streamable HTTP：告别粘性会话

原始 MCP 传输方案——本地进程用 stdio，远程连接用 SSE——假设的是有状态的持久连接。客户端连接服务端，维持会话，在该会话上发送消息。当 MCP 服务端是一个与单个 Claude Code 实例对话的本地进程时，这没有问题。

在生产环境中，这会崩溃。SSE 连接无法在负载均衡器故障转移中存活。水平扩展需要粘性会话——将同一客户端的每个请求路由到同一个服务端实例。粘性会话是自动扩缩的天敌，因为它制造热点：一个实例承担 80% 的流量，另外 9 个闲置。

Streamable HTTP 用无状态请求模型取代了这一切。每个请求都是独立的。服务端不需要在请求之间记住客户端。需要会话状态时，通过 `Mcp-Session-Id` 头部传递——服务端从这个 ID 重建上下文，而非将其保持在内存中。

```
原始 MCP（有状态）:
  Client ──[持久 SSE 连接]──→ Server 实例 #3
  (所有请求通过粘性会话路由到 #3)

Streamable HTTP（无状态）:
  Client ──[POST /mcp]──→ 负载均衡器 ──→ 任意 Server 实例
  Client ──[POST /mcp]──→ 负载均衡器 ──→ 任意 Server 实例
  (Mcp-Session-Id 头部携带上下文；任何实例都能处理)
```

性能差异是可测量的。2026 年 3 月的 MCP 2.1 基准测试显示，在负载下与 SSE 传输相比延迟降低 95%，无状态会话处理成功率 100%。生产部署报告可处理 10,000+ 并发连接，响应时间低于 50 毫秒。

SSE 现已被弃用。规范团队明确选择不添加更多传输类型——保持传输集合小而精（本地用 stdio，远程用 Streamable HTTP）是一个刻意的设计决策。

## Server Cards：无需连接的可发现性

第二个缺口：客户端如何在不先建立连接的情况下知道 MCP 服务端能做什么？

在原始协议中，发现服务端功能的唯一方式是建立连接并调用 `initialize`。这对手动配置 Claude Code 的开发者来说够用。对于需要从注册表发现可用服务端的自动化系统，或索引 MCP 生态的爬虫来说，这行不通。

Server Cards 通过 `.well-known` URL 暴露结构化元数据：

```
GET https://my-mcp-server.example.com/.well-known/mcp-server-card
```

```json
{
  "name": "PostgreSQL Analytics Server",
  "version": "2.1.0",
  "description": "PostgreSQL 数据库的查询优化与 Schema 分析",
  "tools": [
    {
      "name": "explain_query",
      "description": "对查询执行 EXPLAIN ANALYZE 并返回优化建议"
    },
    {
      "name": "suggest_indexes",
      "description": "分析查询模式并建议缺失的索引"
    }
  ],
  "auth": {"type": "oauth2.1", "pkce_required": true},
  "transport": "streamable-http",
  "endpoint": "https://my-mcp-server.example.com/mcp"
}
```

关键属性：Server Cards 是静态的。可以被缓存、被搜索引擎索引、被注册表聚合——无需执行任何服务端代码。浏览器可以展示服务端的功能。注册表可以根据能力重叠度对服务端排名。Agent 可以从数千个服务端目录中选出最合适的一个——所有这些都在建立第一个连接之前完成。

这将发现模型从"连接后询问"转变为"浏览后连接"。Server Cards 对 MCP 服务端的作用，类似于 HTML meta 标签对网页的作用。

## Tasks 原语：管理长时间运行的工作

MCP 最初的请求模型是同步的：客户端发送工具调用，服务端返回结果。对于快速操作没有问题，但对于耗时数分钟或数小时的任务则力不从心。

Tasks 原语（SEP-1686）引入了异步任务生命周期管理：

```
Client                          Server
  │                               │
  ├──[create_task]──────────────→ │  (status: "submitted")
  │←──[task_id, status]──────────┤
  │                               │
  ├──[get_task_status]───────────→│  (status: "working")
  │←──[progress: 45%]────────────┤
  │                               │
  │←──[status: "input-required"]──┤  "需要选择哪个索引？"
  ├──[provide_input]─────────────→│
  │                               │
  │←──[task_complete, result]─────┤  (status: "completed")
```

生命周期状态为：`submitted → working → input-required → completed`（或 `failed`）。`input-required` 状态允许服务端任务暂停并向客户端请求额外信息，然后恢复执行。这使得在单个任务内实现多轮交互成为可能，而无需客户端维持持久连接。

早期生产使用暴露了一些缺口：重试语义（任务中途失败时怎么办？）和过期策略（已完成任务的结果应保留多久？）尚未完全定义。规范团队已将这些确认为下一版本的优先事项。

## OAuth 2.1：信任边界

MCP 最初的认证方案很简陋——API Key 或无认证。对于本地 stdio 服务端够用，但对于 Streamable HTTP 所开启的远程多租户世界则不够。

2026 年规范将 OAuth 2.1 + PKCE（Proof Key for Code Exchange）确定为远程 MCP 服务端的标准认证机制。PKCE 防止授权码拦截——当"客户端"是运行在共享环境中的 AI Agent 时，这一点至关重要。

实际影响：每个远程 MCP 服务端现在都需要 OAuth 提供者。对于之前只接受静态 API Key 的简单工具服务端，这是显著的新增复杂度。生态系统的回应是 MCP 网关服务（Cloudflare、Vercel），它们代替简单服务端处理 OAuth，将已认证的请求代理到仍然使用普通 HTTP 的内部工具。

## 对 Agent 架构的影响

这些变更的累积效果：MCP 服务端现在可以像标准云原生微服务一样部署。它们位于负载均衡器之后，水平扩展，通过 OAuth 认证，并通过静态元数据可被发现。

三种架构模式已经浮现：

**网关模式。** 单个 MCP 网关将多个工具服务端聚合在一个端点之后。Agent 连接到一个网关，网关将工具调用路由到适当的后端。

**注册表模式。** Agent 从可搜索的 Server Cards 注册表中发现 MCP 服务端。在执行任务时，Agent 查询注册表以匹配所需能力，连接到最佳匹配，然后委托工具调用。

**Sidecar 模式。** MCP 服务端作为 sidecar 与 Agent 进程一起运行。Agent 在本地使用 stdio 传输，避免远程认证和网络配置的复杂性。

大多数生产系统使用组合方案：本地 sidecar 用于常用工具（文件系统、Git），远程服务端通过网关用于专业能力（数据库分析、安全扫描），注册表发现用于临时工具选择。

## 未解决的问题：服务端组合

当前规范未涉及的一个问题：将多个 MCP 服务端组合为更高层级的能力。如果 Server A 提供 `read_database`，Server B 提供 `generate_chart`，目前没有标准方式创建链接两者的组合能力。

目前，这种组合发生在 Agent 的推理循环中——模型决定调用 Server A，处理结果，然后调用 Server B。这可以工作，但让组合逻辑对基础设施层不可见。你无法缓存组合操作、预验证工具链或优化服务端之间的数据传输。

MCP 的价值已经清晰：它将碎片化的自定义工具集成生态变成了标准基础设施层。2026 年的变更使该层达到了生产级别。它是否成为 Agent 的 TCP/IP，取决于组合问题能否在生态再次碎片化之前得到解决。
