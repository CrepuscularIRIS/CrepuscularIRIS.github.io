Long-context language models present a fundamental engineering tension: the quadratic complexity of self-attention means that every time you double the context window, you quadruple the compute. Models that handle 1M tokens with standard attention would be economically undeployable — the inference cost alone would preclude any real-world use.

Kimi K2.5 sidesteps this by stacking three architectural decisions that each independently reduce the attention complexity burden, and combining them with a reasoning approach borrowed from test-time compute scaling research. The result is a model that processes 1M tokens at lower inference cost than GPT-4o on 128K, while outperforming it on long-document benchmarks.

This post breaks down each component technically.

## The Quadratic Problem in Concrete Numbers

Standard self-attention over a sequence of length $n$ requires $O(n^2)$ memory and compute for the attention matrix. At 128K tokens, this is already expensive — roughly 16 billion attention operations per layer. At 1M tokens, the same calculation requires 1 trillion operations per layer. That's a 64× increase in compute for an 8× increase in context length.

No amount of hardware scaling makes this viable for production inference. The architectural choices have to change.

The three techniques Kimi K2.5 combines — sparse attention, dynamic depth routing, and improved positional encodings — each attack a different part of this problem.

## Sparse Attention: Not All Tokens Need to Attend to All Others

The core observation motivating sparse attention is that for most practical long-context tasks, any given token only needs to attend to a small subset of the context. A sentence in the middle of a legal document doesn't need direct attention to every word in every other paragraph — it needs local context (nearby sentences) and global anchors (key definitions, headings, named entities).

Kimi K2.5 uses a **Mixture of Sliding Windows** approach: rather than a single fixed window size, the model routes each attention head to one of several sliding window scales depending on the input content. 

```python
class MixtureOfSlidingAttention(nn.Module):
    def __init__(self, d_model: int, num_heads: int, window_sizes: list[int]):
        super().__init__()
        self.heads_per_window = num_heads // len(window_sizes)
        self.windows = nn.ModuleList([
            SlidingWindowAttention(
                d_model=d_model,
                num_heads=self.heads_per_window,
                window_size=w,
            )
            for w in window_sizes
        ])
        # Router: predict which window scale is useful for this token position
        self.router = nn.Linear(d_model, len(window_sizes))

    def forward(self, x: torch.Tensor, position_ids: torch.Tensor) -> torch.Tensor:
        # Compute routing weights per token
        routing_logits = self.router(x)          # [B, T, num_windows]
        routing_weights = torch.softmax(routing_logits, dim=-1)

        # Weighted sum of outputs from each window scale
        outputs = []
        for i, window_attn in enumerate(self.windows):
            out = window_attn(x, position_ids)   # [B, T, d_model]
            outputs.append(out * routing_weights[..., i:i+1])

        return sum(outputs)
```

The key insight: code tends to have high local density (bracket matching, variable scope) and benefits from narrow windows; narrative text benefits from paragraph-level (medium) windows; document-level reasoning tasks (legal, scientific) need wider windows to bridge distant references.

By making the window size a soft, learned decision rather than a fixed hyperparameter, the model allocates attention bandwidth to where it's actually needed.

The complexity reduction is significant. A 4096-token sliding window over 1M tokens requires $O(n \cdot w)$ attention operations where $w = 4096$, compared to $O(n^2)$ for full attention. At 1M tokens this is roughly a 250× reduction in attention compute.

## Dynamic Depth Routing: Matching Compute to Task Complexity

The second architectural innovation addresses a different inefficiency: uniform depth. In a standard 32-layer transformer, every input passes through all 32 layers regardless of whether the task requires it. Simple factual lookups don't need the same depth of processing as multi-step mathematical reasoning.

Kimi K2.5 implements **dynamic depth routing**: a lightweight gating mechanism that decides, per-token and per-layer, whether to execute the full transformer block or skip it. The practical effect is that easy tokens exit the network early, concentrating compute on the positions that actually require deep processing.

| Task Type | Effective Layer Utilization | Relative Inference Cost |
|-----------|---------------------------|------------------------|
| Simple factual retrieval | ~40% | 0.6× |
| Code completion | ~75% | 0.85× |
| Complex multi-step reasoning | 100% + reflection | 2–4× |
| Long-document summarization | ~60% | 0.75× |

The table shows approximate numbers from the K2.5 technical report — the actual routing decisions are content-dependent and vary at token granularity. The aggregate effect across a realistic workload mix is roughly a 30% reduction in inference compute compared to always executing all layers.

This is structurally similar to mixture-of-experts routing, but applied to depth rather than width. The model doesn't add parameters — it decides how many existing parameters to actually use for a given token.

## Positional Encodings: Extrapolating Beyond Training Length

Long-context models face a position encoding problem: even if the architecture can efficiently process 1M tokens, positional encodings trained on 128K sequences don't generalize reliably to longer positions. The model has never seen position 500,000 during training.

The standard fix — training on longer sequences — is expensive. Kimi K2.5 instead uses **NTK-aware interpolation** on rotary positional embeddings (RoPE), which allows the model to extrapolate to context lengths significantly beyond what it was trained on with minimal quality degradation.

The mathematical basis: standard RoPE encodes position $m$ in dimension $2i$ as:

$$\theta_{m,i} = m \cdot \theta_{base}^{-2i/d}$$

When position $m$ exceeds the training length $L_{train}$, the rotation angles become out-of-distribution. NTK-aware scaling applies a learned interpolation factor $\alpha$ to the base frequencies:

$$\theta_{m,i} = m \cdot \left(\alpha \cdot \theta_{base}\right)^{-2i/d}$$

The scaling factor $\alpha$ is set such that the highest-frequency dimensions of the positional encoding don't saturate within the extended context window. Lower-frequency dimensions (which encode coarse positional structure) are left largely unchanged; higher-frequency dimensions (which encode fine-grained local position) are dampened.

The result is that the model can "read" positions it was never trained on without catastrophic confusion, at the cost of slightly reduced precision for very fine-grained positional distinctions beyond the training length.

## What Long Context Actually Means (And Doesn't)

A common misconception: a model that can process 1M tokens can also reliably *remember* anything in that context at arbitrary distance.

This is wrong in an important way. Attention-based transformers are processors, not memories. When generating token $t$, the model attends over all previous tokens — but the signal from token $1$ has been progressively transformed and compressed through every layer of the network. Information from very early in the context doesn't arrive at the generation step with the same fidelity as information from nearby context.

The practical implication: long-context models perform well on tasks that require *finding and synthesizing* information from distant positions (e.g., "find all mentions of X across this document and summarize their relationship"). They perform worse on tasks that require *verbatim recall* of precise details from the beginning of a very long context.

Kimi K2.5 addresses this with a **hierarchical key extraction** step during preprocessing: before the main forward pass, a lightweight module scans the input and inserts explicit "anchor" tokens at semantically important positions (key definitions, section headings, entity introductions). These anchors receive artificially boosted attention weights during the main pass, giving the model an addressable index into the long context.

## Benchmark Results in Context

On LongBench-v2, K2.5 reports:

| Model | Max Context | LongBench-v2 Score | Relative Inference Cost |
|-------|------------|-------------------|------------------------|
| GPT-4o | 128K | 62.3 | 1.0× |
| Claude 3.5 Sonnet | 200K | 64.1 | 1.2× |
| Kimi K2.5 | 1M | 67.8 | 0.7× |
| Gemini 1.5 Pro | 2M | 65.2 | 1.5× |

The 0.7× inference cost figure deserves scrutiny. This is relative cost *per token processed*, and the 1M context means the absolute cost of a max-context K2.5 query is still much higher than a typical 128K GPT-4o query. The efficiency advantage matters most for workloads that need long context — if your median query is 10K tokens, K2.5's architectural optimizations don't help much.

Where the efficiency gains are real: document-level analysis tasks (legal, scientific, financial), codebase-level reasoning (understanding a large repo as a whole), and multi-document synthesis. For these tasks, K2.5 can process the full context in a single pass rather than requiring chunking or retrieval augmentation.

## Engineering Implications

**When to use K2.5 vs. RAG:** For document collections where you can precompute embeddings and the query is relatively narrow, retrieval-augmented generation with a shorter-context model is usually cheaper. K2.5's long context becomes cost-effective when the query requires synthesizing information across many distant positions that are hard to retrieve independently, or when the latency cost of multi-step retrieval is unacceptable.

**Prompt structure matters more at long context:** The hierarchical anchor mechanism works better when the prompt itself is structured — clear section headers, explicit key definitions early in the document. Unstructured walls of text stress the attention pattern in ways that structured documents don't.

**For research applications:** 1M-token context opens genuinely new research workflows. Processing a full conference proceedings, an entire codebase, or a multi-year experimental log in a single context window changes what kinds of synthesis questions are tractable. The bottleneck shifts from "can the model process this?" to "how do I formulate the right query?"

## What Comes Next

The three techniques in K2.5 — sparse attention, dynamic depth, extended positional encoding — are individually well-understood. The contribution is their combination and the engineering required to make them work together reliably at production scale.

The next frontier is architectures that combine long-context attention with explicit external memory. Current long-context models still have soft limits: the attention pattern becomes increasingly noisy at the far end of a 1M-token context, and no amount of positional encoding engineering fully fixes this. Hybrid architectures that use attention for local/medium-range context and retrieval for global/distant references are the most promising direction for pushing beyond current practical limits.

The research questions around long-context evaluation are also underexplored. LongBench-v2 is a reasonable benchmark, but it doesn't fully capture the failure modes that matter most in practice — tasks where the model appears to answer correctly but has silently dropped critical information from the early context. Building evaluation suites that expose these failure modes is as important as building better architectures.
