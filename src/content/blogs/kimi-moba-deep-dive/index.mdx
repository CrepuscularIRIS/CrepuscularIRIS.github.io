The quadratic attention problem is old news. Longformer gave us local windows plus global tokens. BigBird mixed sliding windows, random attention, and global tokens. Linformer projected keys and values to a lower rank. Each approach traded some amount of capability for scalability, and none of them fully escaped the need to hand-engineer the sparsity pattern.

Moonshot AI's MoBA (Mixture of Block Attention, arXiv:2502.13189) takes a different approach: let the model decide which parts of the context to attend to, using the same routing logic that makes Mixture of Experts work — but applied to attention blocks instead of FFN experts. No trainable router parameters. No fixed sparsity patterns. Just a top-k selection over block-averaged key vectors.

The result is a method that was deployed in Kimi's production environment on the day the paper was published, handles 200K+ token inputs in the live service, and achieves up to 40× speedup over full attention at 32K sequence length.

## Why Fixed Sparsity Patterns Break Down

The intuition behind Longformer is sound: most tokens in long documents need to attend to nearby context, and a few special tokens need global visibility. But "most tokens" and "nearby context" are approximations. In practice, the relevant context for a given query might be a paragraph from 50,000 tokens ago, or a specific clause from a contract in a different section.

Fixed patterns can't model this. If the relevant KV blocks aren't in the local window or the global token set, they're invisible to the attention head, and the information is permanently lost from that position's perspective. This is why Longformer and BigBird, despite their efficiency, tend to underperform full attention on tasks requiring long-range retrieval — the sparsity pattern doesn't know where the important stuff is.

The correct fix is data-driven sparsity: let each query dynamically select which key-value blocks to attend to, based on the content of the query itself. This is the core idea behind MoBA.

## The MoBA Architecture

The setup is straightforward once you see the MoE analogy.

In a standard MoE layer, the input is a token embedding, and the router uses a learned linear projection to compute a score over N experts, then activates the top-k. Each expert is a separate FFN. The token only flows through its k assigned experts.

In MoBA, the input is a query vector, and the "router" computes a score over M blocks of the KV context, then activates the top-k. Each block is a contiguous chunk of key-value pairs. The query only attends to its k assigned blocks.

```
Standard Full Attention:
  Q: (N, d)
  K, V: (N, d)
  Output: Attention(Q, K, V)  —  O(N²) computation

MoBA Sparse Attention:
  Q: (N, d)
  K, V: (N, d) → split into M blocks of size B = N/M

  For each query q_i:
    score_j = q_i · mean(K_block_j)   # scalar score per block, j=1..M
    selected = argtop_k(scores)        # select k blocks
    output_i = Attention(q_i, K[selected], V[selected])

  Overall complexity: O(N · k · B) ≈ O(N) when k, B << N
```

The key design decision is the **parameter-free gate**: instead of learning a separate router network (linear layer + softmax over M experts), MoBA uses the dot product between the query and each block's mean key vector. This is a natural measure of relevance — if the block's average key is close to the query in embedding space, the block likely contains relevant context.

This sidesteps the main failure mode of learned MoE routers: routing collapse, where the router consistently assigns all tokens to a few popular experts, leaving the rest undertrained. Since MoBA's gate has no trainable parameters, there's nothing to collapse.

## Implementation: Naive vs. Efficient

The open-source repository (github.com/MoonshotAI/MoBA) provides two implementations that are worth understanding at different levels:

### Naive Implementation (Research Use)

The naive version constructs a sparse attention mask and feeds it to FlashAttention. It's readable and correct:

```python
import torch
import torch.nn.functional as F
from einops import rearrange

def moba_naive(
    q: torch.Tensor,           # (batch, heads, seq_len, d_head)
    k: torch.Tensor,           # (batch, heads, seq_len, d_head)
    v: torch.Tensor,           # (batch, heads, seq_len, d_head)
    num_blocks: int,           # M: number of KV blocks
    top_k: int,                # k: blocks to attend per query
) -> torch.Tensor:
    B, H, N, D = q.shape
    block_size = N // num_blocks
    
    # Reshape keys into blocks, compute per-block mean key vectors
    # k_blocks: (B, H, M, block_size, D)
    k_blocks = rearrange(k, 'b h (m bs) d -> b h m bs d', m=num_blocks, bs=block_size)
    # mean_keys: (B, H, M, D)
    mean_keys = k_blocks.mean(dim=3)
    
    # Score each query against each block's mean key
    # q: (B, H, N, D), mean_keys: (B, H, M, D)
    # scores: (B, H, N, M)
    scores = torch.einsum('bhnd,bhmd->bhnm', q, mean_keys)
    
    # Top-k block selection per query
    # topk_indices: (B, H, N, k)
    _, topk_indices = scores.topk(top_k, dim=-1)
    
    # Build boolean mask: (B, H, N, N)
    # query i can attend to position j iff j's block is in topk_indices[i]
    mask = torch.zeros(B, H, N, N, dtype=torch.bool, device=q.device)
    for block_idx in range(num_blocks):
        block_start = block_idx * block_size
        block_end = block_start + block_size
        # Which queries selected this block?
        selected = (topk_indices == block_idx).any(dim=-1)  # (B, H, N)
        mask[:, :, :, block_start:block_end] |= selected.unsqueeze(-1)
    
    # Standard attention with the sparse mask
    scale = D ** -0.5
    attn = torch.einsum('bhnd,bhmd->bhnm', q * scale, k)
    attn = attn.masked_fill(~mask, float('-inf'))
    attn = F.softmax(attn, dim=-1)
    return torch.einsum('bhnm,bhmd->bhnd', attn, v)
```

This is O(N·M) for the scoring step and O(N·k·B) for the attention step — already better than O(N²) when `k·B << N`, but the mask construction adds overhead that makes this unsuitable for production.

### Efficient Implementation (Production Use)

The efficient version operates at the CUDA kernel level, skipping the mask entirely. Instead of constructing a dense N×N mask, it directly gathers the selected KV blocks and runs FlashAttention on the gathered tensors:

```python
# Pseudocode for the efficient kernel logic
# (actual implementation in CUDA via flash-attn ≥ 2.6.3)

def moba_efficient_kernel(q, k, v, block_indices):
    """
    block_indices: (N, k) — pre-computed top-k block indices per query
    
    Instead of masking, we gather selected KV tiles directly:
    """
    outputs = []
    for i, query in enumerate(q):  # (d_head,) per query
        selected_k = k[block_indices[i]]  # (k * block_size, d_head)
        selected_v = v[block_indices[i]]  # (k * block_size, d_head)
        # Flash attention on the gathered k·B tokens
        out_i = flash_attn(query.unsqueeze(0), selected_k, selected_v)
        outputs.append(out_i)
    return torch.stack(outputs)
```

The 40× speedup cited in the paper comes from this kernel operating on 32K sequences with top-k=3 and 2048 blocks — each query attends to only 3 × (32768/2048) = 48 tokens instead of 32768. The reduction in KV cache reads is ~99.5%, and since memory bandwidth is the bottleneck for long-context inference, the speedup is roughly proportional.

```python
    }

```

Three findings from the ablations worth flagging:

**Block size matters less than you'd expect.** Varying B from 16 to 64 tokens has relatively small impact on downstream task accuracy. The routing quality (can the gate find the right blocks?) dominates over the granularity of what's in each block.

**k=3 is the inflection point.** Increasing top-k from 1 to 3 gives large accuracy gains; going from 3 to 8 gives diminishing returns. This mirrors the finding in MoE literature that top-2 routing is usually sufficient for FFN experts.

**Per-layer independent routing outperforms cross-layer shared routing.** Different layers learn to route to different blocks, suggesting each layer extracts qualitatively different information from the long context.

## Full Attention / Sparse Attention Toggle

One underappreciated engineering feature: the same model weights can switch between full attention and MoBA sparse attention at inference time by simply changing top-k.

```python
class MoBALayer(nn.Module):
    def __init__(self, d_model, num_heads, num_blocks=2048, default_top_k=3):
        super().__init__()
        self.num_blocks = num_blocks
        self.default_top_k = default_top_k
        self.attn = MultiHeadAttention(d_model, num_heads)
    
    def forward(self, x, top_k=None, mode="sparse"):
        if mode == "full" or top_k is None and mode != "sparse":
            # Standard full attention — equivalent to selecting all blocks
            return self.attn(x)
        else:
            k = top_k or self.default_top_k
            return moba_efficient(x, x, x, self.num_blocks, k)

# Precision calibration on a small batch: use full attention
model.set_mode("full")
calibration_output = model(calibration_batch)

# Production inference on 200K token inputs: use MoBA
model.set_mode("sparse", top_k=3)
production_output = model(long_context_input)
```

This toggle is valuable for production deployment: you can verify model correctness with full attention on small inputs, then deploy with MoBA for long-context efficiency without maintaining two separate model weights.

## How MoBA Compares to Competing Approaches

| Method | Sparsity Type | Trainable Router | Complexity | Long-Range Recall |
|--------|--------------|------------------|------------|-------------------|
| Longformer | Fixed (window + global) | No | O(N) | Limited by window size |
| BigBird | Fixed (window + random + global) | No | O(N) | Better, still fixed |
| MoE Attention | Learned routing | Yes | O(N·k·B) | Data-driven |
| **MoBA** | **Data-driven, parameter-free** | **No** | **O(N·k·B)** | **Data-driven** |
| GQA/MQA | Full attention, shared KV | No | O(N²/H) | Full |

The distinctive position MoBA occupies: data-driven sparsity without the training instabilities of a learned router. It's not the fastest approach for short sequences (GQA wins there), but at 100K+ tokens it dominates on the efficiency/recall tradeoff.

## Production Impact

The fact that MoBA shipped to production on paper publication day is not a footnote — it's evidence that the design was constrained by deployment realities, not just benchmark targets.

A method that achieves 98% compute reduction but requires architectural changes that break existing inference infrastructure doesn't actually help a production team. MoBA's compatibility with FlashAttention and its drop-in replacement semantics (same interface as standard attention, just with routing arguments) made deployment tractable.

Moonshot AI has reported that MoBA's production deployment directly enabled the 200K context window in Kimi's live service, and that the engineering lessons from that deployment influenced subsequent model iterations. VMoBA (arXiv:2506.15465), a variant adapted for video diffusion models, extends the same routing principle to 3D spatiotemporal attention — suggesting the approach generalizes beyond text.
