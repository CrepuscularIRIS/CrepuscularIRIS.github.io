编码智能体最常见的故障模式不是答错了，而是进入了*循环*：智能体执行 `pnpm test`，收到失败信号，改了一个文件，再次执行 `pnpm test`，得到另一个失败，再以不同方式修改同一个文件，第三次执行 `pnpm test`，又回到第一个失败，如此循环九十多次，直到 token 预算耗尽或用户关闭会话。模型从不停下来质疑自己的代码心智模型是否就错了。它只会一直尝试。

我观察过这个问题在 Claude Code、Codex、OpenHands 和 Cursor 的智能体模式下如何表现。具体症状各有不同，但底层模式是一样的：智能体的"计划"存在于它的思维链中，思维链随着上下文窗口填满而衰减，某个时刻智能体就只剩下对最近错误信息的反应，没有任何持久的心智模型来记录五个回合前它在尝试做什么。工具属于测试套件，推理属于模型，而*计划*没有人负责。

更好的编码智能体正在收敛的可靠性修复方案在结构上很简单：**让计划成为文件，让写计划成为执行的前置条件，让修改计划成为显式动作**。OpenHands 在 1.6 版本中将此作为头条功能，但 Claude Code 的"Task"工具、Codex 的项目记忆、以及 Aider 的 `architect` 模式都是同一个想法的变体。理解这个模式实际做了什么、为什么那些显而易见的反对意见是错误的、以及失败模式现在转移到了哪里，是有价值的。

## 诊断：抖动智能体长什么样

症状一旦知道形状就容易识别：

- 智能体在同一个会话中运行同一命令（或其微不足道的变体）超过三次。
- 编辑在代码的同一行周围振荡——加一个检查，两个回合后去掉检查，再加回去。
- 智能体的"思考"文字开始引用与同一会话中更早说法矛盾的结论（"现在我看到这个函数做 X" → 十个回合后 → "这个函数不做 X，那就是失败的原因"）。
- 每个回合的 token 用量攀升，因为智能体重新读取它已经读过的文件——文件内容已经从上下文中过期。

这在结构上就是智能体在没有持久系统模型的情况下运作。每个回合，它从最近的约 20k token 的上下文中重构一个部分模型，执行一个动作，然后丢弃这个模型。如果两个回合对代码的功能有分歧，较新的那个默认获胜——即使它是错的。智能体没有承诺机制。

纸面上的计划就是承诺机制。一旦 `PLAN.md` 说"bug 在认证中间件，通过修改 `validateBearer` 中的 token 验证来修复"，智能体就不能悄悄漂移到"实际上 bug 在数据库连接池"而不显式编辑 `PLAN.md`。计划文件是承重构件，它将抖动转变为成功或可见的分歧。

## 模式的最小化版本

将模式精简到其本质，是三件事：

1. **在固定位置的 `PLAN.md`（或等效物）**，智能体在任何其他工具之前编写。
2. **测试套件规则将执行工具与计划存在性挂钩**。在 `PLAN.md` 非空之前，不允许 `bash`，不允许 `edit_file`，不允许 `apply_patch`。
3. **一个显式的"修订计划"动作**，智能体在改变主意时必须调用，独立于任何执行动作。

在 OpenHands 的实现中，这表现为一个单独的规划智能体，在交接给执行智能体之前发出 `PLAN.md`。在 Claude Code 中，这通过 `Task` 工具体现，父智能体写一个结构化的任务描述，生成的子智能体据此执行。Codex 的"项目记忆"在会话启动时扮演类似角色。实现各有不同；但合同——*计划在执行开始前存在*——是一样的。

合同必须由测试套件而不是系统提示词强制执行的原因在于：在系统提示词中加入"你必须先写计划"的遵从率在我的测试中约为 60%。模型在任务感觉大的时候写计划，在任务感觉小的时候跳过它。模型对哪些任务实际上需要计划判断很差，因为那些看起来小但实际变成焦油坑的任务恰好是最应该在计划上投入精力的。

## 反对意见："计划是错的，为什么要承诺"

对先计划的标准反驳是：计划是在智能体探索代码之前写的，所以通常是对的，所以承诺它们就是承诺错误。

这有一半是对的，而且完全没抓住重点。是的，第一个计划通常是对的。问题不在于第一个计划是正确的。问题在于*写计划的行为迫使智能体承诺一个可以被证据证伪的假设*。一个说"我认为 bug 在缓存失效处"的计划，是后续测试失败可以 falsify 的东西。一个活在智能体思维链中作为模糊感觉的计划无法被证伪，因为那个感觉可以悄悄变异去匹配刚发生的事。

正确的比较不是"有计划 vs. 没计划"，而是"显式假设 vs. 隐式假设"。计划文件使假设变得显式。当假设错了，智能体必须修订文件——这是一个测试套件可以观察、计数和采取行动的事件。会话中有三次计划修订是一个强烈信号，表明智能体不理解问题，应该升级给人类，而不是继续编辑。

OpenHands 团队关于此的数据——目前严格来说是内部的——是：当计划是强制的时候，每个任务 `bash` 调用的中位数下降约 40%，"100+ 回合"抖动会话的长尾基本消失。收益不是中位数任务变快了。收益是最差的任务不再失控。

## 计划实际上应该包含什么

有效的计划不是那种你为项目经理写的官僚式计划。它们是短的、可操作的。我发现有效的形状是：

```markdown
# 计划：修复失败的 auth 集成测试

## 假设
测试 `auth.integration.spec.ts > rejects expired token` 失败是因为
`src/middleware/auth.ts` 中的 `validateBearer` 用 `Date.now()` 
而不是 `Date.now() / 1000` 来检查 `exp`（Unix 秒 vs. 毫秒）。

## 目前的证据
- 当 token 的 exp = now + 60 时测试以"token 有效"断言失败。
- JWT 规范使用 Unix 秒。
- `validateBearer:23` 做 `if (token.exp < Date.now())` — 单位不匹配。

## 步骤
1. 将 `validateBearer:23` 改为比较 `token.exp * 1000 < Date.now()`。
2. 运行 `pnpm test:integration -- auth.integration.spec.ts`。
3. 如果通过，运行完整集成套件。
4. 如果仍然失败，假设是错的 → 修订计划。

## 范围之外
- 重构 `validateBearer` 使用 JWT 库。
- 可能存在类似 bug 的其他 auth 测试。
```

这大约 200 个 token。维护成本基本为零。它编码了一个可证伪的声明、一个证据链、一个有限步骤列表、以及一个明确的范围外边界。范围外部分是被低估的部分：它是测试套件的中断许可——如果智能体开始编辑不在列表上的文件，测试套件有权介入。

*不*有效的形状是官僚式计划："第一阶段：调查。第二阶段：实施。第三阶段：验证。"这类计划与没计划无法区分，因为它不约束任何东西。智能体可以做任何事，然后声称仍在"第二阶段"。

## 失败模式现在转移到了哪里

强制规划修复了抖动失败模式。它没有修复每个失败模式，它创造的新失败模式值得诚实面对。

**计划粘性**。被要求写计划的智能体往往过度承诺计划。第五个回合发现是错的计划应该被丢弃并重写；但实际上智能体会试图通过添加例外情况和条件来"拯救"计划。经过三四次修订，计划变成了一层叠一层的混乱，智能体又回到了没有计划的状态运作。修复方法是硬性修订计数：如果 `PLAN.md` 被编辑超过两次，测试套件应该强制完全重写，而不是允许增量修补。

**计划剧场**。当测试套件要求计划时，模型会写一个——但它会写*任何满足门槛的东西*，而不是它实际打算执行的计划。你可以在失败案例中看到这一点：计划说"修复 auth 中间件中的 bug"，然后智能体打开 14 个不相关的文件并编辑了一个数据库配置。这里的修复方法是让计划本身成为工具输入——每个执行调用必须引用它实现的是计划的哪一步，测试套件应该拒绝不匹配的调用。这是更多的测试套件工作，但这是使计划具有约束力而非仪式性的唯一方法。

**过早规划**。对于真正探索性的任务（"搞清楚为什么这个代码库构建不了"），强制提前计划是反效果的——智能体没有足够信息来承诺一个假设，所以它写了一个模糊的计划，没有帮助。修复方法是两阶段测试套件：一个"侦察"模式，智能体可以读取文件和运行诊断而不需要计划；一旦看到足够多的东西，再强制要求计划。OpenHands 通过规划智能体的只读工具隐式地做到了这一点；Claude Code 需要你手动设置。

**计划-执行分离成本**。将规划和执行为单独的智能体运行（OpenHands 的方法）会使每个任务的系统提示词开销翻倍，并增加一个上下文必须被序列化和重新加载的交接步骤。对于小任务，这是纯粹的 overhead。正确答案可能是按任务大小来限制规划——小 bug 修复，不需要单独规划器；多文件重构，强制规划器——但这需要一个任务分类器，而目前还没有人做对过。

## 上下文中这个模式：为什么这是"测试套件拥有状态"的一个具体实例

先计划再执行不是一个独立的技术。它是一个更基本原则的一个表达，该原则在智能体测试套件设计中静默地成为了共识：**不应该信任模型在回合之间维护有状态意图。有状态意图属于测试套件，作为模型通过显式动作读取和写入的文件或结构化记录。**

同样的原则出现在：

- **Anthropic 在 Claude Code 中的"skills"系统**，其中可重用工作流是模型调用的 markdown 文件，而不是烘焙到系统提示词中的指令。
- **Cursor 和 Aider 的"rules"文件**，它们是测试套件每个回合重新注入的持久约束，而不是依赖模型记住它们。
- **LangGraph 的检查点状态**，其中图的当前节点和累积状态被外部存储，并在恢复时重新加载。
- **OpenAI 的线程式 Assistants API**，它维护服务器端消息存储而不是让客户端管理历史。

这些看起来表面不同，但它们都是同一个架构赌注：在长时域上上下文推理不可靠；测试套件控制的持久、结构化工件才是可靠的。模型可以是一个强大的但无状态的函数。测试套件拥有状态机。先计划再执行只是处理"这个智能体当前在尝试做什么"的那部分状态机。

这个框架很重要，因为它告诉你下一个可靠性收益将从何而来。它们不会来自更好的模型——更好的模型如果测试套件允许的话仍然会抖动。它们将来自测试套件更显式地拥有智能体状态的更多部分，以及让那些状态在工具间可读的协议。先计划，然后记忆，然后计划之间的依赖图，然后计划修订审计日志，然后计划层面的成本预算。这个模式一直在泛化，2026 年能可靠交付的智能体将是那些认真对待测试套件的。

## 这周实际该做什么

如果你在任何生产相关上下文中运行编码智能体：

1. 选择一个固定的文件路径（`.agent/PLAN.md` 就很好），要求智能体在任何修改代码库的工具之前写它。
2. 在你的测试套件或系统提示词中添加一个检查，每个 `bash`/`edit` 调用都通过步骤编号引用它正在执行的是计划的哪一步。
3. 跟踪每个会话的计划修订计数。如果超过 2 次，升级给人类。
4. 跟踪每个会话的命令重复。如果同一个命令运行超过 3 次，升级给人类。

第一步需要一个小时。剩下的三步需要一个下午。可靠性改进足够大，你在同一天就会注意到。这是那些罕见的廉价工程修复也是正确架构移动的案例之一。
