SWE-bench Verified 存在污染问题。2024 年 12 月之后在 GitHub 数据上训练的模型见过测试集——不是完全一样的补丁，而是让任务变得可解决的仓库、issue 和周围代码。当 Nebius 在 2026 年 4 月发布 SWE-rebench 时，他们不仅仅提出了一个新的基准测试。他们证明了静态代码评估基准测试的整套方法论是失效的，并提出了一个结构性修复方案：从训练截止日期之后的活跃仓库中持续生成任务。

结果是横跨 Python、JavaScript、TypeScript、Java、Go、Rust、C++ 和 C# 的 3.2 万个多语言任务，全部来自 2025 年 1 月之后修改的仓库。每一项任务都保证晚于当前前沿模型的训练数据。这是第一个设计上可处理的代码基准测试——为再生而建，而非为持久而存。

## 污染机制

SWE-bench 的污染不是简单的答案记忆问题。机制更微妙，也更难检测。

在 2024 年底之前的 GitHub 数据上训练的模型见过 SWE-bench 测试集中的仓库。它未必见过用作标准补丁的特定 bug 修复提交。但它见过代码库结构、编码模式、测试基础设施，而且经常见过 issue 和 PR 中描述 bug 的讨论。当模型遇到 SWE-bench 任务时，它工作在一个熟悉的环境中，而非一个全新的环境。

这造成了一种信息不对称，使基准分数膨胀，却没有对应提升真实世界能力。一个"解决"了 SWE-bench 任务的模型可能正在做以下任何一件事：

1. **真正的解决问题**：理解 bug，推理代码库，生成正确的修复。这是基准测试打算测量的内容。
2. **对熟悉代码的模式匹配**：识别仓库并应用训练数据中学到的关于该代码库中 bug 通常如何修复的模式。
3. **隐式检索**：在训练数据中见过类似的问题或讨论，这些描述了修复方法，即使精确的提交不在训练集中。

问题是：当只测量测试套件的通过/失败时，(2) 和 (3) 与 (1) 无法区分。一个从训练中"深度了解"Django 代码库的模型，将以高于其实际 bug 修复能力的比例解决 Django 相关的 SWE-bench 任务——如果是在一个陌生的代码库中。

证据：多个模型在来自常见训练语料中频繁出现的仓库的 SWE-bench 任务与来自不太常见仓库的任务之间显示出统计学上显著的性能差异。差距从 8 到 15 个百分点不等——足以改变排行榜排名。

## SWE-rebench 的设计

SWE-rebench 通过四个结构性决策解决污染问题：

### 1. 截止日期后任务生成

每一项任务都来自 2025 年 1 月 1 日之后的提交。由于当前前沿模型没有早于此日期的训练数据，任务在发布时即保证无污染。更重要的是，生成管道被设计为持续运行——当被追踪的仓库中出现新提交时，新任务会自动生成。

这是关键转变：基准测试不是固定产物，而是按时间表产生新任务的管道。当模型最终在 2025 年数据上训练时，SWE-rebench 会从 2026 年提交中重新生成。

### 2. 多语言覆盖

SWE-bench 仅支持 Python。SWE-rebench 覆盖 8 种语言：

```
Language     | Tasks  | Repos Tracked | Avg. Repo Size
-------------|--------|---------------|----------------
Python       | 8,200  |      850      |    45K LoC
TypeScript   | 5,400  |      620      |    38K LoC
JavaScript   | 4,100  |      510      |    32K LoC
Java         | 3,800  |      440      |    55K LoC
Go           | 3,200  |      380      |    28K LoC
Rust         | 2,900  |      290      |    22K LoC
C++          | 2,400  |      260      |    68K LoC
C#           | 2,000  |      210      |    42K LoC
-------------|--------|---------------|----------------
Total        | 32,000 |    3,560      |
```

多语言覆盖之所以重要，是因为真实代码库是多语言的。一个在 Python 上表现卓越但在 TypeScript 上失败的智能体，对大多数生产仓库都没有用处。SWE-rebench 迫使模型展示跨语言能力，而非为单一语言优化。

### 3. 污染追踪

SWE-rebench 发布了污染指数：对每个模型-任务对，估计任务源仓库出现在模型训练数据中的概率。估计基于：

- 已知训练数据截止日期（来自模型卡片）
- 仓库创建和修改时间戳
- GitHub 星标数和分叉率（作为爬取可能性的代理指标）
- 发布训练数据组成时的显式包含/排除列表

这不会消除污染——它量化了污染。研究人员可以报告按污染概率分层的分数，更清晰地展示真实能力与训练熟悉度之间的差异。

### 4. 基于变异的难度层级

原始 bug 修复任务的难度差异很大，这使得总分难以解释。一个模型得 40% 可能是在解决所有简单任务同时在所有困难任务上失败，或者以中等可靠性解决多样化混合任务。

SWE-rebench 引入变异测试作为难度校准机制。每个任务关联一个变异分数——即金补丁存活的测试套件变异的百分比。高变异分数表示补丁在功能上精确（几乎任何 token 变化都会使其失效）。低变异分数表示许多变体也能通过测试，表明任务有多个有效解决方案或测试覆盖薄弱。

```python
# 变异分数示例
# 金补丁：修复分页中的 off-by-one 错误
def get_page(items: list, page: int, size: int) -> list:
    start = page * size  # Bug: 应该是 (page - 1) * size
    return items[start:start + size]

# 变异：改变运算符
def get_page(items: list, page: int, size: int) -> list:
    start = (page + 1) * size  # 变异: - → +
    return items[start:start + size]
# 此变异未能通过测试 → 推高变异分数

# 变异：改变常量
def get_page(items: list, page: int, size: int) -> list:
    start = (page - 2) * size  # 变异: 1 → 2
    return items[start:start + size]
# 此变异未能通过测试 → 推高变异分数

# 高变异分数 = 精确修复，强测试
# 低变异分数 = 许多补丁都能通过，测试薄弱
```

任务根据变异分数、代码复杂度指标（圈复杂度、文件数、依赖深度）以及贡献者数据中的人类解决时间估计被分为三个难度层级。

## 早期结果揭示了什么

第一批 SWE-rebench 评估包含了 8 个智能体系统。结果与 SWE-bench 排名的差异证实了污染假设。

**分数压缩。** 最佳与最差智能体之间的差距从 SWE-bench Verified 的约 25 个百分点缩小到 SWE-rebench 的约 12 个百分点。前排模型下降幅度大于后进模型，这与前排模型更多受益于训练数据重叠的情况一致。

**语言依赖的排名变化。** 在 SWE-bench（仅 Python）上排名前三的若干模型，在跨所有语言评估时跌至中游。Python 分数保持可比；多语言聚合将其拉低。这表明优化针对的是 Python 特定的代码模式，而非通用代码推理。

**难度层级反转。** 在 SWE-bench 最难任务（通常来自知名、高星标仓库）上得分最高的模型，在 SWE-rebench 困难层级（来自不太知名的仓库）上跌至第四。这是最清晰的信号，表明仓库熟悉度在人为地抬高分数。

**错误模式转移。** 在 SWE-bench 上，最常见的失败模式是"生成看似合理但不正确的补丁"。在 SWE-rebench 上，最常见的失败模式转变为"未能定位相关代码"——这表明在 SWE-bench 上，模型受益于对熟悉代码库中 bug 倾向发生位置的隐性知识。

## 更深层的问题：我们在衡量什么？

SWE-rebench 解决了污染问题。它没有解决更根本的问题：代码基准测试实际上在衡量什么能力？

一个 SWE-bench/SWE-rebench 任务捆绑了多种能力：

1. **仓库理解**：从有限上下文理解代码库结构、约定和架构
2. **Bug 定位**：识别哪些文件、函数和行与报告的问题相关
3. **补丁生成**：编写修复 bug 而不引入回归的代码
4. **测试解释**：充分理解测试套件以生成能通过的补丁
5. **工具使用**：使用搜索、文件读取和执行工具导航代码库

单一通过/失败分数将这五种能力压缩为一个数字。一个擅长仓库理解但生成错误补丁的智能体，与一个甚至找不到相关文件的智能体得分相同（都是 0）。

SWE-rebench 的难度层级部分解决了这个问题，确保层级内任务需要相似的能力分布。但该领域需要更细粒度的能力分解——分别针对定位准确性、正确定位条件下的补丁正确率，以及工具使用效率的分数。

### Code Arena：对抗性替代方案

PropelCode 的 Code Arena 采用了完全不同的方法。它不用精挑细选的 bug 修复任务，而是在随机采样的编程问题上运行一对一智能体对决，使用改编自国际象棋的 Elo 评分系统。

优势：
- **对抗性配对**减少了刷分——当你不知道自己会面对哪个任务时，无法针对特定任务优化
- **Elo 评分**提供带置信区间的相对排名，而非绝对百分比
- **实时问题**来自活跃仓库，从结构上消除了污染

劣势：
- **任务分布偏差**——随机采样的问题可能不代表真实开发工作的难度分布
- **Elo 不稳定性**——每个模型的小样本量在早期轮次产生宽置信区间
- **缺乏能力分解**——与 SWE-bench 存在相同的单分数问题，只是统计属性更好

理想评估结合 SWE-rebench 的结构化任务生成（受控难度、多语言、污染追踪）与 Code Arena 的对抗性配对（一对一比较、Elo 置信区间）。单独任何一个都无法捕捉全貌。

## 对智能体开发的启示

如果你在构建代码智能体，SWE-rebench 改变了若干开发决策：

### 不要为 SWE-bench 优化

SWE-bench 分数不再是可信的开发目标。如果你的智能体架构是为最大化 SWE-bench 性能而调优的——针对流行仓库的特定检索策略、为 Python Django/Flask 模式优化的提示模板、为 SWE-bench 测试基础设施校准的评估循环——这些优化可能无法转移到未受污染的任务上。

### 在陌生仓库上测试

污染问题延伸到了基准测试之外。如果你的智能体内部评估套件使用知名开源仓库，你的开发反馈循环存在相同的信息不对称。在私有仓库、最近创建的仓库、或你的模型较弱语言的仓库上测试。

### 投资定位而非生成

SWE-rebench 的错误分析表明，在未受污染的任务上，主要瓶颈从"生成正确补丁"转变为"找到正确的代码"。这表明智能体架构应更多投资于代码搜索、语义导航和仓库理解，而非补丁生成质量——大多数前沿模型在这方面已经足够。

### 从第一天起就多语言

在非 Python 语言中的性能下降并非固有的模型限制——而是基准测试驱动开发的产物，这些开发忽略了其他语言。尽早构建多语言工具使用和测试基础设施，防止 Python 特定假设的累积。

## 基准测试生命周期

SWE-rebench 建立了一个其他 AI 基准测试应遵循的模式：**基准测试应是再生管道，而非固定数据集。**

生命周期：

```
1. 从截止日期后数据生成任务
2. 发布任务 + 污染指数
3. 评估模型
4. 发布带污染分层的成果
5. 模型在新数据上训练（任务变得受污染）
6. 从更新的截止日期后数据重新生成任务
7. 重复
```

这比维护静态基准测试成本更高。它需要持续的策展、用于任务验证的持续计算，以及污染追踪基础设施。但静态基准测试有半衰期——它们在 12-18 个月内测量真实能力，然后越来越测量训练数据重叠。再生管道永远保持有效。

维护动态基准测试的成本低于基于污染分数做出开发决策的成本。SWE-rebench 使这种权衡变得明确。
