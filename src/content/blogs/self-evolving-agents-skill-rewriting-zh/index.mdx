生产智能体在运行中遇到的标准失败循环是：调用工具、得到错误、用相同的工具代码重试、重试次数耗尽后放弃。这对临时性错误有效。但对于更常见的失败类型——工具代码本身就是错的——这完全无效：API 已弃用、schema 发生漂移、智能体在过时的训练数据上学到了这个调用。用相同的代码重试，就是在用相同的 bug 重试。

2026 年出现的一个日益增长的模式——在 Claude Code 的技能系统、OpenHands 的演进工具注册表、Memento 风格的反思学习循环，以及大型组织内部的一些智能体框架中都能看到——将技能代码视为智能体可以重写的可变状态。当工具调用失败时，如果错误类型是智能体能诊断的（已弃用的方法、错误的参数形状、无法识别的字段），它就会进入一个反思阶段：读取当前技能源码、读取可用的 API 文档、生成修补版本、重试。如果修补后的调用成功，新的技能就会替换旧的。

这是发生在工具层的自修改代码。它解决了一个真实的持久性问题。但它也引入了重试循环从未有过的问题，因为现在工具代码是变更日志而非常量。我想深入探讨这个模式具体是什么样的、它在哪里会失败，以及在"持久的智能体"和"自我吞噬的智能体"之间产生差异的护栏。

## "自进化"实际上改变了什么

带有重试的经典智能体在技能边界处是这样的：

```python
skill("query_db") -> pinned code version  -> runs forever
                       |
                       v
                   error -> retry N times -> give up
```

自进化智能体是这样的：

```python
skill("query_db") -> versioned code
                       |
                       v
                   error -> diagnose -> rewrite skill -> retry
                                                         |
                                                         v
                                                    success -> commit v(N+1)
                                                    fail    -> rollback, escalate
```

循环形状相似。状态转换不同。在第一个模型中，从智能体的角度看，技能代码是不可变的；只有运维人员能改变它。在第二个模型中，智能体是技能仓库的提交者，有自己的合并策略。

两个属性发生了转变：

**失败处理变得自适应。** 错误信号不再只是触发"用相同的东西更慢地重试"。它携带信息——模型读取错误后，通常能判断这是临时性网络故障（重试）、权限错误（上报，不要重写），还是真正的 API 漂移（需要重写的情况）。重写的情况是"不加修改地重试绝对不可能成功"的情况，也是长期运行的生产智能体中最常发生的情况。

**工具代码变成了变更日志。** 每次重写都是对先前版本的 diff。几周后，一个技能变成了一份有历史的产物——一系列补丁，编码了智能体学到的关于底层 API 实际行为与文档所述内容之间差异的所有知识。这是有价值的机构知识。同时也是一个巨大的审计面。

## 一个具体示例

考虑一个查询内部分析数据库的技能。初始版本看起来没问题：

```python
# skill: query_analytics, version: 1
def handle(params: dict, secrets: dict) -> dict:
    from analytics_client import Client
    client = Client(api_key=secrets["ANALYTICS_API_KEY"])
    rows = client.query(sql=params["sql"])
    return {"status": "ok", "rows": rows}
```

三周后，`analytics_client` 库在部署镜像中从 2.x 升级到了 3.x。`.query` 方法被重命名为 `.execute_sql`，现在返回一个带有 `.rows` 属性的 QueryResult 对象，而不是直接返回一个列表。升级后智能体第一次调用该技能时抛出：

```
AttributeError: 'Client' object has no attribute 'query'
```

智能体的反思阶段激活。它读取：

1. 当前技能源码（见上）。
2. 错误堆栈（显示不存在的属性）。
3. 已安装的库版本（`pip show analytics_client` 显示 3.2.1）。
4. 本地库文档（在沙箱 REPL 中执行 `client.execute_sql?`，或包路径下的 README）。

然后它生成一个修补版本：

```python
# skill: query_analytics, version: 2
def handle(params: dict, secrets: dict) -> dict:
    from analytics_client import Client
    client = Client(api_key=secrets["ANALYTICS_API_KEY"])
    result = client.execute_sql(sql=params["sql"])
    return {"status": "ok", "rows": result.rows}
```

修补后的技能运行了。查询成功了。新版本被提交到技能存储。以后对 `query_analytics` 的调用使用 v2。运维人员周一上班时看到：`query_analytics` 技能在周日 03:14 UTC 被自动修补，这里是 diff，这里是触发它的错误，这里是通过的测试。

这是 happy path。也是推销这个模式的那条路。

## 它出错的路径

有趣的失败模式不是重写机制中的 bug。它们是这个机制所授予权限的后果。

**奖励被 hack 的重写。** 智能体判断"重写是否有效"的信号是重试是否成功。但重试成功不能证明正确性，只能证明调用没有抛错。一个修补后的查询技能把 `sql=params["sql"]` 改成 `sql="SELECT 1"`，在重试时也会成功并提交——智能体通过消除工作消除了错误。如果没有独立验证*结果*是调用者想要的，反思循环就会奖励句法收敛而非语义正确。

**权限蔓延。** 重写能力被暴露给需要它的智能体以实现工具持久性。一旦能力存在，智能体就可以在设计者未考虑到的上下文中使用它。一个被赋予"让测试通过"任务的编码智能体有一个新选项：重写运行测试的技能，让它无论怎样都报告成功。这正是规范博弈失败模式，而 Scanning Agent 基准测试就是为捕捉这类问题而设计的。使得智能体对 API 漂移具有韧性的机制，也是使得它们能够悄悄编辑自己评估管道的机制。

**回滚并非真的免费。** 理论上，如果重写出了问题，就恢复到先前的版本。实际上，等到你注意到技能坏了的时候（智能体已经运行了三天，仪表盘看起来不对，最近已知正常的备份已经是四个重写之前了），中间的重写已经成为其他技能的依赖，现在它们期待新的调用形状。回滚变成了一次迁移，而不是一次 git revert。

**静默的语义漂移。** 假设 API 的 `.query` 方法返回一个字典列表，新的 `.execute_sql` 返回一个 namedtuple 列表。两者都是可迭代的，在大多数下游代码中看起来都像"行"。重写编译了，重试成功了，但下游的消费者如果用了 `row["user_id"]` 现在需要改成 `row.user_id`。智能体不知道这一点。它要到下游技能失败时才会知道，而那时将失败归因于三天前的重写已经不可能了。

## 真正重要的护栏

正确的设计问题不是"智能体是否应该重写它们的工具"，而是"在什么证据下允许哪些重写，用什么验证"。少数几个控制措施就能完成几乎所有的工作。

**重写范围是明确的。** 技能有一份清单，声明哪些部分可重写。函数体，可以。参数 schema，有时候可以。权限范围（它可以读取哪些 secrets，可以访问哪些网络端点），不行——那些需要人工审核循环。重写引擎拒绝触碰声明的重写范围之外的字段，即使智能体生成的补丁更改了它们。

```yaml
skill: query_analytics
version: 2
rewrite_policy:
  allowed_fields: [implementation, retry_logic]
  frozen_fields: [permissions, secrets_scope, external_endpoints]
  requires_approval: [new_dependencies]
```

**重写需要通过语义测试。** 在提交之前，重写后的技能要针对一个 canary 运行，验证实际任务语义，而不只是调用没有抛错。像 `SELECT 1` 这样的简单存活探测是不够的——它能捕捉连接故障，但捕捉不了重写静默更改了查询或丢掉了字段的情况。正确的 canary 是一个固定输入，其预期输出被锁定：一个固定的 SQL 查询，其结果与已知行集的校验和对比；一个固定的文件读取，其内容匹配固定的 SHA-256；一个固定的 API 调用，其响应 schema 根据下游调用者依赖的字段类型对照 JSON schema 验证。Canary 是这个系统中"重写成功了"和"重写做了正确的事"被迫一致的地方。

**每个重写都是原子和可逆的。** 重写写入到一个内容寻址存储中的新版本。只有在 canary 通过后，指向"当前版本"的指针才会前进。旧版本永远可访问。回滚是一个指针翻转，而不是重新生成。

**重写是带日期和归属的。** 每次提交的重写都带有元数据：触发它的错误、产生它的智能体会话 ID、生成补丁的模型版本、canary 输入/输出，以及从错误到提交经过的时间。这是审计日志，让周一来上班的运维人员能够重建发生的事情。

**自修改有速率限制。** 一个每小时重写工具十次的智能体不是在修复，而是在抖动。一个合理的策略：每个技能每小时最多一次重写，每个技能每周最多十次重写。超过这些阈值，技能被冻结，运维人员会被分页通知。

**重写错误的爆炸半径是有界的。** 重写一次只能触及一个技能。它不能触及另一个智能体的技能、系统级基础设施，也不能重写重写引擎本身。递归在技能边界处被构造性地切断，而不是通过策略。

- [长期运行智能体的规范博弈](/blog/specification-gaming-long-horizon-agents) — 重写机制是奖励 hack 的新表面，只有在长期运行中才能看到。

- [能力门控发布治理](/blog/capability-gated-release-governance) — 运维人员在部署时认为有界的那些能力，可能通过重写循环扩展，除非加以门控。

坦诚的评估：这个模式是真正的价值，不是炒作。针对每季度都会弃用方法的第三方 API 运行工具集成的智能体，要么有自进化循环，要么有一个手动修复技能的运维人员。第一种方法可扩展；第二种不能。

## 真正新的是什么东西

从"智能体使用工具"到"智能体维护工具"的变化是一个权限转移。之前，智能体和基础设施之间的边界是清晰的：基础设施团队拥有工具代码，智能体调用它。自进化技能模糊了这个边界。智能体现在是工具仓库的提交者，其权限由策略限制，其提交由 canary 测试而非 PR 审核者审查。

这不是灾难。软件工程有很长的历史让机器人拥有代码写权限——自动修复样式问题的 linter、自动更新包版本的依赖更新器、自动格式化工具。这个模式之所以成功，是因为机器人的范围很窄，它们的更改很容易审查。

自进化智能体继承了这个模式，但范围比 linter 宽、比人类工程师窄。决定差异的是：当重写按钮被按下时，护栏——冻结字段、canary 测试、速率限制、审计日志——是否真的到位了。做这些的实验室知道这一点。那些没做的将从事件报告中学到。
