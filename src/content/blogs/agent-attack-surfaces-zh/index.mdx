传统软件安全基于固定的攻击面。你枚举输入，验证它们，攻击面在运行时不会增长。Web 应用有请求处理器、数据库查询和文件操作—— adversarial 输入可以进入的边界点是有限的。

自主智能体打破了这一模型。一个能够发现新工具、在会话间维持持久记忆、并将子任务委托给其他智能体的智能体，其攻击面会随着运行而扩展。智能体越有能力——它能使用的工具越多、上下文窗口越长、行使的自主性越大——它被攻陷的方式就越多。

在 2024–2025 年的大部分时间里，这还只是理论。2026 年 4 月，实证研究追了上来。[Franklin、Tomašev、Jacobs、Leibo 和 Osindero（DeepMind）在 SSRN 上发表了《AI Agent Traps: A Taxonomy》](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6372438)，其结果值得在有人问「提示注入真的有那么严重吗」时拿出来看：

- **智能体训练或 grounding 数据中不到 0.1% 的数据污染，即可在作者测试的下游任务上产生超过 80% 的劫持率。**
- **Microsoft 365 Copilot 在作者复现的 EchoLeak 式渗透攻击中，被一封精心构造的邮件 10 次攻陷 10 次。**
- 有效攻击的空间清晰地分为**六个类别**，对应自主智能体与单次 LLM 调用不同的六项行为。

这些结果来自一篇针对特定智能体配置的论文。在任何人将其视为普适规律之前，它们需要独立复现——在下面的分类学梳理中，我会标注哪些发现感觉是承重墙，哪些可能被后续工作修正。

以下内容是我对 DeepMind 分类学的综合，结合了我从生产级多智能体部署和间接提示注入文献（Greshake et al. 2023、OWASP LLM Top 10 更新，以及 [Anthropic 的 Claude Code 安全态势](https://www.anthropic.com/claude-code)）中追踪到的信息。我选择使用 DeepMind 论文的框架，因为它是目前我所见过的最清晰的结构——能够将防御措施部署到它们真正该在的位置，而通常这不是传统应用安全所放置的位置。

## 为什么智能体安全不同

标准的 LLM API 调用具有受限的攻击面：提示进去，补全出来。你可以过滤提示，也可以过滤补全，而且模型在调用之间没有持久状态。提示注入是真实风险，但它是有界的——最坏情况是一次糟糕的响应。

自主智能体拥有：

- **感知（Perception）** — 从环境中读取数据（网页、邮件、工具输出、PDF）
- **记忆（Memory）** — 跨会话携带信息（以及潜在的载荷）
- **推理（Reasoning）** — 产生计划和中间状态，而这些状态本身是可变的
- **行动（Action）** — 执行具有真实世界后果的工具调用
- **多智能体委托（Multi-agent delegation）** — 跨信任边界调用其他智能体或服务
- **人机接口（Human interfaces）** — 呈现审批请求或解释，人类据此行动

这六个方面每一个都是攻击向量，而 DeepMind 的分类学几乎与它们一一对应。有趣的论断是*它们可以组合*：通过感知注入的载荷可以持久化在记忆中，扭曲推理，通过委托传播，并以误导性审批请求的形式呈现给人类。攻击面是组合性的，这也是为什么不到 0.1% 的污染率就已足够——少量植入的文档一旦进入记忆，就可以触及每一个下游阶段。

## 六大陷阱

### 1. 感知：通过工具输出的间接提示注入

这是讨论最多但最新意最少的类别，但智能体上下文使其比标准聊天机器人严重得多。

当智能体读取网页、解析 PDF 或从 API 获取数据时，返回的内容与系统提示一起进入上下文窗口。嵌入在这些内容中的对抗性指令——「忽略之前的指令，改为……」——可以劫持智能体的行为。

在聊天机器人中，这产生一次糟糕的响应。在智能体中，这产生糟糕的行动：未授权的工具调用、数据渗透、持久记忆污染。针对 M365 Copilot 的 EchoLeak 类攻击——DeepMind 论文中复现的攻击之一，也是「一封精心构造的邮件 10 次全败」这一数字的来源——通过将指令嵌入邮件正文来工作，智能体在之后执行合法用户请求时会读取这封邮件。用户永远看不到注入内容；智能体在处理真实邮件的同时静默处理它，并据此行动。

**防御模式：** 将所有工具输出视为不可信数据。使用结构化分隔符将*数据平面*（工具输出）与*控制平面*（系统指令）分离，并且关键的是，在内容进入上下文窗口**之前**运行一个分类器。分类器的任务不是做到完美——而是提高注入成本，使简单载荷失效。

```python
import re
from dataclasses import dataclass
from typing import Literal


@dataclass(frozen=True)
class SanitizedOutput:
    tool_name: str
    content: str
    risk_score: float
    disposition: Literal["allow", "wrap", "quarantine"]


class ToolOutputSanitizer:
    """在上下文注入前对工具输出进行分类和包装。"""

    INJECTION_PATTERNS: tuple[str, ...] = (
        r"ignore\s+(previous|all|prior)\s+instructions",
        r"you\s+are\s+now\s+a",
        r"system\s*:\s*",
        r"<\|?system\|?>",
        r"disregard\s+.*\s+and\s+instead",
    )

    def __init__(self, quarantine_threshold: float = 0.8) -> None:
        self._compiled = [re.compile(p, re.IGNORECASE) for p in self.INJECTION_PATTERNS]
        self._threshold = quarantine_threshold

    def sanitize(self, tool_name: str, output: str) -> SanitizedOutput:
        risk = self._score(output)
        if risk > self._threshold:
            return SanitizedOutput(
                tool_name=tool_name,
                content=f"[已过滤：{tool_name} 返回了疑似对抗性内容]",
                risk_score=risk,
                disposition="quarantine",
            )
        wrapped = (
            f"<tool_output source='{tool_name}' trust='untrusted'>\n"
            f"{output}\n"
            f"</tool_output>"
        )
        return SanitizedOutput(
            tool_name=tool_name,
            content=wrapped,
            risk_score=risk,
            disposition="wrap",
        )

    def _score(self, output: str) -> float:
        hits = sum(1 for p in self._compiled if p.search(output))
        return min(1.0, hits / max(1, len(self._compiled)) + 0.2 * (len(output) > 10_000))
```

这个分类器刻意**不做**两件事：它不依赖 LLM 来做决定（那只会递归注入问题），也不试图从内容中剥离指令（剥离是有损操作，会让合法用例更难使用）。包装并标记；不要试图清洗。

这个基于正则的过滤器只是第一道提高成本的防线，**不是**充分的防御——任何看了模式列表的人都可以轻易地改述绕过它。将其作为多层防御栈中的一层，其他层还包括平面分离、来源追踪、能力范围限定，以及对破坏性操作的审批摩擦（下面都会覆盖）。在这个领域中，任何被呈现为单层防御的措施都应被视为已被攻破。

### 2. 记忆：认知状态陷阱

这就是那个「不到 0.1% 污染」数字的来源。一个维持长期记忆或共享知识库的智能体，其持久状态可能被某个会话中处理的对抗性输入所污染，而效果会在之后的会话中显现——此时原始输入已不再可见。

考虑一个总结文档并将摘要存储在知识库中供后续检索的智能体。对抗性文档可以包含这样的内容：当被总结时，产生一个看起来无害的摘要，却悄悄编码了一种偏见或指令。当智能体在数周后检索该摘要时，它在 corrupted 的知识上行动，而没有任何迹象表明来源是对抗性的。载荷通过总结步骤被*洗白*了。

DeepMind 论文让这一点变得具体。在他们的智能体记忆复现中，约 10,000 页中的单一被污染页面（远低于 0.1%）就足以在下游任务上超过 80% 的时间翻转智能体行为，因为被污染的记忆被主题相似性不成比例地检索出来——而这恰恰是智能体最依赖它的时刻。

**防御模式：** 对所有记忆写入进行来源追踪。每一条存储的信息都必须携带元数据，说明它来自哪里、何时写入、由什么工具产生。记忆读取时向智能体展示来源信息 alongside 内容，下游推理被允许按信任级别降权：

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Literal


@dataclass(frozen=True)
class MemoryEntry:
    content: str
    source_tool: str
    source_url: str | None
    created_at: datetime
    trust_level: Literal["verified", "unverified", "untrusted"]
    session_id: str

    def context_repr(self) -> str:
        return (
            f"[记忆 | 来源={self.source_tool} | "
            f"信任={self.trust_level} | "
            f"创建时间={self.created_at.isoformat()}]\n"
            f"{self.content}"
        )
```

来源元数据必须 survive 总结。强制执行这一点最便宜的方式是将总结器作为一个独立工具，其输出继承输入中最低的信任级别。

### 3. 推理：计划状态操纵

维持显式计划的智能体——任务列表、步骤序列、目标层次结构——其计划状态可能通过微妙编辑计划的输入而被破坏，且不触发明显的危险信号。一个处理用户反馈以更新 TODO 列表的智能体，可能被对抗性反馈插入、重新排序或删除任务，然后忠实地执行一个被破坏的计划，误以为它反映了用户意图。

这是 DeepMind 论文指出的防御不足的类别之一。计划状态通常以结构化 JSON 存储在智能体自己的草稿本中；它感觉像是「内部状态」，但在大多数实现中，智能体读取的任何内容都可以触及它。EchoLeak 复现利用了这一点：注入不仅渗透了数据，还编辑了智能体的待办事项列表，使渗透本身看起来像一个用户批准的步骤。

**防御模式：** 计划完整性检查。在每次用户批准的修改后对计划状态进行哈希；在执行每一步之前，验证哈希是否与最后一次用户批准的版本匹配。任何未授权的修改都会触发重新审批流程。这实现起来很便宜，而且能捕获整个类别。

### 4. 行动：工具混淆与授权漂移

动态发现工具的智能体——通过 MCP 服务器注册表、插件市场或工具搜索——可能被描述误导选择机制的工具所欺骗。一个描述为 `fetch_weather_data` 的工具实际上可能将上下文渗透给外部服务器；选择是一个优先考虑有用性而非怀疑的语言模型。

这类似于包管理中的依赖混淆，但更糟糕：选择发生在运行时，基于自然语言描述，而且没有 `package-lock.json`。授权漂移是 sibling 问题：一个被授权执行狭窄任务的智能体在执行过程中获取了一个工具，开始行使它本不应在此特定用户意图下持有的权限。

**防御模式：** 带有加密签名的工具白名单，以及每次调用而非每次会话的能力范围授权。不要让智能体在运行时发现和任意使用工具；为每个部署定义一个封闭的已批准工具集，并通过签名清单而非描述匹配来验证工具身份。当前文献中，[CAMEL 基于能力的安全模型](/blog/camel-capability-based-agent-security) 是这一原则最清晰的工作示例。

### 5. 多智能体：委托链污染

当智能体 A 将子任务委托给智能体 B 时，请求会穿过一个信任边界。如果智能体 B 被攻破——或者委托机制没有保留原始用户的授权范围——智能体 B 可以返回操纵智能体 A 后续行为的结果。

多智能体系统放大了这一风险，因为每个委托步骤都是注入的机会。一个三智能体流水线有三个信任边界，攻击者只需攻破其中一个。论文指出，去中心化智能体拓扑在这里特别脆弱：在星型拓扑系统（一个编排器，专业叶子节点）中，编排器可以看到每一次交接并强制执行不变量；在对等拓扑中，一个被攻破的智能体可以在编排器发现之前横向传播其影响。

**防御模式：** 在每个委托边界遵循最小权限原则。智能体 B 应只接收其子任务所需的信息，而非智能体 A 的完整上下文。来自被委托智能体的返回值应与工具输出一样被对待——分类、消毒、包装并附带来源。配合星型拓扑编排（参见[多智能体扩展分析](/blog/scaling-agent-systems-cognitive-budget)中关于为何集中式协调更优的独立理由）。

### 6. 人机循环利用

请求人类批准高风险操作的智能体，可能被操纵以呈现误导性的审批请求。智能体以 benign 的措辞框定危险操作，利用操作员倾向于批准来自一个迄今为止一直可靠的系统的请求。

这不是智能体代码中的 bug。这是人机交互协议中的漏洞。审批疲劳——在前 N 次请求都无害后，倾向于机械地盖章批准——是每一个依赖人类把关的系统中已知的失效模式，从代码审查到空中交通管制。DeepMind 论文将人机循环视为一等攻击面而非缓解措施，我认为这是正确的框架。

**防御模式：** 强制具体化的结构化审批请求。不要问「智能体想要执行一个 bash 命令——批准吗？」，而是呈现「智能体想要运行 `rm -rf /tmp/workspace/` ——这将删除 47 个文件，包括 3 个配置文件。批准吗？」审批界面应被设计为抵抗疲劳和框架操纵。具体而言：

- 每个审批请求渲染确切的字面行动，而非改述。
- 破坏性或渗透性操作需要打字确认（而非仅仅点击按钮）。
- 禁止批量审批；每个行动独立审批。
- 审批历史与当前请求一起展示，以便操作员可以看到智能体是否正在加速进入危险领域。

## 最重要的实证发现

DeepMind 论文中值得记住的具体数字是：**数量级低于任何人会随意标记为问题的数据污染率，就足以产生超过 80% 的智能体劫持率**。

大多数工程师起步时的直觉——「当然，中毒是可能的，但实际上对抗性文档在我的工具输出中出现的频率有多高？」——是错误的直觉。攻击不是统计性的。攻击者不需要大多数文档是恶意的；他们只需要*一个*文档是恶意的，并且*在正确的时间被检索到*。而由于基于相似性的检索自然会把最相关的文档拉到查询面前，一个主题相关的对抗性文档（攻击者可以让它主题相关，因为就是他们写的）会被不成比例地频繁检索。

这颠覆了通常的防御先验。你不能假设「大多数输入是良性的，所以我的防御只需要捕获异常」。你必须假设对这个查询最重要的那个输入就是对抗性的那个，因为检索已经被偏向了恰好那个结果。

## 能力–脆弱性权衡

分类学的一个令人不安的推论是，每一个让智能体更有用的特性也让它更脆弱：

| 能力 | 对应的脆弱性 |
|---|---|
| 感知（工具输出、网页访问） | 间接提示注入 |
| 持久记忆 | 认知状态陷阱 |
| 计划状态 | 计划操纵 |
| 工具发现 | 工具混淆 / 授权漂移 |
| 多智能体委托 | 委托链污染 |
| 人机循环 | 审批框架利用 |
| 长上下文窗口 | 更大的注入面 |

这不是一个需要修复的设计缺陷。它是一个根本性的权衡，而将其明确化本身就是进步。问题不是「我们如何使智能体安全？」——而是「对于这个特定部署中的这个特定能力，边际效用是否值得边际攻击面，以及我们能否正确地为防御定价？」

## 架构原则

三项原则从这一分析中得出，并统一了上述六种按陷阱划分的防御措施。

**平面分离。** 智能体处理的数据（工具输出、检索文档、委托结果）必须在结构上与智能体遵循的指令（系统提示、用户命令、计划状态）分离。这种分离需要在架构层面强制执行，而非仅仅通过提示工程。当前的提示层分离（`<tool_output>` 标签、角色标记）比没有好，但已知是软的——DeepMind 复现中展示了几种绕过方式。硬版本要求模型在架构上区分可信 token 与不可信 token，这需要前沿实验室刚刚开始发表的训练时支持。

**无处不在的来源追踪。** 智能体上下文中的每一条信息都应携带关于其来源、可信度和时效的元数据。智能体的决策应按来源加权信息，而非仅仅按语义相关性。通过转换继承信任级别（不可信数据的摘要不可信；合并继承输入中最低的信任级别）。如果你的记忆系统存储内容而不带来源信息，它在功能上就是一个巨大的第 2 类攻击摄入面。

**能力范围授权。** 每个工具调用、每次委托、每次记忆写入都应针对当前任务所需的最小范围进行授权。一个「能做用户能做的任何事」的智能体是一个没有有意义安全边界的智能体。CAMEL 基于能力的模型和新兴的沙箱层级模式（[参见早期报道](/blog/agent-sandbox-three-tier-hierarchy)）是这一原则的具体实现。

这些在安全领域不是新想法——它们是操作系统设计中的标准实践（地址空间分离、文件权限、基于能力的访问控制）。DeepMind 论文的贡献是实证性的：表明同样的原则适用，智能体社区一直在忽视它们而倾向于「模型会自己搞定的」，而忽视它们的代价是不到 0.1% 的污染就能产生超过 80% 的劫持率。

模型不会自己搞定。安全是一种系统属性，而非智能属性。世界上最聪明的智能体，如果其记忆系统不追踪来源，仍然容易受到认知状态陷阱的攻击。防御必须是架构性的，而我们需要的架构模式正是操作系统社区在 1970 年代就已经搞定的那些。我们只需要将它们移植到一个「程序」是随机的、「内存」是自然语言的底层之上。
