2026 年 4 月初的七十二小时之内，针对长期运行 Agent 的两个记忆架构相继落地，方向却完全相反。Andrej Karpathy 发布了一份 gist，描述他所谓的 *LLM Wiki*：模型被当作一个"编译器"，离线读入论文、转录、推文、Slides 等原始素材，输出一组互相链接的 Markdown 维基页面，之后只要这个话题再次出现，相关页面就被整体喂回 context。三天之后，**MemPalace** 项目以一个 MCP server 形式发布（按公开 writeup 大约二十个工具），给 Agent 一套空间隐喻——翼楼、房间、抽屉、隧道——让模型在查询时主动在记忆图里"漫步"。

这两个设计不可能在强意义上同时正确。它们对**何时做工作、结构存在哪里、运行时模型该做什么**这三件事的回答，互相矛盾。把它们放在一周内并排来看，是我见过的对一个被 Agent 记忆文献系统性回避的问题的最清晰呈现：你是提前**编译**记忆，还是按需**检索**记忆？

这个问题不新。NLP 已经吵过一次，符号知识库 vs 向量数据库，向量一方赢得彻底，"记忆"几乎成了"检索"的同义词。Agent 这个新场景把数学条件改了，问题又被打开了。

## 两个架构的并排对比

Karpathy 的 LLM Wiki 在机制上是一条**编译流水线**。素材进来，一个模型——这里被字面意义地当作离线 transformer 使用——逐份阅读，并输出几个共享 schema 的 wiki 页面（论点、引用、相关页、悬而未决的问题）。页面之间互相链接。后续提问时，要么在 wiki 里 grep 出相关页直接整体加载到 context，要么让模型沿链接自己导航。检索这一步几乎是 trivial 的：wiki 已经按照你将来要问的问题被组织好了，因为它就是为这些问题写的。

MemPalace 是它的对偶。它完全不做离线编译。它给 Agent 一组 MCP 工具——像 `memorize`、`enter_room`、`open_drawer`、`read_diary`、`dig_tunnel` 这样，总共大约二十个——让模型在推理时调用。记忆是一次工具调用；回忆是对 Agent 增量构建出来的空间结构的一次**遍历**。schema 就是空间隐喻本身：翼楼把相关房间归在一起，房间把相关抽屉归在一起，抽屉装少量条目，隧道是两个远处位置之间的显式跨链。查询是一连串 MCP 调用，模型必须**走过去**才能找到要的东西。

两个系统都声称自己实现了"长期 Agent 记忆"，也都能援引 Karpathy 早先对 in-context 记忆的批评作为论据，证明需要更持久的东西。但它们用相反的原语回答同一个问题。Wiki 把结构放进**产物**里，让运行时模型消费一个已经写好的页面；MemPalace 把结构放进**导航 API**里，让运行时模型自己发现正确的节点。

## 为什么这是符号 vs 向量之争的复刻

历史让这个权衡变得清楚。符号知识库——Cyc、ConceptNet、Wikidata——是"编译"路线的极限。先建 schema，再仔细填充，查询引擎就只是一层薄薄的查表。代价提前付：人工（后来半自动）的本体工程。回报是查询确定、可审计、可组合：你能把多个查询串起来，因为你知道每个槽位是什么意思。

向量库把代价挪到了另一头。把一切都嵌入到高维空间，用近似最近邻搜索把"看起来像"的东西捞回来。不需要维护 schema，不需要做策划。代价在查询时付：一是检索的二义性（top-k 邻居互相相关，但不一定对齐到查询本身），二是下游模型要把检索出来的碎片缝成一段连贯文本。

RAG，在它现在的主流形态里，就是带了一层薄重排的向量记忆。它能赢得早期 Agent 记忆市场，原因和向量库赢得 2010 年代后期知识存储市场的原因一样：不需要离线 schema 工作，能扩展到任意语料，模型自身的灵活性能消化检索噪声。2026 年改变的是——长期运行的 Agent 暴露出了一个 RAG 从未被设计来解决的问题。向量检索给你的是**相似度**，不是**角色**。一个已经针对某个话题工作了六个月的 Agent，并不需要五十段相似的段落，它需要**那一页**——它自己以前就这个话题写过、按它当时的方式组织、带着它当时加的显式链接的页面。

这就是 wiki 给你的东西。这也正是 MemPalace 想给你的东西，只是路径不同——MemPalace 让 Agent 在工作过程中**增量地**构建结构，而不是要求一次离线编译。

## 各自的失败模式

Wiki 模式在两个地方塌掉：新鲜度和话题漂移。编译按定义就是快照。如果素材源源不断进来，你需要一套重编译的调度，而任何增量编译都得面对一个事实：新素材有可能**作废**已有页面上的论点。Karpathy 在 gist 里用一句"周期性重新生成 wiki"打发了这件事，至少很诚实。更难的问题是——新素材并没有作废、而是**精化**了某个现有页面，编译流水线得决定是覆盖、版本化还是 fork，没有干净的局部规则。任何维护过真 wiki 的人都知道，这正是当年维基百科编辑们花最多时间互相吵的部分。

Wiki 还在跨页推理这件事上塌掉。加载一个页面没问题，但一个困难问题真正需要的十几个相关页面一起加载，可能直接吃完 context 预算——尤其当模型想沿着引用链跳三跳验证某个论断时。Wiki 给你的是静态链接结构，但不是**自适应的深度控制**。

MemPalace 在另一个方向上塌掉。空间隐喻在头几周很好用：房间、抽屉、隧道映射到一个研究者实际整理思路的方式。但宫殿一旦变大，隐喻就停止 scaling。这时 Agent 在每一次导航上都要付出一次非平凡的推理成本——每个 `enter_room` 都是一次 MCP round-trip 加一次模型调用决定下一步去哪。如果模型需要走过五个房间才到目标抽屉，那就是五次推理之后**实质**推理才开始。Wiki 因为提前付了结构化成本，可以跳过这些导航。

MemPalace 还有一个 wiki 没有的验证问题。当 Agent 往一个抽屉里写东西时，没有任何机制告诉它"这和隔壁抽屉里的东西矛盾"。空间布局不强制语义一致。Wiki 流水线可以在每次编译之后跑一遍一致性检查，因为产物足够小、足够结构化。空间记忆是**堆积式**生长，堆积加上没有全局视图，等于让模型不断累积它自己永远在同一个 context 窗口里看不到的矛盾。

## 两者都没解决的六个问题

两个架构共有一份开放问题清单，gist 和 MCP 的兴奋情绪没怎么提到这部分。值得点名，因为这些问题最后会决定哪种设计赢。

1. **时间追踪。** 两者都不记录某个事实**何时**被加入、**何时**最近被精化。一个六个月前为真、现在已经过时的页面或抽屉，看起来和一个全新的没有区别。
2. **衰减。** 记忆条目按相同权重对待，与访问频率或年龄无关。一个工作中的研究者对某个话题的心智模型是非线性衰减的，wiki 和宫殿都没捕捉到。
3. **整合。** 跨页面或跨抽屉的重复模式应该被合并到一个更高阶的抽象。两者都没有整合 pass。
4. **弃答。** 两者都没有干净的方式让 Agent 说"我不知道，我的记忆里也没有有用的东西"。两者都会返回**某个东西**——一个近似匹配的 wiki 页、一个错的抽屉——把检测漏召的工作丢给模型。
5. **验证。** 两者都不强制 Agent 把召回的条目和源头交叉核对。Karpathy 的 wiki 带着引用，但不坚持要重读。
6. **关系。** 两者都处理两两链接——A 引用 B、A 抽屉到 B 抽屉的隧道——但都不捕捉**有类型**的关系，比如"A 精化 B"或"A 和 B 矛盾"，除非 Agent 自己发明约定。

这些就是 1980-90 年代符号知识库项目持续撞到的那一类问题，换了一身新词。区别在于我们现在有一个原则上**能**自己做整合、衰减、矛盾检测的运行时——只要记忆系统被设计来把这些东西暴露出来。Wiki 和宫殿都还没把这件事接上。

## 一个工作假设：分叉，而不是胜负

强读法是：两个模式都没赢。弱读法——也是更符合本周观察的那一种——是**长期记忆按访问形态分叉**：

- **编译式记忆**（wiki 形态）服务于**深而窄**的召回：Agent 有显式理由长期维持一个连贯长文视图的少数话题。个人研究笔记、一份在写代码库的设计依据、对一个具体的人或项目的画像。
- **导航式记忆**（宫殿形态）服务于**广而关系**的召回：Agent 可能用得到、但不指望长期维护连贯视图的长尾事实。参考资料、什么时候做了什么的 episodic 日志、一百个互不相关任务的工作状态。

两者不可互换，但互补。难的工程问题是**交接**：Agent 在什么时候决定把一个被频繁访问的抽屉**晋升**成一个 wiki 页？又在什么时候把一个陈旧页面**降级**回抽屉？诚实的回答是——还没有原则性规则。朴素规则"访问 N 次就晋升、M 天没访问就降级"和 Web 缓存用了三十年的规则一样，在这里同样脆弱。

我希望两个系统的下一版能给的是一个**显式的晋升-降级回路**，由模型主导，正如一个工作中的研究者会无声地在每天的草稿和长期笔记本之间来回搬条目。Karpathy 在 gist 里隐约提到这一点（模型被允许重构页面）；MemPalace 通过 `dig_tunnel` 隐约提到这一点（模型被允许加新的结构链接）。两者都还没把晋升-降级回路当作一个其他工具可以挂钩的一等对象。

## 这对 harness 意味着什么

如果分叉假设成立，harness 的形状就要变。今天的生产 harness 一般只 ship 一个记忆子系统——通常是向量库加一层 episodic 日志。分叉化的设计会 ship 至少两个：

- 一条**编译流水线**：定义了 schema 和按计划（或事件触发的）重编译。Agent 写原始笔记，流水线转成页面。
- 一个**导航面**：少量的 MCP 工具（memorize、recall、link、traverse）。Agent 在推理时直接用。
- 一个**晋升控制器**：观察前两者，把条目在它们之间双向搬运。

晋升控制器是任何我本周看过的开源 Agent 栈里都还没有的部分。它会把**随时间变聪明**的 Agent 和**随时间变大**的 Agent 区分开。光编译产生一份漂亮但陈旧的档案。光导航产生一座庞大但未整合的宫殿。控制器是闭环。

Karpathy 的 wiki 是闭环的一半。MemPalace 是另一半。2026 年真正有意思的工作，是它们都没 ship 的那一部分。
