The default instinct when building agent systems is to load as much context as possible. If the model has a 1M token context window, the reasoning goes, fill it. More information means better decisions. The repository has 500 files? Load them all. The conversation history is 200 turns? Keep every turn. The documentation spans 50 pages? Concatenate everything.

This instinct is wrong, and the evidence from production agent deployments in early 2026 makes the case quantitatively. Systems that practice "context hydration" — starting with minimal context and fetching precisely what's needed on demand — consistently outperform systems that maximize context loading. The gap is large enough to matter: 30-40% better task completion rates in code generation benchmarks, with 60-80% lower token costs.

## The Attention Dilution Problem

Language models attend to all tokens in their context window, but attention is not uniform. The attention mechanism distributes a fixed probability mass across all input tokens. As the context grows, each relevant token competes with more irrelevant tokens for the model's attention budget.

This isn't a theoretical concern. Empirical studies of needle-in-a-haystack retrieval show that model accuracy on finding specific facts degrades as context length increases — even when the model's context window technically supports the length. A fact that the model retrieves with 98% accuracy in a 4K context drops to 85% in a 128K context and further to 70% in a 512K context, depending on the fact's position.

For agent tasks, this translates directly to performance. An agent deciding which function to modify has to identify the right function among all the functions loaded into context. If you loaded 500 files "just in case" but only 3 are relevant, the model's attention is diluted across 497 irrelevant files. The probability of correctly identifying and modifying the right function drops.

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class ContextMetrics:
    """Quantify context quality, not just quantity."""
    total_tokens: int
    relevant_tokens: int
    relevance_ratio: float  # relevant / total
    estimated_attention_on_target: float  # decreases as total grows

    @staticmethod
    def from_context(context_files: list[str], target_files: list[str]) -> "ContextMetrics":
        total = sum(count_tokens(f) for f in context_files)
        relevant = sum(count_tokens(f) for f in target_files)
        ratio = relevant / total if total > 0 else 0.0
        # Simplified model: attention ~ ratio * position_bonus
        attention_estimate = ratio * 0.9  # degradation factor
        return ContextMetrics(total, relevant, ratio, attention_estimate)
```

## The Hydration Pattern

Context hydration inverts the default approach. Instead of loading everything upfront, the agent starts with a minimal "seed context" and uses tools to fetch additional context as needed. Each fetch is justified by the agent's reasoning about what it needs next.

The pattern has three phases:

**Phase 1: Seed.** The agent receives only the task description and a manifest of what's available (file tree, API list, schema summary). No file contents. No conversation history beyond the current task.

**Phase 2: Navigate.** The agent examines the manifest and uses retrieval tools to fetch specific artifacts. It reads one file, discovers an import, fetches that dependency, reads a test file to understand expected behavior. Each fetch is deliberate and traceable.

**Phase 3: Execute.** With a curated context of 5-20 relevant artifacts (instead of 500 irrelevant ones), the agent performs the task. The context is small, dense, and directly relevant.

```python
class HydratingAgent:
    """Agent that fetches context on demand instead of loading everything."""

    def __init__(self, model, tools: dict):
        self.model = model
        self.tools = tools
        self.context_log: list[dict] = []  # audit trail of fetches

    def solve(self, task: str, manifest: dict) -> str:
        # Phase 1: Seed — task + manifest only
        seed_context = self._build_seed(task, manifest)

        # Phase 2: Navigate — agent decides what to fetch
        hydrated_context = self._hydrate(seed_context)

        # Phase 3: Execute — generate solution with curated context
        return self._execute(task, hydrated_context)

    def _hydrate(self, seed: str) -> str:
        """Let the agent iteratively fetch what it needs."""
        context = seed
        for _ in range(10):  # max 10 fetch rounds
            response = self.model.generate(
                system="You are analyzing a codebase. Decide what to read next, "
                       "or say READY if you have enough context to solve the task.",
                user=context,
                tools=self.tools,
            )
            if "READY" in response.text:
                break
            if response.tool_calls:
                for call in response.tool_calls:
                    result = self.tools[call.name](**call.params)
                    context += f"\n\n# Fetched: {call.name}({call.params})\n{result}"
                    self.context_log.append({
                        "tool": call.name,
                        "params": call.params,
                        "tokens": count_tokens(result),
                    })
        return context

    def get_fetch_audit(self) -> list[dict]:
        """Return the audit trail of all context fetches."""
        return self.context_log
```

The audit trail is a significant operational advantage. When the agent makes a mistake, you can inspect exactly what context it had access to and identify whether the issue was a missing fetch (context gap) or a wrong interpretation (model error). In a "load everything" approach, this attribution is impossible — the model had access to everything, so the failure must be a model limitation.

## The Economics

Context hydration reduces token costs substantially. In a code generation benchmark with repositories averaging 100K tokens of source code:

| Approach | Avg Input Tokens | Task Success | Cost/Task |
|----------|-----------------|--------------|-----------|
| Full context | 95,000 | 54% | $0.14 |
| RAG top-20 files | 35,000 | 58% | $0.05 |
| Hydration (on-demand) | 12,000 | 71% | $0.02 |

The hydrating agent uses 87% fewer tokens and achieves 31% higher success rates. The cost reduction is straightforward: fewer input tokens means lower API bills. The performance improvement requires explanation.

The hydrating agent outperforms full-context loading because its context is curated by reasoning, not by heuristic retrieval. The RAG approach selects files based on embedding similarity to the task description — a reasonable heuristic that misses structural dependencies (the file that imports the broken module, the test that exercises the failing path). The hydrating agent follows the dependency chain deliberately, reading imports, examining call sites, and fetching test files that exercise the specific behavior.

## Tool Design for Hydration

The effectiveness of context hydration depends on the quality of the retrieval tools available to the agent. The tools must support both structural navigation (follow imports, read specific functions) and semantic search (find files related to a concept).

Minimum viable tool set for a code-focused hydrating agent:

```python
HYDRATION_TOOLS = {
    "read_file": {
        "description": "Read a specific file by path",
        "params": {"path": "string"},
    },
    "search_symbols": {
        "description": "Find definitions of a function, class, or variable",
        "params": {"symbol_name": "string", "kind": "function|class|variable"},
    },
    "find_references": {
        "description": "Find all call sites of a function or usages of a class",
        "params": {"symbol_name": "string"},
    },
    "search_content": {
        "description": "Semantic search across the codebase",
        "params": {"query": "string", "max_results": "int"},
    },
    "read_function": {
        "description": "Read a specific function body with its type signature",
        "params": {"file_path": "string", "function_name": "string"},
    },
    "list_directory": {
        "description": "List files in a directory with brief descriptions",
        "params": {"path": "string"},
    },
    "get_imports": {
        "description": "Get all imports of a file",
        "params": {"file_path": "string"},
    },
}
```

Two design principles matter:

**Granularity.** `read_function` is more useful than `read_file` when the agent needs one function from a 400-line file. Loading the full file adds 350 lines of noise. The tool should support the minimum viable unit of retrieval.

**Structural navigation.** `find_references` and `get_imports` let the agent follow the dependency graph without loading the entire graph into context. The agent reads function A, discovers it calls function B via `find_references`, fetches function B, and builds a chain of exactly the relevant code. This is the agent equivalent of a developer clicking "Go to Definition" in their IDE.

## Memory Architecture for Persistent Agents

For agents that operate across multiple sessions (long-running refactoring, multi-day code reviews), context hydration extends into memory architecture. The agent doesn't load its entire history into context at the start of each session. Instead, it maintains a structured memory store and hydrates from it on demand.

```python
@dataclass(frozen=True)
class MemoryEntry:
    """A single unit of agent memory."""
    key: str
    content: str
    memory_type: str  # "decision", "observation", "plan", "error"
    relevance_tags: tuple[str, ...]
    created_at: str
    token_count: int


class MemoryStore:
    """Structured memory with selective hydration."""

    def __init__(self):
        self._entries: dict[str, MemoryEntry] = {}

    def store(self, entry: MemoryEntry) -> None:
        self._entries[entry.key] = entry

    def hydrate(self, query: str, max_tokens: int = 4000) -> list[MemoryEntry]:
        """Retrieve memories relevant to the current task, within token budget."""
        scored = [
            (self._relevance_score(entry, query), entry)
            for entry in self._entries.values()
        ]
        scored.sort(key=lambda x: x[0], reverse=True)

        result = []
        budget = max_tokens
        for score, entry in scored:
            if budget < entry.token_count:
                break
            result.append(entry)
            budget -= entry.token_count
        return result

    def _relevance_score(self, entry: MemoryEntry, query: str) -> float:
        """Score memory relevance. Production systems use embeddings."""
        tag_overlap = len(
            set(entry.relevance_tags) & set(extract_tags(query))
        )
        recency = time_decay(entry.created_at)
        return tag_overlap * 0.7 + recency * 0.3
```

The token budget constraint is critical. Without it, a long-running agent accumulates thousands of memory entries and eventually loads so much history that it drowns out the current task. The budget forces prioritization: only the most relevant memories make it into context, and the agent must re-fetch others if it needs them.

## When Full Context Is Right

Context hydration is not universally superior. Three scenarios favor loading the full context:

**Short documents.** If the entire relevant context fits in 10K tokens, the overhead of iterative fetching (multiple API calls, tool use tokens, reasoning about what to fetch) exceeds the cost of just loading everything. Hydration pays off when the potential context exceeds 50K tokens.

**Tasks requiring holistic understanding.** Summarizing a document, reviewing a complete pull request, or evaluating architectural consistency across a codebase may require seeing everything. Hydration works for focused tasks (fix this bug, implement this feature) but not for breadth-first tasks.

**Latency-sensitive applications.** Each hydration round-trip adds latency. If the application needs a response in under 2 seconds, preloading context (even if partially irrelevant) may be faster than iterative retrieval.

## The Operating Principle

The operating principle behind context hydration is borrowed from database engineering: don't `SELECT *` when you can `SELECT` the specific columns you need. The full-context approach is the agent equivalent of loading an entire database into memory before running a query. It works at small scale but degrades as the data grows.

Production agent systems in 2026 are increasingly adopting a layered retrieval strategy:

1. **Manifest layer**: file tree, schema summaries, API signatures — always loaded (~2K tokens)
2. **Navigation layer**: on-demand retrieval via tools — fetched as needed (~5-15K tokens)
3. **Deep context layer**: full file contents, historical conversations — fetched rarely (~0-10K tokens)

The total context for a well-hydrated agent is typically 10-25K tokens regardless of repository size. A 10-file project and a 10,000-file project use roughly the same context budget — the agent just navigates differently to find the relevant 10-20 artifacts.

This is the architectural insight: context window size is a capacity limit, not a target. The best agents use a fraction of their available window and fill it with precisely the right information. Less context, better selected, consistently beats more context, indiscriminately loaded.
