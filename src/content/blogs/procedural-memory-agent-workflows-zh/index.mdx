当人们列举 Agent 记忆系统的组件时，几乎总是停在两层：短期上下文缓冲区和长期存储——通常实现为向量 RAG。这个分类法借自检索系统，对 Agent 而言是错误的。认知科学家半个世纪以来一直在使用三分法——*情景记忆*（你经历的事件）、*语义记忆*（你知道的事实）和*程序记忆*（你练习过的技能）——因为这三者功能不同，且独立退化。将这个分类法应用于 Agent，缺口显而易见。大多数生产环境中的 Agent 有类似情景记忆的东西（聊天历史、会话日志）和类似语义记忆的东西（文档 RAG）。但几乎没有程序记忆。这是下一个能力前沿，而研究文献已经展示了它的样子。

这篇文章将带你了解什么是程序记忆、最近三年 Agent 研究展示了如何构建它，以及从业者应该将这些成果带入 2026 年代 Agent 系统。

## 三分记忆框架

这一经典区分来自 Endel Tulving 在 1970 年代的工作和 Lawrence Squire 在 1990 年代的完善，但适用于 Agent 的版本更简洁：

| 记忆类型 | 回答的问题 | 访问模式 | 退化特点 |
|-------------|---------|----------------|-------------|
| 情景记忆 | "上次我执行 X 时发生了什么？" | 时间索引，通常有噪声 | 最早的记忆最先丢失 |
| 语义记忆 | "关于 Y 主题什么是真的？" | 内容索引，已整合 | 使用时稳定，不使用则衰退 |
| 程序记忆 | "如何完成 Z 任务？" | 由任务相似性触发，自动执行 | 缓慢；能在睡眠、头部损伤和衰老中存活 |

在人类身上，三者存储在不同脑区（海马体用于情景记忆，皮层网络用于语义记忆，基底神经节和小脑用于程序记忆），且有不同的遗忘曲线。这对 Agent 重要的原因不是脑类比在字面意义上正确——它并不正确——而是*功能*分解是承重的。试图在同一个底层存储"发生了什么"和"如何做事"的系统，最终会两者都做得很差。

标准 Agent 架构将三者压缩了。聊天历史加上 RAG 索引大致是情景加语义。没有程序记忆的位置。当你让一个成功部署到 AWS 五十次的 Agent 执行第五十一次部署时，它从头开始推理 AWS 语义。它"知道怎么做"——曾经执行过一个可靠序列——这一事实无处表示，只能隐式地存在于下一个模型的训练数据中，假设对话甚至被记录了。

## 程序记忆在实践中是什么样的

研究文献已经在收敛到一个具体的定义：程序记忆是一个*工作流*库，按任务类型索引，其中每个工作流记录了先前在类似任务上成功执行的行动和决策序列。四篇论文标记了这一轨迹。

**Voyager**（Wang et al.，[arXiv:2305.16291](https://arxiv.org/abs/2305.16291)，2023）是在 Agent 环境中最早的清晰演示。Voyager 运行在 Minecraft 中，通过编写代码学习——每次解决一个新的子目标（建造工作台、寻找钻石），它将有效的代码作为命名技能存储在技能库中。未来的任务通过文本嵌入相似性检索技能以匹配当前目标。性能曲线引人注目：Voyager 的效能随累积技能库大小扩展，而非随基础模型能力扩展。几小时探索后，复杂的多步目标可以通过组合先前缓存的技能来执行，而非从头重新推理——论文报告 Voyager 获得了比先前基于 LLM 的基线多 **3.3 倍**的独特物品，技能树解锁速度比基线快 **15.3 倍**。

**Reflexion**（Shinn et al.，[arXiv:2303.11366](https://arxiv.org/abs/2303.11366)，2023）采取了不同的角度。Reflexion 存储的不是什么有效，而是哪里出了问题以及为什么。每次失败的任务尝试后，Agent 写一段简短的"反思"——关于错误的一段自然语言笔记，并将这些反思在前置于下一次尝试的上下文中。失败库是一种专注于反模式的程序记忆。Reflexion 的贡献在于表明记忆不需要是正向例子；负向例子以紧凑形式存储，在编码和推理基准上产生了类似的收益。

**ExpeL**（Zhao et al.，[arXiv:2308.10144](https://arxiv.org/abs/2308.10144)，2023）将提取步骤形式化。Voyager 和 Reflexion 都存储 Agent 碰巧产生的任何内容。ExpeL 增加了一个元步骤：在一批 episode 之后，一个 LLM 审查轨迹，提取可泛化的洞察（"当被要求修改一个你未曾见过的文件时，总是先 `cat` 一下"），并将它们写入一个单独的洞察库。关键创新在于区分*经验*（原始轨迹）和*洞察*（提取的规则）。两者都存储，但检索用于规划的是洞察。

**Agent Workflow Memory (AWM)**（Wang et al.，[arXiv:2409.07429](https://arxiv.org/abs/2409.07429)，2024）是迄今为止最清晰的运营化，也是最直接适用于 2026 年代生产 Agent 的。AWM 从 Agent 轨迹中归纳出可复用的*工作流*——步骤序列上的自然语言抽象，例如"回答关于 Reddit 帖子的提问：(1) 导航到 URL，(2) 展开回复，(3) 提取评论树，(4) 总结。"每个工作流存储为文本描述加上它所泛化的抽象步骤轨迹。在新任务上，AWM 检索相关工作流并在规划前将它们插入 Agent 的上下文。在 Mind2Web 基准上，AWM 报告相对于无记忆基线有约 **24.6%** 的相对任务成功率提升，WebArena 上的绝对收益相当；跨域泛化实验展示了更大的定性故事。头条结果：在线工作流归纳（在评估期间构建库）优于离线归纳，这违反直觉——意味着 Agent 在基准测试期间通过从自身更早的尝试中学习而变得更好。

四篇论文的共同线索：程序记忆不是你检索的文本块。它是一种*结构化产物*——代码、反思、洞察规则或工作流模板——在成功信号后将某个 episode 晋升为该产物。

## 为什么 RAG 不是程序记忆

诱惑在于指着向量数据库说"我们有程序记忆，我们检索过去的轨迹。"这行不通，而且失败原因值得明确说明，因为这种失败模式很常见。

**粒度不匹配。** RAG 检索的是段落。程序知识是一个*计划*——带有依赖关系的步骤序列。当你检索先前轨迹的前 k 个段落时，你得到的是彼此无根基的推理片段。Agent 然后以产生非 Sequitur 的方式将这些片段拼接进自己的计划。你想检索的是整个结构化工作流，这意味着要将其存储为整个结构化工作流，而非碎片。

**缺失成功偏差。** 对轨迹的朴素情景 RAG 检索的是*最相似*的过去 episode，无论该 episode 是否成功。你的一半检索将是失败，而这正是程序记忆应该提供的反面。四篇论文都过滤成功或明确从失败中提取洞察。这个过滤器是承重的。

**没有整合。** 情景记忆是有噪声的。程序记忆需要被*蒸馏*——当你做过某事五十次时，你不想在库里有五十个轨迹，你想要一个工作流。RAG 存储一切并检索相似的东西。程序记忆在激进压缩后存储少量内容。区别等同于记录每次通勤和学习驾驶之间的差异。

**错误的检索键。** 轨迹文本上的向量相似性检索的是*看起来像*当前任务的轨迹。你想要的是*完成了*当前任务类型的轨迹。任务类型索引——在归纳的工作流抽象而非原始轨迹文本上检索——是自然的键，而这正是 AWM 的检索路径所优化的。

## 2026 年代的生产形态

最近的 Agent 平台已经收敛到一种具体的实现模式。你可以在 Claude Code 的 Skills、Codex CLI 的提示配方和 LangGraph 的检查点加工具库模式中看到它。模式大致如下：

1. **磁盘上的技能库。** 明文 Markdown 文件，每个描述一个工作流。文件名或 YAML 元数据提供检索键。这不是向量数据库；是一个 grep 兼容索引的文件系统。
2. **晋升步骤。** Agent 完成一个任务后，单独的一遍——有时是同一个模型，有时是审查模型——决定该轨迹是否应该成为新技能或修改现有技能。输出是对技能库的 diff。这是 RAG 跳过的整合步骤。
3. **任务开始时的检索步骤。** 在 Agent 规划前，系统按任务类型匹配查找相关技能并注入提示中。技能注入到一个特定的、文档化的上下文位置，以便规划器显式引用。
4. **驱逐策略。** 技能有使用计数器和失败计数器。N 个会话未触发或有 K 次失败的技能被降级到冷归档或移除。库保持精简。

这是作为文件系统而非嵌入索引的程序记忆。选择是刻意的。文件系统支持的技能是可审计的（你可以阅读它们）、可编辑的（你可以在重新应用前纠正一个坏的技能）和可 diff 的（你可以看到库是如何演化的）。嵌入索引没有这些属性。

实现成本是真实的：你需要一个晋升管道、一个检索策略、一个驱逐策略和命名约定。但能力收益是实质性的。有程序记忆的 Agent 将更少的上下文预算花费在重新推导它们已经推导过的计划上。它们在重复任务上变得更好，无需任何模型更新。而且库本身是一个可转移的资产——维护良好的技能库实际上是一个成熟 Agent 部署一半的价值。

## 缺失的部分

开放问题是可见的。有四个值得指出。

**技能组合。** Voyager 通过让 LLM 调用技能函数来组合。 AWM 通过在上下文中列出工作流并让 LLM 插值来组合。两者都没有关于*何时*技能应该组合 vs. 何时应该归纳新技能的真正理论。当前的启发式方法是经验的。

**跨 Agent 技能转移。** 如果 Agent A 发现了一个有用的工作流，Agent B 是否应该继承它？直觉上是的，但溯源问题很难（工作流是否只因为 A 的上下文而成功？），安全问题更难（受损的技能是否会传播？）。A2A + AP2 能力令牌工作与此相关：带有溯源的签名技能可以在 Agent 之间安全地旅行。还没有人这样做。

**失败触发的撤回。** 当一个技能开始失败时，你想自动删除或标记它。四篇论文都处理得很粗糙。Reflexion 存储失败但不将其与技能库成员资格绑定。AWM 在已发布的基准中没有撤回机制。这是一个明显的下一步。

**评估。** 大多数面向记忆的 Agent 评估在固定基准上测量任务成功率。程序记忆是关于*随时间在类似任务上改进*，这是另一种测量。AWM 的在线归纳结果是暗示性的，但限于一个基准。我们没有的是一个专门测试跨多样化任务流的累积技能获取的基准，而这正是真实生产 Agent 面临的。

## 要点

三分记忆框架是一个承重抽象，不只是神经科学的借用。混淆程序记忆与情景记忆的 Agent 检索错误的东西，整合失败，重复它们已经学会做的work。2023-2024 年的论文确立了程序记忆在以正确结构实现时产生真实收益——显式工作流表示、成功过滤的晋升、任务类型检索、积极驱逐。2026 年代的生产形态正在收敛到文件系统支持的技能库而非向量数据库，出于审计和可编辑性的原因。

对于在生产中运行 Agent 的任何人，实际的下一步行动：审视你的记忆架构，问自己当 Agent 做同样的任务第五十次时会发生什么，如果答案是"它从头开始"，你就有了一个程序记忆缺口。文献已经有了关于如何填补它的答案。
