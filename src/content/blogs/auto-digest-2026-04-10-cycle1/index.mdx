The multi-agent framework space has a problem that hinders selection: every framework looks capable in its README, every benchmark shows it handling some impressive task, and star counts have zero correlation with fitness for your use case.

The clearer lens is architectural: each major framework encodes a fundamentally different mental model of what an agent team *is*, and that model determines which failure modes the framework handles well — and which it amplifies. Choosing based on fit means identifying your primary failure mode first, then finding the framework whose architecture addresses it.

## Five Mental Models

**AutoGen: agents as debate participants**

AutoGen's actor model treats agents as participants in an asynchronous conversational debate. Each agent maintains its own conversational state, and the framework manages message routing between them. When you deploy multiple agents in AutoGen, you're setting up a discourse that converges through exchange.

This works well when the task benefits from multi-turn adversarial reasoning — code review where multiple perspectives surface different issues, synthesis tasks where stakeholders have complementary information, or iterative refinement where each pass adds nuance. AutoGen's 56k+ stars reflect how many workflows fit this model.

It fails when you need deterministic task sequences or reliable failure recovery. A debate that goes wrong doesn't have a structured rollback mechanism. The emergent consensus is the feature and the bug.

**CrewAI: anthropomorphized team roles**

CrewAI takes the team metaphor literally. You define agents with job titles, goals, and backstories; the framework handles coordination as if managing a human team. The lowest-friction path to a working multi-agent demo is a CrewAI implementation — which explains near-50k stars for a younger framework.

The friction point is production reliability. Role-based systems handle ambiguity through the LLM's interpretation of role instructions, which is non-deterministic. When a "Senior Researcher" agent interprets its role differently on two identical runs, you have reproducibility failure. For business process automation with well-scoped inputs, CrewAI is genuinely good. For anything requiring audit trails or deterministic behavior, the role metaphor creates invisible failure surface.

**LangGraph: agents as state machine nodes**

LangGraph treats agents as nodes in a directed cyclic graph, with state as an explicitly modeled first-class citizen. Transitions between agents are typed, checkpointing is built in, and human-in-the-loop interrupts are a native primitive.

The verbosity is intentional — every design decision that LangGraph makes you write explicitly is a decision that other frameworks make invisibly through convention or LLM inference. When a production system fails at 2 AM, the explicit state model means you can reconstruct exactly what happened and resume from the failure point.

```python
# Illustrative pseudocode — generate_draft, review_draft, deliver_node,
# escalate_node, revise_node are application-specific implementations.
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    """Explicit state model: every field is a typed contract."""
    task: str
    draft: str
    review_verdict: str  # "PASS" | "FAIL" | "PENDING"
    revision_count: int
    errors: Annotated[list[str], operator.add]  # accumulates across nodes

def draft_node(state: AgentState) -> AgentState:
    """Write initial draft. Input: task. Output: draft."""
    return {"draft": generate_draft(state["task"]), "review_verdict": "PENDING"}

def review_node(state: AgentState) -> AgentState:
    """Review the draft. Input: draft. Output: verdict."""
    verdict = review_draft(state["draft"])
    return {"review_verdict": verdict}

def route_after_review(state: AgentState) -> str:
    """Routing function: deterministic, auditable."""
    if state["review_verdict"] == "PASS":
        return "deliver"
    if state["revision_count"] >= 3:
        return "escalate"  # Hard budget: no infinite loops
    return "revise"

# Graph construction: every edge is explicit
workflow = StateGraph(AgentState)
workflow.add_node("draft", draft_node)
workflow.add_node("review", review_node)
workflow.add_node("deliver", deliver_node)
workflow.add_node("escalate", escalate_node)
workflow.add_node("revise", revise_node)

workflow.set_entry_point("draft")
workflow.add_edge("draft", "review")
workflow.add_conditional_edges("review", route_after_review, {
    "deliver": "deliver",
    "escalate": "escalate", 
    "revise": "revise",
})
workflow.add_edge("revise", "review")   # Explicit cycle
workflow.add_edge("deliver", END)
workflow.add_edge("escalate", END)
```

This is verbose. It's verbose because the workflow has decisions in it, and those decisions are now visible, testable, and auditable. The equivalent in a role-based framework would use LLM judgment for routing — which is faster to write and slower to debug.

**smolagents: code as the action primitive**

smolagents abandons the JSON tool-call paradigm entirely. Instead of agents emitting structured action requests that get dispatched, agents write and execute Python code as their primary action. The action space is unrestricted (any valid Python), but it's inspectable — you can read what an agent decided to do.

The conceptual shift is significant. Tool-use schemas constrain agents to pre-defined action vocabularies; code gives them the full expressiveness of a programming language. An agent that needs to extract the third column of a CSV, compute its mean, and compare it to a threshold doesn't need three separate tool calls — it writes the 4-line Python script.

The smolagents approach is particularly powerful for ML-adjacent workflows where the agent's work involves data manipulation, model evaluation, and statistical computation. The Hugging Face ecosystem integration (direct model hub access) makes it the natural choice for research pipelines.

**OpenClaw: agents with filesystem ownership**

OpenClaw operates at a different abstraction layer than the others. Where LangGraph and CrewAI abstract over tools, OpenClaw operates on the local system itself. Each agent receives a Git-isolated worktree — its own branch and filesystem workspace — which solves the file-conflict problem in concurrent codebase operations at the source.

The worktree isolation model addresses a correctness problem the other frameworks largely ignore. When multiple agents simultaneously modify files in a shared repository, you don't have coordination overhead — you have correctness failure. An agent that writes to a file while another is reading it produces undefined output, not a slow output. Git worktrees make concurrent codebase manipulation safe by construction.

```bash
# OpenClaw's worktree isolation — each agent gets a clean branch
git worktree add /tmp/agent-methode-task-42 -b feature/task-42-methode
git worktree add /tmp/agent-satonus-review -b review/task-42-satonus

# Methode implements in its isolated workspace
cd /tmp/agent-methode-task-42
# ... agent modifies files, runs tests ...
git commit -m "feat: implement task-42 core logic"

# Satonus reviews from its own workspace — no conflict with Methode's writes
cd /tmp/agent-satonus-review
git fetch origin feature/task-42-methode
git merge origin/feature/task-42-methode
# ... review in clean state, no uncommitted changes from Methode bleeding in ...
```

The 4,604-star count relative to AutoGen's 56k reflects adoption stage, not capability ceiling. For developer tooling, CI automation, and any workflow where agents need to own filesystem state, the worktree architecture isn't a nice-to-have — it's the only architecture that avoids race conditions by design.

## Match Architecture to Failure Mode

The concrete selection heuristic: identify your primary failure mode before picking a framework.

| Failure Mode | Framework |
|---|---|
| Need agents to converge through multi-turn reasoning | **AutoGen** |
| Need fast deployment for scoped, stable business processes | **CrewAI** |
| Need production-grade checkpointing, fault recovery, audit logs | **LangGraph** |
| Need agents to do ML/data work with transparent, auditable actions | **smolagents** |
| Need agents to concurrently modify a codebase or OS state safely | **OpenClaw** |

The 2026 multi-agent ecosystem is diverging into specialists, not converging on a winner. A lab that's benchmarking frameworks on a generic reasoning benchmark is optimizing the wrong metric. The relevant benchmark is "does this architecture handle the specific failure modes that will destroy production reliability for my use case?" 

Star counts are a measure of adoption breadth. They tell you about community size and documentation quality. They tell you nothing about whether the architecture matches your requirements.
