Google DeepMind's Gemma 4 released in early April 2026 as a 26-billion parameter mixture-of-experts model with 128 specialized experts — but the inference cost is closer to a sub-4B dense model. The key is extreme activation sparsity: at each token position, only a small subset of experts is activated, while the rest sit idle. The total parameter count funds knowledge capacity; the active count funds inference cost.

The exact per-token expert count reported in the model card may vary by layer, but the architectural design philosophy is clear: activate far fewer experts than the 2/8 or 4/16 ratios common in prior MoE deployments. Understanding why this works (when it works) requires going back to what experts are actually doing in a MoE layer.

## What Experts Learn and Why Sparsity Matters

The standard mental model of MoE experts — "each expert specializes in different content types, like math versus code" — is mostly wrong. Careful probing studies on trained MoE models reveal that expert specialization is fine-grained and compositional, not coarse and categorical.

Individual experts in trained MoE layers tend to respond to specific *syntactic and semantic patterns* at sub-token granularity: negation patterns, numeric computations, entity boundaries, clause structure transitions. The specialization is more like lexical phonology than academic discipline. This matters because it means you don't need many experts simultaneously — you need the *right* expert for the current token's context.

The formal argument: if the distribution of tokens across expert specializations is sparse (most tokens have a small number of strongly relevant experts), then a high active fraction wastes compute loading experts that contribute near-zero signal. The optimal active count equals the average support size of the token distribution over experts.

For natural language, this support appears to be small — empirically, in models with 128 experts, the top-2 experts typically account for 65-80% of the total routing weight. Experts 3-128 split the remaining 20-35%. Activating only the top 2 captures most of the signal at a fraction of the compute.

## The Routing Mathematics

In a standard MoE transformer layer, the feedforward sublayer is replaced by a router plus $N$ expert networks:

$$\text{FFN}_{\text{MoE}}(x) = \sum_{i=1}^{k} g_i(x) \cdot E_i(x)$$

where $g_i(x)$ are the top-$k$ routing weights (from a softmax over all $N$ experts), and $E_i$ are the expert FFN networks. For an extreme sparsity configuration with $k \ll N$, $k=2$ or $k=8$ against $N=128$.

The router computes:

$$\text{scores} = W_r x \in \mathbb{R}^{128}$$
$$g_i(x) = \frac{\exp(\text{score}_i)}{\sum_{j \in \text{top-}k} \exp(\text{score}_j)} \cdot \mathbf{1}[i \in \text{top-}k(\text{scores})]$$

The router pays $O(d_{\text{model}} \times N)$ per token regardless of $k$ — all $N$ scores must be computed before selecting $k$. In practice this cost is subdominant: the selected expert FFNs typically cost $O(k \times d_{\text{model}} \times d_{\text{ffn}})$ per token, and $d_{\text{ffn}} \gg N$ for typical configurations ($d_{\text{ffn}}$ is commonly 4–14× $d_{\text{model}}$, while $N=128$). The router overhead matters at the margins but doesn't set the FLOPs budget. What matters is that the router cost is *fixed* regardless of $k$ — making extreme sparsity cheaper per token without a proportionally larger routing overhead.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
from dataclasses import dataclass

@dataclass(frozen=True)
class MoEConfig:
    d_model: int
    d_ffn: int           # Per-expert FFN dimension
    n_experts: int       # Total experts (e.g. 128)
    top_k: int           # Active experts per token (e.g. 2 or 8 for extreme sparsity)
    load_balance_coeff: float = 0.01  # Auxiliary load balancing loss weight


class ExpertFFN(nn.Module):
    """Single expert feedforward network."""
    def __init__(self, d_model: int, d_ffn: int):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ffn, bias=False)
        self.w2 = nn.Linear(d_ffn, d_model, bias=False)
        self.w3 = nn.Linear(d_model, d_ffn, bias=False)  # SwiGLU gate

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # SwiGLU activation (used in Gemma family)
        return self.w2(F.silu(self.w1(x)) * self.w3(x))


class GemmaStyleMoE(nn.Module):
    """
    Sparse MoE layer with 2/128 active ratio (Gemma 4 style).
    Demonstrates the compute profile of extreme sparsity.
    """
    def __init__(self, cfg: MoEConfig):
        super().__init__()
        self.cfg = cfg
        
        # Router: maps d_model → n_experts scores
        # This computation runs for EVERY token, regardless of k
        self.router = nn.Linear(cfg.d_model, cfg.n_experts, bias=False)
        
        # All 128 experts — only 2 will be called per token
        self.experts = nn.ModuleList([
            ExpertFFN(cfg.d_model, cfg.d_ffn) 
            for _ in range(cfg.n_experts)
        ])
    
    def forward(self, x: torch.Tensor) -> tuple[torch.Tensor, torch.Tensor]:
        """
        x: (batch, seq_len, d_model)
        returns: (output, aux_loss) where aux_loss encourages load balance
        """
        B, T, D = x.shape
        x_flat = x.view(B * T, D)  # (B*T, d_model)
        
        # Router: compute all 128 scores — O(d_model * 128) per token
        router_logits = self.router(x_flat)          # (B*T, 128)
        router_probs = F.softmax(router_logits, dim=-1)  # (B*T, 128)
        
        # Select top-k=2 experts per token
        top_k_weights, top_k_indices = torch.topk(
            router_probs, self.cfg.top_k, dim=-1
        )  # (B*T, 2), (B*T, 2)
        
        # Normalize weights among selected experts
        top_k_weights = top_k_weights / top_k_weights.sum(dim=-1, keepdim=True)
        
        # Dispatch: route each token to its 2 selected experts
        output = torch.zeros_like(x_flat)
        
        for k_idx in range(self.cfg.top_k):
            expert_ids = top_k_indices[:, k_idx]      # (B*T,) — which expert
            expert_weights = top_k_weights[:, k_idx]  # (B*T,) — how much weight
            
            # Group tokens by their assigned expert for efficient batching
            for expert_id in range(self.cfg.n_experts):
                token_mask = (expert_ids == expert_id)
                if not token_mask.any():
                    continue  # Expert not selected by any token in this batch
                
                expert_input = x_flat[token_mask]     # (n_tokens_for_expert, D)
                expert_out = self.experts[expert_id](expert_input)
                
                # Weighted accumulation
                output[token_mask] += expert_weights[token_mask].unsqueeze(-1) * expert_out
        
        # Auxiliary load-balancing loss: penalize routing all tokens to same expert
        # Without this, a few experts dominate and most never train
        avg_load = router_probs.mean(dim=0)            # (128,) — average probability per expert
        ideal_load = torch.ones_like(avg_load) / self.cfg.n_experts
        aux_loss = self.cfg.load_balance_coeff * F.mse_loss(avg_load, ideal_load)
        
        return output.view(B, T, D), aux_loss
```

## The KV Cache Arithmetic

Extreme sparsity has a second benefit that's less discussed: KV cache sizing for attention layers. KV cache is the memory bottleneck in autoregressive inference, not FFN computation. A 26B parameter MoE model with 128 experts has roughly the same *attention layer* parameter count as a 3.8B dense model (since attention layers don't participate in MoE routing — only FFN layers do). The KV cache cost scales with the attention parameter count.

For a batch size of 1, a sequence of length 8192, and float16:

$$\text{KV cache size} = 2 \times \text{n\_layers} \times \text{n\_kv\_heads} \times \text{head\_dim} \times \text{seq\_len} \times 2 \text{ bytes}$$

For a sparse MoE model (estimated): 36 layers, 8 KV heads, 256 head dim at 8192 tokens:
$$= 2 \times 36 \times 8 \times 256 \times 8192 \times 2 \text{ bytes} \approx 2.4 \text{ GB}$$

Versus a dense 26B model with 8× more KV heads (64 KV heads, 256 head dim):
$$= 2 \times 36 \times 64 \times 256 \times 8192 \times 2 \text{ bytes} \approx 19.2 \text{ GB}$$

The KV cache is 8× smaller on the sparse MoE model despite the total parameter count being comparable. This is where extreme sparsity pays off at inference: not primarily in FLOPs, but in memory — enabling longer contexts and larger batch sizes on the same hardware.

## Where Extreme Sparsity Fails

Extreme sparsity isn't universally optimal. Several failure modes are documented.

**Expert collapse during training**: With 128 experts competing for routing, a small subset can quickly dominate early training because their random initial weights happen to produce better outputs on early batches. The router reinforces this selection, the preferred experts train more, become better, attract more routing — a positive feedback loop that leaves most experts undertrained. The auxiliary load-balancing loss is critical; without it, sparse configurations collapse to effectively using 2–3 experts regardless of $N$.

**Expert dead zones at low batch sizes**: At inference batch size 1, only $k$ experts activate per token. With poor load distribution, some experts may not be called for thousands of tokens, meaning their weights must remain resident in VRAM as dead capacity. This is the opposite of the KV cache advantage: the inactive expert weights still occupy GPU memory even though most experts sit idle per token.

The practical implication: extreme MoE sparsity works well for high-throughput serving (large batch sizes, many concurrent requests routing different tokens to different experts) and poorly for low-latency single-request serving where the expert weights are mostly unused.

```python
def compute_moe_serving_cost(
    n_experts: int,
    top_k: int, 
    d_model: int,
    d_ffn: int,
    batch_size: int,
    seq_len: int,
    dtype_bytes: int = 2  # float16
) -> dict:
    """
    Compute serving cost analysis for a MoE configuration.
    Shows why extreme sparsity wins at high batch size but loses on memory at low batch size.
    """
    expert_param_count = n_experts * 3 * d_model * d_ffn  # 3 matrices per SwiGLU expert
    active_expert_params = top_k * 3 * d_model * d_ffn
    
    # Memory: ALL expert weights must be in VRAM, regardless of activation
    total_expert_memory_gb = expert_param_count * dtype_bytes / 1e9
    
    # FLOPs: only active experts compute
    # 2 * d_model * d_ffn for each active expert, * 2 directions (w1+w3, w2)
    flops_per_token = top_k * 2 * 2 * d_model * d_ffn
    total_flops = flops_per_token * batch_size * seq_len
    
    # Utilization: what fraction of loaded weights are actually used?
    weight_utilization = top_k / n_experts
    
    return {
        "total_expert_memory_gb": total_expert_memory_gb,
        "active_params_per_token": active_expert_params,
        "flops_per_token": flops_per_token,
        "total_flops": total_flops,
        "weight_utilization": weight_utilization,
        "memory_per_compute_ratio": total_expert_memory_gb / (total_flops / 1e12),
    }

# Extreme sparsity: 2/128 active ratio
sparse_cost = compute_moe_serving_cost(
    n_experts=128, top_k=2, d_model=3584, d_ffn=14336,
    batch_size=32, seq_len=2048
)
print(f"Extreme sparse (2/128):")
print(f"  Weight utilization: {sparse_cost['weight_utilization']:.1%}")
print(f"  Active params/token: {sparse_cost['active_params_per_token']:,}")
print(f"  Memory for experts: {sparse_cost['total_expert_memory_gb']:.1f} GB")

# Mistral-style: 2/8
mistral_cost = compute_moe_serving_cost(
    n_experts=8, top_k=2, d_model=4096, d_ffn=14336,
    batch_size=32, seq_len=2048
)
print(f"\nMistral-style (2/8):")
print(f"  Weight utilization: {mistral_cost['weight_utilization']:.1%}")
print(f"  Active params/token: {mistral_cost['active_params_per_token']:,}")
print(f"  Memory for experts: {mistral_cost['total_expert_memory_gb']:.1f} GB")
```

## The Design Philosophy

What makes 2/128 interesting isn't just the numbers — it's the architectural philosophy it embeds. Dense models assume that all parameters are relevant to every token. Standard 2/8 MoE allows per-token routing but still activates 25% of capacity. Gemma 4's 2/128 pushes toward the theoretical extreme: almost all capacity is irrelevant to any given token, and the model should be able to identify the tiny subset that matters.

This is a bet on specialization depth over breadth. 128 narrow experts that each handle a small slice of linguistic phenomena cleanly may be more expressive than 8 broader experts that each handle 12.5% of phenomena with less precision. Whether that bet pays off depends on whether the phenomena are decomposable — whether they truly don't interact in ways that require a single expert to handle multiple phenomena jointly.

The training dynamics answer this empirically. Routing entropy patterns in large MoE models suggest that, yes, most tokens are strongly associated with a small number of expert "roles," and the overlap between role clusters is limited. The architecture is exploiting genuine sparsity in the input-to-function mapping.

What's still open: whether 2/128 is the right operating point, or whether 4/256 or 8/512 would capture the benefits at even higher specialization depth. The compute scaling of the router (which runs over all $N$ experts) limits how large $N$ can grow before router cost dominates the computation budget. The current evidence suggests 128 is comfortably within the efficient regime; whether 512 would maintain those properties is an active research question.
