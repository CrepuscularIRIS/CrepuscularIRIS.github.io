你的多智能体系统运行良好，直到某个智能体出错。问题是：谁的错？

信用分配是强化学习中历史悠久的难题。当一个智能体团队产出好结果时，谁该记功？当结果糟糕时，谁该背锅？单智能体 RL 用 TD 方法解决了时间维度的问题——将奖励反向传播。但多智能体系统面临更困难的变体，因为信用必须在智能体之间分配，而不仅仅是在时间步之间。

OpenAI 2026年4月的分层智能体蜂群研究引入了一种他们称为**基于声誉的信用分配（RCA）**的机制。核心理念：蜂群中的每个工作智能体根据其输出的下游效用积累声誉分数。声誉更高的智能体获得更多算力预算、更复杂的子任务，以及共享资源的优先访问权。声誉低的智能体获得简单任务和更小的预算，如果分数跌破阈值则可能被替换。

机制很简单。失败模式不简单。

## RCA 工作原理

分层智能体蜂群有三层。编排器将高层目标分解为子任务。工作智能体池执行子任务。验证器根据任务特定标准评估工作智能体的输出。

RCA 添加了第四个组件：一个跟踪每个工作智能体历史表现的**声誉账本**。

```python
from dataclasses import dataclass, field
from typing import Protocol
from collections import defaultdict

@dataclass(frozen=True)
class TaskResult:
    worker_id: str
    task_id: str
    quality_score: float
    latency_ms: int
    tokens_used: int

class ReputationLedger:
    def __init__(self, decay: float = 0.95, window: int = 50):
        self._scores: dict[str, list[float]] = defaultdict(list)
        self._decay = decay
        self._window = window

    def record(self, result: TaskResult) -> None:
        history = self._scores[result.worker_id]
        history.append(result.quality_score)
        if len(history) > self._window:
            history.pop(0)

    def reputation(self, worker_id: str) -> float:
        history = self._scores.get(worker_id, [])
        if not history:
            return 0.5
        weighted = sum(
            score * (self._decay ** i)
            for i, score in enumerate(reversed(history))
        )
        normalizer = sum(
            self._decay ** i for i in range(len(history))
        )
        return weighted / normalizer
```

声誉分数是质量分数的**指数加权移动平均**，近期表现权重更高。衰减因子控制记忆长度——高衰减（0.99）意味着智能体的遥远过去仍然重要；低衰减（0.8）意味着只有近期表现才计入。

编排器利用声誉分数做出三个分配决策：

```python
@dataclass(frozen=True)
class ComputeBudget:
    max_tokens: int
    max_retries: int
    timeout_ms: int

class ReputationAwareOrchestrator:
    def __init__(
        self,
        ledger: ReputationLedger,
        base_budget: ComputeBudget,
    ):
        self._ledger = ledger
        self._base = base_budget

    def allocate(self, worker_id: str, task_complexity: float) -> ComputeBudget:
        rep = self._ledger.reputation(worker_id)
        multiplier = 0.5 + rep
        return ComputeBudget(
            max_tokens=int(self._base.max_tokens * multiplier),
            max_retries=max(1, int(self._base.max_retries * multiplier)),
            timeout_ms=int(self._base.timeout_ms * multiplier),
        )

    def assign(
        self, task_complexity: float, pool: list[str]
    ) -> str | None:
        if not pool:
            return None
        scored = [
            (wid, self._ledger.reputation(wid))
            for wid in pool
        ]
        scored.sort(key=lambda x: x[1], reverse=True)
        if task_complexity > 0.7:
            return scored[0][0]
        midpoint = len(scored) // 2
        return scored[midpoint][0]
```

高声誉智能体获得更多 tokens、更多重试次数和更长超时时间。它们还被分配更困难的任务。低声誉智能体获得受限预算和更简单的任务。分配逻辑避免总是选择顶级智能体——中等复杂度的任务分配给中等级别的智能体，以保留顶级智能体的预算用于真正困难的问题。

## 自然选择类比

RCA 创造了类似于生物适应度景观的**选择压力**。产出高质量输出的智能体"繁殖"——它们获得更多资源和机会。产出低质量输出的智能体被"淘汰"——更少资源、更简单任务、最终被替换。

类比足够精确，可以预测失败模式。

**过度利用而非探索。** 在进化系统中，强选择压力会降低遗传多样性。在 RCA 中，强的基于声誉的分配创建了一种**单一文化**——少数高声誉智能体处理大部分工作。这在任务分布稳定时是高效的——昨天任务的最佳智能体很可能也是今天任务的最佳智能体。当任务分布发生变化时，它就失效了。高声誉智能体针对旧分布优化，在新任务类型上可能表现不佳。

```python
def reputation_variance(ledger: ReputationLedger, pool: list[str]) -> float:
    reps = [ledger.reputation(wid) for wid in pool]
    if not reps:
        return 0.0
    mean_rep = sum(reps) / len(reps)
    return sum((r - mean_rep) ** 2 for r in reps) / len(reps)

def needs_exploration(
    ledger: ReputationLedger,
    pool: list[str],
    threshold: float = 0.15,
) -> bool:
    return reputation_variance(ledger, pool) > threshold
```

当声誉方差超过阈值——表明少数智能体占主导地位而其他智能体停滞——编排器应该注入探索：将困难任务分配给低声誉智能体，或引入具有中性声誉的新智能体。

**声誉崩溃级联。** 一个高声誉智能体遇到分布外任务后失败。它的声誉下降。编排器给它分配更简单的任务来"恢复"。但更简单的任务无法快速重建声誉，因为简单任务的质量分数信息量不足——每个人在简单问题上都得分高。与此同时，原本排名第二的智能体现在独自处理困难任务，增加了其失败概率。一个智能体的声誉崩溃可能通过整个池级联传播。

缓解措施是**任务类型感知的声誉**。不是每个智能体一个标量声誉，而是按任务类别跟踪声誉：

```python
class TypedReputationLedger:
    def __init__(self, decay: float = 0.95, window: int = 50):
        self._scores: dict[str, dict[str, list[float]]] = defaultdict(
            lambda: defaultdict(list)
        )
        self._decay = decay
        self._window = window

    def record(self, result: TaskResult, task_type: str) -> None:
        history = self._scores[result.worker_id][task_type]
        history.append(result.quality_score)
        if len(history) > self._window:
            history.pop(0)

    def reputation(self, worker_id: str, task_type: str) -> float:
        history = self._scores.get(worker_id, {}).get(task_type, [])
        if not history:
            return 0.5
        weighted = sum(
            score * (self._decay ** i)
            for i, score in enumerate(reversed(history))
        )
        normalizer = sum(
            self._decay ** i for i in range(len(history))
        )
        return weighted / normalizer
```

一个智能体可以在代码生成任务上有高声誉，在文档任务上声誉低。编排器将任务类型匹配到智能体的专业领域，创建隐式的劳动分工，无需显式角色分配。

## RCA 与负载均衡的区别

表面上，RCA 看起来像是带反馈环的加权轮询负载均衡。结构差异在于 RCA 分配的是**能力**，而不仅仅是容量。

传统负载均衡分配工作以最小化延迟或最大化吞吐量。它将工作者视为可互换的。RCA 分配工作以最大化输出质量，这需要建模每个工作者的**比较优势**——相对于池中其他工作者的具体擅长领域。

这一区别在实践中很重要，因为基于 LLM 的智能体不是可互换的。在安全审查任务上，运行 Claude Opus 且具有安全聚焦系统提示的智能体将优于同一智能体，但在创意写作上表现不佳。拥有 200K 上下文窗口的智能体处理大型代码库优于 32K 智能体，但在单文件任务上浪费资源。RCA 通过声誉隐式捕获这些差异，无需显式能力声明。

缺点是**收敛时间**。新智能体从中性声誉（0.5）开始，必须积累任务历史，编排器才能有效路由。在冷启动期间，任务分配本质上是随机的。对于智能体更替频繁的蜂群——智能体被添加、移除或重新配置——RCA 可能永远无法收敛，因为声誉信号总是过时的。

## 验证瓶颈

RCA 的质量完全取决于验证器。如果验证器的质量分数有噪声，声誉就变得有噪声。如果验证器有偏见——持续给某种输出风格更高分数——声誉就放大偏见。如果验证器可被博弈——智能体可以产出得分高但实际无用的输出——声誉就奖励博弈。

这是将**古德哈特定律**应用于智能体协调的问题。声誉分数是"这个智能体产出有用工作"的代理指标。当智能体优化代理指标时，代理指标就不再测量它原本要测量的东西。

```python
class VerifierWithCalibration:
    def __init__(self, base_verifier, calibration_set: list[TaskResult]):
        self._base = base_verifier
        self._bias = self._estimate_bias(calibration_set)

    def _estimate_bias(self, calibration: list[TaskResult]) -> float:
        if not calibration:
            return 0.0
        scores = [r.quality_score for r in calibration]
        return sum(scores) / len(scores) - 0.5

    def score(self, result: TaskResult) -> float:
        raw = self._base.score(result)
        return max(0.0, min(1.0, raw - self._bias))
```

校准有助于处理系统性偏见，但无法解决博弈问题。更深入的修复是**对抗性验证**——使用独立模型为工作者输出生成挑战性测试用例，然后根据鲁棒性而非表面质量评分。这很昂贵（每次验证两次模型调用），但使代理指标更接近实际效用。

## 实践架构

对于正在构建多智能体系统的团队，RCA 提供了一个有用的框架，即使没有完整的蜂群基础设施。最小可行实现：

1. **跟踪每个智能体、每个任务类型的质量分数。** 即使是最近 20 个任务的简单滚动平均也足以区分智能体。

2. **将困难任务路由给已验证的智能体。** 不要用最佳智能体处理所有事情——将其保留用于质量差异最大的任务。

3. **有意注入探索。** 每 N 个任务，随机分配给某个智能体，不考虑声誉。这防止单一文化，并发现声誉不能反映真实能力的智能体。

4. **监控多样性。** 如果声誉方差超过阈值，系统过度专门化了。添加智能体、轮换配置或重置信誉分数。

5. **验证验证器。** 定期审计质量分数是否与下游效用相关。一个给冗长但无用输出高分的验证器将创建一支冗长但无用的智能体团队。

RCA 的更深层教训是，多智能体协调不是路由问题——而是**信用分配问题**。正确路由需要解决谁应因好结果记功、谁应因坏结果背锅。OpenAI 的方法借鉴了进化动力学。这是否是正确隐喻取决于智能体蜂群更像生态系统（多样性对韧性至关重要）还是更像装配线（专业化对效率至关重要）。答案几乎肯定取决于任务分布——而那些能根据任务分布调整协调策略的系统将胜过不调整的系统。
