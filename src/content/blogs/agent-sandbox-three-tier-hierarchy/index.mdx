When an AI agent runs `pip install` followed by `subprocess.run(user_generated_command)`, the question of what happens next depends entirely on the isolation boundary around that process. In April 2026, the answer has stratified into three distinct tiers, each encoding a different assumption about trust.

This stratification didn't happen by committee. It emerged from production incidents. The MITRE ATLAS OpenClaw investigation in February 2026 documented multi-stage attack chains where agents вҖ” operating within standard Docker containers вҖ” escalated from configuration injection to host filesystem access. The attack surface wasn't the model. It was the runtime.

## Tier 1: Firecracker MicroVMs вҖ” Kernel-Level Isolation

Firecracker, originally built by AWS for Lambda and Fargate, creates lightweight virtual machines that share the host's hardware but run their own Linux kernel inside KVM. Each microVM gets a dedicated kernel instance with sub-100ms cold start times and less than 5 MiB memory overhead.

The security model is straightforward: the agent cannot reach the host kernel. Firecracker intentionally excludes USB controllers, graphics cards, sound devices, and every other non-essential device from the VM's device model. The attack surface is the minimal VMM interface вҖ” roughly 50,000 lines of Rust вҖ” versus the millions of lines in a full hypervisor like QEMU.

The practical implication: if an agent achieves arbitrary code execution within the MicroVM вҖ” which is the expected case, not the edge case, since code execution is the agent's job вҖ” it still cannot access the host filesystem, other agents' workspaces, or the network (unless explicitly permitted). The blast radius is the VM itself.

E2B, the most widely deployed agent sandbox platform as of April 2026, runs Firecracker microVMs as its isolation primitive. Each sandbox gets a fresh VM with a configurable filesystem snapshot, network policy, and resource limits. The SDK is minimal:

```python
from e2b_code_interpreter import Sandbox

# Each sandbox is an isolated Firecracker MicroVM
with Sandbox() as sandbox:
    # Agent code runs inside the VM
    execution = sandbox.run_code("""
        import subprocess
        # This runs inside the MicroVM, not on the host
        result = subprocess.run(['ls', '/'], capture_output=True, text=True)
        print(result.stdout)
    """)
    print(execution.logs.stdout)
    # MicroVM is destroyed on exit вҖ” no state leaks
```

The cold start penalty is real but manageable. E2B reports sub-200ms sandbox creation for pre-warmed snapshots. For agents that execute dozens of code blocks per task, the amortized overhead is negligible compared to the LLM inference time.

## Tier 2: gVisor вҖ” User-Space Kernel Interception

gVisor sits between containers and VMs. Instead of running a separate kernel, it interposes a user-space kernel (called Sentry) that intercepts system calls before they reach the host kernel. The agent process thinks it's running on a normal Linux system. Sentry handles the calls in user space, drastically reducing the host kernel's exposure.

Google uses gVisor internally for Cloud Run and GKE Sandbox. Gemini CLI adopted it for agent sandboxing in early 2026 вҖ” every shell command the agent generates runs through gVisor's syscall filter.

The tradeoff versus Firecracker: gVisor doesn't provide hardware-level isolation. A vulnerability in the Sentry process could theoretically lead to host kernel access, because Sentry itself runs in user space on the host. In practice, gVisor's attack surface is smaller than a full container runtime but larger than a MicroVM's VMM interface.

Where gVisor wins: filesystem performance. Because it doesn't virtualize block devices, file I/O is significantly faster than MicroVM-based sandboxes. For agents that read and write thousands of source files per task вҖ” the typical coding agent workload вҖ” this matters.

Northflank processes over 2 million isolated workloads monthly using a combination of gVisor and Kata Containers (which are MicroVM-based). Their production data shows that gVisor adds roughly 5-15% overhead on compute-bound tasks versus native containers, while Firecracker MicroVMs add 10-25%.

## Tier 3: Hardened Containers вҖ” Trust but Verify

Standard Docker containers share the host kernel. A container escape vulnerability вҖ” and there have been several вҖ” gives the attacker host-level access. For untrusted agent workloads, this is insufficient.

Hardened containers add layers: seccomp profiles that restrict available syscalls, AppArmor or SELinux mandatory access controls, dropped Linux capabilities, and read-only root filesystems. The agent runs in a container that looks normal from the inside but has a drastically reduced ability to interact with the host.

```python
# Docker SDK configuration for a hardened agent container
container_config = {
    "image": "agent-runtime:latest",
    "security_opt": [
        "seccomp=agent-restricted.json",  # ~40 allowed syscalls
        "apparmor=agent-profile"
    ],
    "cap_drop": ["ALL"],
    "cap_add": ["NET_BIND_SERVICE"],  # minimal capability
    "read_only": True,
    "tmpfs": {"/tmp": "size=512m"},
    "network_mode": "none",  # no network by default
    "mem_limit": "2g",
    "cpu_period": 100000,
    "cpu_quota": 50000,  # 50% of one CPU
}
```

This works when the agent executes code you've reviewed and trust. It does not work for arbitrary code generation, which is what coding agents do. The seccomp profile can block dangerous syscalls, but a sufficiently creative agent (or a sufficiently creative prompt injection) can find syscall combinations that the profile didn't anticipate.

Docker's own response to this gap: Docker Sandboxes, released in early 2026, gives each sandbox a dedicated MicroVM with a private Docker daemon. The agent cannot access the host daemon, other containers, or files outside the workspace. Effectively, Docker acknowledged that containers alone aren't sufficient and wrapped them in a VM.

## Choosing the Right Tier

The decision matrix is simpler than the technology:

| Trust Level | Isolation Tier | Use Case |
|---|---|---|
| **Zero trust** вҖ” agent runs arbitrary generated code | Firecracker MicroVM | Coding agents, code interpreters, untrusted plugins |
| **Low trust** вҖ” agent runs curated tools with controlled inputs | gVisor | CLI agents with tool allowlists, managed agent platforms |
| **Verified trust** вҖ” agent executes reviewed, deterministic code | Hardened containers | CI/CD pipelines, pre-approved automation scripts |

Most production agent systems in April 2026 use a hybrid. The orchestrator runs in a hardened container (it only makes API calls and reads files). Sub-agents that execute generated code run in MicroVMs. Tool-calling agents that invoke pre-defined functions run behind gVisor.

## Network Egress: The Forgotten Attack Surface

Isolation without network control is incomplete. An agent inside a perfectly isolated MicroVM can still exfiltrate data if it has unrestricted outbound network access. The default for most sandbox platforms вҖ” and the default that production teams override immediately вҖ” is to allow all egress.

The 2026 pattern: default-deny egress with explicit allowlists. The agent can reach the LLM API endpoint and approved tool endpoints. Everything else is blocked at the network policy level, not at the application level.

```yaml
# Network policy for an agent sandbox (Kubernetes NetworkPolicy)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: agent-sandbox-egress
spec:
  podSelector:
    matchLabels:
      role: agent-sandbox
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 10.0.0.0/8  # internal API gateway only
      ports:
        - port: 443
          protocol: TCP
```

This is the lesson from the MITRE ATLAS investigation: the documented attack chains relied on network egress to download secondary payloads. Blocking egress by default would have stopped the chain at the first stage.

## The Cost of Getting This Wrong

Early production data from 2026 deployments suggests that agent sandboxing dramatically reduces security incidents compared to unrestricted access. The residual incidents come from misconfigured egress rules and over-permissive filesystem mounts вҖ” not from sandbox escapes themselves.

The cost of getting it right is measured in latency (sub-200ms for MicroVMs, sub-50ms for gVisor) and in operational complexity (managing VM images, seccomp profiles, network policies). For any system where agents execute generated code in production, this cost is not optional.

The hierarchy is clear: MicroVMs for untrusted code, gVisor for controlled execution, hardened containers for trusted automation. Docker without additional isolation is not an option for autonomous agents in 2026. The threat model has moved past it.
