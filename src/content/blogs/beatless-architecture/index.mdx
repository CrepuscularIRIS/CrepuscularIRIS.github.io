## 引言：为什么需要 Beatless？

在使用 AI 助手的过程中，我发现一个核心问题：**单会话的上下文限制和状态丢失**。

当我让 AI 完成一个复杂任务时：
1. 任务可能跨越多天
2. 需要多个专业 Agent 协作
3. 中间状态需要持久化
4. 需要优先级调度和错误恢复

现有的对话模式无法解决这些问题。于是我设计了 **Beatless**——一个个人 AI 任务调度系统。

## 核心设计理念

### 1. 任务即真相（Task as Source of Truth）

```yaml
# TASKS.yaml 是唯一任务真相源
meta:
  schema_version: 4
  shift_id: SHIFT-20260407-1329
  
tasks:
  - id: BT-RESEARCH-20260407-001
    title: "深度调研：神经网络架构"
    status: in_progress
    assigned_agent: gemini-researcher
```

所有状态、进度、上下文都存储在结构化任务中，而非对话历史。

### 2. 队列驱动的调度

```
Backlog → Ready → In Progress → Review → Done
                ↓         ↓
             Blocked  Dead Letter
```

- **Backlog**: 待办任务池
- **Ready**: 可执行任务
- **In Progress**: 执行中（上限 3 个并行）
- **Review**: 等待验收
- **Done**: 已完成
- **Blocked**: 阻塞（等待外部输入）

### 3. Agent 专业化分工

| Agent | 角色 | 专长 | 超时 |
|-------|------|------|------|
| **Lacia** | 调度中枢 | 路由、收敛、回执 | 600s |
| **Methode** | 执行引擎 | 代码实现、产出 | 1800s |
| **Satonus** | 评审验收 | 质量控制、文档化 | 600s |
| **Gemini-Researcher** | 学术研究 | 论文分析、理论推导 | 900s |
| **Codex-Builder** | 工程实现 | 代码、架构、复现 | 1800s |

### 4. Compass 协议：分阶段执行

复杂任务强制分阶段：
```
Research → Implement → Review
```

- **Research**: 收集信息、分析可行性
- **Implement**: 执行实现
- **Review**: 独立验收

Builder 和 Validator 必须分离，避免自说自话。

### 5. Agent Mailbox 系统：轻量级跨 Agent 邮件层

> v4 新增。基于 per-workspace 文件收件箱的轻量级跨 Agent 邮件系统，作为长期记忆与群聊之间的中间层。

#### 设计目标

- **无中央注册表**：每个 workspace 自主管理自己的 inbox
- **尽力投递（Best-effort）**：不保证送达，不阻塞
- **无推送通知**：Agent 按需拉取，不产生消息风暴
- **Token 高效**：每次只读自己 inbox 的 1-3 封最新邮件

#### 目录结构

```
~/.openclaw/
├── workspace-<agent>/
│   └── .agent-mailbox/
│       ├── inbox/      # 未读/待处理邮件
│       └── archive/    # 已消费邮件
```

#### 邮件文件命名规范

```
<stamp>--<sender>--<priority>--<slug>.md
```

示例：`20260407130000--methode--warn--task-overdue.md`

字段说明：
- `stamp`：YYYYMMDDHHMMSS 时间戳
- `sender`：发送方 Agent 短名
- `priority`：`info` | `warn` | `critical`
- `slug`：短可读标识符

#### 邮件格式

```text
Title: cron list 异常
From: methode
Created-At: 2026-04-07T13:00:00+08:00
Priority: warn
Tags: ops,rollback

`openclaw cron list` 当前报错，先不要依赖其输出。
建议改用：openclaw cron status
```

#### 消费流程

```
Agent 启动 → 检查 inbox/ → 读取最新 1-3 封邮件
  → 压缩为短工作摘要 → 判断：忽略 / 本次使用 / 写入记忆
  → 归档已处理邮件（保持 inbox 精简）
```

处理决策：
- **`critical`**：立即读取，影响当前行为
- **`warn`**：读取并在与当前工作相关时采纳
- **`info`**：快速浏览，仅在有用时保留

> ⚠️ **安全边界**：Mailbox 仅用于本地轻量级上下文共享，不实现自动任务执行、不支持 webhook 回调、不保证送达、不支持云同步。

#### 使用场景

- 跨 Agent 低噪音运营提示（"FYI" 类通告）
- 临时警告，不适合写入长期记忆
- 协调性提示，不属于持久化记忆但需要跨 Agent 可见
- 尽力投递的扇出广播，无聊天投递负担

## 控制平面：Budgeted Proactive Control

> v4 新增。Beatless v4 引入**预算感知主动控制**理念，控制平面不直接执行任务，而是通过 assist / silent / escalate / rollback 四种动作对执行平面进行调控。

### 控制哲学

传统架构中，调度器（Scheduler）仅做**路由**——把任务分发到合适的 Agent 即结束。

Beatless v4 的控制平面更进一步：**持续感知执行状态，按预算主动介入**，而不是被动等待结果。

```
用户输入
    ↓
Lacia（控制平面）
    ├── assist  → 补充上下文、提示、资源
    ├── silent  → 观察，不干预
    ├── escalate → 提升优先级/资源/人工介入
    └── rollback → 触发检查点回滚
    ↓
Methode / Codex-Builder（执行平面）
```

### 四种控制动作

| 动作 | 触发条件 | 效果 |
|------|----------|------|
| **assist** | 执行平面输出证据不足、上下文接近耗尽 | 补充背景、约束、参考路径 |
| **silent** | 执行正常，预算充足 | 仅记录，不干预，不打断 |
| **escalate** | 任务阻塞超限、错误扩散风险 | 提升优先级、引入备选 Agent、通知人工 |
| **rollback** | 检测到执行偏离（checkpoint 偏差） | 恢复到上一个稳定检查点，重新调度 |

### 预算感知

控制动作受预算约束，不自由超出：

```yaml
control_plane:
  budget:
    max_assist_per_shift: 12      # assist 动作上限
    max_rollback_per_shift: 3      # rollback 动作上限
    escalate_requires_approval: true  # escalate 需人工确认
  checkpoints:
    interval_tasks: 3             # 每 N 个任务保存一个检查点
    interval_seconds: 1800         # 或每 30 分钟
```

### 检查点（Checkpoint）

每个检查点记录：

```yaml
checkpoint:
  id: CK-20260407-001
  timestamp: 2026-04-07T13:30:00+08:00
  task_id: BT-IMPL-20260407-003
  phase: implement
  context_snapshot: ./.context-snapshots/CK-20260407-001.md
  artifacts:
    - path: ./output/partial-result.md
      checksum: sha256:abc123
  rollback_target: CK-20260407-000  # 上一个稳定检查点
```

Rollback 时，控制平面：
1. 加载 `rollback_target` 的 context_snapshot
2. 丢弃当前 partial artifacts
3. 将任务重新放回 Ready 队列
4. 可选：派遣不同 Agent 重新执行

### Escalation 路径

```
Escalate L1（自动）→ 提升并行度上限，引入第二个 Agent 协助
Escalate L2（半自动）→ 通知人类操作员，等待确认
Escalate L3（人工）→ 停止自动调度，等待人工介入
```

## 上下文管理：Evidence-Preserving Compression

> v4 新增。传统上下文压缩是"遗忘式"的——超出窗口时直接丢弃旧内容。Beatless v4 采用**证据保留压缩**思路：优先保留有决策依据价值的证据，降低冗余叙事。

### 上下文价值估计

每个上下文单元（含 4 类）按价值评分：

| 类型 | 示例 | 基准价值 |
|------|------|----------|
| **Decision** | "因为 X 原因选择了 Y 方案" | 高（不可再生） |
| **Artifact** | 生成的代码块、报告段落 | 高（生成成本高） |
| **Evidence** | 搜索到的论文摘要、实验数据 | 中（可重新获取但有成本） |
| **Narrative** | 过程描述、过渡语句 | 低（纯文本，可压缩） |

### 压缩策略

```
上下文接近窗口上限时：
1. 标记所有 Decision 和 Artifact 为 P0（保留）
2. 标记 Evidence 为 P1（有条件保留）
3. 标记 Narrative 为 P2（优先压缩或丢弃）
4. 对 P1 项目检查：是否可从外部重新获取？
   - 能 → 丢弃（保存查询路径而非结果）
   - 否 → 保留
5. 对 P2 项目：保留关键摘要，丢弃过程叙事
```

### 实现示例

```python
def compress_context(messages, token_budget):
    """Evidence-preserving compression"""
    scored = [(classify(m), m) for m in messages]
    
    # P0: 全部保留
    p0 = [m for c, m in scored if c == "decision" or c == "artifact"]
    
    # P1: 保留不可重获取的 evidence
    p1 = [m for c, m in scored 
          if c == "evidence" and not re retrievable(m)]
    
    # P2: 保留摘要，丢弃叙事
    p2_summaries = [summarize(m) for c, m in scored if c == "narrative"]
    
    kept = p0 + p1 + p2_summaries
    
    if count_tokens(kept) > token_budget:
        # 最后裁剪：优先保留最早出现的 P0
        kept = prune_to_budget(kept, token_budget, priority="decision_first")
    
    return kept
```

### 与 Mailbox 的协同

Context snapshot 不写入 Mailbox（避免重复读取），而是写入专用上下文快照文件：

```
.context-snapshots/
├── CK-20260407-001.md   # 检查点快照
├── CK-20260407-002.md
```

这些文件由控制平面管理，执行 Agent 按需读取，不主动推送。

## 系统架构

### 数据流

```
用户输入
    ↓
Lacia（控制平面）
    ├── Mailbox 检查（1-3 封最新邮件）
    ├── Budget 检查
    ├── Checkpoint 检查
    └── 控制动作（assist / silent / escalate / rollback）
    ↓
读取 TASKS.yaml
    ↓
决策：路由到哪个 Agent？
    ↓
sessions_spawn（创建子会话）
    ↓
Agent 执行（可接收 assist 信号）
    ↓
生成报告
    ↓
更新 TASKS.yaml + Checkpoint
    ↓
Receipt（回执）
```

### 核心组件

#### 1. 任务队列系统 (TASKS.yaml)

```yaml
meta:
  schema_version: 4
  shift_id: SHIFT-20260407-1329
  updated_at: '2026-04-07T13:00:00+08:00'
  owner: CrepuscularIRIS
  orchestrator: lacia

policies:
  compass_protocol:
    enabled: true
    phases: [research, implement, review]
  
  budgets:
    max_tasks_per_shift: 18
    max_parallel_subtasks: 3
    max_expert_calls_per_shift: 6
    # v4 新增：控制平面预算
    control:
      max_assist_per_shift: 12
      max_rollback_per_shift: 3
      escalate_requires_approval: true

queues:
  backlog: []
  ready: []
  in_progress: []
  review: []
  done: []
  blocked: []

# v4 新增：检查点记录
checkpoints: []
```

#### 2. 调度引擎 (Lacia)

决策树：
```python
# v4: 扩展了控制平面逻辑
if review queue non-empty:
    dispatch_satonus()           # 优先验收
elif in_progress has artifacts:
    # v4: 检查是否需要 assist
    if context_value_low() and assist_budget > 0:
        send_assist_signal()
    push_to_review()             # 推进到验收
elif ready non-empty:
    route_and_dispatch()         # 执行就绪任务
elif blocked with retries left:
    if rollback_budget > 0 and checkpoint_available():
        rollback_to_checkpoint()  # v4: 优先尝试 rollback
    else:
        retry_and_promote()
elif escalation_pending:
    escalate()                   # v4: 处理 escalation 队列
else:
    check_mailbox()              # v4: 轻量 mailbox 检查
    heartbeat_ok()
```

#### 3. Agent 路由规则

| 任务类型 | 首选 Agent | 备选 Agent |
|----------|-----------|-----------|
| 日常编码 | claude-generalist | codex-builder |
| 开源复现 | codex-builder | gemini-researcher |
| 学术证明 | gemini-researcher | claude-architect-opus |
| 架构决策 | claude-architect-opus | - |

#### 4. Receipt 系统

每个任务完成必须输出：
```markdown
DONE: 已完成项
DOING: 进行中项
BLOCKED: 阻塞项
NEXT: 下一步行动
VERDICT: PASS | FAIL | PARTIAL
ASSIST_USED: yes/no      # v4 新增：是否接收了 assist
ROLLBACK_COUNT: 0        # v4 新增：本任务 rollback 次数
```

#### 5. Mailbox 协议（通信层）

> v4 新增。基于 Agent Mailbox 的轻量级跨 Agent 通信协议，补充主动投递（sessions_spawn）之外的异步协调能力。

**Mailbox 协议设计**：

| 字段 | 语义 |
|------|------|
| **TTL** | 邮件有效期；过期后自动归档（建议 ≤ 24h） |
| **ACK** | 是否需要发送方确认送达（默认 no-ack） |
| **Checkpoint Ref** | 可选：关联一个检查点 ID，便于追踪上下文状态 |

```yaml
# 带 ACK 和 TTL 的邮件示例
Title: 任务 BT-IMPL-20260407-003 接近上下文上限
From: methode
Created-At: 2026-04-07T13:30:00+08:00
Priority: warn
TTL: 7200                      # 2 小时内有效
ACK: false
Checkpoint-Ref: CK-20260407-002
Tags: context,assist-request

建议在下一个 checkpoint 保存后触发 assist 信号，
补充 Lacia 的调度上下文。
```

**ACK 机制**（可选）：
- 发件方在邮件中加入 `ACK: request`
- 收件方处理后写入 `ACK: delivered:<收件方ID>:<时间戳>`
- 发送方可在 archive 中验证 ACK 状态

> 注意：ACK 是尽力而为的，不保证可靠性。收件方不在线时 ACK 不会补发。

## 关键设计决策

### 1. 为什么用 YAML 而不是数据库？

- **可读性**：人类可以直接阅读和编辑
- **版本控制**：天然支持 Git 版本管理
- **可移植性**：纯文本，任何环境都能解析
- **透明性**：任务状态完全可见

### 2. 为什么强制分阶段（Compass）？

**反模式**：让同一个 Agent 既做研究又做实现
- 容易陷入确认偏误
- 缺乏独立验证
- 质量难以保证

**正解**：Research → Implement → Review 分离
- Research 提供客观信息
- Implement 专注执行
- Review 独立验收

### 3. 为什么限制并行任务数？

```yaml
budgets:
  max_tasks_per_shift: 18
  max_parallel_subtasks: 3
  max_expert_calls_per_shift: 6
```

- **认知负荷**：人类无法同时跟踪太多并行任务
- **资源限制**：API 调用有速率限制
- **质量控制**：并行过多导致每个任务质量下降

### 4. 为什么引入 Mailbox 而非直接用会话消息？

- **解耦**：发送方不阻塞等待回复
- **无风暴**：没有实时推送，不产生交叉消息轰炸
- **Token 高效**：每个 Agent 只读自己 inbox，不消费他人 token 预算
- **Advisory 性质**：邮件内容为建议而非指令，Agent 有最终判断权

### 5. 为什么 Rollback 需要检查点而不能简单重试？

- **上下文重建**：重试若不恢复上下文，Agent 可能走回同一条错误路径
- **Artifact 隔离**：部分完成的 artifact 可能有中间状态污染
- **可审计**：检查点记录了状态快照，便于事后复盘

## 使用场景

### 场景 1：个人研究项目管理

管理多个并行的研究想法：
- 神经网络架构研究
- 可视化效果探索
- 代码实现验证

Beatless 自动分配 Agent、跟踪进度、生成报告。

### 场景 2：长期任务追踪（带 Rollback）

跨越多天的复杂任务：
- Day 1: Research 阶段
- Day 2: Implement 阶段（若偏离 → rollback 到 checkpoint）
- Day 3: Review 和迭代

任务状态持久化在 TASKS.yaml，随时可恢复。

### 场景 3：多 Agent 协作（带 Mailbox 协调）

并行派遣多个 Agent：
- Gemini-Researcher：理论分析 → 发现阻塞点 → 发送 `warn` 邮件到 Methode
- Codex-Builder：代码实现 → 收到邮件后调整实现方向
- Satonus：独立验收 → 完成后发送 `info` 邮件通知 Lacia

Mailbox 提供了无需中央协调的松散协作机制。

## 局限性与未来改进

### 当前局限

1. **人工介入点**：任务创建和最终验收仍需人类
2. **错误恢复**：复杂错误仍需人工诊断（Rollback 只能恢复状态，不能修复逻辑 bug）
3. **上下文限制**：子 Agent 的上下文仍有限制（Evidence-Preserving Compression 是缓解而非根治）
4. **Mailbox 不保证送达**：最佳 effort 机制，不适合关键告警

### 未来方向

1. **自动任务分解**：大任务自动拆分为子任务
2. **学习优化**：根据历史数据优化路由策略
3. **多模态支持**：集成代码、图像、音频处理
4. **分布式执行**：跨机器并行执行
5. **Checkpoint 自动化**：基于 LLM 判断自动插入检查点
6. **Mailbox 可靠性提升**：引入持久化 ACK 和重试机制

---

## 结语

Beatless v4 的核心演进：**从任务调度系统升级为具备控制平面的自主 Agent 编排系统**。

不再是：
> "帮我做这个..."
> → "创建一个任务，分配给合适的 Agent，跟踪进度，验收结果。"

而是：
> "创建一个任务，控制平面持续感知执行状态，按预算主动 assist / rollback，确保任务在约束内收敛。"

从**工具使用**到**工作流编排**，再跃迁到**预算感知的主动控制**。
