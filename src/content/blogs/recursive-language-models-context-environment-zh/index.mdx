提及"长上下文"，多数人脑中浮现的画面是：把更多 token 塞进提示词，然后祈祷注意力机制撑住。过去两年，每家前沿实验室都在推这条曲线——128K、200K、1M、2M——而曲线的形状出奇一致：前几万 token 准确率平滑上升，随后进入平台期，然后在超过标称窗口四分之一处坠崖。最能说明这一点的基准测试（BrowseComp-Plus、HashHop、大规模 Needle-in-a-Haystack）恰恰是前沿实验室已经不再主动引用的那些。

2026 年 4 月，MIT CSAIL 的一篇论文——Zhang、Kraska、Khattab 的《Recursive Language Models》（[arXiv:2512.24601](https://arxiv.org/abs/2512.24601)），配有一篇通俗易懂的解读发布在 [Prime Intellect 的博客](https://www.primeintellect.ai/blog/rlm)——提出了一种不同的思路。不是让根 LLM 消化更长的上下文，而是给它一个 Python REPL，把"上下文"作为 REPL 中的变量加载。根模型从不直接读取长输入。它编写代码来 grep、切片、摘要，并对片段调用子 LLM。完整上下文成为模型所**导航的环境**，而非消费的流。

报告的数据足够亮眼，这个框架值得拆解。单基准配置、10M+ token 环境（BrowseComp-Plus 风格的深度研究，具体数据集划分）下，论文报告 GPT-5 作为 RLM 根时准确率从约 24% 跃升至 62%，RLM 脚手架使用更小的根模型即可匹配 GPT-5-Pro。这些是特定设置下的单一研究数据——在任何人将其视为普适结论之前，需要独立复现。但比分数更重要的是，这个架构暗示了**长上下文能力从何而来、应该如何训练**。

## 核心操作：`llm_batch(x, prompt)` 作为原语

用一个代码块呈现全貌。根模型接收用户问题和一个语料库的**句柄**——而非语料库本身。

```python
# 这段代码运行在根模型的 REPL 内，而非之前。
# `context` 在 REPL 环境中绑定为 Python 变量。
# 根模型从未见过 `context` 的 token。

docs = context.load("internal_wiki")           # ~10M tokens，类型为 list[str]
# 过滤类 grep，不消耗任何 LLM token：
candidates = [d for d in docs if "refund" in d.lower()]
# 子 LLM 调用——每次调用作用于一个片段，独立运行：
summaries = llm_batch(
    candidates,
    prompt="用一句话摘要任何退款政策主张。"
           "如无相关内容则返回 ''。",
)
# 聚合，仍在 Python 中：
evidence = [s for s in summaries if s]
```

根模型现在决定如何处理 `evidence`。它可能直接综合出最终答案，可能用不同关键词重新 grep，也可能再次对 summaries 调用 `llm_batch` 并附上交叉引用提示。关键在于，每一步的成本都是可衡量且局部的：`context.load` 是 I/O，列表推导是 CPU，而 `llm_batch(xs, prompt)` 的成本大约是 `len(xs) * per_call_tokens`——REPL 周边的调度器可以轻易并行化所有这些操作。

使这三点成为 LLM 原语而非框架模式的关键在于：

1. **上下文作为变量，而非前缀。** 加载文档不消耗根模型窗口的任何 token。根模型读取一个句柄、一个长度和一些元数据。它通过编写代码来决定是否进一步读取。
2. **子 LLM 调用是批量的、提示携带的。** `llm_batch` 从根模型视角看是纯函数——输入列表、并行调用、结果返回——这意味着根模型无需推理隐藏的每调用状态。
3. **单一 REPL 会话持久化变量。** 第一步中的 `evidence` 在第二步时仍然绑定。这正是 RLM 递归而非"带工具的智能体"的本质：根模型逐步在语料库上构建自己的数据结构，并判断这些结构是否已足够好到可以从其中回答问题。

这三点各自单独存在于工具调用型智能体中。RLM 所做的是将它们固定在一个具体、简洁的接口上——`context`、`llm_batch`、一个持久的 Python 会话——并论证这个接口值得**训练进模型**而非靠提示词绕过去。

## 为什么"上下文作为环境"打破了长上下文税

标准长上下文失败模式的本质并非模型遗忘了读过什么，而是它在**与问题无关的 token 上支付了二次（或近似二次）的注意力成本**。一个 2M token 的提示词，其中 99% 的 token 与问题无关，但根模型仍然要计算每对 token 之间的注意力模式。最好的情况是模型学会了忽略其中大部分——但计算成本与不忽略时相同。

RLM 让这个成本对模型可见。根模型永远看不到无关 token。过滤在任何 token 进入任何注意力矩阵之前完成。摘要由并行的子 LLM workers 完成，每次调用的上下文是受限的。

基准测试的故事以一种特定方式反映了这一点。论文报告称，RLM 包装器随着语料库规模增长产生接近单调的准确率提升，而裸长上下文 LLM 在约 200K token 处开始坠崖。绝对数字取决于你查看哪个基准，但 BrowseComp-Plus、HashHop 和 HELM 较长配置中的曲线形状是一致的。底层原因是：RLM 的有效上下文等于根模型 Python 代码选择物化的内容——这随着模型的代码编写能力增长，而非随着其原生注意力预算增长。

还有一个在各类解读中被低估的第二个效应：**失败模式从幻觉转变为异常**。一个在干草堆中找不到事实的长上下文 LLM 通常会编造一个。grep 返回空结果的 RLM 根得到一个空列表。空列表是控制流信号；幻觉的事实不是。错误面从"自信地给出错误答案"转移到"困在缺乏证据的状态"——这在检测和绕过上要容易得多。

## 为什么它不同于 RAG、工具调用和 CodeAct

三种模式看起来表面相似，值得区分。

**RAG。** 检索器产生 top-k 段落，LLM 读取它们。检索器是固定的（一个独立的 embedding 模型）且无状态；LLM 无法决定"用不同关键词重新检索"，除非由编排器显式重新调用。RLM 将检索器和推理器合并到同一模型中，用任意代码操作语料库变量。检索函数是根模型选择编写的任何 Python。

**工具调用型智能体。** React 风格的智能体调用 `search(q)` 这类工具并读取简短返回结果。每次工具调用消耗根模型上下文窗口的 token——结果被注入会话。在一个 100 步的工作流中，上下文窗口充满了工具 I/O 和智能体自己的推理，最终撞上 RLM 试图规避的那个长上下文失败模式。RLM 的子 LLM 调用将返回值返回给**变量**，而非上下文窗口。根模型的窗口保持很小。

**CodeAct / 可执行动作。** CodeAct 风格的智能体已经通过编写 Python 来调用工具。这是最近的近亲。区别是哲学层面的：在 CodeAct 中，Python 是**表达工具调用的更清晰方式**。在 RLM 中，Python 是 LLM 群体的控制平面——`llm_batch` 是原语，根模型的工作是编写决定生成哪些子调用的代码。没有 `llm_batch` 的 CodeAct 智能体无法并行化对 10M token 语料库的推理。有 `llm_batch` 但没有外部工具的 RLM 仍然可以解决深度研究任务，因为子 LLM **就是**工具。

换一种说法：RLM 将 LLM 群体视为计算织物，教一个 LLM 在其上调度工作。工具调用教一个 LLM 将任务委托给非 LLM 服务。

## 训练中需要什么

开箱即用的 GPT-5 可以作为 RLM 根——论文展示了这一点——但其性能受限于 REPL 代码的编写质量。有两个训练增量将"如果你仔细提示就能配合 RLM 的模型"与"将 RLM 作为一等公民接口训练的模型"区分开来：

**成本感知规划。** 根模型的工作是在正确回答的同时最小化所有子 LLM 调用的总 token 数量。这是一个**在成本预算下进行程序综合**的问题。仅对答案正确性做朴素 RL 微调会收敛到"对每个文档调用 llm_batch，检查一切"——这是正确的但成本无穷大。训练需要一个同时惩罚错误答案和浪费子 LLM 调用的奖励函数。Prime Intellect 恰好集成了这种形态——其 `verifiers` 库中的 `RLMEnv`——这就是为什么他们的解读以 RL 集成为重点而非基准分数。

**REPL 原生输出。** 一个训练为在长期运行的 REPL 会话中输出 Python 的模型，与训练为一次性输出代码的模型有本质不同。变量命名、增量精炼、"让我通过打印中间结果来检查工作"——这些都是从标准代码补全预训练中不会涌现的习惯。RLM 论文的消融实验表明，在 10M+ token 任务上，少样本提示 RLM 根与微调一个 RLM 根之间的差距是巨大的，而这个差距几乎完全体现在根模型编写增量、可观测代码的能力上。

**子 LLM 校准。** 当子 LLM 返回"未提及退款政策"时，根模型应该信任多少？训练根模型对子 LLM 输出进行概率加权、在并行调用间要求交叉验证、或回退到更细粒度重新读取，这是一个在单调用 LLM 训练中不存在的新 RL 信号轴。这就是"智能体调用自身"框架的意义所在——根模型本质上是在对自己的输出进行自监督错误纠正。

## 训练时 vs 推理时的上下文折叠

RLM 论文中一个更有趣的推论，以及与前沿实验室发展方向最相关的部分，是**RLM 将长上下文能力转化为需要训练的东西，而非扩展进权重的的东西**。当前的前沿做法是向预训练投喂更长的序列，祈祷位置嵌入能内插，最后评估。RLM 的做法是保持每次调用的上下文小，训练模型在推理时将长上下文折叠成结构化数据。

这有两个值得关注的连锁效应。

首先，它解耦了模型规模与上下文长度。一个写出好 REPL 代码的 Haiku 级根模型，可以回答需要 10M token 语料的问题，而裸上下文的 Opus 无法做到，因为 Opus 将计算浪费在无用的注意力上，而 Haiku 在跑 grep。有趣的前沿变成"能良好编排的最小模型"，而非"能读取 2M token 的最大模型"。

其次，它重新框定了 KV-cache 压缩军备竞赛。当前 KV-cache 压缩的工作（本博客[在其他文章中](https://zakura-world.com/blog/kv-cache-compression-long-context-economics)已涵盖）试图让 1M token 提示词的 serving 成本可控。RLM 说：不要 serving 那个 1M token 提示词。并行 serving 十个 10K token 提示词，由一个小型根模型编排。经济学方向不同：每次查询调用更多次，但每次调用成本大幅降低，且可并行化程度高得多。

## 这对评估意味着什么

更重要的含义是方法论的。每个假设模型线性读取所有 token 的长上下文基准测试，测量的都是错误的量。一个通过读取 30K token 定向片段正确回答 10M token 问题的 RLM 根，并非在"作弊"——它在做任何称职的人类研究员会做的事。但它在任何只衡量最终答案正确性的基准上，与消化了全部 10M token 的裸长上下文模型得分无法区分。

正确的基准形态将不得不包含**token 预算准确率曲线**——系统在不同的子 LLM 调用总预算下表现如何——以及根模型生成的 REPL 代码的**程序质量指标**。这两者都借自程序综合评估，而非 NLP 评估。这才是持久的观察：长上下文 LLM 评估正在与程序综合评估收敛，因为长上下文推理正在与程序综合收敛。

## 现在该做什么

如果你在 2026 年构建智能体基础设施，实用的要点很具体。

1. **将语料库作为变量暴露，而非提示词前缀。** 智能体应接收句柄（文档 ID、数据库连接、文件系统路径）并按需程序化加载。"给你 500K token，尽力而为"的扁平提示词是越来越差的默认——它在无关 token 上支付注意力成本，并堵死了程序化过滤的门。
2. **在你的智能体运行时中构建 `llm_batch` 形态的原语。** 对同质输入的并行子 LLM 调用是调度收益最大、而同步 ReAct 循环服务最差的 workload。
3. **开始衡量 token 预算，而不只是准确率。** 如果你的智能体可以用 20K token 或 2M token 回答一个问题，这些是成本不同、产品不同的东西。成本-质量曲线才是基准。
4. **将 REPL 会话视为一等公民状态。** 在工具调用间维护持久 Python（或 JS/Lua）会话的智能体，比每次从零重新提示的智能体具有严格更强的表达能力。如果你的运行时间在调用间重置状态，你已经丢掉了一半的 RLM 收益。

RLM 论文谨慎地没有过度声明：这是一篇论文，数字需要独立复现，然后才有人应该在其基础上押注生产系统。但数字背后的架构声明——**上下文是环境，不是流；LLM 是调度器，不是读者**——是长上下文 AI 两年来一直指向的方向。前沿实验室现在必须回答的问题是：他们是继续将原生注意力窗口推向 10M，还是转向让模型更好地形成为调度器，以适应永远无法塞进任何窗口的工作负载。

我的赌注：两者都要，但第二个才会改变栈的形态。
