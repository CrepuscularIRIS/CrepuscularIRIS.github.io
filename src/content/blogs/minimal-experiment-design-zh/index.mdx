大多数机器学习实验的失败，不是因为假设错了，而是因为实验本身无法区分对结果的多种合理解释。

你在 CIFAR-10 上跑了一个新架构，比基线高出 2%。这究竟是因为架构改动确实有效？是因为你为新变体调了更好的超参数？是因为基线本身就欠拟合？还是因为跨种子的随机方差本来就能解释这 2%？或者以上全是？

没有精心设计的实验，你无法剥离这些因素。不剥离它们，你基于这个结果写的论文经不起审视——任何仔细思考过这些替代解释的审稿人，都会要求你提供那些你未曾运行的控制实验。

这篇文章提供了一套框架，帮助你在投入 GPU 算力之前，就设计出能回答核心问题的实验。示例取自残差连接变体——在这个领域里，"这是否比标准残差有改进？"是一个很容答错的问题——但这套结构适用于任何架构假设。

## 每个架构比较背后的三个假设

当你提出一个架构改动时，你隐含地做出了至少三个断言：

1. **H1（核心假设）**：结构改动使得基线在原则上无法做到的事情成为可能。
2. **H2（训练假设）**：新架构至少和基线一样易于训练——它能收敛，不会不稳定，且超参数敏感性在可控范围内。
3. **H3（成本假设）**：如果改进是真实的，那么它所带来的计算量、内存和实现复杂度的增加是值得的。

最小化实验设计的关键条件是每个假设对应一个实验。只验证 H1 而不验证 H2 和 H3，论文还没准备好。只验证 H2 和 H3 而不验证 H1，那是在做基准测试，不是假设检验。

在过去的三年里，我审阅了大约 200 篇 NeurIPS/ICML/ICLR 投稿。最常见的失败模式只有一个：作者跑了大量真实基准实验（验证 H2+H3），但从未隔离出机制本身（H1）。当审稿人问"你怎么知道收益来自你的特定机制而不是额外的参数？"时，反驳拿不出答案。

## 为 H1 设计：核心机制

验证核心假设最干净的方法是，构造一个任务，使得所提出的机制成为必要条件，而基线无法成功完成。

对于残差连接变体，核心主张通常是："动态层聚合使网络能够比固定权重求和更高效地路由信息。"其机制是：基于内容对先前层输出进行选择性注意力。

为此机制量身设计一个任务：

```python
class SelectiveDepthDependencyTask:
    """
    需要选择性访问特定深度特征的合成任务。
    标准残差（固定聚合）应该表现困难，因为它们无法
    选择性地放大相关深度层的贡献。
    """
    def __init__(self, n_layers: int = 12, embed_dim: int = 256):
        self.n_layers = n_layers
        self.signal_depth_sampler = torch.distributions.Categorical(
            probs=torch.ones(n_layers) / n_layers
        )

    def generate_batch(self, batch_size: int = 64):
        x = torch.randn(batch_size, self.embed_dim)
        signal_depth = self.signal_depth_sampler.sample((batch_size,))
        signal_content = torch.randn(batch_size, self.embed_dim)

        return {
            "input": x,
            "signal_depth": signal_depth,
            "signal_content": signal_content,
        }
```

关键洞察：如果机制有效，模型应该能够检索信号内容，而无论它被注入到哪一层。标准残差连接只有在信号恰好位于紧邻的若干层中时才能高效访问（远处的信息会通过加法链被稀释）。基于注意力的残差则可以直接查询信号。

这是一个**构造任务**，不是真实基准。这是刻意为之的。真实基准同时测量很多因素；构造任务只测量一个。如果你提出的机制在一个专门设计用来需要它的任务上都行不通，那论文还没准备好。

### 构造任务陷阱

一个细节需要注意：构造任务可能太简单了。如果你的合成任务完美匹配所提出架构的归纳偏置，100% 的解决率并不能证明机制有用——它只能证明你设计了一个架构注定会通过的测试。任务还需要具有可信的实用性。问问自己："从业者是否真的会遇到这种结构的问题？"如果答案是否定的，那合成结果就是一个重言式。

更强的设计包括一个负控制：一个机制**不应该**有帮助的任务。如果所提出的架构在不需要选择性深度访问的任务上也同样占优，那么机制解释很可能是错的——你测量的是一般性的容量优势（更多参数、更好的初始化），而不是特定的结构主张。

## 为 H2 设计：训练动态

架构论文往往低估了训练不稳定性。一种方法在作者手中达到 SOTA——经过数月调参后——可能在不同的硬件、学习率或数据预处理下表现脆弱。

最小的 H2 实验是在多个种子下进行学习率扫描：

```python
def h2_training_stability_sweep(
    model_factory: Callable[[], nn.Module],
    learning_rates: list[float],
    seeds: list[int],
    dataset: str = "cifar10",
    epochs: int = 50,
) -> dict:
    results = {}

    for lr in learning_rates:
        for seed in seeds:
            torch.manual_seed(seed)
            model = model_factory()
            optimizer = torch.optim.AdamW(model.parameters(), lr=lr)

            train_losses, grad_norms = [], []
            failed_at_epoch = None

            for epoch in range(epochs):
                loss, grad_norm = train_epoch(model, optimizer, dataset)

                if torch.isnan(torch.tensor(loss)) or torch.isinf(torch.tensor(loss)):
                    failed_at_epoch = epoch
                    break

                train_losses.append(loss)
                grad_norms.append(grad_norm)

            results[(lr, seed)] = {
                "final_loss": train_losses[-1] if train_losses else None,
                "failed_at_epoch": failed_at_epoch,
                "grad_norm_trajectory": grad_norms,
            }

    return results
```

你从 H2 扫描中想要得到的是：所提出的架构至少在和基线相同的超参数范围内保持稳定。如果一种架构要求 lr=1e-5，而基线在 lr=1e-3 下工作正常，那它不是一个实用的贡献。

梯度范数轨迹比最终损失更有信息量。一个模型因为梯度范数大 10 倍而只能在 1/10 的学习率下训练，这意味着存在一个隐藏的超参数依赖，会在实际应用中反噬。

### 审稿人实际检查什么

三个我见过的、能在审稿阶段扼杀架构论文的问题：

1. **"你是否对两种架构进行了同等程度的调参？"** 如果你为提出的方法跑了网格搜索，但基线用的是默认超参数，那么比较就是有混淆的。报告每个变体的调参预算。

2. **"更大规模下会发生什么？"** 一种在 12 层 / 256 维下有效的方法可能在 48 层 / 1024 维时崩溃。如果你无法运行大规模实验，至少展示 2-3 个模型规模下的扩展趋势。

3. **"改进是否在噪声范围内？"** 跨种子标准差为 1.2% 时，0.5% 的准确率提升不是一个结果。是噪声。报告置信区间，而不只是均值。

## 为 H3 设计：计算/准确率帕累托

如果一个架构只好了 1% 但慢了 3 倍，它不会对领域做出贡献。H3 实验必须刻画这个权衡。

正确的框架是**帕累托前沿**：在给定的计算预算下，每个变体能达到的最佳准确率是多少？

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class ExperimentPoint:
    model_variant: str
    flops_per_forward: float   # 实测，而非估算
    params: int
    val_accuracy: float
    training_hours: float      # 在特定硬件上，明确说明

def compute_pareto_frontier(
    points: list[ExperimentPoint],
) -> list[ExperimentPoint]:
    pareto = []
    for p in points:
        dominated = any(
            q.val_accuracy >= p.val_accuracy and
            q.flops_per_forward <= p.flops_per_forward and
            q != p
            for q in points
        )
        if not dominated:
            pareto.append(p)
    return sorted(pareto, key=lambda x: x.flops_per_forward)
```

一个贡献在帕累托前沿上，当在某个计算预算下，它的准确率高于所有替代方案。如果所提出的架构在任何地方都被帕累托支配——比某种基线变体组合更慢且更差——它不是一个贡献，无论在哪个单一基准上发生了什么。

### 内存维度

FLOPs 与准确率的权衡是必要的，但不够充分。内存消耗决定了什么能在真实硬件上运行。一个在 FLOPs 上帕累托最优但需要 2 倍 GPU 内存的方法（相同 batch size 下），对大多数从业者来说是无法部署的。将峰值内存与 FLOPs 一同报告——两者有时会急剧分化（注意力机制就是典型例子：计算高效但内存消耗大）。

## 四格设计矩阵

最小完整实验有四个格子：

| | 合成任务 (H1) | 真实基准 (H2+H3) |
|---|---|---|
| **基线** | 在机制特定任务上的准确率 | CIFAR-10 / 语言模型困惑度 / 下游任务 |
| **提出的方法** | 用提出架构运行相同任务 | 相同基准 |

合成任务测试 H1（机制）。真实基准测试泛化。基线数据建立公平测量的基准。

这个矩阵能捕获单基准评估会遗漏的内容：一种在真实基准上显示大增益但合成机制测试没有改进的架构，其收益可能来自基准中隐含的归纳偏置，而不是所声称的机制。论文主张与经验证据没有对齐。

## 方差控制实用规则

架构比较实验经常在方差控制不足的情况下运行。最低要求：

**匹配参数量。** 如果所提出的架构添加了参数（例如注意力残差中的注意力投影层），基线应该用相同的参数预算但以不同方式使用（例如更深或更宽）。参数差异比机制更频繁地解释性能差异。

**使用相同的数据流水线。** 变体之间任何数据增强、归一化或加载的差异都会引入混淆。最安全的方法是在单个训练脚本中实现两个变体，共享所有数据流水线代码。

**报告跨种子的方差。** 单次运行使用一个随机种子不是结果。报告至少 3 个种子的均值 ± 标准差。对于 1-2% 的差异要在 95% 置信度下有意义，通常需要 5 个以上种子。

**在对比运行期间测量梯度范数。** 不要在单独的实验中测量——就在实际的对比运行期间。梯度范数揭示两种架构是否在相似的训练机制下运行，还是其中一个获得了软性的学习率优势。

## 可复现实验的工具

上面的框架只有在实验可复现时才有用。有两个工具选择可以消除最常见的复现失败：

**使用 Hydra 进行配置管理。** 每个实验应该由一个配置文件完全指定。Hydra 的覆盖系统让你在不修改代码的情况下扫描超参数，并且它会与结果一起自动保存解析后的配置：

```yaml
# conf/experiment/h1_mechanism_test.yaml
model:
  variant: attention_residual
  n_layers: 12
  embed_dim: 256
training:
  lr: 1e-3
  epochs: 50
  seeds: [42, 123, 456, 789, 1024]
task:
  type: selective_depth_dependency
  batch_size: 64
```

```bash
# 运行扫描
python run_experiment.py --multirun \
  model.variant=standard_residual,attention_residual \
  training.lr=1e-4,3e-4,1e-3,3e-3
```

**使用 W&B 或 MLflow 进行实验追踪。** 将指标、配置和梯度范数记录到追踪服务器。关键规范：将*配置哈希*作为运行标签记录，这样你总能追溯一个结果到产生它的确切配置。

## 何时停止实验

实验设计中最容易被忽视的技能是知道何时有足够的证据。三个停止标准：

1. **H1 为负。** 机制在合成任务上没有帮助。停止。真实基准结果（如果有的话）可以用你声明的机制之外的东西来解释。在运行更多实验之前修改假设。

2. **H2 显示不稳定。** 架构在基线成功范围超过 50% 的学习率范围内失败。停止。在评估准确率之前修复训练动态。

3. **H3 显示帕累托支配。** 提出方法在每个计算预算下都更慢且准确率不更高。停止。当前形式下这不是一个贡献。

诱惑总是再跑一个实验——一个不同的数据集、不同的规模、不同的超参数。除非你对下一个实验为何会改变结论有一个特定的假设，否则抵制这种诱惑。无方向的实验就是这样让你在论文上烧掉数月，而它早在第一周就已经死了。

## 完整稿件展示什么

架构论文的报告标准：

1. 合成任务结果（H1：机制被确认或推翻）
2. 学习率扫描结果（H2：稳定性被刻画，范围被陈述）
3. 帕累托前沿图（H3：效率权衡被量化）
4. 混淆部分：明确说明实验**没有**控制什么，以及你为什么相信那些混淆不会解释结果

第四部分是最有价值的，也是最常被省略的。一个坦诚的混淆部分比没有混淆部分更有力——它表明作者理解其实验的局限性，这是审稿人信任其结论的前提条件。

目标不是跑更多实验。是跑最少的实验，使得假设和证据对一个持怀疑态度的读者清晰可辨。在大多数架构论文中，这意味着比通常提交的更少但设计得更好的实验。
