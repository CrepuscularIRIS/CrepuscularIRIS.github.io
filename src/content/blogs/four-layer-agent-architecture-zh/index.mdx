在过去半年构建和迭代 AI 编程 Agent 配置的过程中，我发现了一个尚未被系统性记录的模式。我接触过的每一套成熟的 Agent 配置——无论是我自己的 Claude Code 配置、类似的 Codex CLI 方案，还是 Gemini CLI 工作区——最终都会收敛到四个不同的层次，每个层次解决一个不同的关注点。这四层分别是：Rules、Skills、Agents 和 Hooks。

这不是一个框架提案，而是对一个客观现象的描述：当你认真对待 Agent 辅助开发并需要引入结构化管理时，这四层会自然浮现。它们之所以出现，是因为它们解决的是四个不同的问题——把其中任意两个混为一谈，都会带来随时间持续累积的维护负担。

## 四层架构

### 第一层：Rules——始终生效的策略约束

Rules 是适用于每一次交互的全局约束。它们不教 Agent 如何做事，而是定义在所有任务中"什么是可接受的"。

```
~/.claude/rules/
  common/
    coding-style.md    # immutability, file size limits, error handling
    security.md        # no hardcoded secrets, input validation, OWASP
    testing.md         # 80% coverage minimum, TDD workflow
    git-workflow.md    # conventional commits, branch strategy
  python/
    coding-style.md    # PEP 8, type hints, ruff/black
    testing.md         # pytest, coverage config
  golang/
    coding-style.md    # gofmt, accept interfaces return structs
    testing.md         # table-driven tests, race detector
```

核心设计决策是：Rules 按**特异性分层**。通用规则（common）建立普适原则——不可变性、错误处理、安全底线。语言特定规则在其基础上扩展具体的工具和模式。例如，Python 的 coding-style.md 会引用通用版本的不可变性原则，然后追加 `ruff check .` 和 `mypy src/` 等具体验证命令。Go 的 testing.md 会引用通用的 80% 覆盖率要求，然后追加 `go test -race ./...` 的具体执行方式。

Rules 回答的是："所有产出必须满足什么标准？"它们在每次对话中都会被加载到上下文中，占用固定的 Token 预算。一套设计良好的规则集总计不超过 2,000 Token——足以建立约束，又不会挤占工作记忆。

**反模式识别**：区分策略与过程。"每次编辑后都要运行测试"属于 Rule——这是一个适用于所有场景的标准。"以下是如何为数据库连接编写 pytest fixture"则不属于——那是一个特定任务类型的详细知识，应该归入 Skills。如果你发现自己的 Rules 文件超过了 500 Token，很可能你在 Rules 里混入了过程性指令。

### 第二层：Skills——按需加载的深度知识

Skills 是当检测到特定任务类型时才被激活的详细参考资料。与 Rules（始终加载）不同，Skills 是有选择性地加载的，以节省上下文窗口空间。

一套生产级配置可能包含 30 多个 Skills，涵盖：

```
Research:     research-ideation, citation-verification, results-analysis
Writing:      ml-paper-writing, review-response, writing-anti-ai
Development:  git-workflow, bug-detective, architecture-design, tdd-workflow
Plugin:       skill-development, hook-development, mcp-integration
Design:       frontend-design, ui-ux-pro-max, web-design-reviewer
```

每个 Skill 是一份自包含的文档，通过 frontmatter 声明其激活条件：

```yaml
---
name: bug-detective
description: >
  Debugging and error investigation. Use when the user reports
  errors, exceptions, unexpected behavior, or asks to investigate.
---
```

Agent 会将用户请求与 Skill 描述进行语义匹配，并在响应前加载匹配的 Skills。这就是与 Rules 的关键区别：Skills 是**按需加载**的，而非始终存在。

实际运行中的 Skill 激活模式：

```
User: "Fix this authentication bug"

Agent 语义匹配:
  - bug-detective skill → match (keyword: bug, fix, error)
  - security-review skill → match (keyword: authentication)
  - tdd-workflow skill → 可能适用 → 加载

Context 构成:
  [Rules: ~2K tokens] + [3 Skills: ~8K tokens] + [Code: variable]
```

一个 3,000 Token 的 TDD 工作流 Skill 只在编写测试时才占用上下文。一个 5,000 Token 的论文写作 Skill 只在起草学术内容时才加载。这种按需加载机制是 Skills 与 Rules 的核心分界线——它让一个拥有 30+ Skills 的配置在简单任务上只消耗 2K Token 的配置开销，而非一次性加载全部 80K+ Token 的知识库。

Skills 回答的是："面对这类特定任务，我应该如何处理？"这就像从书架上抽出一本参考手册——你不可能随时携带所有手册。

**Skill 设计原则**：

1. **自包含**：每个 Skill 不应依赖其他 Skill 同时被加载
2. **描述精确**：描述必须包含足够的关键词来确保正确的语义匹配
3. **渐进式披露**：先概述核心原则（~500 Token），再展开详细参考（~2000-5000 Token）
4. **行为指导而非知识百科**：Skill 应该告诉 Agent "做什么"，而不是"世界是什么样的"

### 第三层：Agents——专用的执行上下文

Agents 是拥有独立上下文窗口、工具访问权限和行为约束的隔离子进程。它们适用于那些需要关注点分离的任务——要么因为任务足够大以至于需要独立上下文，要么因为任务需要不同于主编排器的视角。

```python
# Conceptual agent registry
AGENTS = {
    "code-reviewer":        {"focus": "quality, security, maintainability"},
    "build-error-resolver":  {"focus": "minimal diffs to fix build failures"},
    "tdd-guide":            {"focus": "test-first methodology, coverage"},
    "architect":            {"focus": "system design, module boundaries"},
    "refactor-cleaner":     {"focus": "dead code removal, consolidation"},
    "bug-analyzer":         {"focus": "execution flow tracing, root cause"},
}
```

Agents 与 Skills 的本质区别在于：Agents 运行在**隔离的上下文**中。当编排器启动一个 code-reviewer Agent 时，该 Agent 获得自己的对话、加载自己的 Rules 和 Skills、独立地操作代码库，然后返回结构化的结果。编排器的上下文不会被 reviewer 的中间推理过程所污染。

这在规模化场景下至关重要。一次审查 15 个文件的 Code Review 会产生数千 Token 的分析内容。如果这些分析发生在主上下文中，就会挤掉下一个任务所需的工作记忆。Agent 隔离机制保持了编排器上下文的整洁。

**何时启用 Agent 而非 Skill？**

判断标准不是任务复杂度，而是**上下文隔离的需求**：

| 信号 | 选择 | 原因 |
|------|------|------|
| 任务会产生大量中间推理 | Agent | 避免中间结果污染主上下文 |
| 任务需要不同的视角 | Agent | 独立上下文产生独立判断 |
| 任务需要并行执行 | Agent | 多个独立上下文可以同时运行 |
| 任务可以在当前上下文内完成 | Skill | 无需隔离开销 |
| 任务结果需要立即影响后续推理 | Skill | Agent 返回的是摘要，不是完整推理链 |

并行执行是 Agents 的另一个关键能力。当任务相互独立时，多个 Agent 可以同时启动：

```
# 三个独立审查，同时启动
Agent 1: security-reviewer 审查认证模块
Agent 2: code-reviewer 审查 API 端点
Agent 3: refactor-cleaner 扫描死代码

# 编排器聚合结果
→ 总耗时 ≈ max(Agent1, Agent2, Agent3)，而非三者之和
```

**Agent 自动触发规则**：

在实践中，某些事件应该自动触发 Agent 调用，而不需要用户明确请求：

| 事件 | 触发 Agent | 原因 |
|------|-----------|------|
| 代码编写完成 | code-reviewer | Review 结论不应干扰实现上下文 |
| 构建失败 | build-error-resolver | 错误诊断是一种不同的推理模式 |
| 新功能请求 | tdd-guide | TDD 脚手架适合在全新上下文中搭建 |
| 架构决策 | architect | 设计探索需要足够空间评估备选方案 |
| Bug 报告 | bug-analyzer | 深度执行流追踪会消耗大量上下文 |

### 第四层：Hooks——事件驱动的确定性自动化

Hooks 是由系统事件触发的 Shell 命令。它们弥合了 Agent 对话模式与生产工作流所要求的确定性自动化之间的鸿沟。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "node security-guard.js",
        "description": "Block writes containing secrets or dangerous patterns"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "ruff check --fix $FILE && black $FILE",
        "description": "Auto-format Python files after modification"
      }
    ],
    "Stop": [
      {
        "command": "node session-summary.js",
        "description": "Generate work log on session end"
      }
    ]
  }
}
```

Hooks 回答的是："有哪些事情应该自动发生，而不依赖于 Agent 的决策？"

这是 Hooks 与其他三层的根本区别。Rules 告诉 Agent 什么是可接受的——但 Agent 可能忘记检查。Skills 告诉 Agent 如何处理任务——但 Agent 可能选择不同的方法。Agents 执行专项工作——但编排器可能忘记调用它们。而 Hooks **无论 Agent 做出什么决定都会执行**。

一个阻止写入机密信息的 security-guard Hook 不依赖于 Agent 记得去检查机密信息。一个每次文件编辑后运行的格式化 Hook 不依赖于 Agent 记得去格式化。这种自动化是**结构性的**，而非**认知性的**——它发生在 Agent 的决策循环之外。

三种 Hook 类型对应一个完整的生命周期：

- **PreToolUse**：操作前的验证——安全检查、参数清洗、危险操作拦截
- **PostToolUse**：操作后的清理——代码格式化、Lint 检查、类型检查
- **Stop/SessionEnd**：会话结束时的收尾——工作总结、临时文件检测、状态保存

## 为什么是四层，不是三层或五层

每一层应对一种不同的失效模式：

| 缺少该层 | 失效模式 | 后果 |
|---|---|---|
| 无 Rules | Agent 在跨会话过程中逐渐偏离标准 | 代码风格不一致、安全检查遗漏 |
| 无 Skills | Agent 在专业任务上只能给出浅层建议 | TDD 流程不完整、论文结构混乱 |
| 无 Agents | 复杂多步任务导致主上下文溢出 | 第五步输出质量远不如第一步 |
| 无 Hooks | 确定性检查依赖于 Agent 的记忆 | 偶尔忘记格式化、偶尔忘记安全检查 |

**合并层次的后果**：

将 Rules 合并到 Skills 中，安全标准就不再始终生效——当 security Skill 未被触发加载时，Agent 可能忘记检查机密信息。将 Agents 合并到主上下文中，意味着每个子任务的推理过程都在争夺同一份 Token 预算——15 个文件的 Code Review 分析会挤掉后续任务所需的工作记忆。将 Hooks 合并到 Rules 中，则意味着自动化依赖于 Agent 对指令的解读和遵守，而非确定性执行。

你或许会主张需要第五层——比如用于跨会话持久化的"Memory"层。但在实践中，Memory 是一个正交的关注点，它横切所有四层，而非与它们并列。Rules 以文件形式持久化，Skills 以文件形式持久化，Agent 配置以文件形式持久化，Hooks 以配置文件形式持久化。对话级的 Memory（上一次会话发生了什么）则是一种完全不同的机制——它更接近数据库，而非架构层。

## Token 预算架构

驱动这一设计的核心约束是上下文窗口。即使 Opus 4.6 拥有 1M Token 的可用容量，上下文也不是免费的——每一个用于配置的 Token，都是一个无法用于代码、推理或工具输出的 Token。更关键的是，上下文窗口的后半段存在注意力衰减：越靠后的内容，被模型正确引用的概率越低。

```
Layer       | 加载时机           | 典型预算       | 持久化方式
------------|-------------------|--------------|-----------
Rules       | 始终加载           | 1,500-2,500  | 文件
Skills      | 按需加载           | 2,000-5,000/个 | 文件
Agents      | 启动时加载          | 独立上下文      | 子进程
Hooks       | 事件触发           | 0 (外部执行)   | 配置 JSON
```

这种分层加载机制意味着，一个简单任务（"重命名这个变量"）只产生约 2K Token 的配置开销。而一个复杂任务（"用 TDD 实现认证功能、Review 它、并验证构建"）可能通过 Rules + 3 个 Skills 使用约 15K Token 的配置，同时还会启动 2-3 个拥有独立上下文的 Agents——每个 Agent 内部也加载自己的 Rules 和相关 Skills。

总的上下文预算分配：

```
简单任务:  Rules(2K) + Code(variable)                    → 配置开销 ~2K
中等任务:  Rules(2K) + Skills(8K) + Code(variable)       → 配置开销 ~10K
复杂任务:  Rules(2K) + Skills(8K) + 3 Agents(各自独立)    → 主上下文 ~10K
                                                          + 3个子上下文各 ~10K
```

## 从零构建你自己的四层架构

如果你正在从零开始搭建 Agent 辅助开发工作流，建议按以下顺序逐步构建：

**第一步：从 Rules 开始。** 写下适用于项目所有工作的 5-10 条约束，总计控制在 2,000 Token 以内。常见的起点：编码风格（不可变性、文件大小限制）、Git 工作流（Conventional Commits）、安全基础（禁止硬编码密钥）、测试要求（80% 覆盖率）。

**第二步：在触及深度瓶颈时添加 Skills。** 当你发现 Agent 在你关心的主题上给出的建议过于浅层时——例如 TDD 流程只是"先写测试再写代码"而缺少具体的 pytest fixture 组织方式——就为其创建一个 Skill。每个 Skill 应当自包含，并能通过描述关键词来精确匹配触发条件。

**第三步：在上下文溢出时引入 Agents。** 信号是多步任务开始失去连贯性——Agent 忘记了早先的决策、自相矛盾，或者第五步的输出质量明显不如第一步。这意味着上下文窗口已经饱和。为那些产生大量中间推理的步骤启动专用 Agent。

**第四步：最后添加 Hooks。** Hooks 用于自动化那些你发现自己（或 Agent）经常遗忘的事情。每当你心想"我本该格式化那个文件"或"我本该检查有没有机密信息"，那就是一个等待被写出的 Hook。

四层模式并不限定具体工具。它可以与 Claude Code 的 `.claude/` 目录配合，可以与 Codex CLI 的配置配合，可以与 Gemini CLI 的设置配合，也可以与任何将策略与执行分离的 Agent 框架配合。

关键在于分离：策略（Rules）与知识（Skills）与执行（Agents）与自动化（Hooks）。将其中任意两者合并，你会在日常使用一周内感受到痛苦。保持它们分离，你就拥有了一套能随项目规模持续扩展的 Agent 配置架构。
