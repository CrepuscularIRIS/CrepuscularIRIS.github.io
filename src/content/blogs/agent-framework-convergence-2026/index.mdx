In the first two weeks of April 2026, three events in agent orchestration individually look like footnotes and together reveal a structural shift. Microsoft released Agent Framework v1.0, the product of merging Semantic Kernel and AutoGen into a single runtime. CrewAI disclosed that its platform processes 12 million agent executions per day. And a fresh crop of minimal hierarchical frameworks — CEO / Manager / Worker in three hundred lines of Python — keep hitting thousands of GitHub stars within days of launch.

The surface observation is that agent frameworks are popular. The deeper observation is that they are converging on the same abstractions, despite starting from radically different premises about how multi-agent systems should work. And once you notice the convergence, a second question becomes more pressing than "which framework should I pick?": which layer of the stack is actually the stable interface, and which layer is going to be commoditized?

The answer is not flattering to the framework vendors. The stable interface is the protocol layer — MCP, A2A, AP2, OpenTelemetry GenAI. The framework is rapidly becoming an implementation detail.

## The Five Patterns That Won

Look at every major agent framework shipping in April 2026 — Microsoft Agent Framework, CrewAI, LangGraph, Anthropic's Managed Agents, OpenAI's Swarm successors — and you find the same five orchestration patterns implemented under different names:

| Pattern | Microsoft AF | CrewAI | LangGraph | Managed Agents | Description |
|---------|-------------|--------|-----------|----------------|-------------|
| Sequential | Pipeline | Process: sequential | Linear graph | Task chain | Agents execute in fixed order |
| Concurrent | Parallel dispatch | Process: parallel | Fan-out / fan-in | Parallel tasks | Independent agents run simultaneously |
| Handoff | Agent handoff | Delegation | Edge routing | Agent transfer | One agent passes control to another |
| Group chat | Group chat manager | Process: consensual | Multi-node cycle | Council | Agents deliberate to reach consensus |
| Hierarchical | Orchestrator agent | Manager delegation | Supervisor node | Supervisor / worker | Top-level agent decomposes and delegates |

These are not superficially similar. They are structurally identical. A "sequential pipeline" in Microsoft AF and a "sequential process" in CrewAI produce the same execution graph — a linear DAG whose nodes are agent invocations and whose edges are typed messages. The difference is in the API surface, the docs style, and the tooling around it. The computational model is the same.

Convergence did not happen by coordination. It happened because the design space of orchestration is smaller than it appeared in 2024-2025, when every framework tried to differentiate through novel execution models (voting, auctions, blackboard architectures, dynamic team formation). Most of those models turned out to be research artifacts. The patterns that survive are the ones that have counterparts in distributed systems literature — the same five patterns you see in workflow engines like Airflow, Temporal, or Step Functions.

The reason for the overlap is not mysterious: multi-agent orchestration is a distributed-workflow problem with LLMs at the nodes. The LLM novelty does not change the orchestration primitives. The orchestration layer looks like Temporal-with-LLM-nodes, and Temporal has had thirty years of distributed systems research telling it what primitives matter.

## What Microsoft's Merge Actually Reveals

The Agent Framework v1.0 merge is the most architecturally informative event of the quarter because it forced Microsoft to publicly commit to which abstractions they thought would survive. Merging Semantic Kernel and AutoGen required finding the common layer between two frameworks with opposite philosophies:

**Semantic Kernel** started as an enterprise SDK — strongly typed, plugin-oriented, designed for embedding LLM calls inside .NET / Java / Python applications. Its mental model: the LLM is a function you call from application code.

**AutoGen** started as a Microsoft Research project — conversation-oriented, designed for multi-agent interaction experiments. Its mental model: agents are autonomous entities that send messages to each other.

Microsoft's merged answer is the **agent runtime** — a process that manages agent lifecycle, message routing, state persistence, and retry semantics, independent of how individual agents are implemented. The agents themselves are actors; the runtime is the mailbox and scheduler.

```python
# Illustrative sketch only — not a runnable API.
# The real framework uses autogen_agentchat with AssistantAgent,
# GroupChat, and a RoutedAgent base class; the snippet below
# compresses several layers for readability.
from agent_framework import AgentRuntime, Agent, Message

class ResearchAgent(Agent):
    async def handle(self, message: Message) -> Message:
        results = await self.tools.search(message.content)
        return Message(role="research", content=results)

class WriterAgent(Agent):
    async def handle(self, message: Message) -> Message:
        draft = await self.llm.generate(
            f"Write based on this research: {message.content}"
        )
        return Message(role="writer", content=draft)

runtime = AgentRuntime()
runtime.register(ResearchAgent(), WriterAgent())
runtime.set_pattern("sequential")  # or "handoff", "group_chat", etc.

result = await runtime.run("Analyze the latest MCP protocol changes")
```

Four design decisions in the merged framework are worth making explicit because they represent bets about what production agent infrastructure needs:

1. **Streaming by default.** Every agent interaction is a stream, not a request-response. This enables real-time observability, early termination, and interactive human-in-the-loop patterns.
2. **Checkpointing built in.** The runtime periodically snapshots agent state to durable storage, so a crashed workflow resumes from the last checkpoint rather than restarting. This is the same pattern Temporal uses — treat the agent as a durable function.
3. **Human-in-the-loop as a first-class pattern.** Any step can be configured to pause and wait for human approval. The pause is not an exception — it is a first-class runtime state, with named edges for `approve`, `reject`, `modify`.
4. **MCP for tools, A2A for inter-framework communication.** Tools are MCP servers. Communication between agents in different frameworks uses A2A. Microsoft's framework is the first major SDK to treat both protocols as load-bearing infrastructure rather than optional integrations.

The runtime-centric design is the key move. Both Semantic Kernel's "functions" and AutoGen's "agents" collapse into the same thing: handlers on a runtime. This is the actor model, rediscovered for the third time in computing history. The novelty is not the model — Erlang had this in 1986 — it is that LLM agents happen to fit the actor model almost perfectly. Each agent has private state, processes messages one at a time, can spawn other agents, and communicates only via explicit messages. If you were designing an LLM orchestration runtime from scratch and did not know about actors, you would rediscover them.

## CrewAI's Scale Numbers, Read Skeptically

CrewAI's published 12-million-daily-executions figure is worth reading carefully, because the distribution of those executions matters more than the headline. An approximate breakdown, inferred from CrewAI's published case studies, developer-community discussion, and conference talks — not a precise internal metric — roughly maps to:

- **Sequential**: ~60% of workflows. The most common pattern is "Agent A does research, Agent B writes, Agent C reviews." Simple, predictable, debuggable.
- **Hierarchical**: ~25% of workflows. A manager agent decomposes a complex task and delegates to specialist agents. Common in customer support and content generation.
- **Concurrent**: ~10% of workflows. Independent tasks executed in parallel for latency reduction.
- **Group chat / consensus**: ~5% of workflows. Multiple agents deliberate on a decision. The least common in production because it is the hardest to debug when it fails.

The takeaway: production agent systems overwhelmingly use the simplest orchestration pattern that works. The sophisticated patterns — consensus, dynamic routing, adaptive hierarchies — are research artifacts that rarely survive contact with production requirements like latency budgets, cost controls, and debuggability. Every agent engineer has lived this: the group-chat pattern works beautifully in the demo, and two weeks into production you are trying to figure out why three agents are in a polite disagreement loop that is burning $40/hour.

A complementary data point: the [multi-agent scaling illusion](/blog/multi-agent-scaling-illusion) result formalizes why this distribution is not an accident. Without independent information sources between agents, the expected-utility upper bound of a multi-agent system equals that of a single optimal agent. The simpler patterns — sequential, hierarchical — preserve information asymmetry by construction (each agent has a distinct role and distinct inputs). The consensus patterns routinely destroy it by having all agents read the same context and vote.

## Why the Simplicity Signal Is Real

The rapid adoption of minimal hierarchical frameworks in early 2026 is the market signal that framework builders should take seriously: developers do not want more abstraction layers, they want fewer. The most-starred recent entrants share a three-concept API:

- **Planner**: top-level agent that receives the goal.
- **Manager**: mid-level agent that breaks goals into tasks.
- **Worker**: leaf agent that executes a single task.

No plugin system, no middleware, no lifecycle hooks. The planner-manager-worker hierarchy maps directly to how humans think about delegation, which means developers can reason about the system without learning framework-specific concepts. The tradeoff is flexibility — these minimal frameworks cannot express cyclic workflows, dynamic agent spawning, or conditional routing beyond a couple of levels. But for the 60% of workflows that are sequential delegation chains, simplicity is a feature, not a limitation.

The structural lesson is familiar from every other piece of enterprise infrastructure: the abstractions that win are the ones that map cleanly to the mental model of the user who is *already* under time pressure. Abstractions that require a mental retooling lose to ones that match how people already think, even when the retooled abstractions are technically superior.

## Protocol Convergence: MCP + A2A + AP2 as Infrastructure

The more significant convergence is at the protocol layer, and this is where framework builders should be most nervous. The major frameworks in the April 2026 landscape — Microsoft Agent Framework, CrewAI, LangGraph, Anthropic's Managed Agents — are all either shipping or in active integration with three interop protocols as infrastructure:

- **MCP (Model Context Protocol)** for tool access. The [MCP 2026 roadmap](/blog/mcp-2026-roadmap-tasks-primitive) added the Tasks primitive (SEP-1686), turning stateless tool calls into first-class long-running operations with explicit input / output schemas. The 2025-03-26 spec revision added OAuth 2.1 authorization with dynamic client registration, closing the multi-tenant gap.
- **A2A (Agent-to-Agent Protocol)** for cross-framework communication. [A2A v1.0](/blog/a2a-v1-signed-cards-ap2-mandates) shipped Signed Agent Cards — JWS signatures (RFC 7515) over an RFC 8785 JSON-canonicalized Agent Card describing each agent's capabilities, input schema, and authentication requirements. Discovery is now authenticable.
- **AP2 (Agent Payments Protocol)** for financial actions. AP2's signed Mandate structure gives an agent an interoperable capability token — "this agent may execute up to $X until date Y" — that any counterparty can verify independently.

An Agent Card looks like this:

```json
{
  "name": "research-agent",
  "version": "1.0.0",
  "capabilities": ["web_search", "document_analysis"],
  "input_schema": {
    "type": "object",
    "properties": {
      "query": { "type": "string" },
      "depth": { "type": "integer", "default": 3 }
    }
  },
  "auth": {
    "type": "oauth2",
    "token_endpoint": "https://auth.example.com/token"
  },
  "signature": {
    "alg": "ES256",
    "kid": "issuer-key-2026-q2",
    "signature": "..."
  }
}
```

The practical consequence is that you can now build a multi-agent system where some agents run on Anthropic's infrastructure, some on Azure's, and some on your own servers, and they communicate through standardized, signed protocols. *Framework choice becomes an implementation detail rather than an architectural commitment.* That sentence is the one every framework vendor is trying to pretend is not true.

Layer above this a fourth convergence: **OpenTelemetry GenAI semantic conventions** for tracing. The semantic-convention draft that stabilized in early 2026 defines attributes (`gen_ai.system`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.agent.name`, `gen_ai.operation.name`) that every framework is adopting for spans. Once the traces look the same across frameworks, the observability stack becomes framework-agnostic too. Langfuse, Arize, Phoenix, Datadog all ingest the same schema.

This is the stack in 2026:

```text
┌────────────────────────────────────────────────┐
│  Observability: OTel GenAI semantic convs      │
├────────────────────────────────────────────────┤
│  Trust / identity: A2A Signed Cards + AP2      │
├────────────────────────────────────────────────┤
│  Tool interop: MCP (Tools, Tasks, Resources)   │
├────────────────────────────────────────────────┤
│  Agent runtime: Microsoft AF / CrewAI / …      │
├────────────────────────────────────────────────┤
│  Model: Claude / GPT / Gemini / Open-weight    │
└────────────────────────────────────────────────┘
```

The framework layer is sandwiched between the model layer and a thick protocol layer. Both sandwich layers are standardized (models via OpenAI-compatible APIs for years, protocols via MCP / A2A now). Being sandwiched between two standardized layers is exactly the position where a layer becomes commodified.

## What Gets Given Up in Convergence

Convergence is not free. Each framework lost something distinctive when it adopted the common abstractions, and it is worth naming what was given up:

- **Semantic Kernel** gave up the "LLM as function call" metaphor. The merged Agent Framework treats agents as long-running actors with identity and state. Integrations that used to be a single `semantic_function.invoke()` call are now agent registrations with lifecycle hooks.
- **AutoGen** gave up its open-ended conversation model. The merged runtime enforces typed messages and explicit patterns. Research-style "let the agents figure it out" is still possible through the `group_chat` pattern, but it is no longer the default.
- **CrewAI** gave up the option to invent novel coordination primitives. The public API is now pinned to the five standard patterns. CrewAI's distinctive "process" abstraction has become a thin wrapper over the same graph the other frameworks produce.
- **LangGraph** gave up purity. LangGraph's original pitch was "your agent is a state machine, explicitly." To compete in the 60%-sequential world, LangGraph had to add higher-level wrappers that hide the graph for simple cases — undoing some of the clarity that was its selling point.

Every convergence story looks like this. The frameworks that survive are the ones that accept the cost of matching the shared mental model. The ones that insist on distinctive abstractions become niche.

## What This Means for Agent Engineers

**If you are choosing a framework today.** The orchestration patterns are equivalent across frameworks. Choose based on your existing infrastructure and team: Microsoft stack → Agent Framework, Python-native with high-volume crews → CrewAI, graph-oriented workflows with explicit state → LangGraph, minimal dependencies → one of the planner/manager/worker micro-frameworks. Do not spend weeks on a comparison matrix. The decision is reversible.

**If you are building agent infrastructure.** Invest in MCP, A2A, AP2, and OpenTelemetry GenAI integration rather than framework-specific abstractions. The frameworks are converging; the protocols are the stable interface. Code you write against MCP will still work when you swap frameworks. Code you write against a framework's proprietary "workflow DSL" will not.

**If you are designing agent systems.** Start with sequential orchestration and add complexity only when you have evidence that sequential is insufficient. The production data is clear: most workflows do not need sophisticated patterns, and the simpler patterns are dramatically easier to debug, monitor, and operate. Reach for hierarchical when a single agent's context is overloaded. Reach for concurrent when latency dominates cost. Reach for group-chat essentially never — the [multi-agent scaling illusion](/blog/multi-agent-scaling-illusion) tells you why.

**If you are evaluating agent capabilities.** Benchmark the agent's task performance, not the framework's feature list. A simple agent with good prompts, good tools, and a sequential pipeline will outperform a sophisticated multi-agent consensus system with mediocre prompts every time. The framework-layer differences disappear in the production eval; the prompt, tool, and memory differences stay.

**If you are thinking about the two-year horizon.** The interesting research questions have moved from "how do we orchestrate agents?" to "how do we make orchestrated agents reliable, observable, secure, and cost-effective?" — questions that framework convergence makes easier to answer because the evaluation criteria are becoming standardized. The innovation is shifting up the stack (evaluation, alignment, capability gating) and down the stack (inference efficiency, scheduling, cost attribution). The middle — the orchestration framework itself — is being commodified, and that is the natural endpoint of any maturing infrastructure layer.
