The skills system in Claude Code is the part of the platform that most users underuse. The typical pattern: install a skill, trigger it once, find it marginally helpful, move on. The gap between this experience and what skills can actually do is about 10x — and the gap is entirely in how the skill is written, not how the skill is invoked.

This post covers four things: what a skill actually is architecturally (the answer is more interesting than "a Markdown file"), what separates skills that produce reliable behavior from skills that produce vague improvements, three production patterns from the OpenClaw configuration, and how skills interact with the Managed Agents session/harness model that launched in April 2026.

## What a Skill Is, Architecturally

A skill is a Markdown file injected into Claude's context when invoked. The directory structure:

```
~/.claude/skills/
├── git-workflow.md       # Activated by /git-workflow
├── code-review.md        # Activated by /code-review
├── python-patterns.md    # Activated by /python-patterns
└── ml-paper-writing.md   # Activated by /ml-paper-writing
```

When you call `/code-review`, Claude Code reads the skill file and prepends it to the conversation context before responding. The model's behavior changes because its context changed — not because the model changed.

This framing — skills as **context-window policy documents** — makes two things clearer:

**First**, skills are in direct competition with everything else in the context window. A 3,000-word skill is 3,000 tokens that can't hold tool results, retrieved files, or conversation history. The discipline required to write good skills is partly the discipline of compression: encoding maximum behavioral specificity in minimum tokens.

**Second**, skills compose by concatenation, not by some smarter mechanism. When you activate three skills simultaneously, the model sees three policy documents followed by your question. There is no semantic merging. Conflicts between skills are resolved the same way conflicts in any long context are resolved: by proximity to the end (more recent wins) and by specificity (more specific wins). Designing skills with composability in mind means designing for predictable conflict resolution.

## Skills in the Managed Agents Model

The April 2026 Managed Agents launch introduced a new way to think about where skills fit in the execution stack. Managed Agents separates agent execution into three layers: the **session** (durable event log), the **harness** (stateless execution loop), and the **sandbox** (isolated code execution environment).

Skills live in the harness layer. They're assembled into context at harness startup and injected into every model call within that session. In a managed deployment, this means skills can be version-pinned to a session: the harness records which skill versions were active, making it possible to reproduce behavior from any past session.

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class HarnessConfig:
    """Harness configuration for a managed agent session."""
    model: str
    skills: tuple[str, ...]           # skill slugs, in load order
    skill_versions: dict[str, str]    # slug → git SHA or semver
    max_skill_tokens: int = 8_000     # token budget for injected skills
    rules_paths: tuple[str, ...] = ()  # always-active rules (separate from skills)


def assemble_system_context(config: HarnessConfig) -> str:
    """Build the system context from rules and skills.
    
    Called once per session; result is logged in the session event stream
    for reproducibility.
    """
    parts = []

    # Rules load unconditionally
    for rules_path in config.rules_paths:
        parts.append(load_file(rules_path))

    # Skills load in order, within token budget
    token_count = 0
    for slug in config.skills:
        content = load_skill(slug, config.skill_versions.get(slug))
        skill_tokens = count_tokens(content)
        if token_count + skill_tokens > config.max_skill_tokens:
            log.warning(f"Skill {slug} skipped: would exceed token budget")
            break
        parts.append(content)
        token_count += skill_tokens

    return "\n\n---\n\n".join(parts)
```

The significant implication: in a managed agent deployment, the skill set is part of the session definition. Changing which skills are active is a harness configuration change, not just a prompt change — it's tracked, versioned, and reproducible. This makes skills a first-class part of the agent's behavioral specification rather than an informal "I usually invoke these."

## The Quality Gap

Most skills shared online are weak. They look like:

```markdown
# Code Review
Review this code for quality and security issues. Be thorough.
```

This is marginally better than asking Claude to review code directly. A strong skill differs in three specific ways:

**1. Encodes non-obvious judgment, not just intent**

Weak: "Check for security issues."
Strong: "Flag `eval()` or `exec()` with string inputs as CRITICAL. Flag SQL string concatenation as CRITICAL. Flag missing CSRF tokens on state-changing endpoints as HIGH. Flag missing rate limiting on auth endpoints as HIGH. Ignore XSS risks flagged by ruff's S-series rules — those are handled automatically."

The model knows what security issues are. What it doesn't know is your severity calibration, your stack's specific failure modes, and which automated tools already handle which concerns. Skills should encode that, not re-state what the model already knows.

**2. Specifies output structure exactly**

```markdown
## Output Format

For each finding:

**[SEVERITY]** `file.py:line` — Description

Severity: CRITICAL (ship-blocking) | HIGH (fix before merge) | MEDIUM (this sprint) | LOW (eventually)

End with one-line verdict: APPROVE | REQUEST_CHANGES | BLOCK
```

Without an explicit format, the model produces useful but inconsistent output. With a format, you can parse, filter, and route the output programmatically — the skill's output becomes an interface, not just text.

**3. Negative constraints are as important as positive ones**

```markdown
## Do NOT do these
- Don't suggest renaming variables unless the current name is genuinely misleading
- Don't add docstrings to code you didn't modify
- Don't refactor beyond what was requested
- Don't flag style issues that ruff/black enforce automatically
- Don't add error handling for cases that can't occur given the call site
```

These constraints are often worth more than the positive instructions. They prevent the model from doing unhelpful things it would otherwise do by default — scope creep, over-engineering, re-doing work that automated tools already cover.

## Three Production Skills Worth Understanding

### 1. Harness-Enforced Git Workflow

The `git-workflow` skill encodes Conventional Commits and branch naming, but more importantly it encodes a *verification sequence* that must complete before any commit happens.

**Pre-commit sequence (mandatory — refuse if any step fails):**
1. `ruff check .` — fix all errors
2. `mypy src/` — fix all type errors
3. `pytest -x` — all tests pass
4. `git diff --cached` — confirm no `.env` files or secrets staged

Only after all four pass: construct and execute the commit command.

**The behavior this produces**: Claude will refuse to write a git commit until it has confirmed the sequence completed successfully. This is stronger than a reminder — it's a conditional on the action. In sessions where you're moving fast and want to skip checks, the skill creates friction precisely where friction belongs.

In the Managed Agents model, this skill participates in the Builder agent's harness configuration. The Reviewer agent has a different skill set — one that includes the diff-aware code review skill but explicitly excludes the commit skill (the Reviewer never commits). This role separation, enforced through skill configuration rather than prompts, makes the architectural constraint explicit and auditable.

### 2. ML Experiment Documentation

ML reproducibility fails not at the seed-setting step but at the decision-recording step. Three months later, `dropout=0.3` in the config with no context about whether it was principled or arbitrary.

```markdown
# Experiment Documentation Skill

After every training run, generate an experiment card:

**ID**: {timestamp}_{model_name}_{key_variant}
**Hypothesis**: What specifically are you testing? (one sentence, falsifiable)
**Changed from baseline**: Only what changed, not the full config
**Expected outcome**: Concrete prediction with direction and magnitude
**Actual result**: Primary metric values with units
**Interpretation**: Did the result match the hypothesis? What does the gap mean?
**Next experiment**: What does this result suggest trying?

## Anti-patterns
- "Ran experiment with config X" is not a hypothesis
- Don't record wall-clock time unless it's the variable being studied
- Don't copy-paste the full config — only the delta matters
```

The card format enforces hypothesis-driven experimentation structurally. You cannot fill in "Changed from baseline" and "Hypothesis" as empty strings and still have a useful card. The skill doesn't remind you to write a hypothesis; it makes the hypothesis a required field.

### 3. Diff-Aware Code Review

Standard code review skills work at the file level. This one works at the *change* level:

```markdown
# Code Review — Diff-Aware

## Determine context first:
1. Is this a bug fix, new feature, or refactor? (from commit message)
2. What is the scope? (from files changed count and directories)
3. What was the stated goal? (from PR description or issue)

## Review scope must match change scope:
- Bug fix: Focus on the fix and potential regressions. Don't critique unrelated code.
- New feature: Check correctness, edge cases, integration points.
- Refactor: Verify behavior preservation. Check that tests cover changed paths.

## ML/data code specifically:
- Data leakage: Is train/val split happening before or after featurization?
- Gradient flow: Any `.detach()` calls that shouldn't be there?
- Seed: Is random seed set and logged at experiment start?
- Paths: Hardcoded dataset paths should be config values
- Magic numbers: Any non-config constants in training loops?

## Severity
CRITICAL: Data leakage, incorrect gradient flow, loss function bugs
HIGH: Missing seed, hardcoded paths, untested edge cases
MEDIUM: Type annotations, naming issues
LOW: Style
```

The ML-specific section encodes failure modes from real mistakes: data leakage from split-after-featurization, gradient bugs from accidentally detached tensors, loss scale issues from incorrect reduction. These aren't textbook security patterns — they're a knowledge transfer from past debugging sessions into future reviews.

## Composition and Token Budget

Every active skill consumes context. Stack ten 2,000-word skills and you've used 20K tokens before your actual conversation starts. Three practices that make skill composition work:

**Skills should be narrow and purpose-specific.** Instead of one "Python development" skill, use:
- `python-patterns.md` — idioms and anti-patterns (~400 tokens)
- `python-testing.md` — pytest patterns specifically (~300 tokens)
- `python-security.md` — security rules (~200 tokens)

Each is invoked when relevant, not loaded by default. The total token cost is lower, and each skill can be updated or replaced independently.

**Hooks can manage loading automatically.** PostToolUse hooks can check the current task type (Python file edited → load python-patterns, git operations → load git-workflow) and load skills automatically. This moves skill selection from a manual discipline to an enforced one — the harness handles it rather than relying on the user to remember which skills to activate.

**The right mental model is specialization, not coverage.** Skills are not a knowledge base you're trying to make comprehensive. They're specialists called in for specific work. You don't want your database expert in the room for every frontend review. You want them when it's database work, with specific focus on the decisions they know how to evaluate.

## The Skill Development Loop

Good skills come from frustrating interactions. The development loop:

1. **Identify the miss**: Claude produced unhelpful output or missed something obvious. Write down exactly what it missed.

2. **Encode as a rule**: "When reviewing ML code, explicitly check for data leakage at the train/val split point" — specific, not general.

3. **Add the negative**: "Don't suggest architectural changes during a bug fix review — that's scope creep." The negative constraint prevents the model from substituting a different unhelpful behavior.

4. **Test on a real case**: Invoke the updated skill on the actual scenario that produced the frustrating interaction. Does it catch the issue?

5. **Generalize carefully**: Extract the principle, not the specific instance. "Flag missing random seed at experiment start" generalizes better than "check line 47 of train.py."

The `/skill-improver` and `/skill-quality-reviewer` commands in the OpenClaw configuration automate parts of this loop — running the skill against a test case and having a separate session critique its output against the stated goals. In a managed agent setup, this loop becomes more systematic: the skill improvement agent can replay past sessions where the skill underperformed and test whether the updated skill would have caught the issue.

## Where Skills Sit in the Larger Picture

Skills are, in Anthropic's framing, a primitive form of harness engineering: constrained, policy-governed AI behavior where the model operates within explicit boundaries defined outside the model itself.

The Claude Code skills architecture makes this accessible without infrastructure. You don't need a policy server or a governance layer. You need well-written Markdown files and the discipline to invoke them consistently.

The ceiling on this approach is how well you can articulate your own expertise as structured rules. That turns out to be a useful exercise independent of the skill itself — writing a skill forces implicit knowledge explicit, which is half the value even if the skill never gets invoked. The act of encoding "what I know that the model doesn't" clarifies both what you actually know and where your knowledge is vague.

With the Managed Agents infrastructure, skills get a new property: they're part of the session definition, tracked alongside model version and harness configuration. The evolution from "a Markdown file I sometimes invoke" to "a versioned component of a reproducible agent session" doesn't require rewriting any skills. It just requires taking seriously what skills already are: behavioral specifications for a specific type of work.
