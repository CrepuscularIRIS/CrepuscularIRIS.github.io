The skills system in Claude Code (OpenClaw in my configuration) is the part of the platform that most users underuse. Most people install a skill, trigger it once, and move on. But skills are the mechanism that turns a general-purpose AI assistant into a specialized, disciplined engineering partner — one that knows your project's conventions, enforces your workflow constraints, and handles specific task types with reproducible quality.

This post covers the architecture of the skills system, how to write skills worth using, and three production patterns that have meaningfully changed how I work.

## What a Skill Actually Is

A skill is a Markdown file that gets injected into Claude's context when invoked. That's it. The magic is in the structure.

```
~/.claude/skills/
├── git-workflow.md       # Activated by /git-workflow
├── code-review.md        # Activated by /code-review
├── python-patterns.md    # Activated by /python-patterns
└── ml-paper-writing.md   # Activated by /ml-paper-writing
```

When you call `/code-review`, Claude Code reads the skill file and prepends it to the conversation context before responding. The skill functions as a dense system-prompt injection: it can contain rules, checklists, examples, anti-patterns, domain knowledge, and structured output templates.

This sounds simple, but the implications are significant. Skills are:

- **Composable**: Multiple skills can be active simultaneously
- **Versionable**: They're just text files in your dotfiles repo
- **Shareable**: The same skill works for any project, any language
- **Overridable**: Project-level skills in `.claude/skills/` take precedence over global ones

The skill mechanism is essentially a *knowledge injection* system. You're not modifying the model. You're changing what the model knows about what you want.

## The Quality Gap Between Weak and Strong Skills

Most skills people share online are weak. They look like this:

```markdown
# Code Review
Review this code for quality and security issues. Be thorough.
```

This is barely better than just asking Claude to review code directly. A strong skill does several specific things differently:

**1. Encodes non-obvious judgment, not just intent**

Weak: "Check for security issues"  
Strong: "Flag any use of `eval()` or `exec()` with string inputs as CRITICAL. Flag SQL string concatenation as CRITICAL. Flag missing CSRF tokens on state-changing endpoints as HIGH. Flag missing rate limiting on auth endpoints as HIGH."

The model knows what security issues are. What it doesn't know is *your severity calibration* and *your specific stack's common failure modes*. Skills should encode that.

**2. Specifies output structure precisely**

```markdown
## Output Format

For each finding, output:

**[SEVERITY]** `file:line` — Description

Severity levels: CRITICAL (ship-blocking) | HIGH (fix before merge) | MEDIUM (fix this sprint) | LOW (when convenient)

End with a one-line verdict: APPROVE | REQUEST_CHANGES | BLOCK
```

Without an explicit format, Claude produces useful but inconsistent output. With a format, you can parse, filter, and act on the output programmatically.

**3. Contains anti-patterns to avoid**

```markdown
## Do NOT do these
- Don't suggest renaming variables unless the current name is genuinely misleading
- Don't add docstrings to code you didn't modify
- Don't refactor beyond what was requested
- Don't flag style issues that ruff/black already enforce
```

These negative constraints are often more important than the positive instructions. They prevent the model from doing unhelpful things it would otherwise do by default.

## Three Production Skills Worth Understanding

### 1. The Harness-Enforced Git Workflow

My `git-workflow` skill encodes Conventional Commits, branch naming, and PR structure — but more importantly, it encodes the *workflow sequence* that must be followed before any commit happens.

The skill defines four sections: Pre-Commit Sequence, Commit Message Format, Types/Scopes, and Never Do.

**Pre-Commit Sequence (mandatory):**
1. Run `ruff check .` — fix all errors before proceeding
2. Run `mypy src/` — fix all type errors before proceeding
3. Run `pytest -x` — all tests must pass
4. Verify no `.env` files or secrets are staged

Only after these pass: construct the commit message.

**Commit Message Format:**

```text
type(scope): short summary

optional body — explain WHY, not WHAT
```

Types: `feat` | `fix` | `refactor` | `docs` | `test` | `chore` | `perf`  
Scopes: `data` | `model` | `config` | `trainer` | `utils` | `workflow`

**Never Do:** use `--no-verify`, commit to main, include unrelated changes in a commit.

The "harness" aspect is that Claude will refuse to write a commit command until it has verified the sequence was followed. This sounds minor, but it changes commit discipline entirely — the skill functions as an enforced checklist, not just a reminder.

### 2. The ML Experiment Documentation Skill

In ML research, the thing that kills reproducibility is not forgetting to set seeds (you'll remember eventually). It's forgetting to record *why* you made a specific decision. Three months later, the config says `dropout=0.3` and you have no idea if that was principled or arbitrary.

My `experiment-doc` skill fires after every training run and generates a structured experiment card:

```markdown
# Experiment Documentation Skill

After every training run, generate:

## Experiment Card Template

**ID**: {timestamp}_{model_name}_{key_hyperparam}
**Hypothesis**: What specifically are you testing? (one sentence)
**Changed from baseline**: List *only* what changed
**Expected outcome**: What do you predict, and why?
**Actual result**: Metric values (val_loss, accuracy, etc.)
**Interpretation**: Did it match the hypothesis? Why or why not?
**Next experiment**: What does this result suggest to try next?

## Anti-patterns to Avoid
- Don't write "ran experiment with config X" — that's not a hypothesis
- Don't record wall-clock time unless it's the variable being studied  
- Don't copy-paste the full config — record only the delta
```

The card format enforces hypothesis-driven experimentation. You can't fill in "Hypothesis" and "Changed from baseline" without having thought about what you're actually testing. The skill makes implicit reasoning explicit.

### 3. The Context-Aware Code Review

Standard code review skills work at the file level. This one works at the *change* level and is context-aware:

```markdown
# Code Review — Diff-Aware

## Before reviewing, determine:
1. Is this a bug fix, new feature, or refactor? (check git commit message)
2. What is the scope of the PR? (check files changed)
3. What was the stated goal? (check PR description or issue)

## Review scope should match change scope
- Bug fix: Focus on the fix itself and potential regressions. Don't critique unrelated code.
- New feature: Check for correctness, edge cases, and integration points.
- Refactor: Verify behavior preservation. Check that tests cover the changed code paths.

## For ML/data code specifically:
- Flag any hardcoded dataset paths (should be in config)
- Flag magic numbers in training loops (should be named constants or config)
- Flag missing type annotations on public functions
- Flag any mutable default arguments in dataclass or function definitions
- Verify random seed is set and logged at experiment start

## Severity for this codebase
CRITICAL: Data leakage, incorrect gradient flow, loss function bugs
HIGH: Missing seed, hardcoded paths, untested edge cases
MEDIUM: Type annotations, naming conventions
LOW: Style, readability
```

The ML-specific section encodes failure modes I've actually hit: data leakage from train/val split done wrong, gradient flow bugs from accidentally detached tensors, incorrect loss reduction causing scale issues. These aren't generic security best practices — they're a knowledge transfer from past mistakes into future reviews.

## The Skill Development Loop

Writing a good skill takes iteration. The pattern that works:

1. **Start with a frustrating interaction**: Claude did something unhelpful or missed something obvious. Write down what it missed.

2. **Encode the miss as a rule**: "Next time, when reviewing ML code, explicitly check for data leakage in train/val splits."

3. **Add the negative**: "Don't suggest architectural changes in a bug fix review — that's scope creep."

4. **Test on a real case**: Invoke the skill on the actual scenario that inspired it. Does it catch the issue?

5. **Generalize carefully**: Extract the principle, not the specific instance. "Flag missing random seed at experiment start" generalizes better than "check line 47 of train.py for seed setting."

The `/skill-improver` and `/skill-quality-reviewer` commands in my config automate parts of this loop — running the skill against a test case and critiquing its own output against the stated goals.

## Skill Composition and Load Management

One limitation worth naming: every active skill consumes context. Stack ten 2000-word skills and you've eaten 20K tokens before your actual conversation starts. This matters in long sessions.

The discipline I've adopted: skills should be **narrow and purpose-specific**, not comprehensive. Instead of one giant "Python development" skill, I have:

- `python-patterns.md` — idioms, anti-patterns (400 words)
- `python-testing.md` — pytest patterns specifically (300 words)  
- `python-security.md` — security-specific rules (200 words)

Each is invoked when relevant, not loaded by default. The harness enforces this through hooks that check which task type is active before loading skills automatically.

The right mental model for skills is not "knowledge base" — it's "expert colleague who specializes in this exact thing." You don't ask your database expert to sit in on every frontend review. You call them when it's database work.

## Where This Leads

The skills system is essentially a primitive form of the larger pattern DeepMind calls harness engineering: constrained, policy-governed AI behavior, where the model operates within explicit boundaries defined outside the model itself.

The Claude Code skills architecture makes this accessible without infrastructure. You don't need a policy server or a governance layer. You need well-written Markdown files and the discipline to invoke them consistently.

The ceiling on this approach is how well you can articulate your own expertise as structured rules. That turns out to be a useful exercise in itself — writing a skill forces you to make implicit knowledge explicit, which is half the value even if the skill is never used.
