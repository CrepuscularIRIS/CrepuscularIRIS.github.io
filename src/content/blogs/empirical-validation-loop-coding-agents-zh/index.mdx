第一代自主编程智能体遵循的模式在理论上看起来非常合理：阅读 Bug 报告、分析代码库、生成补丁（Patch）、提交。智能体对补丁的信心是其内部推理能力的函数——如果其思维链（Chain-of-Thought）推导出修复方案是正确的，它就会提交。在这个过程中没有外部验证。智能体对正确性的主观认知与实际的正确性之间，仅仅通过模型自身的推理质量这一环节进行弱耦合。

这种模式在早期基准测试（Benchmark）中表现得惊人地好，因为那些 Bug 大多是单文件修改、有着明确的错误消息和显而易见的根本原因。但当 Bug 变得复杂时，这种方法就失效了。在 SWE-bench Verified 上，第一代模式的解决率徘徊在 25-30% 左右。智能体虽然能识别出正确的文件和函数，但生成的补丁往往会引入回归（Regression）、遗漏边缘情况，或者只是修复了表象而留下了病灶。

实证验证循环（Empirical Validation Loop, EVL）是打破这一瓶颈的关键范式。其核心思路非常朴素：在修复 Bug 之前，智能体必须先复现（Reproduce）它。复现过程——即一个能准确捕捉 Bug 行为的、失败的测试用例——成了衡量成功的唯一标准。智能体不再基于对正确性的"自我感觉"提交代码，而是当且仅当复现测试通过时才提交。智能体的推理用于生成候选方案，但最终的裁判是实证结果。

2025 年底，首批采用 EVL 模式的智能体将 SWE-bench Verified 的解决率推升至 40% 以上；而到了 2026 年 4 月，目前最顶尖的智能体解决率已突破 50%。

## 核心流程

EVL 包含五个必须严格按顺序执行的步骤：

**第一步：调研。** 智能体阅读 Bug 报告，识别相关文件，并追踪产生 Bug 的代码路径。这与第一代方法相同。智能体在此阶段构建关于"发生了什么"以及"为什么"的心智模型。

**第二步：复现。** 智能体编写一个测试（或脚本），使其以 Bug 报告中描述的方式失败。这是最关键的一步。复现必须具备针对性：它必须因报告的 Bug 而失败，并在 Bug 修复后通过。如果复现脚本因为其他无关原因失败，那么它不仅毫无用处，甚至会误导智能体验证错误的修复方案。

```python
def test_csv_export_unicode_handling():
    """Reproduction for issue #4821: CSV export crashes on non-ASCII names."""
    dataset = Dataset([
        {"name": "Muller", "score": 95},
        {"name": "Muller", "score": 87},  # umlaut
        {"name": "Takahashi", "score": 91},  # CJK
    ])

    # This should not raise UnicodeEncodeError
    output = dataset.export_csv(encoding="utf-8")

    assert "Muller" in output
    assert "Takahashi" in output
```

**第三步：验证失败。** 智能体运行复现测试并确认其确实失败。如果测试通过了，说明复现是失败的——要么 Bug 已经修复，要么测试没有捕捉到真实的故障点。此时智能体必须回到第二步。

**第四步：定向修复。** 在确认复现成功后，智能体开始生成修复方案。修复的目标被限定为：能够让该复现测试通过的最小改动。智能体可能会生成多个候选补丁，并逐一进行测试。

**第五步：验证。** 智能体运行完整的测试套件（不仅仅是复现测试），以确保修复方案没有引入回归。如果复现测试通过了但其他测试失败了，该修复方案会被拒绝，智能体回到第四步重新尝试。

```
调研 → 复现 → 验证失败 → 修复 → 验证
  ↑                               |
  └────────── 遇到回归时重试 ──────┘
```

## 为什么复现能改变一切

复现测试是驱动整个循环的动力。没有它，智能体对于修复是否正确缺乏客观标准；有了它，智能体就有了一个具体的、可执行的谓词来区分"已修复"和"未修复"。

这从根本上改变了智能体的优化目标。第一代智能体优化的是**看似合理的补丁**——即在模型的推理看来是正确的改动。而 EVL 智能体优化的是**能通过特定测试的补丁**。这种区别在处理疑难 Bug 时至关重要：对语言模型来说"看起来正确"的修复，并不总是能解决实际问题的修复。

此外，复现测试约束了智能体的搜索空间。像"CSV 导出在非 ASCII 字符下崩溃"这样的报告是模糊的——它可能是 Locale 问题、编码问题、缓冲区溢出或第三方库的 Bug。而复现测试将 Bug 的特定行为编码化：在导出路径中，当输入包含 ASCII 范围外的字符时抛出 `UnicodeEncodeError`。修复方案必须解决这个特定的失败，而不是空泛的"Unicode 处理"概念。

这种约束是高效的。一个试图"修复 Unicode 处理"的智能体可能会重构整个导出模块；而一个试图让 `test_csv_export_unicode_handling` 通过的智能体，会在导出函数的 `open()` 调用中添加一个显式的 `encoding` 参数。后者的修复更小、更安全，也更可能是正确的。

## 复现难题

编写高质量的复现测试是 EVL 中最难的一步。智能体必须对 Bug 有足够深入的理解，才能写出精准的测试——既不能太宽泛（导致错误的修复也能通过），也不能太狭隘（无法捕捉 Bug 的本质行为）。

在实践中常见的失败模式包括：

**同义反复式复现：** 智能体写的测试断言的是代码的"现状"而非"预期行为"。测试失败是因为智能体写错了断言，而不是捕捉到了 Bug。当智能体"修复"Bug 时，它实际上是在强迫代码适配那个错误的断言。

**对报告过拟合：** 智能体完全照搬 Bug 报告中的输入编写测试。这虽然能解决特定案例，但可能无法捕捉底层模式。一个只能处理报告中特定输入、但在同类其他输入下依然失败的补丁，虽然能通过复现测试，却并没有真正修复 Bug。

**环境依赖：** 复现测试依赖外部状态——特定的数据库 Schema、报告者机器上的文件，或者与测试环境不一致的库版本。测试因为错误的原因失败，导致针对错误根源的修复方案被错误地验证。

最优秀的 EVL 实现通过生成多个复现变体来解决这些问题：一个完全匹配报告，一个将输入模式泛化，一个测试边界条件。修复方案必须通过所有变体。

## 实证验证 vs. 基于推理的验证

EVL 代表了编程智能体确立正确性方式的哲学转变。基于推理的方法假设，只要模型足够强大，就能通过分析来判断补丁是否正确。而实证方法则假设，运行代码永远比推理代码更可靠。

两种方法都有局限性。基于推理的验证可以处理编写复现代价极高的情况——如性能 Bug、竞态条件或 UI 问题。而实证验证则能解决仅靠推理会出错的情况——如复杂的内部状态交互、平台特定的行为或第三方库的边缘情况。

实践结论是：实证验证应作为默认选项，而基于推理的验证应作为复现不可行时的退路。这与经验丰富的人类开发者的工作方式高度一致：先写一个捕获 Bug 的测试，不断修改代码直到测试通过，最后通过人工评审确保改动没有产生意料之外的影响。

```python
class EVLAgent:
    """Simplified EVL agent loop."""

    async def fix_issue(self, issue: Issue, repo: Repository) -> Patch | None:
        # Step 1: Research
        context = await self.analyze_issue(issue, repo)

        # Step 2: Reproduce
        reproduction = await self.write_reproduction(issue, context)
        if reproduction is None:
            return await self.fallback_reasoning_fix(issue, context)

        # Step 3: Verify failure
        result = await repo.run_test(reproduction)
        if result.passed:
            return None  # Bug not reproducible

        # Step 4-5: Fix and validate (up to 3 attempts)
        for attempt in range(3):
            patch = await self.generate_fix(issue, context, reproduction, attempt)
            repo.apply_patch(patch)

            repro_result = await repo.run_test(reproduction)
            if not repro_result.passed:
                repo.revert_patch(patch)
                continue

            suite_result = await repo.run_test_suite()
            if suite_result.passed:
                return patch

            repo.revert_patch(patch)

        return None
```

## SWE-bench 上的变化

SWE-bench Verified 追踪了来自 12 个流行 Python 仓库的 500 个真实 GitHub Issue。该基准测试要求智能体在干净的环境中生成能够通过相关测试的补丁。这使得它与 EVL 模式天然契合——基准测试本身的成功标准就是实证性的。

解决率的变化轨迹说明了一切：

**EVL 普及前（2025 年中）：** 顶尖智能体在 25-30% 左右。智能体能生成看似合理的补丁，但在边缘情况、回归测试和跨文件修改上屡屡受挫。失败的补丁通常看起来是对的，但破坏了智能体未测试的其他功能。

**EVL 采用初期（2025 年底）：** 首批实现"复现优先"工作流的智能体达到了 35-40%。提升几乎完全来自于对回归的拦截——智能体并没有生成更好的补丁，它们只是在提交前拒绝了那些坏补丁。

**EVL 成熟期（2026 年 4 月）：** 最优秀的智能体达到了 50-55%。额外的提升来自迭代修复——智能体生成方案、运行复现、分析失败原因、再生成精炼后的方案。多轮"修复-测试-分析"循环解决了单次尝试绝对无法搞定的复杂 Bug。

剩余 45-50% 未解决的问题主要集中在 EVL 杠杆作用较小的类别：需要理解未声明需求的 Issue（报告未完整说明预期行为）、跨仓库的 Issue，以及复现过程依赖外部服务的 Issue。

## 更广泛的启示

EVL 是一个通用原则的具体实例：通过实证验证输出的自主系统，性能始终优于仅依赖自我评估的系统。这一原则不仅限于编程。

在科研自动化领域，通过源文档验证论点的智能体比仅靠自身知识评估准确性的智能体表现更好。在数据分析领域，针对已知测试案例验证转换逻辑的智能体比仅靠自检结果的智能体产出更可靠。

共同点在于：当系统的推理和输出是由同一个模型生成时，自我评估是不可靠的。模型在推理上的盲点会直接变成评估上的盲点。而外部验证——运行测试、核对来源、与事实对比——打破了这种相关性带来的恶性循环。

对于编程智能体而言，EVL 模式意味着测试套件不再仅仅是最终输出的质量关卡。它是智能体推理循环的活跃参与者：测试结果指导下一次修复尝试，而复现测试则是将智能体对 Bug 的理解以可执行的形式编码化。测试，就是智能体与代码库之间达成的契约。
