OpenAI quietly changed how they charge for GPT-5.4. Instead of per-token pricing, Pro users now get "Reasoning Minutes" — approximately 40 minutes of reasoning time per 5-hour window. The marketing framing is about simplicity. The technical framing reveals something more interesting: OpenAI is admitting that the dominant cost of frontier inference is no longer token generation but *thinking*.

This isn't a billing change. It's a disclosure about architecture.

## What Reasoning Minutes Actually Measure

Token-based pricing assumes that cost scales linearly with output length. For autoregressive generation — the model produces one token at a time, each requiring one forward pass — this is approximately true. The cost of generating 1,000 tokens is roughly 10x the cost of generating 100 tokens.

Reasoning models break this assumption. GPT-5.4's o-series reasoning mode performs internal search before committing to an answer. The model doesn't just generate the next token — it explores multiple solution paths, evaluates their likelihood of being correct, and selects the best one. This internal search is invisible in the output but dominates the compute budget.

A simple question ("What's the capital of France?") requires minimal internal search. A complex mathematical proof requires extensive search — potentially thousands of internal evaluation steps before producing a single output token. Under token-based pricing, both queries generate similar-length outputs and cost similar amounts. Under reasoning-minute pricing, the proof costs 100x more because the *thinking* takes 100x longer.

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class InferenceProfile:
    """Cost profile for different query types under reasoning-time pricing."""
    query_type: str
    output_tokens: int
    reasoning_steps: int     # internal search evaluations
    reasoning_time_ms: int   # wall-clock time for thinking
    generation_time_ms: int  # wall-clock time for token output
    
    @property
    def thinking_ratio(self) -> float:
        """Fraction of total time spent thinking vs. generating."""
        total = self.reasoning_time_ms + self.generation_time_ms
        return self.reasoning_time_ms / total if total > 0 else 0.0

# Approximate profiles for GPT-5.4 reasoning mode
PROFILES = [
    InferenceProfile("factual_lookup",     50,    10,    200,    500),
    InferenceProfile("code_generation",   200,   500,   5000,   2000),
    InferenceProfile("math_proof",        150,  5000,  60000,   1500),
    InferenceProfile("multi_step_agent",  300, 10000, 120000,   3000),
]

for p in PROFILES:
    print(
        f"{p.query_type:25s} "

# factual_lookup            tokens=  50  think_ratio=28.6%  reasoning_steps=    10

```

For reasoning-heavy tasks, the model spends 97%+ of compute on *thinking* and less than 3% on *generating output*. Token-based pricing charges for the 3%. Reasoning-minute pricing charges for the 97%. This is the honest pricing model.

## The Search Tree Inside the Model

What does "reasoning" look like computationally? The internal architecture of o-series models is not publicly documented. But the design space is constrained by what we know from academic predecessors and inference-time scaling research. One plausible computational pattern — not confirmed, but consistent with observed behavior — resembles **Monte Carlo Tree Search (MCTS) over the token space**.

The model maintains a search tree where each node is a partial solution and each edge is a possible next reasoning step. At each node, the model:
1. **Selects** a promising branch (using a learned value function)
2. **Expands** it by generating the next reasoning step
3. **Evaluates** the expanded node (is this path leading to a correct answer?)
4. **Backpropagates** the evaluation to update parent nodes

This is the same algorithmic skeleton as AlphaGo's game-playing search, adapted to natural language reasoning. The "reasoning time" is the time spent traversing this tree — expanding nodes, evaluating positions, backtracking from dead ends.

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class ReasoningNode:
    """A node in the internal reasoning search tree."""
    state: str                      # partial reasoning chain
    value: float                    # estimated probability of reaching correct answer
    visit_count: int = 0
    children: list["ReasoningNode"] | None = None
    parent: Optional["ReasoningNode"] = None
    
    @property
    def ucb_score(self) -> float:
        """Upper Confidence Bound for tree search selection."""
        import math
        if self.visit_count == 0:
            return float("inf")
        parent_visits = self.parent.visit_count if self.parent else 1
        exploitation = self.value
        exploration = math.sqrt(math.log(parent_visits) / self.visit_count)
        c = 1.41  # exploration constant
        return exploitation + c * exploration


def reasoning_search(
    query: str,
    model_fn,        # generates next reasoning step
    value_fn,        # evaluates partial solution quality
    max_iterations: int = 1000,
    confidence_threshold: float = 0.95,
) -> str:
    """Simplified MCTS-style reasoning loop.
    
    The real implementation runs on specialized inference hardware
    with batched tree traversal. This captures the algorithmic structure.
    """
    root = ReasoningNode(state=f"Question: {query}\nReasoning:", value=0.0)
    
    for _ in range(max_iterations):
        # SELECT: traverse tree using UCB scores
        node = root
        while node.children:
            node = max(node.children, key=lambda n: n.ucb_score)
        
        # EXPAND: generate possible next reasoning steps
        candidates = model_fn(node.state, n_candidates=3)
        node.children = [
            ReasoningNode(
                state=f"{node.state}\n{step}",
                value=0.0,
                parent=node,
            )
            for step in candidates
        ]
        
        # EVALUATE: assess each expanded node
        for child in node.children:
            child.value = value_fn(child.state)
            child.visit_count = 1
        
        # BACKPROPAGATE: update ancestor values
        best_child = max(node.children, key=lambda n: n.value)
        current = best_child
        while current.parent:
            current.parent.visit_count += 1
            current.parent.value = max(
                c.value for c in (current.parent.children or [current])
            )
            current = current.parent
        
        # TERMINATE: if high-confidence solution found
        if best_child.value > confidence_threshold:
            return best_child.state
    
    # Return best path found within budget
    best = root
    while best.children:
        best = max(best.children, key=lambda n: n.value)
    return best.state
```

The `max_iterations` parameter is the reasoning budget. More iterations → deeper search → better answers → more compute. Reasoning minutes directly cap this parameter. A 40-minute budget allows roughly 100K-1M search iterations depending on model size and hardware — enough for complex proofs but not enough for open-ended research problems.

## Why This Changes the Application Landscape

Token pricing created a specific optimization pressure: minimize output length. Applications were designed to request concise responses, use structured output formats, and avoid verbose reasoning chains. The implicit assumption: shorter = cheaper.

Reasoning-minute pricing inverts this. The cost is dominated by thinking, not output. A model that thinks for 60 seconds and produces 50 tokens of answer costs the same as one that thinks for 60 seconds and produces 500 tokens. The optimization pressure shifts from "minimize output" to "minimize reasoning complexity."

This changes which applications are economically viable:

| Application | Token Pricing | Reasoning Pricing | Winner |
|---|---|---|---|
| Chatbot (simple Q&A) | Cheap | Cheap | Tie |
| Code generation | Moderate | Moderate | Tie |
| Mathematical reasoning | Moderate (reasoning tokens billed) | Expensive (deep search) | Depends on depth |
| Long document generation | Expensive (many tokens) | Cheap (shallow reasoning) | Reasoning pricing |
| Agent workflows | Moderate | Very expensive (deep planning) | Token pricing |
| Summarization | Moderate | Cheap | Reasoning pricing |

The most affected category is **agent workflows**. An autonomous agent that plans 20 steps ahead, evaluates contingencies, and reasons about failure modes consumes enormous reasoning budgets even if its output is a single shell command. Under token pricing, this was hidden — the agent produces short outputs, so it looks cheap. Under reasoning pricing, the true cost is exposed.

## The Compute Cost of Certainty

There's a deeper principle here: **certainty is expensive**. A model that's 70% confident in its answer requires minimal search — pick the most likely path and commit. A model that needs to be 99% confident requires extensive search — explore many paths, verify consistency, rule out failure modes.

The relationship between confidence and compute follows an inverse-error pattern — each doubling of search iterations roughly halves the remaining error rate. This means compute scales as `1 / (1 - confidence)`, which grows slowly at first but explodes near certainty:

```python
import math

def estimated_reasoning_cost(
    base_difficulty: float,    # 0-1 scale, task complexity
    target_confidence: float,  # desired confidence in answer
    base_compute: float = 1.0, # compute for 50% confidence
) -> float:
    """Estimate reasoning compute as a function of desired confidence.
    
    Model: each doubling of compute (search iterations) approximately
    halves the remaining error rate. This produces inverse-error scaling:
    cost ~ 1 / (1 - confidence), not logarithmic in confidence.
    
    The result: going from 90% to 99% costs ~10x more than 50% to 90%.
    """
    if target_confidence >= 1.0:
        raise ValueError("Cannot achieve 100% confidence with finite compute")
    if target_confidence <= 0.5:
        return base_compute * base_difficulty
    
    # Each "doubling" of compute halves error rate
    # error_rate = 1 - confidence
    # doublings_needed = log2(0.5 / error_rate) = log2(0.5 / (1 - confidence))
    error_rate = 1.0 - target_confidence
    doublings = math.log2(0.5 / error_rate)
    
    return base_compute * base_difficulty * (2.0 ** doublings)

# Examples
for conf in [0.50, 0.80, 0.90, 0.95, 0.99, 0.999]:
    cost = estimated_reasoning_cost(0.5, conf)
    print(f"  confidence={conf:.1%}  relative_cost={cost:.1f}x")
#   confidence=50.0%  relative_cost=0.5x
#   confidence=80.0%  relative_cost=1.3x
#   confidence=90.0%  relative_cost=2.5x
#   confidence=95.0%  relative_cost=5.0x
#   confidence=99.0%  relative_cost=25.0x
#   confidence=99.9%  relative_cost=250.0x
```

Going from 90% to 99% confidence costs 10x more compute. Going from 99% to 99.9% costs another 10x. This is why reasoning-minute pricing makes certain applications prohibitively expensive: tasks that require high certainty (formal proofs, safety-critical decisions, financial analysis) consume disproportionate reasoning budgets.

The practical implication for system design: build confidence-aware routing. Route easy queries to fast models with minimal reasoning. Route hard queries to reasoning models. Route certainty-critical queries to reasoning models with extended budgets. The cost difference between these tiers can be 100x or more.

## What Anthropic and Google Are Doing Differently

Anthropic hasn't adopted reasoning-minute pricing for Claude. Their extended thinking feature exposes the reasoning chain as visible output — the thinking tokens are priced the same as output tokens, just at a higher rate. This has a different economic implication: users can *see* the reasoning and decide whether to truncate it. The feedback loop is tighter.

Google's approach with Gemini 3.1 is hybrid: a reasoning budget is allocated per query based on the model's estimate of task difficulty. Simple queries get minimal budgets; complex queries get more. The user doesn't control the budget directly — the model allocates it adaptively. This is arguably the most efficient approach but gives users the least control.

The three pricing models reflect three philosophies:
- **OpenAI (reasoning minutes):** "You're buying thinking time. Spend it wisely."
- **Anthropic (visible thinking tokens):** "You can see the reasoning. Pay for what you use."
- **Google (adaptive allocation):** "We'll figure out how much thinking is needed."

For developers building on these APIs, the choice of pricing model affects application architecture. OpenAI's model encourages batch processing (amortize the reasoning budget across many queries). Anthropic's model encourages streaming (read the reasoning in real-time, stop early if the answer is obvious). Google's model encourages fire-and-forget (let the system optimize).

## The Inference Scaling Law

The underlying observation: inference-time compute scales performance logarithmically, just like training-time compute. More thinking produces better answers, but with diminishing returns. The "reasoning minutes" framing is OpenAI's acknowledgment that they've found the inference scaling law and are pricing accordingly.

This has a structural consequence for the industry. Training scaling — throwing more GPUs at training — has been the primary competitive axis since GPT-3. Inference scaling — throwing more compute at thinking — is becoming the second axis. A model that trains on 10x less data but thinks 100x harder at inference time can match a larger model's performance on reasoning tasks.

The labs that figure out efficient inference-time search — better tree search algorithms, better value functions, better pruning strategies — will have a cost advantage that compounds over every API call. This is why the pricing change matters: it signals that OpenAI believes inference-time compute is the new battleground.
