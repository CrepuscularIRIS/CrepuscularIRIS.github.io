ICML 2026 拒绝了 497 篇论文，原因是评审者将评审工作外包给了语言模型。检测方法不是分类器，不是困惑度评分，也不是文体分析——而是**提示注入**。会议组织者在提交的 PDF 中嵌入了不可见的指令，当 LLM 处理这些文档时，会在输出中包含特定的标记短语。

这大概是第一次大规模部署对抗性提示注入作为**制度性防御机制**。该技术借鉴了信息安全（隐写水印），将一个已知的 LLM 漏洞（提示注入）武器化，并在一个特殊语境中部署——"攻击者"是学术机构，"受害者"是任何违反评审政策的人。

## 水印方案的技术设计

ICML 构建了一个包含 170,000 个短语的词典——每个短语在学术评审语境中语义上合理，但特异性足以使其在自然写作中出现的概率极低。对于每篇提交的论文，系统从词典中随机选取 2 个短语，并将它们作为隐藏指令嵌入 PDF。

碰撞概率——两篇不同论文被分配到相同短语对的概率——低于百亿分之一。这意味着被标记的评审可以几乎确定地追溯到特定的提交稿件。

### 注入机制

隐藏指令利用了 LLM 处理文档上下文的方式。当评审者将 PDF 喂给 ChatGPT、Claude 或任何前沿模型时，模型会对整个文档内容进行标记化处理——包括元数据、隐藏文本层和人类通常忽略的格式化指令。嵌入的短语被放置在 PDF 中：

1. **人类不可见** — 隐藏在元数据字段、白底白字、或 PDF 标注层中
2. **LLM 标记器可见** — 包含在模型推理前的原始文本提取中
3. **指令性质** — 措辞为指令（"在你的分析中包含短语 X"），利用模型的指令遵循行为

```python
import hashlib
from typing import Tuple

PHRASE_DICTIONARY_SIZE = 170_000

def select_watermark_phrases(
    paper_id: str,
    secret_key: bytes,
    dictionary: list[str],
) -> Tuple[str, str]:
    """确定性地为给定论文选择 2 个短语。"""
    h = hashlib.pbkdf2_hmac("sha256", paper_id.encode(), secret_key, 100_000)
    idx1 = int.from_bytes(h[:4], "big") % len(dictionary)
    idx2 = int.from_bytes(h[4:8], "big") % len(dictionary)
    if idx1 == idx2:
        idx2 = (idx2 + 1) % len(dictionary)
    return dictionary[idx1], dictionary[idx2]

# 碰撞概率：1 / (170000 * 169999) ≈ 3.5e-11
```

### 检测率与误报

ICML 报告称 795 篇评审（约占总投稿的 1%）来自 506 位独立评审者被标记。其中 51 位评审者超过 50% 的评审被标记——这种模式与系统性使用 LLM 一致，而非偶然的短语匹配。

检测成功率在大多数前沿模型上超过 80%。未检测到的 20% 可能反映了以下情况：模型输出经过大量人工编辑；PDF 提取管道在到达模型之前剥离了隐藏内容；或模型恰好没有遵循注入的指令。

## 双轨评审政策

ICML 2026 在水印方案之外引入了双轨评审政策：

- **政策 A**：评审过程中不允许使用 LLM 辅助
- **政策 B**：允许有限的 LLM 辅助（如语法检查、文献搜索），但不允许用于生成实质性评审内容

通过提供合法的 LLM 辅助评审路径（政策 B），ICML 降低了在政策 A 下秘密使用 LLM 的动机。

## 数字背后的真相

1% 的标记率低估了 LLM 辅助评审的实际普遍程度。ICLR 2026 的独立分析估计 21% 的评审显示 AI 生成的统计特征。如果 ICML 的评审者群体有类似行为，1% 的检测率大约捕获了每 20 例实际违规中的 1 例——那些既懒到使用完整 PDF 管道、又粗心到没有编辑掉异常短语的人。

这对理解该检测方案的目的很重要：它不捕获所有违规者，它捕获最恶劣的案例。威慑效应可能延伸得更远——知道水印*存在*（ICML 公开了该方案）会抑制 LLM 的使用，即使对那些可能逃避检测的评审者也是如此。

## 提示注入悖论

有一个值得深思的悖论。AI 安全社区多年来一直在警告提示注入作为一种**漏洞**——恶意文档劫持 LLM 行为的方式。而 ICML 的水印方案*就是*提示注入，只不过由受信任的机构出于合法目的部署。

这创造了一个令人不安的先例：

**如果提示注入是合法的制度工具，还有谁可以使用它？** 雇主在与员工共享的文档中嵌入监控指令？政府在机密材料中插入追踪短语？技术机制完全相同——只有制度语境和意图不同。

**防御手段破坏了自身的基础。** ICML 的方案之所以有效，恰恰是因为 LLM 容易受到提示注入的影响。如果模型变得对注入具有鲁棒性（安全社区正在积极追求的目标），水印方案就会失效。ICML 的检测能力与 LLM 安全性呈负相关。

**对抗性不对称。** 当前方案捕获的是天真的违规者。理解提示注入的评审者可以在将 PDF 喂给 LLM 之前预处理文档以剥离隐藏内容。PDF 清理工具并不复杂。

```python
import fitz  # PyMuPDF

def sanitize_pdf_for_review(input_path: str, output_path: str) -> None:
    """
    在将 PDF 喂给 LLM 之前剥离隐藏文本、元数据和标注层。
    这不是建议——而是展示检测方案中的对抗性不对称。
    """
    doc = fitz.open(input_path)
    clean_text_parts = []
    for page in doc:
        blocks = page.get_text("blocks")
        for block in blocks:
            x0, y0, x1, y1, text, block_no, block_type = block
            if block_type == 0 and len(text.strip()) > 0:
                clean_text_parts.append(text.strip())
    doc.close()
    
    clean_text = "\n\n".join(clean_text_parts)
    with open(output_path, "w") as f:
        f.write(clean_text)
```

## 对研究社区的启示

水印方案是一个更深层问题的症状：评审系统依赖于研究者的自愿无偿劳动，而这些研究者的时间越来越紧张。LLM 提供了一条明显的捷径。问题在于社区的回应应该是检测与惩罚、结构性改革，还是二者的结合。

**检测有其极限。** 如上所示，该方案主要捕获粗心的违规者。更重要的是，即使完美的检测也不能解决根本原因。

**结构性替代方案已经存在：**

- **有偿评审** — 支付报酬降低评审者偷工减料的动机
- **更短的评审格式** — 要求更少的文本但更结构化的评估
- **人机协作评审流程** — 明确设计让 LLM 处理文献搜索和格式检查，人类提供实质性评估
- **开放评审** — 通过透明度而非监控来创建问责制

ICML 的水印方案是精巧的工程。它是否代表了对 AI 辅助评审的正确制度回应——这是一个涉及检测精度、评审者隐私、制度信任，以及同行评审作为制度可持续性之间权衡的独立问题。
