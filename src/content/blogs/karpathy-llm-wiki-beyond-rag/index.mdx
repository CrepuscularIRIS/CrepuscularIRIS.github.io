Andrej Karpathy published a post in early April proposing what he calls the **LLM Wiki** paradigm — a direct challenge to how nearly every production AI system handles memory today. The argument is surgical: RAG doesn't actually *learn*. It retrieves, re-reads, and re-reasons from raw documents on every single query. That's not a knowledge base. That's a filing cabinet with a fast lookup.

The LLM Wiki idea is deceptively simple but architecturally significant. Instead of storing raw source documents and querying them, you deploy an LLM agent that *compiles* those documents into a living, structured, hyperlinked Markdown wiki — and keeps it up to date.

## The Core Problem With RAG

Standard RAG pipelines work like this: embed documents into vectors, store them, then at query time retrieve the top-k chunks and stuff them into the context window. The model then reasons over raw text it has never "processed" before.

This creates a structural inefficiency. Every query restarts from zero. The model can't build compound understanding across documents. Contradictions between sources aren't resolved — they're just both retrieved and left for the model to sort out in-context, which it often doesn't. Staleness is invisible: a chunk from an outdated document looks identical to a fresh one.

For simple factual Q&A over a bounded document set, this works fine. For long-horizon reasoning — maintaining a codebase context, tracking a research project, understanding a customer over months — it breaks down.

## The LLM Wiki Architecture

Karpathy's proposal inverts the relationship between retrieval and reasoning. The LLM isn't called at query time to reason over raw data. It's called *continuously* to maintain a structured knowledge base:

```
Raw Sources → LLM Librarian Agent → Structured Markdown Wiki → Query Interface
```

The agent's job is not to answer questions. Its job is to be a research librarian: read new documents, update the wiki, resolve conflicts, add cross-links, prune stale entries. Query answering then becomes trivially cheap — you're reading compiled knowledge, not reasoning over raw text.

A minimal version of this looks like:

```python
import anthropic
from pathlib import Path

client = anthropic.Anthropic()

WIKI_PATH = Path("./knowledge-wiki")

def ingest_document(doc_text: str, doc_source: str) -> None:
    """Ingest a new document into the wiki, updating relevant pages."""
    
    # Read the current wiki state
    existing_pages = {
        p.stem: p.read_text() 
        for p in WIKI_PATH.glob("*.md")
    }
    wiki_index = "\n".join(f"- {name}" for name in existing_pages)
    
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=4096,
        messages=[{
            "role": "user",
            "content": f"""You are a research librarian maintaining a structured wiki.

Current wiki pages:
{wiki_index}

New document from {doc_source}:
{doc_text}

Your tasks:
1. Identify which existing wiki pages this document updates, contradicts, or extends
2. Create new pages if needed for genuinely new topics
3. For each page to create or update, output the full page content

Output format:
<page name="PageName">
Full markdown content here, including links to related pages like [[RelatedPage]]
</page>

Resolve any contradictions explicitly. Link related concepts. Prune claims that are superseded."""
        }]
    )
    
    # Parse and write updated pages
    import re
    pages = re.findall(r'<page name="([^"]+)">(.*?)</page>', response.content[0].text, re.DOTALL)
    for page_name, content in pages:
        (WIKI_PATH / f"{page_name}.md").write_text(content.strip())
    
    print(f"Updated {len(pages)} wiki pages from {doc_source}")
```

The key shift: the LLM is doing *editorial work*, not answering questions. The output is structured knowledge that persists and compounds.

## The Self-Healing Linter

The more interesting part of Karpathy's proposal is a **linting mechanism**. Periodically — not on every ingestion, but as a background maintenance pass — the LLM scans the wiki for:

- **Contradictions**: Page A says X, Page B says not-X
- **Orphan pages**: Topics referenced by nothing else
- **Stale facts**: Dated claims that newer pages have superseded
- **Missing links**: Topics that should cross-reference but don't

```python
def lint_wiki() -> list[str]:
    """Scan the wiki for contradictions, staleness, and missing links."""
    
    all_pages = {
        p.stem: p.read_text() 
        for p in WIKI_PATH.glob("*.md")
    }
    
    # Feed entire wiki to the model for cross-page analysis
    wiki_dump = "\n\n---\n\n".join(
        f"# {name}\n{content}" 
        for name, content in all_pages.items()
    )
    
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=8192,
        messages=[{
            "role": "user", 
            "content": f"""Audit this knowledge wiki for quality issues.

{wiki_dump}

Find and report:
1. Direct contradictions between pages (cite both pages and the conflicting claims)
2. Circular or missing cross-links that should exist
3. Pages that appear outdated given what other pages say
4. Topics referenced but not yet documented

For each issue, output:
<issue type="contradiction|orphan|stale|missing" pages="Page1,Page2">
Description of the issue and suggested resolution
</issue>"""
        }]
    )
    
    # Parse issues and return them for review or auto-fix
    import re
    issues = re.findall(r'<issue[^>]*>(.*?)</issue>', response.content[0].text, re.DOTALL)
    return [i.strip() for i in issues]
```

Run this nightly. Auto-fix low-risk issues (missing links, orphan cleanup). Flag contradictions for human review or a more careful resolution pass.

## Why This Beats RAG for Long-Horizon Tasks

The practical advantages compound over time:

**1. Contradiction resolution happens once, at write time.** With RAG, if your document corpus has two papers that disagree, every query that retrieves both documents sees the contradiction. With a wiki, the librarian agent resolves it when the second paper is ingested. Queries never see the raw conflict.

**2. Cross-document synthesis is baked in.** The wiki can have a "MoE Training Approaches" page that synthesizes insights from 20 papers. RAG can't do this — it can only retrieve, not synthesize at index time.

**3. Staleness becomes explicit.** Wiki pages can carry modification timestamps and provenance. A RAG chunk from 2024 doesn't know it's been superseded by a 2026 paper. A wiki page can explicitly say "this was the consensus as of March 2026; see [[Updated Findings]] for revisions."

**4. Query latency collapses.** Reading a wiki page is fast. Embedding, retrieving, then reasoning over 10 raw chunks is slow. For high-frequency queries over stable knowledge, the wiki approach is dramatically more efficient.

## Where RAG Still Wins

This isn't a wholesale replacement. RAG has genuine advantages in specific regimes:

- **High document velocity**: If you're ingesting 10,000 new articles per day, you can't maintain a wiki at that rate without massive compute. RAG scales to large corpora more easily.
- **Verbatim retrieval requirements**: Legal, regulatory, and compliance contexts often require exact original text. A wiki that paraphrases and synthesizes loses provenance.
- **Cold-start freshness**: If you need to answer questions about a document ingested 5 minutes ago, RAG is ready. The wiki takes time to compile.

## The Hybrid Architecture: Wiki + RAG

The real answer isn't choosing between the two. It's building a tiered system that uses each where it excels.

```python
from dataclasses import dataclass
from enum import Enum

class RetrievalTier(Enum):
    HOT_WIKI = "hot_wiki"      # Compiled, high-value, frequently accessed
    WARM_INDEX = "warm_index"  # Summarized, searchable, less critical
    COLD_RAG = "cold_rag"      # Raw documents, vector retrieval

@dataclass
class WikiResult:
    content: str
    confidence: float

@dataclass
class KnowledgeRouter:
    """Route queries to the appropriate retrieval tier."""
    
    wiki_path: Path
    vector_store: VectorStore
    summary_index: SummaryIndex
    
    def query(self, question: str) -> str:
        # Check hot wiki first — compiled knowledge, fast lookup
        wiki_results = self._search_wiki(question)
        if wiki_results and wiki_results.confidence > 0.85:
            return wiki_results.content
        
        # Fall back to warm index — summarized documents
        summary_results = self.summary_index.search(question, top_k=5)
        if summary_results and summary_results[0].score > 0.80:
            return self._synthesize(summary_results, question)
        
        # Cold RAG — raw document retrieval as last resort
        raw_chunks = self.vector_store.search(question, top_k=10)
        return self._reason_over_chunks(raw_chunks, question)
    
    def _search_wiki(self, question: str):
        """Search compiled wiki pages with keyword + semantic matching."""
        pages = list(self.wiki_path.glob("*.md"))
        # Combine keyword matching with embedding similarity
        candidates = []
        for page in pages:
            content = page.read_text()
            score = self._relevance_score(question, content)
            if score > 0.5:
                candidates.append((page, content, score))
        
        candidates.sort(key=lambda x: x[2], reverse=True)
        if candidates:
            return WikiResult(
                content=candidates[0][1], 
                confidence=candidates[0][2]
            )
        return None
```

The three tiers serve different access patterns:

| Tier | Content | Update Frequency | Query Latency | Use Case |
|------|---------|-----------------|---------------|----------|
| Hot Wiki | Compiled, cross-linked Markdown | On ingestion + nightly lint | &lt;100ms | Core domain knowledge, frequently asked questions |
| Warm Index | Document summaries with metadata | Daily batch | ~500ms | Broader context, supporting evidence |
| Cold RAG | Raw document chunks | On ingestion | ~2s | Long-tail queries, verbatim retrieval |

The wiki handles maybe 20% of your documents — the high-value, frequently accessed, slowly changing core knowledge. The warm index covers another 30% — useful context that doesn't need full editorial treatment. Cold RAG handles the remaining 50% — the long tail of documents you might need someday.

## The Cost Question

The obvious objection: running an LLM to compile every ingested document is expensive. True. But the cost analysis isn't straightforward.

A RAG system that handles 1,000 queries/day, each retrieving 10 chunks and reasoning over them, runs 1,000 LLM calls per day. A wiki system that ingests 50 new documents/day runs 50 compilation calls plus maybe 10 lint calls — but then serves queries from compiled Markdown with zero LLM calls for cached knowledge.

```
RAG daily cost:     1,000 queries × 10K tokens/query = 10M tokens
Wiki daily cost:    50 ingestions × 5K tokens + 10 lint × 20K tokens = 450K tokens
Wiki query cost:    Free (reading Markdown files)

Break-even point:   ~45 queries/day over the same knowledge base
```

Above 45 queries/day, the wiki is cheaper. For high-traffic knowledge bases — customer support, internal documentation, research literature — the economics favor compilation heavily.

The real cost isn't API tokens — it's the engineering investment in building the compilation pipeline, the linting system, the tiered routing, and the provenance tracking. This is where most teams give up and default to RAG. RAG is a pip install. A wiki system is a project.

## Provenance and Trust

One underappreciated advantage of the wiki approach: provenance is explicit and auditable. Every wiki page can track its sources:

```markdown
# MoE Training Approaches

## Summary
Mixture of Experts models achieve parameter efficiency by activating only a 
subset of experts per input token. The dominant approach uses top-k routing 
with auxiliary load balancing losses.

## Key Findings
- Top-2 routing with capacity factor 1.25 is the current standard [[Switch Transformer]]
- Expert parallelism scales to 2048 experts with acceptable communication overhead [[GShard]]
- Extreme sparsity (1 active out of 256) shows surprising effectiveness [[Gemma 4 Analysis]]

## Sources
- [Switch Transformers](https://arxiv.org/abs/2101.03961) — Fedus et al., 2021
- [GShard](https://arxiv.org/abs/2006.16668) — Lepikhin et al., 2020
- [Gemma 4 blog analysis](/blog/gemma-4-extreme-moe-sparsity) — internal, 2026-04-12

Last updated: 2026-04-12 | Confidence: HIGH | Next review: 2026-05-12
```

With RAG, provenance is implicit in the chunk metadata — if you remembered to store it. With a wiki, provenance is a first-class citizen of the knowledge itself. When a claim is questioned, you trace it to its source page, which traces it to its source document.

## Implementation Lessons

Having built a system along these lines for this blog's research pipeline, three non-obvious lessons:

**1. The librarian prompt matters more than the ingestion prompt.** The quality of wiki compilation depends heavily on how you instruct the LLM to handle conflicts, decide granularity, and maintain cross-links. A generic "summarize this document" prompt produces bad wiki pages. A specific "you are a research librarian maintaining a knowledge base for an AI researcher" prompt produces dramatically better ones.

**2. Incremental updates beat full rewrites.** When a new document arrives, don't regenerate every potentially affected page. Identify the 2-3 pages most affected and update only those. Full-wiki regeneration is expensive and introduces unnecessary drift.

**3. The linting pass is the secret weapon.** Ingestion quality varies — sometimes the librarian misses a connection or introduces a subtle inconsistency. The periodic linting pass catches these. Think of it as continuous integration for knowledge: run the quality checks nightly, fix what's broken, log what needs human review.

## The Filing Cabinet Is Not a Library

Karpathy's framing is more philosophical than prescriptive — an argument about what knowledge bases *should* be, not a complete system design. But the core insight holds: retrieval systems that never synthesize their inputs are leaving intelligence on the table.

The filing cabinet — RAG — is good at storage and lookup. The library — a compiled, maintained, cross-linked knowledge base — is good at *understanding*. The librarian — the LLM that maintains the wiki — is the missing piece that makes the transition possible.

Most production systems in 2026 still use filing cabinets. The ones that are building libraries are seeing compounding returns that filing cabinets can't match.

The filing cabinet is not a library. A library has a librarian.
