Most multi-agent systems in production today are prompt cascades: Agent A generates text, Agent B receives that text as input, Agent C does something with B's output. The flow is implicit, encoded in prompt templates and string interpolation. When it works, it looks like magic. When it fails, you have no idea where the failure happened, no way to resume from a checkpoint, and no mechanism to inject human oversight at the right moment.

The April 2026 convergence of Google's Agent-to-Agent (A2A) protocol and LangGraph v2.0's state machine primitives marks a shift in how the industry builds multi-agent workflows. The core idea: treat agent orchestration as a graph problem with explicit state, not a chain problem with implicit context.

## Why Chains Break

A chain is a sequence of function calls where each function's output feeds the next function's input. In agent terms: prompt Agent A, take its response, embed it in Agent B's prompt, take B's response, embed it in C's prompt.

Three failure modes dominate chain-based orchestration:

**Error propagation without attribution.** When Agent C produces garbage, the root cause might be Agent A's hallucination that Agent B faithfully passed along. In a chain, there's no structured way to trace which agent introduced the error. You're debugging by reading interleaved prompt logs.

**No resumability.** If the chain fails at step 4 of 7, you restart from step 1. In a 15-minute workflow that costs $2 in API calls, re-running the entire chain because step 6 failed is operationally expensive and architecturally lazy.

**Binary human oversight.** In a chain, HITL (human-in-the-loop) means stopping the entire pipeline, showing the human the current state, and restarting. There's no concept of "approve this specific decision and continue" without building it from scratch each time.

## The DAG Alternative

A directed acyclic graph (DAG) represents agent workflows as nodes (agents or tools) connected by edges (state transitions). Each node reads from and writes to an explicit state schema. The graph runtime manages execution order, parallelism, and checkpointing.

```python
from dataclasses import dataclass, field
from typing import Literal


@dataclass
class WorkflowState:
    """Explicit state schema — every field is typed and versioned."""
    task_description: str
    localized_files: list[str] = field(default_factory=list)
    proposed_fix: str | None = None
    fix_compiles: bool = False
    tests_pass: bool = False
    human_approved: bool = False
    retry_count: int = 0
    status: Literal["localizing", "fixing", "verifying", "reviewing", "done", "failed"] = "localizing"


def localize_node(state: WorkflowState) -> WorkflowState:
    """Static analysis to find relevant files. Pure function on state."""
    files = run_static_analysis(state.task_description)
    return WorkflowState(
        **{**state.__dict__, "localized_files": files, "status": "fixing"}
    )


def fix_node(state: WorkflowState) -> WorkflowState:
    """Generate a fix using the localized context."""
    fix = agent_generate_fix(state.task_description, state.localized_files)
    return WorkflowState(
        **{**state.__dict__, "proposed_fix": fix, "status": "verifying"}
    )


def verify_node(state: WorkflowState) -> WorkflowState:
    """Run compiler and tests against the proposed fix."""
    compiles = check_compilation(state.proposed_fix)
    passes = run_tests(state.proposed_fix) if compiles else False
    new_status = "reviewing" if passes else (
        "fixing" if state.retry_count < 3 else "failed"
    )
    return WorkflowState(
        **{
            **state.__dict__,
            "fix_compiles": compiles,
            "tests_pass": passes,
            "status": new_status,
            "retry_count": state.retry_count + (0 if passes else 1),
        }
    )
```

The state schema is the contract. Each node is a pure function from state to state. The graph runtime decides which node to execute next based on the current state. This buys you three properties that chains lack.

## Time-Travel Checkpointing

Because every state transition is explicit and the state schema is serializable, the runtime can persist state to a database after each node execution. When a workflow fails at step 4, you inspect the serialized state at step 3, fix the issue (or modify the state manually), and resume from step 3 without re-running steps 1 and 2.

This is "time-travel debugging" applied to agent workflows. The implementation is straightforward when your state is a typed dataclass:

```python
import json
from datetime import datetime


class CheckpointStore:
    """Persist workflow state after each node for resumability."""

    def __init__(self, db_connection):
        self.db = db_connection

    def save(self, workflow_id: str, node_name: str, state: WorkflowState) -> None:
        self.db.execute(
            "INSERT INTO checkpoints (workflow_id, node_name, state, created_at) "
            "VALUES (%s, %s, %s, %s)",
            (workflow_id, node_name, json.dumps(state.__dict__), datetime.utcnow()),
        )

    def load_latest(self, workflow_id: str) -> tuple[str, WorkflowState]:
        row = self.db.execute(
            "SELECT node_name, state FROM checkpoints "
            "WHERE workflow_id = %s ORDER BY created_at DESC LIMIT 1",
            (workflow_id,),
        ).fetchone()
        return row["node_name"], WorkflowState(**json.loads(row["state"]))

    def resume(self, workflow_id: str) -> WorkflowState:
        """Resume workflow from last checkpoint."""
        node_name, state = self.load_latest(workflow_id)
        return graph_runtime.execute_from(node_name, state)
```

In production, this eliminates the most expensive operational failure mode: long-running agent workflows that fail near the end and must restart from scratch. A 30-minute code migration workflow that checkpoints after each file can resume from the last successfully processed file.

## The A2A Protocol Layer

Google's Agent-to-Agent (A2A) protocol, released alongside LangGraph v2.0, standardizes how agents discover and delegate tasks to each other. Where MCP (Model Context Protocol) standardizes agent-to-tool communication, A2A standardizes agent-to-agent communication.

The key abstraction is the **Agent Card** — a JSON document that describes an agent's capabilities, input/output schemas, and authentication requirements:

```json
{
  "name": "code-reviewer",
  "description": "Reviews code changes for bugs, security issues, and style violations",
  "input_schema": {
    "type": "object",
    "properties": {
      "diff": { "type": "string", "description": "Unified diff of changes" },
      "context_files": { "type": "array", "items": { "type": "string" } },
      "review_focus": {
        "type": "string",
        "enum": ["security", "correctness", "style", "all"]
      }
    },
    "required": ["diff"]
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "issues": { "type": "array" },
      "approval": { "type": "boolean" },
      "summary": { "type": "string" }
    }
  },
  "auth": { "type": "oauth2", "scopes": ["read:code"] }
}
```

The orchestrating agent discovers available agents by querying a registry of Agent Cards, selects the right agent for each subtask based on capability matching, and delegates with typed inputs. The delegated agent returns typed outputs that feed directly into the workflow state.

This replaces the ad-hoc pattern of "paste Agent A's output into Agent B's system prompt and hope the format is compatible." Type safety at agent boundaries catches integration errors at delegation time, not three agents downstream.

## HITL Gates as Graph Nodes

Human-in-the-loop oversight in a DAG is a node like any other. The node pauses execution, presents the current state to a human reviewer, and resumes when the human provides a decision.

```python
def human_review_node(state: WorkflowState) -> WorkflowState:
    """Pause for human approval. State is persisted; execution halts."""
    checkpoint_store.save(workflow_id, "human_review", state)

    # In practice, this triggers a webhook to a review UI
    # The human sees: task description, proposed fix, test results
    # The human can: approve, reject, or modify the fix

    # Execution resumes when the human responds
    decision = wait_for_human_decision(workflow_id)

    if decision.action == "approve":
        return WorkflowState(**{**state.__dict__, "human_approved": True, "status": "done"})
    elif decision.action == "modify":
        return WorkflowState(
            **{**state.__dict__, "proposed_fix": decision.modified_fix, "status": "verifying"}
        )
    else:
        return WorkflowState(**{**state.__dict__, "status": "failed"})
```

The architectural advantage: HITL gates are positioned in the graph based on risk, not pipeline position. High-risk decisions (deploying to production, modifying database schemas, sending external communications) get HITL gates. Low-risk decisions (reformatting code, running tests) proceed automatically. The graph structure makes this placement explicit and auditable.

## Conditional Routing and Parallel Execution

DAGs support branching and parallelism natively. The routing function examines the current state and returns the next node(s) to execute:

```python
def route(state: WorkflowState) -> list[str]:
    """Determine next node(s) based on current state."""
    match state.status:
        case "localizing":
            return ["localize_node"]
        case "fixing":
            return ["fix_node"]
        case "verifying":
            return ["verify_node"]
        case "reviewing":
            return ["human_review_node"]
        case "failed" if state.retry_count < 3:
            return ["localize_node"]  # re-localize with broader search
        case "failed":
            return ["escalation_node"]
        case "done":
            return []  # terminal state

    # Parallel execution: return multiple nodes
    # e.g., ["security_review", "style_review", "correctness_review"]
    # Runtime executes all three concurrently and merges results
```

Parallel branches are particularly useful for review workflows: run security review, style review, and correctness review concurrently, then merge their findings into a single review state. The DAG runtime handles fan-out and fan-in without the orchestrating agent managing concurrent promises.

## When DAGs Are Overkill

Not every agent workflow needs graph-based orchestration. Simple request-response patterns — a single agent with tool access solving a bounded problem — work fine as chains or even bare API calls. The overhead of state schemas, checkpointing, and graph runtimes is justified when:

- The workflow has more than 3 steps
- Failures are expensive to retry from scratch
- Human oversight is required at specific decision points
- Multiple agents need to coordinate with typed contracts
- The workflow runs for minutes or hours, not seconds

For a coding assistant that answers questions in a single turn, a DAG is unnecessary complexity. For an agent that performs multi-file refactoring across a repository, with compilation checks, test runs, and human review of the migration plan — the DAG pays for itself on the first failure that doesn't require a full restart.

## The Convergence

The April 2026 landscape shows three complementary standards crystallizing:

| Layer | Protocol | Purpose |
|-------|----------|---------|
| Agent ↔ Tool | MCP | Standardized tool discovery and invocation |
| Agent ↔ Agent | A2A | Typed delegation with capability matching |
| Workflow ↔ State | LangGraph/DAG | Deterministic execution with checkpointing |

MCP tells agents what tools exist. A2A tells agents what other agents can do. The DAG runtime decides when each agent runs and manages the state between them. Together, they replace the implicit, fragile prompt cascade with an explicit, debuggable, resumable workflow.

The shift from chains to graphs is the same shift that happened in CI/CD a decade ago: from shell scripts that pipe output between commands to DAG-based systems (Airflow, GitHub Actions) that checkpoint, retry, and parallelize. Agent orchestration is catching up with the operational maturity that data engineering figured out years ago.

The next generation of agent systems will not be distinguishable by which model they use. They'll be distinguishable by whether their orchestration is a graph you can inspect, checkpoint, and resume — or a chain you can only restart from scratch.
