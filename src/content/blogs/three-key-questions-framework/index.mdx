Reading ML papers at conference pace produces a specific kind of cognitive debt: you accumulate claimed improvements faster than you can evaluate them. A new residual variant. An attention modification. A routing mechanism. Each comes with a table showing it's 1-2% better than baselines on four standard benchmarks.

Most of these claims do not replicate cleanly. The improvement is real in the paper's experimental setup — but the setup has hidden degrees of freedom that explain the gap. The architecture contributes less than the authors believe, and the contribution comes from somewhere other than the mechanism they describe.

Three questions have proven consistently useful for separating signal from noise. They apply to any architectural proposal, but they're sharpest when the claim is a modification to an existing architecture (residual connections, attention mechanisms, normalization strategies) rather than an entirely new paradigm.

---

## Question 1: Is This Changing Weight Allocation or Changing What's Being Modeled?

The first axis separates **connection weight modifications** from **modeling object changes**. This distinction matters because they have fundamentally different failure modes and different upside ceilings.

**Connection weight modifications** change how information is aggregated, routed, or combined within an existing computational graph. The inputs and outputs remain the same; the transformation changes. Examples:

- Attention Residuals (fixed weights → Softmax-weighted aggregation over layer history)
- Mixture-of-Experts routing (dense FFN → sparse expert selection)
- Learnable layer scaling (fixed unit residual weights → per-layer gain parameters)

**Modeling object changes** change what the network is asked to represent. The computational structure may look similar, but the model is now capturing a fundamentally different distributional assumption. Examples:

- Variational Autoencoders (direct density estimation → latent variable modeling with explicit priors)
- Neural ODEs (discrete layer stacking → continuous dynamics in depth-space)
- Diffusion models (direct generation → score function estimation and reverse SDE)

The practical test: if the baseline model could approximate the proposed model's behavior at test time — just with the right learned weights — you're looking at a connection weight modification. If the baseline model *cannot in principle* represent what the proposal represents, you're looking at a modeling object change.

```python
# Connection weight modification: baseline can express this at initialization
# Attention Residuals uses Softmax weights α_i over layer history h_i
# At initialization, α_i ≈ uniform → equivalent to standard residual / L
# At convergence, α_i = f(content) → beyond baseline expressivity, but
#     the transition is smooth and the baseline was "almost there"

h_attn_residual = sum(alpha_i * h_i for i in range(current_layer))

# Modeling object change: baseline cannot express this distribution
# A VAE models p(x) = ∫ p(x|z)p(z)dz — requires explicit latent structure
# A standard autoencoder models p(x|enc(x)) — cannot approximate the integral
# These are not related by a weight change; they model different things

z_mean, z_logvar = encoder(x)
z = reparameterize(z_mean, z_logvar)  # requires stochastic computation graph
x_recon = decoder(z)
kl_loss = -0.5 * sum(1 + z_logvar - z_mean**2 - exp(z_logvar))
```

**Why this matters for evaluation**: Connection weight modifications are correctly evaluated by asking whether the learned weights actually differ from initialization — and whether the delta is positive. If you train Attention Residuals and find the learned α_i are approximately uniform, the model isn't using the proposed mechanism even if benchmarks improve. (Benchmark improvement from a parameter count increase is the most common confound for this class of modification.)

Modeling object changes require different evaluation: you need to show the new modeling assumption is *correct for the domain*, not just that the model achieves better metrics. A VAE's value comes from learning a meaningful latent space, not from reconstruction accuracy alone.

**The skeptic's conclusion from Q1**: A connection weight modification with benchmark improvements is a weak claim unless the paper shows (a) the learned weights actually differ from the uniform initialization, and (b) the improvement isn't explained away by the extra parameters.

## Question 2: Is This Block-Level or Does It Commit to Continuity?

The second question asks where a proposed mechanism sits on the **discretization spectrum**: from fully block-local (operates within fixed computational groups, no structure beyond the block) to fully continuous (operates over a smooth manifold with no discrete boundaries).

This matters because most "novel" architectures are implicit compromises on this spectrum — and the paper rarely tells you what they gave up by landing where they did.

**Block-local mechanisms** operate within a fixed-size chunk of layers, tokens, or time steps. The "block" is a design choice: 4 layers, 8 tokens, 512ms of audio. Within the block, the mechanism can be arbitrarily sophisticated. Across blocks, interaction is by fixed residual or concatenation.

Examples: Transformer blocks (attention within context window, no cross-window interaction in base GPT), Block Attention Residuals (attention-weighted aggregation *within* each block of B layers, fixed residual *across* blocks), sliding window attention.

**Continuous mechanisms** commit to smooth variation over the dimension in question. Neural ODEs parameterize depth as a continuous variable; memory-augmented architectures treat time as a continuous signal; CDMN-style proposals treat layer depth as a coordinate in a learned manifold.

```python
# Block-level: interaction bounded by block_size
class BlockAttnRes(nn.Module):
    def __init__(self, block_size: int = 4):
        self.block_size = block_size  # hard boundary
    
    def forward(self, h_current: torch.Tensor, 
                history: list[torch.Tensor],
                layer_idx: int) -> torch.Tensor:
        # Can only attend within the current block — hard constraint
        block_start = (layer_idx // self.block_size) * self.block_size
        block_history = history[block_start:layer_idx]  # O(B) items max
        
        if not block_history:
            return h_current
        
        # Attention within block is arbitrary; boundary is fixed
        return self._attend(h_current, torch.stack(block_history, dim=0))

# Continuous: depth treated as a coordinate
class NeuralODELayer(nn.Module):
    def __init__(self, dynamics_fn: nn.Module):
        self.dynamics = dynamics_fn  # dh/dt = f(h, t)
    
    def forward(self, h0: torch.Tensor, 
                t_span: tuple[float, float] = (0.0, 1.0)) -> torch.Tensor:
        # Depth t ∈ [0,1] is a continuous variable
        # No block boundaries; integration is smooth
        from torchdiffeq import odeint
        t = torch.linspace(*t_span, steps=50)
        return odeint(self.dynamics, h0, t)[-1]
```

**The practical tradeoff** between block-level and continuous: block-level is computationally tractable and often achieves most of the theoretical benefit at a fraction of the cost. Continuous mechanisms are theoretically principled but introduce optimization instability, ODE solver overhead, and adjoint method complexity.

The question for evaluation is whether the block boundary is a principled choice or a concession. If a paper introduces a mechanism that requires continuity in principle — cross-layer attention over the full depth — but implements it at block level, the block size is a critical hyperparameter that needs ablation. Papers routinely omit this ablation.

**Ablation you should require**: If the mechanism is block-level, what happens as block size varies? Does performance increase monotonically with block size (suggesting you want continuity but can't afford it) or does it peak at a specific size (suggesting the block is the right inductive bias)? These are different claims about what the mechanism is actually doing.

## Question 3: Is This Architecture Insight or Product Disclosure?

The third question is uncomfortable but necessary: **is the paper reporting a scientific finding, or is it documenting an engineering decision that was made for other reasons?**

Major labs — OpenAI, Anthropic, Google, Moonshot AI — publish papers about the architectures they've already deployed. These papers often contain genuine scientific insight. But they also contain decisions that were made for reasons that are not scientific: available hardware, team expertise, inference cost, compatibility with existing serving infrastructure.

A paper reporting a genuinely novel architecture finds the mechanism, demonstrates it, and concludes that it works. A product disclosure paper describes what was built, presents evaluations that show it's competitive, and implicitly argues that the engineering choices were reasonable.

The signals that distinguish them:

**Architecture insight signals:**
- The mechanism is tested on a *new synthetic task* designed specifically to require it (not just better scores on standard benchmarks)
- The paper includes experiments where the mechanism *fails* and discusses why
- The ablations vary the mechanism's core parameters (block size, attention temperature, routing k) and show non-monotonic behavior that reveals something about the mechanism's nature
- The reported result generalizes: other groups can reproduce it without tuning

**Product disclosure signals:**
- Benchmark improvement reported without a mechanism-specific test
- "We trained on 5T tokens with 128 A100s" — compute details that are opaque to most readers but legitimate to the lab
- Ablations that vary the full system (model size, data scale) rather than the specific mechanism
- Authors cannot clearly state what experiment would *falsify* the main claim

```python
# Test for "what would falsify this?":
# A well-formed architectural claim should specify:

@dataclass
class ArchitecturalClaim:
    mechanism: str  # "dynamic layer aggregation"
    hypothesis: str  # "attention-weighted residuals reduce PreNorm dilution"
    falsifiable_prediction: str  # "attention weights should deviate from uniform
                                 #  at convergence; if they don't, the mechanism
                                 #  isn't active regardless of benchmark results"
    confounds_ruled_out: list[str]  # ["parameter count matched", 
                                    #  "lr sweep run for both variants",
                                    #  "gradient norms compared"]

# Most papers can state mechanism and hypothesis.
# Very few explicitly state falsifiable_prediction.
# Fewer still document confounds_ruled_out.
```

The way to apply Q3 to a specific paper: identify the three most important engineering decisions in the paper (block size, warmup schedule, initialization scheme, routing algorithm). For each, ask: *would the paper's main result survive if this decision were changed?* If the answer is "probably not, but we didn't test it," the paper is more product disclosure than architecture insight — which is fine, but changes how much you should trust the mechanism claim versus the benchmark claim.

## Applying All Three in Practice

These questions are most useful applied together on a paper you're deciding whether to base your own work on. A quick evaluation template:

**Q1 (Weight allocation vs. modeling object):**
- [ ] What exactly changes in the computational graph?
- [ ] Could the baseline model express this behavior with different weights?
- [ ] Does the paper measure whether the proposed mechanism is actually active (e.g., weight statistics)?

**Q2 (Block-level vs. continuous):**
- [ ] Where is the mechanism on the discretization spectrum?
- [ ] Is the block boundary principled or pragmatic?
- [ ] Did the authors ablate block size / discretization granularity?

**Q3 (Architecture insight vs. product disclosure):**
- [ ] What experiment would falsify the main claim?
- [ ] Which engineering decisions are load-bearing, and were they varied?
- [ ] Can the result be reproduced without the paper's specific compute setup?

A paper that scores well on all three — demonstrates the mechanism is active, ablates the discretization granularity, and specifies what would falsify the claim — is trustworthy. A paper that scores poorly on all three — shows benchmark improvements without mechanism validation, doesn't ablate the block boundary, and reports results from a unique compute configuration — is worth reading for awareness but not building on.

Most papers fall somewhere in between. The checklist makes it legible where they fall and what verification experiments you'd need to run yourself before trusting the contribution.
