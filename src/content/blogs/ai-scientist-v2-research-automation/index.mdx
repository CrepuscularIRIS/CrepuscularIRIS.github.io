A paper submitted to an ICLR 2026 workshop passed peer review without human authors. The system that produced it — the AI Scientist v2, developed by Sakana AI — ran the full research pipeline autonomously: literature search, hypothesis generation, experimental implementation, result analysis, and manuscript writing. The reviewers evaluated it as they would any submission.

The workshop acceptance is a milestone, but the framing of "AI replaces researchers" misses what's actually interesting about it. The AI Scientist v2 is not a replacement for scientific judgment. It's a system that reveals which parts of the research process are automated away (execution), which parts are made more legible (the hypothesis-experiment-result loop), and which parts remain fundamentally resistant to automation (knowing what questions are worth asking).

## What the AI Scientist v2 Actually Does

The system architecture is a pipeline of LLM agents with tool access, organized around an iterative refinement loop. The v1 design (published 2024) used a fixed pipeline: generate idea → write code → run experiment → write paper. The v2 advance is introducing **tree search** over the hypothesis space: rather than committing to a single idea and executing it, the system explores a tree of possible research directions, evaluates each branch by running small-scale experiments, and allocates compute budget to the most promising branches.

```
Idea Generation (LLM) 
    ↓
Feasibility Filter (LLM + tool)     ← prunes branches before they're expensive
    ↓
Implementation (code-writing agent)
    ↓
Experiment Runner (sandboxed subprocess)
    ↓
Result Analyzer (LLM + stat tests)  ← interprets numerical results
    ↓
Branch Evaluator (novelty + significance scores)
    ↓
Tree search: expand promising branches, prune weak ones
    ↓
Paper Writer (structured output agent)
    ↓
Self-Review (LLM acting as reviewer)
    ↓
Revision Loop
```

The key engineering contribution in v2 is the **branch evaluator**: a scoring function that combines estimated novelty (distance from retrieved similar papers in embedding space) with estimated significance (magnitude of the experimental effect relative to baseline variance). This score determines which branches to expand in the tree search, which prevents the system from committing all compute to a single direction that might not pan out.

A minimal version of the branch evaluation logic:

```python
import numpy as np
from dataclasses import dataclass

@dataclass
class BranchEvaluation:
    novelty_score: float      # 0-1: 1 = maximally novel
    significance_score: float  # 0-1: 1 = effect clearly exceeds noise
    expected_value: float      # combination used for tree expansion decisions

def evaluate_branch(
    hypothesis: str,
    result: dict,
    paper_embeddings: np.ndarray,  # embeddings of N retrieved similar papers
    hypothesis_embedding: np.ndarray,
    baseline_variance: float,
) -> BranchEvaluation:
    """
    Evaluate a research branch for tree search expansion decisions.
    
    novelty_score: how far the hypothesis is from existing work 
                   (cosine distance in embedding space)
    significance_score: how clearly the experimental result exceeds noise
                        (effect size normalized by baseline variance)
    """
    # Novelty: cosine distance from nearest neighbor in paper embedding space
    # Normalize to unit sphere so dot product equals cosine similarity in [-1, 1]
    paper_norms = np.linalg.norm(paper_embeddings, axis=1, keepdims=True)
    hyp_norm = np.linalg.norm(hypothesis_embedding)
    paper_unit = paper_embeddings / (paper_norms + 1e-8)
    hyp_unit = hypothesis_embedding / (hyp_norm + 1e-8)
    similarities = paper_unit @ hyp_unit          # cosine similarities in [-1, 1]
    max_similarity = float(similarities.max())
    # Map cosine similarity [−1, 1] → novelty [0, 1]: similarity=1 ⟹ novelty=0
    novelty_score = (1.0 - max_similarity) / 2.0
    
    # Significance: Cohen's d — effect size normalized by within-condition noise
    effect_magnitude = abs(result["delta_metric"])
    effect_noise = result["std_over_seeds"]        # std of metric across seeds
    cohens_d = effect_magnitude / (effect_noise + 1e-8)
    significance_score = min(1.0, cohens_d / 2.0)  # d=2 → score=1 (large effect)
    
    # Weight novelty and significance equally (tunable)
    alpha = 0.5
    expected_value = alpha * novelty_score + (1 - alpha) * significance_score
    
    return BranchEvaluation(
        novelty_score=novelty_score,
        significance_score=significance_score,
        expected_value=expected_value,
    )
```

The branch evaluator is where most of the system's scientific judgment is encoded — and it's where the system's limitations are most visible. The novelty score measures distance from retrieved papers, but retrieval is imperfect: a hypothesis that's genuinely novel may score low because the embedding space doesn't capture the right similarity. The significance score measures effect size, but effect size isn't the same as scientific value: a large effect on a meaningless metric is not a contribution.

## The Accepted Workshop Paper: What the Reviewers Saw

The ICLR 2026 workshop acceptance exposes a calibration failure in peer review under current conditions. The reviewers scored the paper on standard criteria: novelty, soundness, clarity, significance. They didn't know it was machine-generated. Their scores placed it in the bottom quartile of accepted papers — the acceptance threshold for this workshop was relatively low — but it cleared the bar.

The paper's content was a narrow empirical result in neural architecture search: a small improvement on a benchmark task, with ablations that supported the mechanism claim. The writing was coherent. The experimental methodology was correct. The comparison to baselines was fair.

What was absent: any sense of why this particular experiment was worth running. The paper's "motivation" section reads like a competent summary of prior work without a genuine animating question. The contribution is real in the narrow sense — the numbers support the claim — but there's no argument for why the field should care about *this* problem, in *this* domain, at *this* time.

This isn't a flaw in the AI Scientist's implementation. It's a reflection of what the system optimizes for. The branch evaluator scores novelty (distance from existing papers) and significance (size of measured effect). It does not score *importance*, because importance is not computable from the experimental results alone. Importance requires understanding which problems the field is stuck on, which results would change how practitioners work, and which directions have been exhausted versus merely underexplored.

## The Automation Layer and What It Reveals

The AI Scientist v2's existence clarifies which parts of academic research have always been mechanical and which parts have been doing genuine intellectual work.

**Automated by v2:**

- Literature search and summarization
- Hypothesis reformulation from prior work gaps
- Experiment implementation (Python code, training loops, evaluation)
- Statistical analysis and result formatting
- Structured academic prose generation
- Response to reviewer comments (the v2 includes a revision loop)

These are non-trivial capabilities. PhD students spend significant time on all of them. The fact that they can be automated is evidence that they were, to a meaningful extent, executing on a learned template rather than exercising judgment.

**Not automated (and revealing why):**

- Selecting which problems are worth attacking
- Knowing when a line of research is exhausted vs. when you haven't looked in the right place
- Recognizing that a surprising result falsifies your prior belief rather than confirming it
- Deciding that an existing benchmark is poorly designed and that solving it doesn't constitute progress
- Understanding community norms well enough to know which results practitioners will act on

These require what you might call *scientific taste* — a calibrated prior over the space of questions worth asking, informed by deep immersion in a research community's live debates, failures, and open problems. This is built through reading, discussion, failed experiments, and time spent at the frontier — not through training on papers.

## Implications for Research Practice

The AI Scientist v2 changes research practice in three concrete ways, regardless of whether fully autonomous papers become common.

**Automated literature review becomes standard.** The system's literature ingestion — continuous monitoring of arXiv, classification, summarization, cross-linking — is immediately useful as a component even without the downstream hypothesis generation. Researchers who build pipelines that maintain structured, auto-updated knowledge bases of their subfield will have a significant information advantage over those relying on manual reading.

```python
# Minimal literature monitoring pipeline
import anthropic
from pathlib import Path

client = anthropic.Anthropic()

def ingest_paper_into_knowledge_base(
    paper_text: str,
    paper_id: str,
    kb_path: Path,
) -> None:
    """
    Read a new paper and update the structured knowledge base.
    Extracts: core contribution, methodology, experimental setup,
              claimed limitations, relationship to existing entries.
    """
    existing_entries = "\n\n".join(
        f.read_text() for f in kb_path.glob("*.md")
    )
    
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=2048,
        messages=[{
            "role": "user",
            "content": f"""Update the research knowledge base with this new paper.

Existing KB entries (summary):
{existing_entries[:8000]}  # truncated for context

New paper (ID: {paper_id}):
{paper_text[:12000]}

Output a structured Markdown entry covering:
1. Core contribution (1-2 sentences, specific and precise)
2. How it relates to existing KB entries (which does it extend, contradict, supersede?)
3. Experimental setup (dataset, baseline, metric)
4. Key numbers (results table as-is, not summarized)
5. Stated limitations
6. What this opens (what experiments become tractable that weren't before?)
"""
        }]
    )
    
    (kb_path / f"{paper_id}.md").write_text(response.content[0].text)
```

**Hypothesis generation as a thinking aid.** The most useful near-term application of AI Scientist-style systems is not autonomous paper writing — it's assisted hypothesis generation. A researcher who has spent years in a subfield has deep priors that the system lacks. But the system can enumerate hypotheses that the researcher's priors might filter out prematurely. Running the system's generation step on your current research question, then evaluating its outputs with your domain knowledge, catches genuinely novel directions that expert intuition would have dismissed.

**The evaluation problem becomes central.** If hypothesis generation and experiment execution are automated, the binding constraint on research throughput becomes evaluation design — knowing whether your experimental setup actually tests the hypothesis, and whether the metric you're measuring is the right proxy for the thing you care about. This is exactly the skill that the AI Scientist v2 cannot develop internally. It optimizes within an evaluation framework; it cannot evaluate evaluation frameworks.

## The Workshop Acceptance Is Not the Right Benchmark

The framing of "a machine-generated paper passed peer review" treats peer review as a ground-truth quality signal. It isn't. Peer review is a filter calibrated for a world where the supply of submissions is limited by human research bandwidth. That calibration breaks when submission supply is unconstrained.

The right benchmark for AI-generated research is not "did reviewers accept it?" but "did the result change how practitioners work?" That bar is much higher. Most human-authored workshop papers don't clear it either — but human researchers face bandwidth constraints that force some triage. Autonomous systems face no such constraint: they can generate a workshop paper per GPU-hour and submit to every venue simultaneously.

What actually matters: whether the results in those papers contain genuine new information about the world, as opposed to well-formatted empirical observations on narrow benchmarks. That question is answerable, but the answer requires engagement with the research community, longitudinal tracking of which results get built on, and domain judgment that no current automated system can exercise.

The AI Scientist v2 is a real technical achievement. It is also a diagnostic: it identifies precisely which parts of academic publication have always proceeded by template, and which parts actually required scientific judgment. That boundary is narrower than most researchers would have predicted before reading the workshop paper.
