操作代码仓库的基础模型和操作物理对象的基础模型，底层架构的共性超出大多数人的预期。两者都需要在不确定性下进行多步规划，都需要从失败中恢复，都处理多模态输入——编码 Agent 读文件和终端输出，机器人读摄像头画面和力传感器。核心推理循环是一样的：观察、规划、行动、验证。

区别在于失败的代价。编码 Agent 写了一个坏函数，可以 revert commit。机器人抓握判断失误，直接压碎目标物体。这种失败代价的不对称性解释了为什么具身 AI 落后数字 AI 大约两年——不是因为推理更难，而是因为验证循环更慢、更贵、而且有物理破坏性。

Google DeepMind 在 2026 年 4 月发布的 Gemini Robotics-ER 1.6，配合与 Boston Dynamics 的合作，是弥合这一差距最认真的尝试。

## Gemini Robotics-ER 的核心能力

"ER" 代表 Embodied Reasoning（具身推理），描述三项长期阻碍自主工业机器人的能力：

**1. 模拟仪表读取。** 模型能可靠地从摄像头图像读取模拟仪表、液位计和机械指示器。这比表面看起来更难——模拟仪表有非线性刻度、反光玻璃、视差畸变和环境污染。之前的方法需要逐仪表标定。ER 1.6 通过 zero-shot 推理处理任意仪表。

技术路径利用了基础模型预训练获得的对数字、刻度和空间关系的理解。读取"47.3 PSI"需要理解指针指向弧形上的位置、弧形有带数字标记的刻度线、刻度之间的插值映射到特定数值。

**2. 成功检测。** 模型通过对比操作前后的观察，判断物理动作是否达到预期目标。抓起螺栓——夹爪实际抓住了还是滑脱了？拧紧接头——扭矩达标了还是工具脱离了？

成功检测将开环机器人控制转变为闭环控制。这相当于编码 Agent 写完代码后跑 `pytest` 检查绿灯。

**3. 故障模式分类。** 当成功检测判定动作失败时，模型从故障分类法中判断原因：抓握滑脱、障碍物、对位不准、力不足、工具错误。分类驱动恢复策略。

## 数字 Agent 与物理 Agent 的桥梁

两种 Agent 都遵循四阶段循环：

```
观察 → 规划 → 行动 → 验证
  ↑                      ↓
  └──── 重试/适应 ───────┘
```

编码 Agent 的世界模型是代码仓库，行动空间是文件编辑。具身 Agent 的世界模型是物理场景，行动空间是电机指令。但推理架构相同。

这种收敛解释了 DeepMind 为什么选择在 Gemini 基础上构建 Robotics-ER，而不是从头训练机器人专用模型。规划和推理能力直接迁移。需要新增的是感知管道（实时处理摄像头和传感器输入）和行动空间（将高层意图映射到电机指令）。

## 模拟仪表读取揭示的视觉推理本质

传统 CV 方法读取仪表需要一条多阶段流水线：目标检测 → 实例分割 → 边缘检测 → OCR → 空间推理 → 角度到数值映射。每步都是独立的模型或算法，每步都可能独立失败。误差在各阶段间累积。

Gemini Robotics-ER 跳过了整条流水线。它用和描述照片相同的视觉-语言推理，整体性地处理图像。"这个仪表读数是多少？" 的回答方式和 "这个标牌写的什么？" 一样——通过集成的视觉-语言理解，而非分解的 CV 流水线。

性能差异不是增量的。传统流水线需要逐仪表类型标定，遇到陌生设计就失败。基础模型方法之所以能处理任意仪表，是因为预训练时见过数百万包含仪表、表盘和数字刻度的图像。

## 失败代价差距

最大的障碍不是推理能力，而是物理实验的代价。

编码 Agent 每小时能跑数千次实验。每次实验的成本是零点几美分的算力。反馈循环在毫秒到秒级。

具身 Agent 在受控实验室环境下每小时大约能跑 10-50 次实验，生产环境中更少。每次实验消耗分钟级的机器人时间，并有损坏设备、材料或机器人本身的风险。

这种不对称直接影响训练和评估。编码 Agent 可以在 SWE-bench 上评估——数千个真实世界任务，自动化验证。机器人 Agent 没有同等规模的基准测试。最接近的替代品（RoboSuite、Meta-World）使用模拟环境，无法完全捕捉真实世界操作的物理特性。

DeepMind-Boston Dynamics 合作的一部分赌注是：真实机器人数据比模拟数据更有价值。Boston Dynamics 的机器人在生产环境（仓库、巡检场地）中运行，在规模化地产生真实世界交互数据。

## 对基础模型研究的启示

**从语言到动作的迁移学习是真实的。** Gemini Robotics-ER 使用与 Gemini 3.1 Pro 相同的基础模型，通过微调而非从头训练，意味着语言模型的通用推理能力确实能迁移到物理领域。

**闭环验证是瓶颈，不是规划。** 成功检测能力可能比规划能力更重要。机器人几十年来都能规划操作序列。它们无法可靠做到的是判断计划是否真正执行成功。

**数字-物理收敛的时间线比预期更短。** 如果同一模型架构处理编码任务和操作任务，那一个领域的改进会迁移到另一个。代码生成能力的每一次提升，也在改善机器人规划的推理质量。
