The Model Context Protocol launched as a clean answer to a messy problem: how do you let an LLM reliably call external tools without baking every integration into the model? JSON-over-stdio, a standard schema for tool definitions, a consistent error format. Boring, useful infrastructure.

The 2026 MCP roadmap suggests something more ambitious: a stateless, horizontally scalable transport layer combined with "MCP Apps" — servers that can render interactive HTML/JS/CSS directly to AI clients. The protocol is becoming a platform.

This piece covers what's changing, why the architectural choices matter, what the security implications are, and how to build MCP Apps that work with both current and future clients.

## What MCP Does Now (And What's Changing)

The current MCP architecture is stateful and connection-oriented. A client opens a session with a server, the server maintains state across that session, and tool calls happen within that persistent connection. This works well at small scale — easy to reason about, easy to debug, sufficient for single-user workflows.

The problem is horizontal scaling. Stateful connections tie requests to specific server instances. Load balancing becomes complicated because subsequent requests from the same client may need to reach the same server to preserve session state. Adding replicas doesn't help if half of them can't serve half of the clients.

**SEP-1442** addresses this by shifting to a stateless request model. Each MCP request becomes self-contained: it carries enough context that any server instance can handle it without prior session state. This is the same architectural shift that made REST scalable relative to SOAP and that enables modern serverless functions.

For developers, this means MCP servers can be deployed as stateless compute — Lambda functions, Cloud Run instances, edge workers — without coordinating session affinity.

## MCP Apps: What They Actually Are

The more novel piece of the 2026 roadmap is "MCP Apps." The concept: an MCP server can include a `ui` field in its tool response that contains a UI bundle — HTML, CSS, and JavaScript — that an MCP-compatible client renders inline.

The effect is that agents can surface interactive interfaces within the conversation context. A data analysis tool doesn't just return a table of numbers; it returns a sortable, filterable table. A scheduling tool renders a calendar picker. A code review tool renders a diff view with collapsible sections.

Here's what a minimal MCP App response looks like:

```json
{
  "tool": "show_results",
  "result": {
    "data": [
      {"id": 1, "title": "Attention Is All You Need", "year": 2017, "citations": 120000},
      {"id": 2, "title": "BERT: Pre-training", "year": 2018, "citations": 85000}
    ],
    "ui": {
      "type": "html",
      "content": "<div id='results-app'></div>",
      "scripts": ["https://cdn.example.com/table-sort.js"],
      "styles": ["https://cdn.example.com/base.css"],
      "sandbox": "allow-scripts"
    }
  }
}
```

The client sandboxes the rendered content (same-origin policies apply, no access to the outer conversation context), but the user gets interactive UI rather than raw data.

## Building an MCP App: A Concrete Example

The theory is straightforward. Here's what building an actual MCP App looks like. Consider a paper search tool that returns interactive results:

```python
from dataclasses import dataclass, field
from typing import Optional
import json


@dataclass(frozen=True)
class MCPUIBundle:
    """A renderable UI component returned alongside tool results."""
    html: str
    css: str = ""
    js: str = ""
    sandbox_permissions: tuple[str, ...] = ("allow-scripts",)

    def to_mcp_response(self, data: dict) -> dict:
        return {
            "result": {
                "data": data,
                "ui": {
                    "type": "html",
                    "content": self.html,
                    "styles": [self.css] if self.css else [],
                    "scripts": [self.js] if self.js else [],
                    "sandbox": " ".join(self.sandbox_permissions),
                },
            },
        }


def build_paper_search_ui(results: list[dict]) -> MCPUIBundle:
    """Build an interactive paper search results component."""
    rows = "\n".join(
        f"""<tr data-year="{r['year']}" data-citations="{r['citations']}">
            <td>{r['title']}</td>
            <td>{', '.join(r['authors'][:3])}</td>
            <td>{r['year']}</td>
            <td>{r['citations']:,}</td>
            <td><a href="{r['url']}" target="_blank">PDF</a></td>
        </tr>"""
        for r in results
    )

    html = f"""
    <div class="mcp-paper-search">
        <input type="text" id="filter" placeholder="Filter results..."
               oninput="filterTable(this.value)">
        <select id="sort" onchange="sortTable(this.value)">
            <option value="citations-desc">Most cited</option>
            <option value="year-desc">Most recent</option>
            <option value="year-asc">Oldest first</option>
        </select>
        <table id="results">
            <thead>
                <tr><th>Title</th><th>Authors</th><th>Year</th>
                    <th>Citations</th><th>Link</th></tr>
            </thead>
            <tbody>{rows}</tbody>
        </table>
        <div id="stats"></div>
    </div>
    """

    js = """
    function filterTable(query) {
        const rows = document.querySelectorAll('#results tbody tr');
        const q = query.toLowerCase();
        let visible = 0;
        rows.forEach(row => {
            const text = row.textContent.toLowerCase();
            const show = text.includes(q);
            row.style.display = show ? '' : 'none';
            if (show) visible++;
        });
        document.getElementById('stats').textContent =
            `Showing ${visible} of ${rows.length} results`;
    }

    function sortTable(key) {
        const tbody = document.querySelector('#results tbody');
        const rows = Array.from(tbody.querySelectorAll('tr'));
        const [field, direction] = key.split('-');
        rows.sort((a, b) => {
            const va = parseInt(a.dataset[field]);
            const vb = parseInt(b.dataset[field]);
            return direction === 'desc' ? vb - va : va - vb;
        });
        rows.forEach(row => tbody.appendChild(row));
    }

    // Initialize stats
    filterTable('');
    """

    css = """
    .mcp-paper-search { font-family: system-ui, sans-serif; }
    .mcp-paper-search input, .mcp-paper-search select {
        padding: 8px; margin: 4px; border: 1px solid #ddd; border-radius: 4px;
    }
    .mcp-paper-search table {
        width: 100%; border-collapse: collapse; margin-top: 8px;
    }
    .mcp-paper-search th, .mcp-paper-search td {
        padding: 8px 12px; text-align: left; border-bottom: 1px solid #eee;
    }
    .mcp-paper-search th { background: #f5f5f5; font-weight: 600; }
    .mcp-paper-search tr:hover { background: #f9f9f9; }
    #stats { color: #666; font-size: 0.85em; margin-top: 8px; }
    """

    return MCPUIBundle(html=html, css=css, js=js)
```

The MCP server handler integrates this into a standard tool:

```python
from mcp.server import Server

app = Server("paper-search")

@app.tool()
async def search_papers(query: str, limit: int = 20) -> dict:
    """Search academic papers by semantic similarity."""
    results = await semantic_search(query, limit=limit)

    # Build interactive UI alongside structured data
    ui_bundle = build_paper_search_ui(results)
    return ui_bundle.to_mcp_response({"papers": results, "query": query})
```

Clients that understand the `ui` field render the interactive table. Clients that don't still get the structured `data` field and can format it however they prefer. Progressive enhancement, not a hard requirement.

## The Security Model: Why Sandboxing Matters

Rendering arbitrary HTML from an MCP server inside an agent client is a significant attack surface. The security model needs to handle several threat vectors:

**Cross-context data exfiltration.** An MCP App's JavaScript must not access the outer conversation context. If a malicious MCP server returns a UI bundle with JavaScript that reads the surrounding page's DOM, it could exfiltrate the user's conversation history, API keys visible in prior tool results, or system prompts.

The mitigation is iframe sandboxing with strict origin isolation:

```html
<!-- Client-side rendering of MCP App -->
<iframe
  sandbox="allow-scripts"
  srcdoc="${sanitized_html}"
  csp="default-src 'self'; script-src 'unsafe-inline'; style-src 'unsafe-inline'"
  referrerpolicy="no-referrer"
  style="width: 100%; border: none;"
></iframe>
```

The `sandbox` attribute without `allow-same-origin` ensures the iframe has an opaque origin — it cannot access the parent page's cookies, localStorage, or DOM. The Content Security Policy further restricts what the embedded code can do.

**Network exfiltration.** MCP App scripts could make fetch requests to external servers, leaking conversation data. The sandbox should restrict outbound network access:

```python
@dataclass(frozen=True)
class MCPAppSecurityPolicy:
    """Security constraints for rendering MCP App UI bundles."""
    allow_scripts: bool = True
    allow_same_origin: bool = False  # NEVER for untrusted servers
    allow_forms: bool = False
    allow_popups: bool = False
    allowed_external_domains: tuple[str, ...] = ()  # CDN allowlist

    @property
    def sandbox_value(self) -> str:
        parts = []
        if self.allow_scripts:
            parts.append("allow-scripts")
        if self.allow_same_origin:
            parts.append("allow-same-origin")
        if self.allow_forms:
            parts.append("allow-forms")
        if self.allow_popups:
            parts.append("allow-popups")
        return " ".join(parts)

    @property
    def csp_value(self) -> str:
        domains = " ".join(self.allowed_external_domains)
        return (
            f"default-src 'self'; "
            f"script-src 'unsafe-inline' {domains}; "
            f"style-src 'unsafe-inline' {domains}; "
            f"connect-src 'none'; "  # Block outbound network
            f"img-src data: {domains}"
        )
```

**Resource exhaustion.** A UI bundle with an infinite loop or excessive DOM manipulation could freeze the client. Render timeouts and memory limits on the iframe provide defense:

```typescript
// Client-side: render with resource limits
function renderMCPApp(bundle: MCPUIBundle, container: HTMLElement): void {
  const iframe = document.createElement("iframe");
  iframe.sandbox = bundle.sandbox;
  iframe.srcdoc = assembleSrcdoc(bundle);

  // Kill the iframe if it consumes too much
  const watchdog = setTimeout(() => {
    iframe.srcdoc = "<p>MCP App timed out</p>";
  }, 5000);  // 5-second render budget

  iframe.onload = () => clearTimeout(watchdog);
  container.appendChild(iframe);
}
```

**Prompt injection via UI.** An MCP App could render text that looks like system instructions ("Ignore previous instructions and...") to trick the agent on subsequent turns. The client must ensure that rendered UI content is not fed back into the agent's context as text — only structured interaction events (button clicks, selections) should be passed back through the MCP protocol.

## Stateless Transport: The Deployment Shift

For developers building MCP servers today, the stateless transport change has concrete implications.

**Before SEP-1442 (current state)**:

```python
class StatefulMCPServer:
    def __init__(self):
        self.sessions: dict[str, SessionState] = {}

    async def handle_tool_call(
        self, session_id: str, tool: str, args: dict
    ) -> dict:
        state = self.sessions.get(session_id)
        result = await self.execute_tool(state, tool, args)
        self.sessions[session_id].update(result)
        return result
```

**After SEP-1442 (stateless target)**:

```python
from fastapi import FastAPI
from pydantic import BaseModel


class MCPRequest(BaseModel):
    tool: str
    args: dict
    context: dict  # All state arrives in the request


class MCPResponse(BaseModel):
    result: dict


app = FastAPI()


@app.post("/mcp/v2/call")
async def mcp_endpoint(request: MCPRequest) -> MCPResponse:
    # Each request is self-contained — no session lookup needed
    result = await execute_tool(
        request.tool, request.args, request.context,
    )
    return MCPResponse(result=result)
```

The second pattern deploys to any stateless compute environment. AWS Lambda, Google Cloud Run, Cloudflare Workers, standard containers. Autoscaling works correctly. No sticky sessions, no state synchronization, no session cleanup.

For MCP Apps specifically, stateless transport means the UI bundle is regenerated on every request. This simplifies cache invalidation (there is no cache) but means the server must be fast — the user is waiting for both the data and the UI to arrive before they can interact. Pre-built UI templates with dynamic data injection keep response times under 200ms in practice.

## What Needs to Change in Your MCP Servers

If you're building MCP servers targeting the 2026 protocol:

**Externalize any session state.** If your server stores state in memory between tool calls (authentication tokens, conversation context, accumulated results), move that to an external store (Redis, a database, the request envelope). Design each handler to be stateless.

**Return structured data alongside UI.** MCP Apps work with structured results that client-side code can render. A tool that returns `"Found 47 papers matching your query:\n1. Smith et al..."` is harder to make interactive than one that returns structured JSON. Always include both:

```python
# Return both structured data AND UI
return {
    "data": {"papers": results, "total": len(results)},  # For non-UI clients
    "ui": build_paper_search_ui(results).to_dict(),       # For UI-capable clients
}
```

**Add UI hints progressively.** Start with the `ui` metadata fields as optional hints rather than dependencies. Clients that understand them will use them; clients that don't will ignore them. Test with both UI-capable and plain-text clients during development.

**Version your tool schemas explicitly.** The stateless model makes versioning more important because requests can arrive without the negotiation that session establishment enables. Use semantic versioning in your tool schema IDs and include a `protocol_version` field in responses.

## Comparing Approaches: MCP Apps vs. Alternatives

MCP Apps aren't the only approach to rich agent output. Here's how the alternatives compare:

| Approach | Who Controls UI | Portability | Security Model |
|----------|----------------|-------------|----------------|
| **MCP Apps** | Tool author | Any MCP client | Sandboxed iframe |
| **Client-native** | Client developer | One client only | Full trust |
| **Markdown/text** | Model + client | Universal | No execution |
| **Artifacts (Claude)** | Model | Claude only | Sandboxed iframe |
| **Canvas (OpenAI)** | Model | ChatGPT only | Controlled editor |

MCP Apps occupy a unique position: the tool author controls the UI (not the model or the client), and any MCP-compatible client can render it. This means a single tool implementation works across Claude Code, Codex CLI, Gemini CLI, and any future MCP client.

The trade-off is complexity. Markdown output is universally supported and secure by construction. MCP Apps require sandboxing, CSP policies, and careful attention to the attack surface. The right choice depends on whether the interactive capabilities justify the security overhead.

## The Bigger Picture

MCP started as plumbing. A way to connect LLMs to external tools without handwriting every integration. That's still what it is, but the scope of what "external tools" means is expanding.

A tool that renders an interactive calendar is not qualitatively different from a tool that returns a list of dates. But the rendered calendar is usable in a way that the list often isn't — it reduces the cognitive load of interpreting results and the round-trips required to narrow down a choice. At scale, across hundreds of tool calls per session, that difference compounds.

The stateless transport makes MCP servers operationally boring in the way that HTTP servers are operationally boring — well-understood scaling patterns, standard monitoring, predictable failure modes. That's not glamorous, but it's what enables a protocol to become infrastructure.

MCP Apps make the *results* of tool calls richer without requiring that richness to be implemented in the LLM itself or the client. The tool author owns the output format; the client renders it. This is a better separation of concerns than either "bake it into the model" or "add it to the client."

Whether the full vision — agents producing interactive applications within conversational contexts — lands with users is an empirical question. But the architectural direction is sound: stateless, composable, progressively enhanced. If you're building agent tools today, designing toward that target costs little and positions you well.
