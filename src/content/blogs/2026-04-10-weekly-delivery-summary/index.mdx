## 交付成果一：Beatless v4 架构详解

**文章链接**：[/blog/beatless-architecture](/blog/beatless-architecture)

### 核心创新点

Beatless v4 围绕一个根本性问题而设计：**单会话 AI 的状态丢失与上下文碎片化**。面对跨天任务、多 Agent 协作和错误恢复需求，传统对话模式无法胜任。v4 的架构回应了这一问题：

**1. Agent Mailbox 协议**
- Agent 间通信通过原子化 JSONL 邮箱传递，避免共享内存竞争
- 支持异步投递与优先级队列，解耦发送方与接收方

**2. 控制平面（Control Plane）**
- 将 Agent 行为抽象为四类动作：`assist / silent / escalate / rollback`
- 基于任务状态机驱动调度，而非依赖 LLM 即时判断
- 明确引入成本函数：误报打扰、漏报风险、token 消耗三目标联合优化

**3. 上下文管理**
- `TASKS.yaml` 作为唯一任务真相源（v4 schema），所有 Agent 从此同步状态
- 分层上下文压缩：活跃任务保留完整历史，归档任务仅保留摘要
- 30 分钟心跳周期触发状态快照，保证跨会话连续性

这三项机制共同将 Beatless 从"个人自动化脚本"升级为一个具备持久状态、可推理调度的轻量 Agent OS。

## 交付成果二：2026 研究路线图

**文章链接**：[/blog/multi-agent-research-routes-2026](/blog/multi-agent-research-routes-2026)

### 三条路线优先级

从 Beatless 的工程实践中提炼出三个具备学术价值的研究问题，优先级排序如下：

| 优先级 | 路线 | 核心题目 |
|--------|------|----------|
| 🥇 最高 | 路线1：主动控制平面 | Budgeted Proactive Control for Persistent Agents |
| 🥈 次高 | 路线2：上下文价值估计 | Context Value Estimation for Long-Horizon Agents |
| 🥉 第三 | 路线3：通信/回滚协议 | Communication & Rollback Protocol for Multi-Agent Systems |

**路线1（最优先）** 的核心判断：控制平面问题系统性最强，指标涵盖成功率与一致性，契合顶会偏好的"真实环境评测 + 固定 backbone 系统增益"范式，而非空泛的 AGI 叙事。

**路线2** 稳定性好、资源需求低，适合作为路线1的补充研究，或在路线1推进受阻时的备选。

**路线3** 工程味浓厚，更适合系统会议（如 OSDI/ATC），学术新颖性相对受限。
