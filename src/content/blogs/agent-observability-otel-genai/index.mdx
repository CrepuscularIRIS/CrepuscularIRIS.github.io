When an autonomous agent makes a decision you wish it hadn't, the first thing you reach for is the trace. The expectation, inherited from a decade of microservice debugging, is that the trace will tell you which service called which, with what arguments, in what order, and where time went. OpenTelemetry's GenAI semantic conventions extend the same model to LLM and agent workloads — `gen_ai.system`, `gen_ai.operation.name`, `gen_ai.request.model`, plus token counts and cost. On the surface, an LLM call looks like an HTTP call: a span, attributes, a duration, a status.

The places where the analogy breaks down are the places where agent observability is genuinely a different problem. A failed HTTP call has a small set of well-understood failure modes — timeout, 5xx, schema mismatch — and each one points to a localized fix. A failed LLM call can be syntactically successful (200, valid JSON, non-empty output) and semantically wrong in ways the trace shape was never designed to surface. The interesting work in agent observability happens in the gap between "the call returned" and "the call was correct."

This post walks through three failure modes that microservice-style tracing handles poorly, what the OpenTelemetry GenAI conventions get right and wrong, and what an agent trace actually has to capture to be useful for debugging.

## What Microservice Tracing Assumes

Distributed tracing assumes a few things that are mostly true for HTTP-based microservices:

- A request can be uniquely identified, follows a deterministic path through services, and has a clear "started" / "finished" boundary.
- The interesting failure modes are localized to a span — a service that timed out, a database that returned an error, a downstream API that 500'd.
- Causality runs through service boundaries, captured by `traceparent` headers and parent/child span relationships.
- Re-running the same input produces the same output (modulo intentional non-determinism).

LLM calls violate every one of these in ways that matter for debugging. The same prompt produces different outputs across runs (and not just for sampling reasons — provider-side updates change behavior silently). The "interesting failure" is rarely localized to one span — it's that the model hallucinated a function name three spans back, and the entire downstream chain is well-formed garbage built on top of that hallucination. The causality that matters is *semantic* (this output references that input) rather than *temporal* (this span happened after that span).

This is not OpenTelemetry's fault. The conventions were designed for the world where they grew up. The question is what additional structure agent observability needs on top.

## Failure Mode 1: The Successful Hallucination

An agent calls a tool with a function name and arguments. The trace shows a clean span: `tools/call`, name `get_user_profile`, arguments `{"user_id": "u_8a7f2"}`, duration 47ms, status `success`. The downstream span shows the tool returning a 404. The agent then calls a different tool. The user complains that the agent could not find their profile.

What the trace does not show: the user's actual ID was `u_8a7e2`, with an `e` instead of an `f`, and the agent generated the wrong ID by half-correctly transcribing it from a previous turn. The 404 is not a service failure — the service is working perfectly. The hallucination happened inside the LLM, three steps before the tool call, and it was structurally indistinguishable from a correct call until the tool actually had to look the ID up.

OpenTelemetry GenAI captures the prompt and completion as span attributes (or, depending on configuration, as separate `gen_ai.content.prompt` events). What it does not capture is the *provenance* of values inside the completion. There is no native way to say "this `user_id` token came from attention over tokens X, Y, Z in the input." Provenance attribution at the token level is hard for the model itself to expose, and OTel was not designed to ask for it.

The practical workaround is to make the harness, not the model, responsible for provenance. When a tool argument is generated by the model, the harness can require it to be a verbatim copy of a value present in the input context, rejecting any tool call where the argument is novel. This is the same defense as parameterized SQL queries — it removes a class of failures by removing the model's ability to invent the values. The trace then carries an attribute — `agent.tool.argument_source: "input.message[3].user_id"` — and a hallucinated argument is caught at the harness, not three calls later.

## Failure Mode 2: The Causal Chain Across Sessions

A user reports that an agent, asked the same question on Monday and Tuesday, gave subtly different answers. The Monday trace and the Tuesday trace are both clean — same prompt, same tools, same model, no errors. The difference comes from a third trace from Sunday, in which the agent updated a file the model treats as long-term memory. Tuesday's answer reflects the Sunday update; Monday's does not, because the agent ran before Sunday.

This is causality across sessions, and microservice tracing has no concept of it. Spans do not have a relationship called "this answer depends on a memory written by another trace last week." OpenTelemetry's `link` mechanism — which attaches one span to other spans by reference — is the right primitive but is rarely used for this purpose, partly because there is no convention for *which* prior trace is the relevant one.

The agent observability fix is to log every memory read and write as an explicit span with a stable identifier for the memory artifact (file path, key, embedding ID), and to link a current span to the most recent write span for any memory it reads. This requires the harness to maintain a memory-revision graph at the storage layer, which is a separate component most agent stacks lack. The cost is real: an extra storage write per memory read, plus the index needed to look up the latest writer. The payoff is that "why did this answer change?" becomes a graph traversal instead of a forensic guess.

The OTel GenAI conventions are silent on long-term memory. There is no `gen_ai.memory.*` namespace, no convention for memory-revision IDs, no defined way to express the link from a read to its writer. This is one of the largest gaps in the standard relative to what production agent systems actually need. Filling it requires either an extension to the conventions or a domain-specific overlay (Langfuse, Phoenix, Arize all do this differently). The lack of a standard means trace data is not portable across observability vendors, which is exactly the problem OTel was created to solve.

## Failure Mode 3: The Cost of Being Right

An agent solves a multi-step task correctly. The trace is clean: 47 LLM calls, 12 tool calls, no errors, total wall time 8 minutes. The cost report at the end of the month says the agent ate $14 of API credit on this single task, when a similar task last week cost $0.60. Nothing in the trace, as designed, makes the cost spike legible.

Cost in microservice land is largely about CPU and bandwidth, both of which are roughly proportional to spans and durations — so they show up in the trace shape itself. Cost in LLM land is dominated by tokens, which are not visible in span durations. A span that takes 200ms might cost $0.002 or $0.20 depending on context length, output length, and model tier. OTel GenAI does capture token counts (`gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`) and a cost attribute exists in the latest revisions, but these are per-span. They do not aggregate naturally into "what was the cost contribution of this sub-task to the overall cost?" the way span durations aggregate naturally into critical-path analysis.

The trace structure that helps here is one that reflects the *agent's* decomposition, not the *LLM's* call structure. If an agent has a planning sub-task, an execution sub-task, and a verification sub-task, the spans should nest accordingly, with cost attributes rolled up at each level. This is what the `gen_ai.agent.*` conventions are starting to address — `gen_ai.agent.id`, `gen_ai.agent.name`, span hierarchies that reflect agent boundaries rather than just LLM call boundaries — but the conventions are still maturing. Production deployments today often add their own per-agent span groups on top.

## What OTel GenAI Gets Right

The criticisms above should not obscure what the conventions get correct. The biggest correct call is keeping prompts and completions out of the default span attributes (they are sent as separate events, opt-in). This avoids the privacy and storage disaster of putting potentially-PII content into a sampled-and-stored trace by default. OTel learned the lesson from HTTP body capture in earlier conventions, and that lesson transferred cleanly.

The second correct call is the namespace structure. `gen_ai.system` (e.g., `anthropic`, `openai`, `gemini`), `gen_ai.request.model`, `gen_ai.operation.name` (`chat`, `text_completion`, `embeddings`, `tool.execute`) — these are exactly the attributes you want to filter on when something breaks. They make queries like "show me all tool failures in the last hour for the `production-coding-agent` system" possible without per-vendor query rewrites.

The third correct call is the explicit acknowledgment that conventions are still evolving. The GenAI subgroup publishes a clear "experimental" / "stable" status per attribute, and the cadence of revisions reflects how new the field is. This is the right posture for a domain where the underlying technology shipped major changes between drafting one revision of the conventions and merging it.

## What an Agent Trace Should Capture

Stepping back, an agent trace useful for production debugging needs at least these layers:

**Call layer.** Every LLM call, tool call, and external HTTP request, with model, parameters, token counts, latency, cost. OTel GenAI handles this well. This is the floor.

**Decision layer.** Every branching point the agent took — which tool it picked, which sub-agent it delegated to, which retry path it followed. The hierarchy of spans should reflect the agent's mental model of its own work, not the raw LLM call pattern. This is where vendor-specific extensions live today.

**Provenance layer.** For every value used in a downstream call, where did it come from? Verbatim copy from input? Generated by model? Retrieved from memory? Constructed via templating? OTel does not standardize this, but it is the layer that catches successful hallucinations.

**Memory layer.** Every read and write of long-term state, with a stable artifact ID and a link from reads to the most recent corresponding write. This is the cross-session causality mentioned above. Mostly absent from current standards.

**Cost layer.** Token counts and dollar costs aggregated up the agent's decomposition hierarchy, not just at the call level. Required for cost regression detection — the "why did this task suddenly cost 20x more?" investigation.

A trace stack that captures all five of these is debuggable. A stack that captures only the first one is not. Most production agent observability today sits somewhere in the middle, with vendors filling in the missing layers in incompatible ways.

## The Convergence Question

The interesting structural question is whether agent observability will end up converging on OTel-with-extensions, or whether it will fragment into vendor-specific stacks (Langfuse, Phoenix, Arize, Helicone, Datadog LLM) that are interoperable only through lossy export formats. The microservice analogy is informative: distributed tracing fragmented for years (Zipkin, Jaeger, Datadog APM, NewRelic) before OTel became the lingua franca. The pressure for convergence came from teams running multi-vendor stacks who could not afford to maintain N tracing pipelines.

Agent stacks are already multi-vendor. A single Claude Code session might call Anthropic for planning, OpenAI for a Codex review pass, Gemini for a long-context research dive, plus arbitrary tools wrapped in MCP servers. The case for an OTel-style standard is strong on the same grounds it was strong for HTTP services: nobody can afford to maintain per-vendor observability pipelines, and no single vendor's stack covers the full execution path.

The case against — and the reason adoption has been slower than HTTP-services-OTel — is that agent observability has more domain-specific structure (memory, provenance, cost, decisions) that the GenAI conventions are still figuring out how to express portably. The standard is a moving target, and vendors are building production tools faster than the standard can stabilize. This is the same dynamic that played out in container orchestration in 2015 — innovation outran standardization for several years before convergence happened.

## Closing: Trace First-Principles, Not by Analogy

The temptation, when adding observability to an agent system, is to treat it as a microservice with extra metadata. That gets you the floor — call records, token counts, basic cost — but misses the failure modes that are actually expensive in production: silent hallucinations, cross-session causality, cost regressions hidden by clean traces.

A useful agent trace is structured around the *agent's* decomposition of the work, with explicit provenance for every generated value, memory-revision links across sessions, and cost roll-ups that match the agent's mental model. OTel GenAI is converging on the right primitives for the call and decision layers, lagging on memory and provenance, and silent on the cost-aggregation problem. Production teams need to fill the gaps until the standard catches up, ideally in ways that future-proof against the standard catching up.

The general lesson is one we will see again as autonomous systems mature: borrowing observability primitives from a previous generation of infrastructure gets you most of the way, but the failure modes that actually bite are the ones the previous generation never had. Microservices never had to debug a successful hallucination. Agent stacks have to.
