AI 工程中反复出现的模式：小规模有效的方案，在放大规模后越来越难以管理。更多参数改善语言模型，直到不再改善。更多数据有用，直到引入噪声。更多智能体——当前的前沿——被认为通过并行化智能来提升任务性能。

但实际上很少能提升。2025年底和2026年初成熟的一系列关于智能体系统扩展的理论研究——以《迈向扩展智能体系统的科学》及其配套的 Google Research 评论为基础——形式化了一个任何部署过多智能体系统的人从日志中已经知道的事实：五个智能体回答一个问题，并不会给你五个独立的尝试机会。它们给出的结果大致相当于**一个智能体的判断，被投票了五次**，而且相关错误被放大成了虚假共识。本文从第一性原理出发，逐步推导这个论点，将理论主张与实际基准测试中的经验曲线联系起来，并提出真正让多智能体扩展发挥作用的架构原则。

## 核心论点

这个结论可以简单表述：除非向系统中引入新的外生信息，否则 **N 个智能体的团队通常被一个设计良好的单一智能体所支配**。这里的"支配"有精确含义——在相关错误机制下，集体输出的期望效用被团队中最佳个体智能体的期望效用所上界，除非存在某种机制能真正聚合独立信息。

这对于一直在构建多智能体管道的人来说是不舒服的，但这种不适感是正确的反应。数学推导直接来自两个第一性原理，这两个原理在智能体文献之外是众所周知的，但很少被应用到其中。

## 为什么独立智能体无法聚合智能

考虑当你向五个智能体发送同一个问题，然后聚合它们的输出时会发生什么。聚合只有在智能体提供**真正独立的信号**时才能创造价值。这里的独立性是信息论意义上的：智能体的推理路径必须具有低互信息，实际上是对同一个底层问题运行不同的随机实验。

在实践中，它们做不到。同一家族的语言模型共享训练分布。面对相同的问题陈述，即使加入温度多样性，它们倾向于从相似的输出分布区域采样。随机解码会扰动轨迹，但很少改变全局后验——模型不太可能跳到定性上不同的答案模式。结果是**相关错误**，而非独立估计。

对于每个智能体准确率 $p > 0.5$ 且成对错误相关性 $\rho$ 的智能体，多数投票下的集成准确率由集成的有效样本量决定。在可交换的成对相关性下，标准一阶近似是：

$$N_{\text{eff}} \approx \frac{N}{1 + (N-1)\rho}$$

当 $\rho = 0$ 时，$N_{\text{eff}} = N$，你恢复了经典的孔多塞陪审团定理——准确率随着投票者增加趋近于 1。当 $\rho \to 1$ 时，$N_{\text{eff}} \to 1$，与 $N$ 无关：你有 N 次重复的一张投票。中间机制才是有趣的——而这正是你实际运行五个 Claude 实例或五个 GPT-5 实例处理同一问题时所处的状态。

```python
import numpy as np
from scipy.stats import binom, norm


def majority_vote_accuracy(n_agents: int, per_agent_accuracy: float,
                           correlation: float) -> float:
    """
    Estimate majority vote accuracy under pairwise error correlation.

    rho=0: exact binomial (Condorcet jury theorem).
    rho>0: effective-sample-size approximation under exchangeable
           pairwise correlations. Plug N_eff into a normal approximation
           of the binomial. Underestimates at small N; the qualitative
           plateau behavior is the point.
    """
    p = per_agent_accuracy
    if n_agents == 1:
        return p

    if correlation == 0.0:
        return float(binom.sf(n_agents // 2, n_agents, p))

    n_eff = n_agents / (1 + (n_agents - 1) * correlation)
    mean = n_eff * p
    std = np.sqrt(n_eff * p * (1 - p))
    return float(1 - norm.cdf(n_eff / 2, loc=mean, scale=std))


header = f"{'N':>4} | {'rho=0 (exact)':>14} | {'rho=0.3':>8} | {'rho=0.7':>8}"
print(header)
print("-" * len(header))
for n in [1, 3, 5, 10, 20, 50]:
    print(f"{n:>4} | {majority_vote_accuracy(n, 0.7, 0.0):>14.3f} "
          f"| {majority_vote_accuracy(n, 0.7, 0.3):>8.3f} "
          f"| {majority_vote_accuracy(n, 0.7, 0.7):>8.3f}")

# Indicative output:
# rho=0 rises from 0.70 at N=1 to ~0.99 at N=50
# rho=0.3 saturates near 0.88 by N=10 and moves little after
# rho=0.7 plateaus around 0.80 by N=5 — ~14% lift over a single agent,
# and every additional agent costs an API call for no benefit.
```

这些曲线是朴素多智能体系统的定义性经验事实。当 $\rho = 0$（独立评判者）时，从 $N=1$ 扩展到 $N=20$ 将准确率从约 70% 提升到约 95%。在同家族 LLM 集成典型的高相关性状态下，同样的扩展勉强超过 80%，曲线在 $N=5$ 时几乎平坦。这与文献中大多数多智能体消融表停在 $N=3$ 或 $N=5$ 的观察一致——超过该点后收益落在噪声范围内。

## 虚假共识放大问题

独立性失败会演变成更糟糕的问题：相关智能体可能以高于任何个体智能体表达的置信度，集体收敛于错误答案。

这就是虚假共识失败模式。当五个智能体犯下相同的推理错误——这正是相关样本的行为——聚合会将一致性解读为高置信度信号。多数投票、LLM 综合器或平均值都将一致视为证据。但这种一致是虚假的。智能体们不是从独立先验收敛；它们是同一后验的样本。它们的共同错误在投票计数开始前就已经被烘焙进去了。

```python
# Illustrative. Requires the `anthropic` package and ANTHROPIC_API_KEY set in
# the environment. The point is the aggregation shape, not the transport.
from collections import Counter
from dataclasses import dataclass
from typing import Callable

PROMPT_TEMPLATE = (
    "{question}\n\n"
    "Respond with exactly two lines:\n"
    "ANSWER: <your final answer>\n"
    "CONFIDENCE: <float between 0 and 1>"
)


@dataclass(frozen=True)
class AgentVote:
    answer: str
    confidence: float


def parse_vote(raw: str) -> AgentVote:
    """Extract a structured answer from an LLM response.

    Using a structured schema instead of substring hashing avoids fake
    agreement from minor surface variations in otherwise-identical answers.
    """
    answer = ""
    confidence = 0.0
    for line in raw.splitlines():
        if line.startswith("ANSWER:"):
            answer = line[len("ANSWER:"):].strip().lower()
        elif line.startswith("CONFIDENCE:"):
            try:
                confidence = float(line[len("CONFIDENCE:"):].strip())
            except ValueError:
                confidence = 0.0
    return AgentVote(answer=answer, confidence=confidence)


def run_agent_panel(
    question: str,
    n_agents: int,
    call_llm: Callable[[str], str],
) -> dict:
    """Run N instances of the same model on the same prompt.

    Illustrates how apparent agreement inflates confidence when agents are
    correlated. The returned `agreement_rate` should NOT be treated as a
    calibrated confidence in a correlated ensemble — see discussion below.
    """
    votes = [parse_vote(call_llm(PROMPT_TEMPLATE.format(question=question)))
             for _ in range(n_agents)]

    tally = Counter(v.answer for v in votes if v.answer)
    if not tally:
        return {"majority_answer": None, "agreement_rate": 0.0}

    majority, agree_count = tally.most_common(1)[0]
    return {
        "majority_answer": majority,
        "agreement_rate": agree_count / n_agents,
        "n_agreeing": agree_count,
        "total_votes": n_agents,
        # WARNING: agreement_rate is NOT calibrated confidence. With rho > 0
        # correlated agents produce inflated agreement that systematically
        # overestimates ground-truth accuracy.
    }
```

聚合层没有办法原生地区分"五个智能体因为独立计算得到相同答案而一致"和"五个智能体因为是同一有缺陷分布的五个样本而一致"。因此，原始一致率对于多智能体管道来说是误导性的质量指标。2023-2024 年论文中广泛报道的"自我一致性"收益之所以有效，是因为底层问题有良好结构的答案空间（算术、多选题），即使相关样本也能通过多数投票自我纠正。在开放式推理任务上，同样的技术很快达到瓶颈，甚至可能反转——集成变得比单个智能体**更有把握地确信错误答案**。

## 2025-2026 年的经验图景

理论是锐利的。经验图景更锐利——不过在这里，比在任何其他地方，具体的数字更应该被视为数量级的估计，而非精确值。过去一年发表的消融研究中有三个反复出现的模式。

1. **同家族集成早早达到瓶颈。** 在 SWE-bench Verified 和 GPQA 等推理密集型基准上，单一模型家族抽取的集成往往在少数几个智能体后停止改善。将 $N$ 推到 10-20 范围的论文通常报告与适度 $N$ 相比没有统计上有意义的收益，偶尔还有回退。同家族模型智能体之间的成对错误相关性（通过错误集重叠测量）舒适地高于零，在最难的项目上接近一。

2. **跨家族有帮助，但有限。** 混合 Claude 与 GPT- 和 Gemini 系列模型会降低相关性并提升瓶颈。混合家族集成通常在困难推理上优于任何单一组成模型，但观察到的提升是几个百分点——远低于朴素陪审团计算对完全独立评判者预测的双位数提升。

3. **验证器不对称胜过陪审团对称。** 搭配代码执行验证器和文档搜索验证器的单一生成器，在类似总成本下胜过类似规模的对称集成。这是编码智能体堆栈收敛的模式：写、运行、观察、修改。收益会复合，因为验证器与生成器**处于不同的信息类别**，所以它们的信号天生独立。

精确的百分比不如曲线形态重要。朴素并行化很快饱和，跨家族集成略有帮助，架构不对称才是复合收益所在。

## 多智能体系统真正有效的时候

MIT 的分析不是说多智能体系统无用。它是说它们需要一个特定属性才能产生价值：**外生信息注入**。能为问题带来不同上下文的不同智能体——不同的工具访问、不同的知识库、不同的专业化，或 ground-truth 验证器——才能产生真正的独立性。这条原则区分了有效的多智能体设计与朴素并行化。

有三种值得命名的真正独立性来源，因为架构工作在于识别候选设计提供了哪一种。

**专业化作为独立性来源。** 有代码库读取权限的智能体和有 API 文档读取权限的智能体有不同的信息足迹。它们在跨越两者的问题上的推理将是真正独立的，因为任何一方都不能直接产生另一方的证据。LangGraph 的状态机模型利用了这一点——智能体操作共享状态的不同切片，并贡献其他智能体无法观察到的信息。

**外部验证作为独立性来源。** 运行代码并返回具体输出的代码执行智能体提供的信号与生成器的语言分布不相关。输出不是来自任何 LLM 的 token 后验；它来自执行环境。这就是为什么生成器-验证器模式（AlphaProof 的定理验证器、o-series 的测试时验证器、带单元测试的 best-of-N）比集成不受约束的生成器效果更好。验证器完全处于不同的信息类别。

**角色不对称作为独立性来源。** 当同一模型被部署在不同角色——写作者、批评者、规划者、执行者——每个角色的输出分布都由其角色提示和角色特定上下文塑造。角色不对称作为独立性来源比专业化或验证弱，因为底层模型仍然偏置输出，但与 N 份相同角色的副本相比，它确实产生了可测量地更低的错误相关性。OpenClaw 五智能体架构（Lacia/Methode/Satonus/Snowdrop/Kouka）不是对同一问题运行五个智能体——而是对问题的五个不同**阶段**运行五个智能体，具有不对称的成功标准。

```python
from dataclasses import dataclass
from typing import Callable, Optional
import subprocess


@dataclass(frozen=True)
class SpecializedAgent:
    """An agent with a distinct information source — the thing that
    makes its signal independent of the other agents in the pipeline."""
    name: str
    tools: list[str]                          # what external systems can it call
    context_scope: str                        # what slice of context it sees
    verification_fn: Optional[Callable] = None  # hard ground-truth check, if any


def design_independent_pipeline() -> list[SpecializedAgent]:
    """
    BAD:  5 agents running the same prompt with temperature diversity.
    GOOD: agents with non-overlapping information sources — the agreement
          between them actually means something.
    """
    return [
        SpecializedAgent(
            name="code_generator",
            tools=["write_file", "read_codebase"],
            context_scope="full repository context",
        ),
        SpecializedAgent(
            name="test_runner",
            tools=["execute_code", "read_test_results"],
            context_scope="execution environment only",
            # Hard verifier — test pass/fail is ground truth from outside
            # the LLM's token posterior, and therefore uncorrelated with
            # the generator's errors.
            verification_fn=lambda out: subprocess.run(
                ["pytest", out["test_file"]],
                capture_output=True,
            ).returncode == 0,
        ),
        SpecializedAgent(
            name="doc_checker",
            tools=["read_api_docs", "read_spec"],
            context_scope="API docs and specifications only",
        ),
    ]
```

这条管道是可以扩展的。`code_generator`、`test_runner` 和 `doc_checker` 之间的一致是强有力证据，因为三个智能体在字面上看不到相同的信息。不一致是一个可诊断的失败——三个见证者中哪个错了？——而不是统计伪影。

## 真正有效的扩展模式

多智能体系统沿**专业化深度**而非**人数**有效扩展。问题不是"我应该部署多少个智能体？"而是"哪些信息源目前在我的单一智能体的访问范围之外，什么样的智能体设计能提供真正的独立性？"

在实践中，这意味着：设计良好的单一智能体，如果能访问代码执行、文档搜索、网络浏览和验证函数，通常在推理任务上优于五个无差异的智能体——而且成本更低，因为你为工具调用付费而不是为冗余的 LLM 调用付费。只有当五个智能体真正操作非重叠信息时——不同的数据库、不同的工具、不同的验证机制、管道中的不同阶段——五智能体集成才更好。

推论是，添加智能体的成本效益分析不是"边际准确率 vs 边际 API 支出"。而是"**边际独立信息** vs 边际管道复杂度"。如果新智能体不能回答"我拥有其他智能体所没有的什么？"——没有独立的工具、没有独特的上下文、没有不对称的角色——那它就不是一个智能体。它是相关噪声，却按智能体的价格收费。

虚假共识失败在相关错误情况下使情况比单个智能体更糟：集成以高表面确定性失败，这比单个智能体适当校准的不确定性更难被检测和纠正。在任何将置信度作为信任信号的下游循环中——代码执行、自主浏览、面向客户的聊天——自信地断言错误答案都是一个隐患。这些都是当前智能体系统常见的部署场景。

## 具体的设经验法则

从上面推导出四条在代码审查中足够具体可用的规则：

1. **添加智能体之前，先说明其信息足迹。** 如果你无法命名新智能体有而现有智能体没有的工具、上下文切片、角色或验证器，那你没有得到一个智能体——你得到了一个重复样本。

2. **永远不要将集成置信度报告为一致率。** 在相关性下这是系统性错误校准的。要么对保留集追踪 ground-truth 准确率，要么使用验证器从 LLM 家族外部的信息源产生置信度。

3. **优先选择生成器 + 验证器而非生成器 + 生成器。** 每一次都是。验证器增加的独立信息比第二个生成器更多，成本相当或更低，在本质上任何可定义验证器的任务上都是如此。

4. **根据曲线而非计数来评估集成的预算。** 如果基准显示对于你的模型家族集成在 $N=3$ 达到瓶颈，$N=5$ 就是浪费。停止浪费。

统一的观察是多智能体设计的架构单元不是智能体。它是**信息源**。智能体是暴露信息源给对话图的包装器。从信息源角度思考的设计会复合。从围着桌子坐的人数角度思考的设计会达到瓶颈。2025-2026 年的文献在其更诚实的时刻，正是这个领域以高昂代价学到的东西。
