当前推理时扩展的标准方案，是给单一模型分配更多算力：更长的思维链、更深入的推理路径搜索、迭代式自我验证。Claude 的扩展思考模式、OpenAI 的 o 系列、DeepSeek-R1——都遵循这一范式。一个模型，更多 token，更好的答案。

Meta 在 2026 年 4 月推出的 Muse Spark 引入了一种结构上截然不同的方案。它的 "Contemplating" 模式并非让单一模型花更多时间思考，而是生成多个并行智能体，各自独立推理，然后将它们的输出综合为最终答案。推理时的计算预算投入的是广度，而非深度。

这不是一个无关紧要的实现细节。它改变了推理时计算资源的扩展特性、失败模式和经济结构。这一方案能否超越 Muse Spark 实现泛化，取决于推理多样性、综合质量和并行与串行计算成本结构等一系列问题。

## 三种推理模式，三种架构

Muse Spark 内置三种明确标注的模式，代表三种不同的计算分配策略：

**Instant 模式** 运行方式类似标准自回归模型。单次前向传播，无扩展推理，针对延迟优化。这处理的是模型参数量化知识中可直接检索答案的任务，无需多步推理。

**Thinking 模式** 遵循现已成熟的扩展思维链模式。模型在生成输出前先产出内部推理 token。这在架构上类似于 Claude 的思考模式或 o 系列推理链。计算量随推理深度线性增长。

**Contemplating 模式** 则是那个性存在。多个智能体——据报道根据任务复杂度不同为 3-7 个——并行推理问题。每个智能体可能采用不同的推理策略，或聚焦于问题的不同方面。一个综合步骤将它们的输出合并为最终回复。

关键架构问题是：这些智能体是同一模型的不同提示词实例，还是针对不同推理策略专门训练的不同模型变体，抑或完全不同的东西？Meta 的技术文档在此语焉不详，但性能特征表明至少存在某种架构分化——这些智能体表现出真正不同的推理行为，而不仅仅是温度采样带来的随机变化。

## 为何并行推理产生不同结果

机器学习中集成方法的数学论证早已成熟：如果 N 个模型的错误分布部分独立，它们的组合错误率下降速度会比任何单一模型的改进更快。推理领域同样适用这一原理。

考虑一道需要代数运算和几何直觉的数学题。单一模型串行推理时，可能早早锁定代数方法，沿此思路生成 2,000 个 token 后走进死胡同，回溯（如能识别到死胡同），再尝试几何。串行方法可行，但容易陷入沉没成本困境——模型可能持续优化一个次优方法，而非切换赛道。

并行智能体则无此困扰。智能体 A 走代数路线。智能体 B 走几何路线。智能体 C 尝试数值逼近。如果几何路线在 500 个 token 内成功，而代数路线需要 3,000 个 token 的回溯，并行架构在墙上时钟时间内更快找到解，且因为综合步骤看到全部三种方法并能选择最优解，所以识别得更可靠。

实证证据支持这一点。据报道，Muse Spark 的 Contemplating 模式在受益于多样化推理策略的任务上（CharXiv、HealthBench Hard）优于其 Thinking 模式，而在深度串行推理是瓶颈的任务上（部分竞赛编程问题）表现不佳。这种性能特征与集成假设一致。

## 综合问题

并行推理生成 N 条候选推理链。将它们转换为单一正确答案并非易事。

**多数投票** 是最简单的综合策略。每个智能体产出一个最终答案，出现次数最多的答案胜出。这在答案空间离散时（选择题、数值题）有效，但对开放式生成任务失效——没有任何两个智能体会产生完全相同的文本。

**Best-of-N 选拔** 使用奖励模型对每个智能体的输出评分，选择得分最高者。这很有效，但削弱了集成优势——你是在选拔一条推理链，而非整合多条推理链的洞见。

**交叉注意力综合** 更为复杂。一个综合模型同时关注全部 N 条推理链，生成一条综合各方最强推理的回复。这可能是 Muse Spark 实现的方式，因为其 Contemplating 模式输出显示出整合多推理路径洞见的迹象。

综合策略决定了扩展特性：

```
策略           | 计算成本     | 质量扩展     | 失败模式
----------------|--------------|--------------|--------------------------
多数投票        | O(N)         | sqrt(N)      | 相关联的错误
Best-of-N       | O(N) + O(1)  | log(N)       | 奖励模型错误
交叉注意力综合   | O(N) + O(N^2)| ~线性(N)     | 综合模型瓶颈
```

交叉注意力综合最有前景，但引入了自身的扩展瓶颈：综合模型必须处理全部 N 条推理链，如果这些链很长（每条数千个 token），综合步骤本身就会变得昂贵。对于大 N，连接链上的二次注意力成本可能主导总计算预算。

## 经济结构

串行推理时扩展（更长的思维链）有一个简单的成本模型：更多 token 等于更多成本，延迟与生成的 token 数量成正比。并行多智能体推理改变了两个维度。

**吞吐量与延迟的权衡。** N 个并行智能体可以同时运行在 N 块 GPU 上。如果 GPU 吞吐量充足，墙上时钟延迟等于最慢智能体的延迟加上综合时间，与 N 无关。串行推理延迟随总推理预算线性增长。对于延迟敏感的应用，只要有硬件可用，并行方案严格优于串行。

**成本结构取决于利用率。** 如果按 GPU 秒计费，N 个并行智能体的成本是一个智能体的 N 倍。但如果运行在有空闲容量的预留基础设施上，并行智能体的边际成本接近于零。这意味着多智能体推理对于已有过量配置 GPU 集群的组织（超大规模云商、大型 enterprises）是经济理性的，但对于按 token 付费的 API 消费者来说则成本高昂。

**Token 效率实际上可能改善。** 反直觉地，所有 N 个智能体生成的 token 总数可能少于单一智能体回溯的 token 总数。如果三个并行智能体各生成 500 个 token，综合步骤 200 个 token，总计 1,700 个 token。单一智能体尝试方法 A（800 个 token，失败），回溯到方法 B（600 个 token，成功），产出最终答案（200 个 token）共使用 1,600 个 token——但没有多样性收益且延迟更高。

## 与现有研究的关联

推理时的多智能体 deliberation 连接到多条已建立的研究线索：

**思维树（Yao et al., 2023）** 探索了带回溯的分支推理路径。多智能体 deliberation 可以视为思维树，但其分支不共享中间状态。关键区别在于 Contemplating 模式中的智能体维护独立的工作内存，而思维树在单一模型的上下文窗口内运作。

**辩论（Irving et al., 2018）** 提议将对抗性多智能体交互用作对齐机制：两个智能体各执一词辩论，一个人类裁判（或模型）判定胜负。Contemplating 模式并非对抗性的，但综合步骤扮演了裁判的类似角色——评估多个立场并产出裁决。

**智能体混合（Wang et al., 2024）** 证明将响应的不同层路由至不同模型可提升质量。推理层面的多智能体推理是自然延伸：不同模型处理不同推理策略，而非不同响应组件。

**STILL-2（Kim et al., 2025）** 表明训练模型根据任务难度动态分配推理时计算资源可获得持续收益。多智能体 deliberation 将此扩展到跨推理多样性的计算分配，而不仅仅是推理深度。

## 这种架构何时胜出

基于 Muse Spark 及相关系统报告的性能特征，多智能体推理时计算在特定条件下优于串行推理：

**高推理多样性价值的场景。** 存在多种有效方法且最优方法事先并不明确的问题。多种解法的数学推理、多种候选假设的医学诊断、多种架构选项的代码生成。

**答案验证比答案生成更容易。** 当综合步骤能可靠识别最优推理链时（因为验证答案比找到答案成本更低），并行架构获得完整的集成收益。

**延迟预算紧张而推理深度需求高。** 需要深度推理但无法承受串行延迟的场景。实时决策系统、交互式应用，以及每个步骤必须在时间预算内完成的智能体循环。

串行推理仍然胜出的场景：

**深度、顺序依赖的推理链。** 第 N 步依赖第 N-1 步精确输出的问题，无法以有意义的方式并行化推理路径。长数学证明、单一有效路径的多步逻辑推演。

**计算预算有限。** 当只能负担运行一个模型实例时，带单一更长思维链的串行推理比被截断的并行方案更有效。

## 实现示例

对于有兴趣尝试多智能体推理时计算的研究者和实践者，核心架构可用当前工具实现：

```python
import asyncio
from dataclasses import dataclass

@dataclass(frozen=True)
class ReasoningChain:
    agent_id: str
    strategy: str
    tokens: list[str]
    conclusion: str
    confidence: float

async def deliberate(
    problem: str,
    strategies: list[str],
    model_fn,
    synthesize_fn,
) -> str:
    """运行并行推理智能体并综合结果。"""
    tasks = [
        model_fn(
            problem=problem,
            system=f"使用 {strategy} 的方法推理这个问题。"
                   f"逐步展示你的思考过程。",
        )
        for strategy in strategies
    ]
    chains: list[ReasoningChain] = await asyncio.gather(*tasks)
    return await synthesize_fn(problem=problem, chains=chains)

# 用法：为数学问题配置多样化的推理策略
strategies = [
    "代数运算",
    "几何可视化",
    "数值估计与验证",
    "从答案空间反推",
]
answer = await deliberate(problem, strategies, call_model, synthesize)
```

关键设计决策是策略选择（生成哪些推理方法）和综合质量（如何整合洞见）。两者都是活跃的研究领域。

## 接下来是什么

多智能体编排框架（MCP、Agent Teams）与推理时扩展研究的交汇表明，这是一个早期趋势而非一次性实验。如果 Muse Spark 的架构被证明有效，预计在未来 6-12 个月内 Claude、GPT 和 Gemini 会出现类似模式。

更深层的问题是：推理多样性应该通过多个模型实例实现（成本高，多样性高），还是通过单一模型内的架构机制实现（成本低，多样性低）？MoE 架构已经将不同 token 路由到不同专家——将此扩展到将不同 *推理策略* 路由到不同专家组，就能在单一前向传播内实现类似多智能体的多样性。

目前，推理时的多智能体 deliberation 代表了将算力转化为能力的真正架构创新。并行推理的扩展定律与串行推理不同，而理解这些定律将变得越来越重要——随着推理时计算在 AI 部署总成本中占比不断增加。
