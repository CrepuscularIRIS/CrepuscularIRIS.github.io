When Anthropic open-sourced the Model Context Protocol in late 2024, it solved one problem cleanly: give LLMs a standardized way to call external tools. JSON-RPC over stdio or HTTP, a registry of tool schemas, a consistent error model. MCP became the USB-C of AI tool integration — one protocol, every provider.

But MCP was designed for agent-to-tool communication. When autonomous agents need to discover, negotiate with, and delegate to *other agents*, a different protocol is required. Google's Agent-to-Agent (A2A) specification, which reached v1.0 in 2026, fills this gap. Understanding the boundary between MCP and A2A — and where they overlap — is essential for anyone building multi-agent production systems.

## MCP's Scope and Its Boundary

MCP standardizes how an agent invokes a tool:

```
Agent ↔ MCP Client ↔ Transport (stdio/Streamable HTTP) ↔ MCP Server ↔ Tool
```

The MCP server exposes a manifest of available tools with JSON Schema definitions. The agent calls a tool by name with typed arguments. The server returns structured results. Each MCP server is a self-contained integration — a filesystem server, a Postgres server, a GitHub server.

With the 2026 Streamable HTTP transport, MCP servers can be deployed as stateless microservices behind load balancers. Server Cards (`.well-known/mcp-server-card`) enable discovery without connection. The Tasks primitive handles long-running operations. MCP has become production-grade infrastructure.

What MCP deliberately does not handle: multi-turn interactions between two reasoning systems. When Agent A needs Agent B to analyze a dataset, make judgment calls about anomalies, and return a structured report, that's not a tool call. Agent B needs to reason, possibly use its own tools, ask clarifying questions, and produce output that reflects its own decision-making. This is the problem A2A addresses.

## What A2A Actually Specifies

The core concept in A2A is the **Agent Card** — a machine-readable description of an agent's capabilities, published at a well-known endpoint:

```
GET https://sql-specialist.internal/.well-known/agent-card.json
```

```json
{
  "name": "SQL Specialist Agent",
  "version": "1.2.0",
  "description": "Database migration generation and query optimization",
  "url": "https://sql-specialist.internal",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true,
    "stateTransitionHistory": true
  },
  "skills": [
    {
      "id": "generate-migration",
      "name": "Generate Database Migration",
      "description": "Creates migration SQL from schema diff",
      "tags": ["sql", "migration", "schema"]
    },
    {
      "id": "optimize-query",
      "name": "Query Optimization",
      "description": "Analyzes and rewrites slow queries",
      "tags": ["sql", "performance"]
    }
  ],
  "authentication": {
    "schemes": ["oauth2"]
  }
}
```

The Agent Card serves the same function as MCP's Server Card but for agents: a static metadata document that enables discovery and capability matching without establishing a session. A registry can index thousands of Agent Cards. An orchestrator can select the right specialist for a task at runtime by matching skill tags against task requirements.

## The Task Lifecycle

Where MCP tool calls are request-response, A2A defines a stateful task lifecycle:

```
Orchestrator                    Specialist Agent
    │                               │
    ├──[message/send]───────────────→│
    │                               │ (status: "submitted")
    │←──[task_id, status]───────────┤
    │                               │
    │  ... agent reasons ...        │
    │                               │
    │←──[status: "input_required"]──┤
    │   "Which table should be the  │
    │    migration source?"         │
    │                               │
    ├──[message/send (reply)]──────→│
    │   "The users table"           │
    │                               │
    │←──[status: "working"]─────────┤
    │                               │
    │  ... agent continues ...      │
    │                               │
    │←──[status: "completed"]───────┤
    │   {migration_sql, warnings}   │
```

The `input_required` state is what separates A2A from a glorified RPC call. The specialist agent can pause mid-task and request additional information from the orchestrator, then resume with the new context. This enables collaborative reasoning — the orchestrator and specialist jointly converge on the right output through structured dialogue.

The task states — `submitted`, `working`, `input_required`, `completed`, `failed`, `canceled`, `rejected`, `auth_required` — are deliberately similar to CI/CD pipeline states. They're designed for systems that already have infrastructure for monitoring job pipelines.

## MCP + A2A: The Two-Layer Stack

The relationship between MCP and A2A is complementary, not competitive:

```
┌─────────────────────────────────────────────────┐
│  Orchestrating Agent (Claude Code)               │
│                                                   │
│  Uses A2A to delegate to specialist agents        │
│  ┌──────────────┐  ┌──────────────────────────┐ │
│  │ A2A Client   │  │ MCP Client               │ │
│  │ (agent-to-   │  │ (agent-to-tool)          │ │
│  │  agent)      │  │                          │ │
│  └──────┬───────┘  └──────────┬───────────────┘ │
└─────────┼──────────────────────┼─────────────────┘
          │                      │
          ▼                      ▼
┌─────────────────┐  ┌──────────────────────────┐
│ SQL Specialist  │  │ Filesystem MCP Server    │
│ (A2A Agent)     │  │ (read, write, search)    │
│                 │  └──────────────────────────┘
│ internally uses │  ┌──────────────────────────┐
│ MCP for its     │  │ Git MCP Server           │
│ own tools ──────┤  │ (commit, diff, log)      │
│                 │  └──────────────────────────┘
└─────────────────┘
```

The orchestrator uses A2A to talk to other agents and MCP to talk to tools. The specialist agents internally use MCP for their own tool access. This layering is clean: A2A handles the reasoning-level protocol (task delegation, multi-turn interaction, capability negotiation) while MCP handles the execution-level protocol (tool invocation, resource access).

## Building a Heterogeneous Swarm

The practical value of this two-layer stack: it enables multi-model, multi-provider agent systems without custom middleware for every integration.

```python
from dataclasses import dataclass, field
from typing import Any
import httpx


@dataclass(frozen=True)
class AgentCard:
    """Parsed A2A Agent Card."""
    name: str
    url: str
    skills: tuple[dict[str, Any], ...]
    auth_scheme: str = "oauth2"


@dataclass(frozen=True)
class TaskResult:
    """Result from an A2A task delegation."""
    task_id: str
    status: str
    output: dict[str, Any] | None = None
    requires_input: str | None = None


class A2AOrchestrator:
    """Route tasks to specialist agents discovered via Agent Cards."""

    def __init__(self, agent_urls: list[str], auth_token: str):
        self._agents: dict[str, list[AgentCard]] = {}
        self._auth_token = auth_token
        self._client = httpx.Client()
        self._discover_agents(agent_urls)

    def _discover_agents(self, urls: list[str]) -> None:
        """Fetch Agent Cards and index by skill tags."""
        for url in urls:
            resp = self._client.get(f"{url}/.well-known/agent-card.json")
            resp.raise_for_status()
            data = resp.json()
            card = AgentCard(
                name=data["name"],
                url=data["url"],
                skills=tuple(data.get("skills", [])),
            )
            for skill in card.skills:
                for tag in skill.get("tags", []):
                    self._agents.setdefault(tag, []).append(card)

    def delegate(self, skill_tag: str, task_input: dict) -> TaskResult:
        """Delegate a task to the best agent matching the skill tag."""
        candidates = self._agents.get(skill_tag, [])
        if not candidates:
            raise ValueError(f"No agent found for skill: {skill_tag}")

        agent = candidates[0]
        resp = self._client.post(
            f"{agent.url}/message/send",
            json={"skill": skill_tag, "input": task_input},
            headers={"Authorization": f"Bearer {self._auth_token}"},
        )
        resp.raise_for_status()
        data = resp.json()
        return TaskResult(
            task_id=data["id"],
            status=data["status"]["state"],
            output=data.get("artifacts"),
            requires_input=data["status"].get("message"),
        )
```

The orchestrator doesn't care that the SQL specialist is a fine-tuned Llama running on AWS and the security scanner is GPT-4o on Azure. Both publish Agent Cards. Both accept A2A task delegations. Both internally use MCP for their tools. The heterogeneity is hidden behind the protocol.

## Observability in the Protocol

A2A includes structured observability by default. Every task delegation generates a trace. Every response includes metadata: which tools were invoked, tokens consumed, confidence scores. The orchestrator can log complete execution traces without instrumenting each agent separately.

```json
{
  "id": "task_7f3a2b",
  "status": {
    "state": "completed",
    "timestamp": "2026-04-16T14:23:00Z"
  },
  "artifacts": [
    {
      "name": "migration.sql",
      "parts": [{"type": "text", "text": "ALTER TABLE users ADD COLUMN..."}]
    }
  ],
  "metadata": {
    "duration_ms": 4230,
    "tokens_consumed": 8420,
    "tools_invoked": ["validate_schema", "generate_migration"],
    "warnings": ["Table 'user_sessions' has no index on 'created_at'"]
  }
}
```

When a four-agent workflow produces wrong output, you inspect the trace and identify which delegation failed, rather than debugging emergent behavior across a black-box system.

## What A2A Doesn't Solve

Honest accounting of the gaps:

**Trust chains.** When Agent A delegates to Agent B, which delegates to Agent C, the trust hierarchy is hard to audit. A2A Agent Cards include authentication requirements but don't specify how trust propagates through delegation chains. In practice, each delegation is a separate auth boundary — Agent B authenticates with Agent C independently of Agent A's permissions.

**Context compression.** When delegating a task, the orchestrator must summarize relevant context for the specialist. This summarization is lossy. A specialist working from a compressed context will sometimes miss constraints that were obvious in the full context. The quality of this compression determines the quality of the delegation — and there's no standard way to validate it.

**Discovery latency.** Fetching Agent Cards and negotiating capabilities at startup adds initialization time. For short tasks where you know exactly which agent to call, this overhead isn't worth it. Direct A2A invocation (skipping discovery) is supported but undermines the dynamic routing benefits.

**Conversation recovery.** If a multi-turn A2A task is interrupted (network failure, agent crash), the recovery semantics are underspecified. The `stateTransitionHistory` capability helps — it allows replaying the task state — but production systems still need custom checkpoint logic.

## The Decision Heuristic

Use A2A delegation when the sub-task requires **reasoning**, not just execution. If the task is "run this SQL query," use MCP — it's a tool call. If the task is "analyze this schema and generate a safe migration," use A2A — the specialist needs to reason about constraints, make judgment calls, and possibly ask for clarification.

Good A2A candidates:
- Domain-specific analysis requiring specialized fine-tuning
- Tasks with ambiguous requirements that benefit from multi-turn clarification
- Long-running work that shouldn't block the orchestrator
- Work requiring tool chains the orchestrator doesn't have access to

Good MCP candidates:
- Deterministic operations (file I/O, database queries, API calls)
- Tasks where input and output schemas are fully specified
- Operations that complete in seconds
- Tools the orchestrator uses directly

The two protocols together form a complete communication stack for agent systems: MCP for the tool layer, A2A for the collaboration layer. The boundary between them — reasoning versus execution — is the architectural decision that determines how cleanly your multi-agent system scales.
