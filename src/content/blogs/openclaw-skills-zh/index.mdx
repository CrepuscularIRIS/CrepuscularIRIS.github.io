Claude Code 的技能系统是这个平台被利用得最少的部分。典型模式：安装一个技能、触发一次、发现有点帮助、然后就搁置了。这种体验与技能实际能做的事情之间有 **10 倍**的差距——而这个差距完全在于技能怎么写，而不是怎么调用。

本文涵盖四个方面：技能在架构上究竟是什么（答案比"一个 Markdown 文件"有趣得多）、什么让技能产生可靠行为而不是泛泛改进、来自 OpenClaw 配置的三个生产模式，以及技能如何与 2026 年 4 月发布的托管代理 session/架具模型交互。

## 技能在架构上是什么

技能是一个被调用时注入 Claude 上下文的 Markdown 文件。目录结构：

```
~/.claude/skills/
├── git-workflow.md       # 通过 /git-workflow 激活
├── code-review.md        # 通过 /code-review 激活
├── python-patterns.md    # 通过 /python-patterns 激活
└── ml-paper-writing.md   # 通过 /ml-paper-writing 激活
```

当你调用 `/code-review` 时，Claude Code 读取技能文件并将其添加到对话上下文之前再响应。模型行为改变是因为它的上下文变了——不是因为模型变了。

这个框架——技能作为**上下文窗口策略文档**——让两件事更清晰：

**第一**，技能与上下文窗口中的所有其他内容直接竞争。一份 3000 词的技能就是 3000 个无法容纳工具结果、检索文件或对话历史的 token。写出好技能所需的纪律部分就是压缩的纪律：在最少的 token 中编码最大的行为特异性。

**第二**，技能通过拼接组合，而不是某种更聪明的机制。当你同时激活三个技能时，模型看到的是三份策略文档，然后才是你的问题。没有语义合并。技能之间的冲突解决方式与任何长上下文中的冲突解决方式相同：由接近程度决定（越近的越优先）和由特异性决定（越具体的越优先）。设计可组合的技能意味着为可预测的冲突解决而设计。

## 托管代理模型中的技能

2026 年 4 月托管代理的发布引入了一种新的思考技能在执行栈中位置的方式。托管代理将代理执行分离为三层：**session**（持久事件日志）、**harness**（无状态执行循环）和 **sandbox**（隔离代码执行环境）。

技能存在于架具层。它们在架具启动时组装成上下文，并注入该 session 中的每个模型调用。在托管部署中，这意味着技能可以版本锁定到 session：架具记录哪些技能版本处于活跃状态，使得重现任何过去 session 的行为成为可能。

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class HarnessConfig:
    """托管代理 session 的架具配置。"""
    model: str
    skills: tuple[str, ...]           # 技能 slug，按加载顺序
    skill_versions: dict[str, str]    # slug → git SHA 或 semver
    max_skill_tokens: int = 8_000     # 注入技能 的 token 预算
    rules_paths: tuple[str, ...] = ()  # 始终活跃的规则（与技能分离）


def assemble_system_context(config: HarnessConfig) -> str:
    """从规则和技能构建系统上下文。
    
    每个 session 调用一次；结果记录在 session 事件流中以确保可复现性。
    """
    parts = []

    # 规则无条件加载
    for rules_path in config.rules_paths:
        parts.append(load_file(rules_path))

    # 技能按顺序加载，在 token 预算内
    token_count = 0
    for slug in config.skills:
        content = load_skill(slug, config.skill_versions.get(slug))
        skill_tokens = count_tokens(content)
        if token_count + skill_tokens > config.max_skill_tokens:
            log.warning(f"技能 {slug} 跳过：会超出 token 预算")
            break
        parts.append(content)
        token_count += skill_tokens

    return "\n\n---\n\n".join(parts)
```

重要意义：在托管代理部署中，技能集是 session 定义的一部分。改变哪些技能处于活跃状态是架具配置的改变，而不仅仅是 prompt 的改变——它被跟踪、版本化且可复现。这使得技能成为代理行为规范的第一类组成部分，而不是"我通常调用这些"这种非正式的存在。

## 质量差距

大多数在网上分享的技能都很弱。它们看起来像这样：

```markdown
# 代码审查
审查这段代码的质量和安全问题。要彻底。
```

这比直接让 Claude 审查代码好一点。强技能在三个具体方面不同：

**1. 编码非显而易见的判断，而非仅仅意图**

弱："检查安全问题。"
强："将含字符串输入的 `eval()` 或 `exec()` 标记为 CRITICAL。将 SQL 字符串拼接标记为 CRITICAL。将状态变更端点上缺失 CSRF token 标记为 HIGH。将认证端点上缺失速率限制标记为 HIGH。忽略 ruff 的 S 系列规则标记的 XSS 风险——那些已自动处理。"

模型知道什么是安全问题。它不知道的是你的严重程度校准、你技术栈的具体失败模式，以及哪些自动化工具已经处理了哪些问题。技能应该编码这些，而不是重复模型已经知道的东西。

**2. 精确指定输出结构**

```markdown
## 输出格式

每个发现：

**[SEVERITY]** `file.py:line` — 描述

严重程度：CRITICAL（阻塞发布）| HIGH（合并前修复）| MEDIUM（本冲刺修复）| LOW（最终修复）

以一句话判决结尾：APPROVE | REQUEST_CHANGES | BLOCK
```

没有明确格式，模型产生有用但不一致的输出。有了格式，你可以程序化地解析、过滤和路由输出——技能的输出成为接口，而不仅仅是文本。

**3. 负面约束与正面指令同样重要**

```markdown
## 不要做这些
- 不要建议重命名变量，除非当前名称确实具有误导性
- 不要为你未修改的代码添加文档字符串
- 不要超出请求范围进行重构
- 不要标记 ruff/black 自动强制的样式问题
- 不要为调用站点不可能发生的情况添加错误处理
```

这些约束通常比正面指令更有价值。它们防止模型做默认会做的无帮助事情——范围蔓延、过度工程、重复自动化工具已经覆盖的工作。

## 三个值得了解的生产技能

### 1. 架具强制执行的 Git 工作流

`git-workflow` 技能编码了约定式提交和分支命名，但更重要的是编码了**验证序列**，该序列必须在任何提交发生前完成。

**提交前序列（强制——任何步骤失败则拒绝）：**
1. `ruff check .` — 修复所有错误
2. `mypy src/` — 修复所有类型错误
3. `pytest -x` — 所有测试通过
4. `git diff --cached` — 确认没有 `.env` 文件或密钥被暂存

仅在全部通过后：构造并执行提交命令。

**这产生的行为**：Claude 会拒绝编写 git 提交，直到确认序列成功完成。这比提醒更强——它是行动的条件。在你快速前进而想跳过检查的 session 中，技能在应该有摩擦的地方产生了摩擦。

在托管代理模型中，这个技能参与 Builder 代理的架具配置。Reviewer 代理有不同的技能集——包括差异感知的代码审查技能，但明确排除提交技能（Reviewer 不会提交）。这种通过技能配置而非 prompt 强制的角色分离，使架构约束变得明确且可审计。

### 2. ML 实验文档

ML 可复现性失败不在设置种子的步骤，而在记录决策的步骤。三个月后，`dropout=0.3` 在配置中，但没有上下文说明这是有原则的还是随意的。

```markdown
# 实验文档技能

每次训练运行后，生成一张实验卡：

**ID**: {timestamp}_{model_name}_{key_variant}
**假设**: 你具体在测试什么？（一句话，可证伪）
**相对于基线的变更**：仅记录变更的部分，不是完整配置
**预期结果**：具体预测，包含方向和幅度
**实际结果**：主要指标值及单位
**解读**：结果是否符合假设？差距意味着什么？
**下一个实验**：这个结果建议尝试什么？

## 反模式
- "用配置 X 运行了实验"不是假设
- 除非正在研究的是运行时间，否则不要记录 wall-clock 时间
- 不要复制完整配置——只有 delta 重要
```

卡片格式在结构上强制假设驱动的实验。你不能把"相对于基线的变更"和"假设"留空却仍然得到一张有用的卡。技能不是提醒你写假设；它使假设成为必填字段。

### 3. 差异感知的代码审查

标准代码审查技能在文件级别工作。这个在**变更**级别工作：

```markdown
# 代码审查——差异感知

## 首先确定上下文：
1. 这是 bug 修复、新功能还是重构？（来自提交消息）
2. 范围是什么？（来自变更文件数和目录）
3. 声明的目标是什么？（来自 PR 描述或 issue）

## 审查范围必须匹配变更范围：
- Bug 修复：聚焦修复和潜在回归。不要批评无关代码。
- 新功能：检查正确性、边界情况和集成点。
- 重构：验证行为保持。检查测试是否覆盖了变更的路径。

## ML/数据代码特别关注：
- 数据泄露：train/val 划分是在特征化之前还是之后发生的？
- 梯度流：是否有不该有的 `.detach()` 调用？
- 种子：随机种子是否在实验开始时设置并记录？
- 路径：硬编码的数据集路径应该是配置值
- 魔数：训练循环中是否有非配置的常量？

## 严重程度
CRITICAL：数据泄露、错误的梯度流、损失函数 bug
HIGH：缺失种子、硬编码路径、未测试的边界情况
MEDIUM：类型注解、命名问题
LOW：样式
```

ML 特定部分编码了真实错误的失败模式：在划分后特征化导致的数据泄露、意外分离张量导致的梯度 bug、错误的归约导致的损失尺度问题。这些不是教科书式的安全模式——它们是从过去调试 session 到未来审查的知识转移。

## 组合与 Token 预算

每个活跃技能都消耗上下文。堆叠十份 2000 词的技能，你的实际对话开始前已经用了 20K token。三个使技能组合有效的实践：

**技能应该狭窄且目的特定。** 不要用一个"Python 开发"技能，而用：
- `python-patterns.md` — 惯用法和反模式（约 400 token）
- `python-testing.md` — pytest 模式（约 300 token）
- `python-security.md` — 安全规则（约 200 token）

每个在相关时被调用，而非默认加载。总 token 成本更低，每个技能可以独立更新或替换。

**钩子可以自动管理加载。** PostToolUse 钩子可以检查当前任务类型（Python 文件被编辑 → 加载 python-patterns，git 操作 → 加载 git-workflow）并自动加载技能。这将技能选择从手动纪律转变为强制执行——架具处理它，而不是依赖用户记住要激活哪些技能。

**正确的心理模型是专业化，而非覆盖。** 技能不是你试图使其全面的知识库。它们是被召入进行特定工作的专家。你不会想让数据库专家参与每次前端审查。你想在他们处理数据库工作时让他们参与，且对他们的决策有具体的关注点。

## 技能开发循环

好技能来自令人沮丧的交互。开发循环：

1. **识别失误**：Claude 产生了无帮助的输出或遗漏了明显的东西。写下它具体遗漏了什么。

2. **编码为规则**："审查 ML 代码时，明确检查 train/val 划分点的数据泄露"——具体的，而非泛泛的。

3. **添加负面约束**："在 bug 修复审查期间不要建议架构变更——那是范围蔓延。"负面约束防止模型用另一种无帮助行为替代。

4. **在实际案例中测试**：在产生令人沮丧交互的实际场景中调用更新后的技能。它能catch到问题吗？

5. **谨慎泛化**：提取原则，而非具体实例。"在实验开始时标记缺失的随机种子"比"检查 train.py 的第 47 行"泛化得更好。

OpenClaw 配置中的 `/skill-improver` 和 `/skill-quality-reviewer` 命令自动化了这个循环的部分——针对测试用例运行技能，并让单独的 session 根据声明的目标批判其输出。在托管代理设置中，这个循环变得更加系统化：技能改进代理可以重放技能表现不佳的过去 session，并测试更新后的技能是否本可以catch到问题。

## 技能在更大图景中的位置

用 Anthropic 的话说，技能是一种原始形式的架具工程：受限的、由策略治理的 AI 行为，模型在明确定义的边界内运作，这些边界定义在模型本身之外。

Claude Code 技能架构使这无需基础设施即可实现。你不需要策略服务器或治理层。你只需要写得好的 Markdown 文件和一致调用的纪律。

这个方法的上限取决于你能多好地将自己的专业知识表达为结构化规则。事实证明，这是一项独立于技能本身的有价值的练习——写技能迫使隐性知识显性化，即使技能从未被调用，这本身就是一半的价值。将"我知道而模型不知道的东西"编码出来的行为，澄清了你实际知道什么，也澄清了哪里是你的知识是模糊的。

借助托管代理基础设施，技能获得了一个新属性：它们是 session 定义的一部分，与模型版本和架具配置一起跟踪。从"一个我有时调用的 Markdown 文件"到"可复现代理 session 的版本化组件"的演变不需要重写任何技能。它只需要认真对待技能已经是的东西：特定类型工作的行为规范。
