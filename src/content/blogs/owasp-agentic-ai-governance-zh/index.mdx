2025 年 12 月，OWASP 发布了首个针对自主 AI Agent 的风险分类框架。2026 年 4 月，微软发布了一个覆盖全部十项风险的运行时安全工具包。这四个月的时间差说明了 Agent 安全领域的演进速度——以及大多数生产部署落后多远。

核心发现：基于 Prompt 的安全防护——即在系统提示中嵌入"请不要做危险的事情"之类的指令——根据微软 Agent Governance Toolkit 发布时附带的基准测试，在对抗性测试下的策略违反率为 26.67%。微软的确定性执行层在相同条件下的违反率为 0.00%。这个差距构成了将 Agent 治理视为基础设施而非指令的完整论据。

## 十项风险

OWASP 的 Agentic AI Top 10 首次超越了通用 LLM 风险（Prompt 注入、幻觉），聚焦于模型获得自主权后出现的特有失效模式——当它们持有凭证、调用 API、修改文件系统、并向其他 Agent 委派任务时。

| # | 风险 | 破坏场景 |
|---|------|---------|
| 1 | 过度代理 | Agent 在超出预期范围的工具权限下行动 |
| 2 | 目标劫持 | 对抗性输入在任务中途重定向 Agent 的目标 |
| 3 | 工具误用 | Agent 以语法正确但语义危险的方式调用工具 |
| 4 | 身份滥用 | Agent 继承人类凭证而未缩减权限范围 |
| 5 | 供应链漏洞 | 执行链中的恶意插件、技能或 MCP 服务器 |
| 6 | 不安全代码执行 | Agent 生成的代码在沙箱不足的环境中运行 |
| 7 | 记忆投毒 | 持久化上下文存储被注入对抗性内容 |
| 8 | 不安全的 Agent 通信 | Agent 间消息缺乏认证或完整性验证 |
| 9 | 级联故障 | 一个 Agent 的错误在多 Agent 管道中传播 |
| 10 | 失控 Agent | Agent 追求与运营者意图不一致的目标 |

其中几项风险在单轮 LLM 交互中不存在。目标劫持（风险 2）要求 Agent 跨步骤维护状态——无状态的聊天机器人没有可被劫持的目标。记忆投毒（风险 7）要求持久化存储。级联故障（风险 9）要求多 Agent 拓扑结构。

## 为什么 Prompt 级安全防护失败

2025-2026 年 Agent 安全的默认做法是在系统提示中添加指令："永远不要执行破坏性命令"、"调用工具前始终验证参数"、"拒绝违反安全策略的请求"。

这种方法有三个结构性弱点：

**概率性遵从。** LLM 以概率方式遵循指令。26.67% 的违反率不是某个特定模型的 bug——这是使用自然语言指令作为安全边界的固有属性。在对抗性压力下（精心构造的用户输入、多轮上下文操纵），指令遵从会退化。安全指令与对抗性内容在模型的注意力中竞争，而模型没有确定性机制来始终选择安全指令。

**上下文长度导致的旁路。** 随着 Agent 对话增长，系统提示中的安全指令离模型的注意力窗口越来越远。一条在 2K Token 上下文中可靠工作的安全指令，在 50K Token 时可能失效。执行长周期任务的 Agent——恰恰是最需要强安全防护的 Agent——对此最为脆弱。

**缺乏执行分离。** 判断某个行动是否安全的实体，与决定执行该行动的实体是同一个。没有关注点分离。在安全工程中，最小权限原则要求执行层处于主体无法绕过的位置。

## 微软 Agent 治理工具包

微软的 Agent Governance Toolkit（AGT），2026 年 4 月 2 日发布，MIT 许可证，采用了根本不同的方法：在 Agent 框架与 Agent 行动之间插入确定性策略执行层。

架构包含七个核心组件：

**Agent OS**——策略引擎。Agent 的每一个操作（工具调用、文件写入、API 请求、Agent 间消息）在执行前都必须通过策略引擎。策略以声明式规则表达，而非自然语言指令。引擎以确定性方式评估——一条策略要么允许要么阻止操作，没有概率性的灰色地带。

**Agent Mesh**——Agent 间通信安全。当 Agent 向其他 Agent 委派任务时，Mesh 层执行认证、授权和消息完整性验证。

**Agent Runtime**——动态执行环。借鉴操作系统设计：Agent 在特权环中执行，内环拥有更多能力，外环更少。外环中的被攻破 Agent 无法影响内环操作。

**Agent SRE**——Agent 系统的可靠性工程。断路器、重试预算、超时执行、自动降级。

**Agent Compliance**——自动化治理验证。针对 EU AI Act（2026 年 8 月生效）、Colorado AI Act（2026 年 6 月生效）等法规框架生成合规评分。

**Agent Marketplace**——插件生命周期管理。Ed25519 签名和清单验证，解决供应链风险。

**Agent Lightning**——强化学习训练治理。确保 RL 训练的 Agent 不会在训练中获得超出策略边界的能力。

## 架构对比

Agent 安全的核心设计问题是执行发生在哪里：

```
Prompt 级安全:
  用户 → Agent (安全在系统提示中) → 工具
  └─ Agent 决定是否遵从安全指令
  └─ 同一实体: 决策者 = 执行者

内核级执行:
  用户 → Agent → 策略引擎 → 工具
  └─ Agent 无法绕过策略引擎
  └─ 独立实体: 决策者 ≠ 执行者
```

这直接映射到系统安全中的经典模式：引用监控器（Reference Monitor）。引用监控器必须满足：(1) 始终被调用，(2) 不可篡改，(3) 足够小以便验证。AGT 的策略引擎满足全部三条。

性能开销为亚毫秒级。对于单次 LLM 推理需要 500ms-2s 的 Agent 工作负载，添加微秒级的策略检查可以忽略不计。

## 风险映射：AGT 如何应对每项 OWASP 风险

确定性执行如何具体应对每项风险：

**目标劫持** — 策略引擎中的语义意图分类器检测 Agent 当前行动是否偏离其声明的目标。如果 Agent 被分配"重构认证模块"的任务却试图修改支付处理代码，策略引擎会阻止该操作——无论模型的推理链说了什么。

**工具误用** — 能力沙箱限制每个 Agent 可以访问哪些工具。MCP 安全网关对工具调用参数进行类型检查和范围验证，超出模型自检能力所能提供的范围。

**身份滥用** — 基于去中心化标识符（DID）的身份系统，配合行为信任评分。每个 Agent 获得唯一的加密身份，而非继承人类操作者的凭证。信任分数随时间衰减，在异常行为时重置。

**供应链漏洞** — Ed25519 插件签名和清单验证。加载到 Agent 执行环境中的每个插件必须有可追溯到已知发布者的有效签名链。

**记忆投毒** — 跨模型验证内核（Cross-Model Verification Kernel），采用多数投票机制。当内容写入 Agent 的持久化记忆时，多个独立模型验证内容是否为对抗性注入。需要多数同意才能提交写入。

**失控 Agent** — 环隔离、信任衰减和自动化终止开关。如果 Agent 的行动持续偏离其策略边界，SRE 层通过警告、限流和最终终止逐级升级。

## 仍然缺失的部分

AGT 解决了运行时执行的缺口，但几个问题仍然开放：

**测试 Agent 策略。** 声明策略容易；知道你的策略集是否覆盖了实际风险面很难。目前没有类似属性测试的方法来验证 Agent 策略配置的完备性。

**跨组织 Agent 交互。** AGT 假设你控制完整的 Agent 技术栈。当你的 Agent 与外部 Agent 交互时，治理边界变得模糊。跨组织 Agent 信任是一个未解决的问题。

**监管节奏。** EU AI Act 的高风险 AI 义务于 2026 年 8 月生效。监管框架的演进速度快于合规工具的适配速度。"法规要求什么"和"工具能验证什么"之间的差距将持续数年。

## 更大的模式

Agent 治理正在重复十年前云安全的轨迹：先是"信任应用"阶段（Prompt 级安全），然后转向基础设施级执行（策略引擎、服务网格、身份系统）。26.67% 的 Prompt 级安全失败率是这一代的 SQL 注入——一类因为执行发生在错误层次而存在的漏洞。

2026 年正在浮现的最小可行 Agent 安全技术栈：

1. **Agent 资产清单** — 知道存在哪些 Agent 及其能力
2. **运行时策略执行** — 确定性的，而非概率性的
3. **身份绑定** — 每个 Agent 行动绑定到可审计的身份
4. **故障隔离** — 一个 Agent 的失陷不能级联

IDC 预测到 2028 年将有 13 亿 AI Agent。治理正确的时间窗口正在收窄。
