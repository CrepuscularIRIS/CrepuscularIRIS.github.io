The pitch for multi-agent systems is seductive: decompose complex tasks into subtasks, assign each to a specialized agent, and coordinate them. The demo always works. Production rarely does. The gap between demo and production is what I call the coordination tax — the overhead, failure modes, and emergent pathologies that appear when agents must actually cooperate under real constraints.

This post taxonomizes the five coordination failure modes that matter in practice, draws on the emerging patterns from Claude Code's Coordinator Mode, Microsoft Agent Framework 1.0, and the MCP/A2A protocol convergence, and proposes engineering mitigations that are concrete enough to implement.

## The Five Failure Modes

### 1. Cascading Hallucinations

When Agent A produces a plausible but incorrect intermediate result, Agent B consumes it as ground truth. Agent C builds on B's output. By the time the final result reaches the user, the original hallucination has been laundered through multiple "reasoning" steps, each adding apparent confidence.

This is worse than single-agent hallucination because multi-agent systems create an illusion of verification. If three agents "agree" on an answer, operators assume it was cross-checked. In practice, the agents may have independently reinforced the same error, or — worse — the second and third agents may have deferred to the first agent's output because it appeared confident.

**Mitigation pattern: Adversarial verification layers.** Insert a dedicated verification agent that has no access to intermediate reasoning, only the final claim and the original source material. This agent's sole job is to find counterexamples. The key constraint: the verifier must use a different model or a different prompt structure than the generators. If all agents share the same systematic biases, "verification" is just agreement.

```python
class AdversarialVerifier:
    """Verifier that checks claims against source material only."""
    
    def __init__(self, verifier_model: str, generator_model: str):
        # Must be different models to avoid shared biases
        assert verifier_model != generator_model
        self.model = verifier_model
    
    def verify(self, claim: str, sources: list[str]) -> VerificationResult:
        prompt = f"""Given ONLY the following source materials, determine if 
        this claim is supported, contradicted, or unverifiable.
        
        CLAIM: {claim}
        
        SOURCES:
        {chr(10).join(sources)}
        
        If the claim makes any assertion not directly supported by the 
        sources, classify it as UNVERIFIABLE, not as SUPPORTED.
        """
        # Force conservative classification
        return self._classify(prompt)
```

### 2. Context Handoff Failures

Agent A processes 50,000 tokens of context and produces a 500-token summary for Agent B. That 100x compression necessarily loses information. The question is whether it loses the *right* information. In practice, summaries systematically drop edge cases, caveats, and negative results — exactly the information that downstream agents need to avoid errors.

This is the multi-agent analog of the telephone game, but with a specific bias: LLMs compress toward the expected, the central, and the confident. Uncertainty and qualification are the first things dropped.

**Mitigation pattern: Structured handoff schemas with mandatory uncertainty fields.**

```typescript
interface AgentHandoff {
  // What the agent concluded
  conclusion: string;
  // Confidence: 0-1, with mandatory calibration
  confidence: number;
  // What the agent explicitly could NOT determine
  unresolved: string[];
  // Assumptions the conclusion depends on
  assumptions: string[];
  // Raw evidence that supports the conclusion
  supporting_evidence: string[];
  // Raw evidence that contradicts the conclusion
  contradicting_evidence: string[];
}
```

The key fields are `unresolved`, `assumptions`, and `contradicting_evidence`. These force the upstream agent to explicitly represent what it does not know, which preserves the information that free-form summaries systematically drop.

### 3. Coordination Deadlocks

Agent A waits for Agent B's output. Agent B needs clarification from Agent A. Neither proceeds. In synchronous orchestration frameworks, this is a literal deadlock. In asynchronous frameworks, it manifests as indefinite polling or timeout cascades.

More subtle: Agent A claims it needs Agent B's output, but actually has enough information to proceed with a reasonable default. The agent is "playing it safe" by deferring to another agent — a behavior that single-agent systems don't exhibit because there's no one to defer to.

**Mitigation pattern: Timeouts with fallback heuristics.**

```python
async def coordinate_with_fallback(
    agent: Agent,
    dependency: Agent,
    timeout_seconds: float = 30.0,
    fallback_fn: Callable[[], Result] | None = None
) -> Result:
    try:
        result = await asyncio.wait_for(
            dependency.execute(),
            timeout=timeout_seconds
        )
        return result
    except asyncio.TimeoutError:
        if fallback_fn:
            logger.warning(
                f"Agent {dependency.name} timed out after {timeout_seconds}s. "
                f"Using fallback."
            )
            return fallback_fn()
        raise CoordinationTimeout(
            f"Agent {dependency.name} did not respond within {timeout_seconds}s "
            f"and no fallback was provided."
        )
```

Every inter-agent dependency must have a timeout and a fallback. If you cannot define a reasonable fallback, the dependency is too tight and the agents should be merged.

### 4. Objective Misalignment

Each agent optimizes its local objective. Agent A (the code writer) optimizes for passing tests. Agent B (the reviewer) optimizes for code quality. Agent C (the security scanner) optimizes for vulnerability elimination. Locally optimal decisions produce globally suboptimal outcomes: the code writer writes convoluted code that technically passes tests, the reviewer requests a rewrite, the security scanner flags the rewrite, and the system enters an oscillation.

This is a multi-agent coordination problem that game theory has studied extensively. The standard solution — a shared utility function — is hard to implement when agents are LLMs with natural-language objectives.

**Mitigation pattern: Hierarchical authority with explicit priority ordering.**

```yaml
# orchestrator.yaml
agent_priority:
  1: security    # Security constraints are non-negotiable
  2: correctness # Tests must pass
  3: quality     # Code quality is optimized within security and correctness bounds
  
conflict_resolution:
  # When agents disagree, higher-priority agent wins
  strategy: priority_override
  # But log the conflict for human review
  log_conflicts: true
  # And cap the total resolution rounds
  max_rounds: 3
```

The priority ordering must be explicit and static. Dynamic negotiation between agents about who has priority is the coordination tax itself — don't meta-coordinate to resolve coordination.

### 5. Token Cost Explosion

Multi-agent systems multiply token consumption in ways that are not obvious from single-agent baselines. Each agent receives a system prompt (1,000-5,000 tokens). Each inter-agent message includes context that the receiving agent already has in its own system prompt. Shared context is duplicated N times across N agents.

A 3-agent system with 3,000-token system prompts, processing a 10,000-token task with 5 rounds of coordination, consumes approximately:

```
System prompts:        3 × 3,000  =   9,000 tokens
Task distribution:     3 × 10,000 =  30,000 tokens
Coordination messages: 5 × 3 × 2,000 = 30,000 tokens (est.)
Agent responses:       5 × 3 × 1,500 = 22,500 tokens (est.)
                                      ─────────
Total:                                 91,500 tokens
```

A single agent processing the same task might consume 25,000 tokens. The 3.7x multiplier is the coordination tax in token form.

**Mitigation pattern: KVComm (Key-Value Communication).** Instead of passing natural-language messages between agents, share structured key-value pairs that encode the information density of the exchange without the verbosity of prose.

```python
class KVMessage:
    """Structured inter-agent message that reduces token overhead."""
    
    def __init__(self):
        self.data: dict[str, Any] = {}
    
    def set(self, key: str, value: Any, confidence: float = 1.0):
        self.data[key] = {"value": value, "confidence": confidence}
    
    def to_prompt_fragment(self) -> str:
        """Convert to minimal prompt text for receiving agent."""
        lines = []
        for key, entry in self.data.items():
            conf = entry["confidence"]
            val = entry["value"]
            if conf < 0.8:
                lines.append(f"- {key}: {val} [uncertain]")
            else:
                lines.append(f"- {key}: {val}")
        return "\n".join(lines)

# Usage: Instead of passing a 2,000-token prose summary,
# pass a 200-token structured KV message
handoff = KVMessage()
handoff.set("vulnerabilities_found", 3)
handoff.set("critical_files", ["auth.py", "session.py"])
handoff.set("suggested_fix", "rotate session tokens on privilege change")
handoff.set("confidence_in_root_cause", 0.7)
```

Reports from teams using KVComm-style approaches show up to 70% reduction in inter-agent token consumption. The tradeoff: structured messages require a predefined schema, which reduces the flexibility of agent communication.

## The Framework Convergence

April 2026 brought a structural shift in how multi-agent systems are built. Three developments are worth tracking:

**Microsoft Agent Framework 1.0** (early April 2026) unified Semantic Kernel and AutoGen into a single SDK with a graph-based orchestration model. Agents are nodes in a directed graph rather than participants in a conversation. This is a meaningful architectural change: conversation-centric designs (AutoGen's original model) struggle with complex dependency structures because conversation is inherently sequential. Graph-based designs can express parallel execution, conditional branching, and join operations natively.

**MCP and A2A as complementary protocols.** Both Model Context Protocol (tool/resource layer) and Agent-to-Agent Protocol (networking layer) are now Linux Foundation projects. The separation is clean: MCP handles how an agent interacts with tools and data; A2A handles how agents interact with each other. This two-layer architecture means you can swap orchestration frameworks without rewriting tool integrations, and vice versa.

**Claude Code's Coordinator Mode** introduced hierarchical agent management with shared memory and atomic claim approval. The "claim" mechanism is particularly interesting: before an agent modifies a file, it must claim the file, and the coordinator must approve the claim. This prevents the common failure where two agents independently modify the same file and produce conflicting changes. It is a distributed locking mechanism adapted for LLM agent systems.

## When to Use Multi-Agent vs. Single Agent

The coordination tax is real, and it means multi-agent systems are not always the right architecture. Here are the decision criteria I use:

**Use a single agent when:**
- The task fits in one context window (under 100K tokens of relevant information)
- The task requires coherent reasoning across all its parts (writing a paper, debugging a complex interaction)
- Latency matters more than throughput (interactive use cases)
- The failure cost of coordination errors exceeds the benefit of specialization

**Use multiple agents when:**
- The task naturally decomposes into independent subtasks with minimal information sharing
- Different subtasks require genuinely different capabilities (code generation vs. security analysis vs. test writing)
- You need to process more information than fits in one context window
- You can tolerate higher latency in exchange for parallel execution

**The critical test:** can you define the inter-agent communication schema in advance? If you can specify exactly what information each agent needs from the others, multi-agent orchestration will work. If the agents need to have an open-ended "conversation" to figure out what they need, you're paying the coordination tax for something a single agent with a longer context window would handle better.

## Measuring the Tax

If you deploy a multi-agent system, instrument these metrics from day one:

```python
coordination_metrics = {
    "token_amplification_ratio": (
        "total_tokens_consumed / equivalent_single_agent_tokens. "
        "If > 4x, your coordination overhead dominates the actual work."
    ),
    "handoff_error_rate": (
        "Percentage of inter-agent handoffs where the receiving agent "
        "misinterprets or drops critical information. Measure by comparing "
        "upstream agent output against downstream agent's representation of it."
    ),
    "cascade_depth": (
        "Maximum chain length of agent-to-agent dependencies before "
        "reaching a human-verifiable output. Each link is a potential "
        "hallucination amplifier. Target: <= 3."
    ),
    "conflict_resolution_rounds": (
        "Number of back-and-forth exchanges needed to resolve agent "
        "disagreements. If > 2 on average, your priority ordering is wrong."
    ),
    "fallback_trigger_rate": (
        "How often timeout fallbacks are used. High rates (>20%) indicate "
        "agents are regularly failing to coordinate, which means the "
        "decomposition is too fine-grained."
    ),
}
```

Multi-agent systems are a legitimate architectural pattern for tasks that genuinely decompose into loosely-coupled subtasks. They are not a substitute for building better single agents. The coordination tax is the price you pay for decomposition, and like all engineering tradeoffs, it is only worth paying when the benefits of specialization exceed the costs of coordination. In April 2026, the infrastructure is finally maturing — MCP, A2A, graph-based orchestration — to reduce that tax. But the failure modes are structural, and no amount of protocol standardization eliminates them entirely.
