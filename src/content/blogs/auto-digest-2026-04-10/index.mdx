The first two weeks of April 2026 concentrated more architectural decisions about agent infrastructure than most quarters. Three developments stand out because they change deployment constraints, not just feature sets.

## MCP's Protocol-Level Reset

The Model Context Protocol's shift to stateless HTTP transport (SEP-1442) is the kind of change that looks boring on a changelog and restructures how you deploy agent tools. The core move: replace persistent Server-Sent Events connections with standard HTTP request-response, carrying session state in a resumption token per request.

The practical consequence is that MCP tool servers become deployable like any REST API — behind standard load balancers, scaling horizontally without sticky sessions, compatible with serverless runtimes that scale to zero. The previous SSE requirement meant every agent held a persistent connection to every tool server, coupling the agent tier's scale to the tool tier's connection capacity.

Alongside stateless transport, SEP-1686 introduces a Tasks primitive for long-running tool invocations. Instead of blocking a connection while a test suite runs, the tool server returns a task handle that the agent polls. This is the async job pattern from webhook-based APIs, applied to agent tool use.

For teams running more than a handful of agents against shared tool servers, this is the update that unblocks horizontal scaling.

## Microsoft Agent Framework 1.0 Goes GA

Microsoft reportedly shipped Agent Framework 1.0 as a General Availability release in early April, merging Semantic Kernel and AutoGen into a single SDK. The significant architectural addition is native MCP support for tool discovery, with Agent-to-Agent (A2A) protocol support planned for a near-term release.

A2A matters because it addresses the interoperability problem that's kept multi-agent deployments locked to single frameworks. An agent built with LangGraph can discover and communicate with an agent built on Microsoft's framework, provided both implement A2A. The capability advertisement mechanism lets agents broadcast what they can do, enabling dynamic composition rather than hard-coded orchestration graphs.

The release also includes DevUI, a browser-based debugger for visualizing agent message flows in real time — the kind of tooling that signals the framework is targeting production teams, not just prototype builders.

## Benchmark Mutation Quantifies the SWE-bench Gap

Recent work on agent evaluation introduces "benchmark mutation" — a technique that transforms the clean, well-specified GitHub issues in SWE-bench into noisy, underspecified queries that better approximate real-world bug reports. The result: reported performance overestimation exceeds 50% in several tested configurations.

This isn't surprising if you've deployed coding agents in production, but having the number matters. SWE-bench's evaluation gives agents two forms of strong supervision that don't exist in deployment: a clear issue description (the specification is unambiguous) and a test oracle (pass the test suite and you know you're correct). Removing either degrades performance dramatically.

Related work is shifting evaluation from binary pass/fail to trajectory analysis — asking not just "did the agent produce a correct patch?" but "did the agent reason efficiently?" This means measuring tool invocations, dead-end paths, and fallback strategy usage across the agent's execution trace.

If your team is using SWE-bench numbers to calibrate trust in a coding agent, discount those numbers by roughly half for real-world tasks. Then ask whether the agent's reasoning trajectory is efficient, because token cost on inefficient trajectories is where the real production expense lives.

## Smaller Signals Worth Tracking

**Gemini CLI** shipped dynamic sandboxing using gVisor containers, with Plan Mode now default for complex tasks. The sandbox profile adapts to the agent's execution plan — an approach to least-privilege enforcement worth watching.

**Claude Code** added a `/loop` command for recurring task scheduling and previewed Computer Use — GUI-level OS automation from the CLI. The convergence of terminal agents and robotic process automation is happening faster than expected.

**Gemini's latest Pro models** introduced compositional function calling: combining built-in tools (Search, Maps) with developer-defined tools in a single parallel execution step. Reduces round-trips for multi-domain agent queries.

**OpenAI's latest models** introduced delegation patterns for shared enterprise resources, where the model manages distinct identity contexts when operating across team mailboxes. The IAM implications for multi-tenant agent deployments are significant.

## What Connects These

A pattern: the agent models are roughly converging in capability (all frontier models can code, reason, use tools). The differentiation is shifting to infrastructure — how agents connect to tools, how they're sandboxed, how they're evaluated, how they interoperate.

April 2026's developments are infrastructure decisions. MCP's stateless transport, Microsoft's A2A protocol, Gemini's dynamic sandboxing, and trajectory-based evaluation all address the gap between "agent that works in a demo" and "agent that works in a deployment." The models write the code. The infrastructure determines whether that code ships safely.
