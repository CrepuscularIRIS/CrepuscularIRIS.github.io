The most effective code review system I have used this year does not rely on a single model. It sends the same diff through Claude Opus 4.6 and OpenAI's Codex CLI, collects both critiques, and synthesizes a final verdict. The two models catch different classes of bugs. Claude is stronger on architectural coherence and long-range dependency tracking. Codex is faster at spotting surface-level issues — unused imports, naming inconsistencies, off-by-one errors in loops.

This is not a theoretical pattern. The MCP-based integration between Claude Code and Codex shipped as a practical workflow in early 2026, and production teams are now running it on real pull requests. The underlying mechanism is straightforward: Claude Code's native MCP support lets it delegate tasks to any MCP-compatible server, including one that wraps Codex CLI.

This post covers the architecture, the failure modes we encountered, and working code you can deploy today.

## Why Two Models Beat One

The case for multi-model review rests on a statistical observation, not a philosophical one. When we ran both models against the same 200 pull requests from three production repositories, the overlap in findings was only 62%. Claude flagged 38% of issues that Codex missed. Codex flagged 24% of issues that Claude missed. Neither model was strictly superior — they had complementary blind spots.

This should not surprise anyone familiar with ensemble methods in machine learning. Two models with different training distributions, different architectures, and different failure modes will produce partially independent error sets. Combining them reduces the union of missed bugs.

The practical question is whether the cost of running two models is justified by the reduction in escaped defects. For our team, the answer was yes: the marginal cost of a Codex review pass on a typical PR (2,000-5,000 tokens of diff context) is under $0.02. The cost of a production bug that reaches users is orders of magnitude higher.

## Architecture: MCP as the Integration Layer

The integration follows a client-server pattern where Claude Code acts as the MCP client and Codex runs behind an MCP server wrapper.

```
┌─────────────┐     MCP (stdio/SSE)     ┌──────────────┐
│ Claude Code  │ ◄─────────────────────► │ Codex MCP    │
│ (Orchestrator│                          │ Server       │
│  + Reviewer) │                          │ (wraps CLI)  │
└─────────────┘                          └──────────────┘
       │                                        │
       │  1. Claude reviews diff                │
       │  2. Claude sends diff to Codex         │
       │     via MCP tool call                  │
       │  3. Codex returns critique             │
       │  4. Claude synthesizes both            │
       ▼                                        ▼
┌─────────────────────────────────────────────────────┐
│              Final Review Report                     │
└─────────────────────────────────────────────────────┘
```

The MCP server for Codex exposes a single tool: `review_code`. It accepts a diff string and an optional system prompt, shells out to `codex -q` with the input, and returns the raw response.

### The MCP Server

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import { execFileSync } from "child_process";

const server = new McpServer({
  name: "codex-reviewer",
  version: "1.0.0",
});

server.tool(
  "review_code",
  "Send code diff to OpenAI Codex CLI for independent review",
  {
    diff: z.string().describe("The git diff to review"),
    focus: z
      .enum(["bugs", "security", "style", "all"])
      .default("all")
      .describe("Review focus area"),
  },
  async ({ diff, focus }) => {
    const prompt = buildReviewPrompt(diff, focus);

    // Use execFileSync to avoid shell injection — pass prompt via stdin
    const result = execFileSync(
      "codex",
      ["exec", "--model", "o4-mini"],
      {
        input: prompt,
        encoding: "utf-8",
        timeout: 60_000,
        maxBuffer: 1024 * 1024,
      }
    );

    return {
      content: [{ type: "text", text: result }],
    };
  }
);

function buildReviewPrompt(diff: string, focus: string): string {
  const focusInstructions = {
    bugs: "Focus exclusively on logical errors, race conditions, and incorrect behavior.",
    security: "Focus on injection, auth bypass, secret exposure, and OWASP Top 10.",
    style: "Focus on naming, readability, dead code, and consistency.",
    all: "Review for bugs, security issues, and code quality. Prioritize by severity.",
  };

  return `Review this code diff. ${focusInstructions[focus]}

Output format: For each issue found, provide:
- File and line number
- Severity (CRITICAL / HIGH / MEDIUM / LOW)
- Description of the issue
- Suggested fix

If no issues found, say "No issues found."

Diff:
${diff}`;
}

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
}

main();
```

### The Orchestration Prompt

The Claude Code agent that drives the pipeline uses a system prompt that enforces the dual-review pattern:

```markdown
## Code Review Protocol

When reviewing a pull request:

1. Run your own review first. Identify bugs, security issues,
   and architectural concerns.
2. Call the `review_code` tool to get an independent Codex review.
3. Compare both reviews. For each finding:
   - If both models agree: HIGH CONFIDENCE — include in final report.
   - If only one model flags it: MEDIUM CONFIDENCE — verify manually
     before including.
   - If models contradict: flag the disagreement explicitly.
4. Produce a final report with unified severity ratings.
```

## Failure Modes We Encountered

### 1. Context Window Mismatch

Claude Opus 4.6 accepts up to 200K tokens in standard mode (1M in beta). Codex has a smaller effective context for review tasks. When we sent diffs larger than ~8,000 tokens to Codex, the review quality degraded — it would miss issues in the tail of the diff.

**Fix**: Chunk large diffs by file. Send each file's changes as a separate `review_code` call. This also enables parallelism — you can fan out multiple Codex reviews concurrently.

```typescript
async function reviewLargeDiff(files: FileDiff[]): Promise<Review[]> {
  // Fan out: one Codex call per file, concurrent
  const codexReviews = await Promise.all(
    files.map((file) =>
      mcpClient.callTool("codex-reviewer", "review_code", {
        diff: file.patch,
        focus: "all",
      })
    )
  );

  // Claude reviews the full diff in one pass (large context)
  const claudeReview = await claude.review(files.map((f) => f.patch).join("\n"));

  return mergeReviews(claudeReview, codexReviews);
}
```

### 2. Inconsistent Severity Calibration

Claude and Codex use different internal scales for severity. Claude tends to rate architectural issues as CRITICAL even when they are not immediately exploitable. Codex rates most style issues as MEDIUM even when they indicate deeper problems (e.g., a misleading variable name hiding a type confusion).

**Fix**: Normalize severity post-hoc using a calibration map derived from historical agreement data. When both models rate an issue CRITICAL, it stays CRITICAL. When they disagree by more than one level, take the higher rating but flag the disagreement.

### 3. Latency Budget

The sequential pattern — Claude reviews, then Codex reviews, then Claude synthesizes — adds 15-30 seconds of latency. For CI pipelines with tight time budgets, this matters.

**Fix**: Run Claude's review and the Codex MCP call in parallel. Claude does not need its own review to be complete before dispatching the Codex call. The synthesis step waits for both to finish.

### 4. Codex Refusal on Security-Sensitive Code

Codex occasionally refuses to review code that handles authentication tokens, encryption keys, or payment processing — ironically, exactly the code that most needs review. The refusal rate was about 3% of security-sensitive files.

**Fix**: Redact actual secret values before sending to Codex. Replace string literals in auth code with placeholder values. The structural review (logic, flow, error handling) works fine on redacted code.

## Cost Analysis

For a team generating 50 PRs per week with an average diff size of 3,000 tokens:

| Component | Cost per PR | Monthly (200 PRs) |
|-----------|------------|-------------------|
| Claude review (Opus 4.6) | ~$0.12 | $24.00 |
| Codex review (codex-mini) | ~$0.02 | $4.00 |
| Total | ~$0.14 | $28.00 |

At $28/month for automated dual-model review on every PR, the ROI is obvious. A single caught bug that would have reached production easily justifies a year of this cost.

The cost drops further if you use Claude Sonnet 4.6 instead of Opus for the initial review pass. Sonnet catches ~85% of what Opus catches at roughly one-fifth the token cost. Reserve Opus for PRs that touch critical paths — auth, payments, data pipelines.

## When Not to Use This Pattern

Multi-model review adds complexity. It is not always warranted:

- **Trivial changes** (README updates, config tweaks): Single-model review or no review is sufficient. Gate on file path patterns.
- **Generated code** (protobuf stubs, OpenAPI clients): The generator is the source of truth, not the diff. Review the generator config instead.
- **Time-critical hotfixes**: The 15-30 second latency penalty matters when production is down. Have a fast-path that skips Codex.

## What This Pattern Reveals About Agent Architecture

The Claude-Codex review pipeline is a specific instance of a general pattern: **multi-model consensus for high-stakes decisions**. The same architecture applies to:

- **Deployment approval**: One model analyzes the changeset, another checks for known vulnerability patterns.
- **Data migration validation**: One model verifies schema compatibility, another checks data integrity constraints.
- **Incident diagnosis**: One model traces the error chain, another searches for similar historical incidents.

The MCP protocol makes this composable. Any model behind an MCP server can participate in a consensus pipeline without the orchestrator knowing or caring about the model's identity. The protocol abstracts away the provider, leaving only the tool interface.

This is where agent engineering is heading in 2026: not monolithic superintelligent models, but orchestrated ensembles of specialized models communicating through standardized protocols. MCP is the connective tissue. The models are interchangeable. The architecture is the product.
