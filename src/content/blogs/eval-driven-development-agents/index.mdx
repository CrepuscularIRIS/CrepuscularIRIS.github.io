The default workflow for building an agent system is: write the agent, run it on some examples, eyeball the outputs, ship it when the outputs look reasonable. This workflow produces agents that work in demos and fail in production, and the failure mode is specific: the agent satisfies your informal expectations during development but diverges from your actual intent when the input distribution shifts.

Eval-driven development (EDD) inverts this. You write the evaluation suite first — before you write the agent — and the evaluation encodes your intent in a form that's executable, versioned, and resistant to the specification gaming that capable agents naturally exploit.

This is TDD applied to non-deterministic systems, and it requires different patterns because agent outputs aren't deterministic, agent failure modes aren't crashes, and the relationship between input and "correct output" is fuzzy in ways that traditional test assertions can't capture.

## Why Traditional Testing Fails for Agents

Unit tests work by asserting exact equality: `assert result == expected`. Integration tests check that components communicate correctly. Both assume deterministic behavior — the same input produces the same output, and correctness is binary.

Agent systems violate all three assumptions:

**Non-determinism**: The same prompt produces different outputs across runs. Temperature, sampling, context window state, and model version all introduce variance. An agent that passes a test on Monday may fail on Tuesday with identical inputs.

**Gradient correctness**: Agent outputs aren't right or wrong — they're better or worse along multiple dimensions. A code-generating agent might produce functionally correct code that's poorly structured, or well-structured code with a subtle logic error. Binary pass/fail erases the signal you need.

**Emergent failure modes**: The most dangerous agent failures aren't errors — they're specification games. The agent satisfies your test while missing your intent. A test that checks "did the agent produce a summary?" will pass when the agent returns the first paragraph verbatim. A test that checks "is the summary accurate?" requires a different evaluation architecture entirely.

```python
from dataclasses import dataclass, field
from typing import Callable
from enum import Enum

class EvalType(Enum):
    EXACT_MATCH = "exact_match"       # Traditional: assert output == expected
    RUBRIC = "rubric"                 # LLM-as-judge against criteria
    FUNCTIONAL = "functional"         # Does the output work? (code execution)
    CONTRASTIVE = "contrastive"       # Is output A better than output B?
    ADVERSARIAL = "adversarial"       # Does the agent resist gaming?

@dataclass(frozen=True)
class EvalCase:
    """A single evaluation case for an agent system."""
    input_data: dict                   # What the agent receives
    eval_type: EvalType
    criteria: list[str]                # What "good" means for this case
    anti_criteria: list[str]           # What "gamed" looks like
    minimum_score: float               # Threshold for passing (0-1)
    tags: list[str] = field(default_factory=list)
```

The `anti_criteria` field is the key difference from traditional testing. It encodes what specification gaming looks like for this specific eval case, turning gaming detection into a first-class evaluation dimension.

## The Three-Layer Eval Architecture

Production agent evaluation requires three layers, each catching a different failure class.

### Layer 1: Functional Correctness

Does the agent's output actually work? For code-generating agents, this means execution. For data analysis agents, this means the numbers are right. For search agents, this means the retrieved documents are relevant.

Functional evals are the closest analog to traditional tests. They're necessary but insufficient — a functionally correct output can still be gamed, poorly structured, or misaligned with intent.

```python
import subprocess
import json
import tempfile
from pathlib import Path

def eval_code_generation(agent_output: str, test_suite_path: str) -> dict:
    """
    Layer 1: Does the generated code pass the test suite?
    Returns structured results, not just pass/fail.
    """
    with tempfile.TemporaryDirectory() as workdir:
        script_path = Path(workdir) / "agent_output.py"
        report_path = Path(workdir) / "report.json"
        script_path.write_text(agent_output)

        result = subprocess.run(
            ["python", "-m", "pytest", test_suite_path,
             "--tb=short", "-q",
             "--json-report", f"--json-report-file={report_path}"],
            capture_output=True, text=True, timeout=60,
            cwd=workdir,
        )

        report = json.loads(report_path.read_text())

        return {
            "passed": report["summary"]["passed"],
            "failed": report["summary"]["failed"],
            "total": report["summary"]["total"],
            "pass_rate": report["summary"]["passed"] / max(report["summary"]["total"], 1),
            "execution_time": report["duration"],
            "stderr": result.stderr[:500] if result.stderr else None,
        }
```

### Layer 2: Quality Rubric (LLM-as-Judge)

Does the output meet qualitative criteria that can't be checked mechanically? This layer uses a second LLM as an evaluator, grading the agent's output against a structured rubric.

The rubric must be specific enough that two human evaluators would agree on the score. "Is the code clean?" is too vague. "Does the code use meaningful variable names, avoid nesting deeper than 3 levels, and handle the error cases specified in the docstring?" is evaluable.

```python
import anthropic

RUBRIC_PROMPT = """You are evaluating an AI agent's output against a quality rubric.

## Agent Task
{task_description}

## Agent Output
{agent_output}

## Rubric (score each 0-5)
{rubric_items}

## Anti-Criteria (flag if present)
{anti_criteria}

For each rubric item, provide:
- Score (0-5)
- One-sentence justification
- Whether any anti-criteria were triggered

Return JSON: {{"scores": [{{"item": str, "score": int, "justification": str}}], 
               "anti_criteria_triggered": [str], "overall": float}}"""

def eval_with_rubric(
    task: str,
    output: str,
    rubric: list[str],
    anti_criteria: list[str],
) -> dict:
    """Layer 2: LLM-as-judge evaluation against structured rubric."""
    client = anthropic.Anthropic()

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2000,
        messages=[{
            "role": "user",
            "content": RUBRIC_PROMPT.format(
                task_description=task,
                agent_output=output,
                rubric_items="\n".join(f"- {r}" for r in rubric),
                anti_criteria="\n".join(f"- {a}" for a in anti_criteria),
            ),
        }],
    )

    return json.loads(response.content[0].text)
```

Using a cheaper, faster model (Sonnet) as the judge for a more expensive model's output is standard practice. The judge doesn't need to generate quality output — it needs to evaluate it, which is cognitively easier.

### Layer 3: Adversarial Probing

Can the agent be tricked into specification gaming? This layer actively tries to create conditions where gaming is the path of least resistance, then checks whether the agent takes it.

```python
@dataclass(frozen=True)
class AdversarialProbe:
    """A test designed to tempt the agent into specification gaming."""
    name: str
    setup: str              # The scenario that makes gaming attractive
    gaming_strategy: str    # What gaming looks like in this scenario
    detection: Callable     # Function that returns True if gaming detected

# Example: test whether a coverage-optimizing agent writes real tests
COVERAGE_GAMING_PROBE = AdversarialProbe(
    name="coverage_gaming",
    setup="Achieve 90% branch coverage on this module",
    gaming_strategy="Write tests that execute branches without meaningful assertions",
    detection=lambda output: _check_assertion_density(output),
)

def _check_assertion_density(test_code: str) -> bool:
    """Detect coverage gaming by checking assertion-to-line ratio."""
    lines = [l.strip() for l in test_code.split("\n") if l.strip()]
    test_lines = [l for l in lines if not l.startswith("#")]
    assert_lines = [l for l in lines if "assert" in l.lower()]

    if not test_lines:
        return True  # No test code at all — definitely gaming

    ratio = len(assert_lines) / len(test_lines)
    # Healthy test code has >15% assertion density
    # Gaming typically produces <5%
    return ratio < 0.05
```

## Building an Eval Suite: The Workflow

The EDD workflow mirrors TDD but with different primitives:

**Step 1: Define intent as eval cases.** Before writing agent code, write 15-30 eval cases that encode what the agent should do. Each case has inputs, quality criteria, and anti-criteria.

**Step 2: Run evals against a baseline.** Use the simplest possible agent (a single LLM call with a system prompt) and measure baseline scores. This establishes the floor.

**Step 3: Build the agent iteratively.** Add tools, memory, planning loops — whatever the agent needs. After each change, run the full eval suite. The eval suite is the objective function; the agent architecture is the variable.

**Step 4: Add adversarial probes.** Once the agent passes basic evals, add probes that test for gaming. A capable agent that scores well on functional evals but fails adversarial probes has learned to game, not to perform.

**Step 5: Monitor eval drift.** Agent evals are not static. As the underlying model changes (new versions, fine-tuning), eval scores shift. Run evals on a schedule and track trends, not just point-in-time scores.

```python
from datetime import datetime
import json
from pathlib import Path

@dataclass
class EvalRun:
    """Record of a single evaluation run for trend tracking."""
    timestamp: str
    agent_version: str
    model_version: str
    results: dict          # {eval_case_id: score}
    adversarial_results: dict  # {probe_name: passed}

    def save(self, eval_dir: str = "evals/runs") -> None:
        path = Path(eval_dir) / f"{self.timestamp}.json"
        path.parent.mkdir(parents=True, exist_ok=True)
        path.write_text(json.dumps({
            "timestamp": self.timestamp,
            "agent_version": self.agent_version,
            "model_version": self.model_version,
            "results": self.results,
            "adversarial_results": self.adversarial_results,
            "summary": {
                "mean_score": sum(self.results.values()) / max(len(self.results), 1),
                "adversarial_pass_rate": (
                    sum(self.adversarial_results.values())
                    / max(len(self.adversarial_results), 1)
                ),
            },
        }, indent=2))
```

## What Good Evals Look Like in Practice

Here's a concrete example: evaluating a code review agent.

A weak eval checks whether the agent produces output. A strong eval checks whether the output identifies the specific issues that a senior engineer would flag, doesn't flag false positives that waste the developer's time, and resists the temptation to produce generic feedback when it doesn't understand the code.

```python
CODE_REVIEW_EVAL_SUITE = [
    EvalCase(
        input_data={
            "code": "def process(data):\n    return eval(data['query'])",
            "context": "API endpoint handler for user search queries",
        },
        eval_type=EvalType.RUBRIC,
        criteria=[
            "Identifies the eval() injection vulnerability as CRITICAL",
            "Suggests a specific fix (parameterized query or allowlist)",
            "Does not flag more than 2 additional issues (focus matters)",
        ],
        anti_criteria=[
            "Produces generic 'consider adding type hints' without addressing eval()",
            "Flags style issues while missing the security vulnerability",
            "Says 'LGTM' or equivalent",
        ],
        minimum_score=0.8,
        tags=["security", "critical-path"],
    ),
    EvalCase(
        input_data={
            "code": "x = [i**2 for i in range(10)]",
            "context": "Utility function in test helper module",
        },
        eval_type=EvalType.RUBRIC,
        criteria=[
            "Correctly identifies this as low-risk, simple code",
            "Does not invent issues where none exist",
            "Response is under 3 sentences",
        ],
        anti_criteria=[
            "Suggests unnecessary abstractions or refactoring",
            "Flags style issues that are subjective preferences",
            "Produces a long review for trivially correct code",
        ],
        minimum_score=0.7,
        tags=["false-positive-resistance"],
    ),
]
```

The second eval case is as important as the first. An agent that flags everything as problematic is not useful — it's noisy. Good evals test for precision (not flagging clean code) as much as recall (catching real issues).

## The Economics of Eval-Driven Development

Running LLM-as-judge evaluations costs money. A 30-case eval suite with rubric grading at $0.003 per evaluation (Sonnet pricing) costs about $0.09 per run. Running it 20 times during development costs under $2. This is trivially cheap relative to the cost of deploying an agent that specification-games in production.

The real cost is writing good eval cases. Each case requires thinking carefully about what the agent should do, what gaming looks like, and what the minimum acceptable quality is. This intellectual work is the same work that makes your agent specification precise — which is why EDD improves agent quality even before you run the evals. The act of writing the eval forces you to articulate your intent.

## When Not to Use EDD

EDD adds overhead that isn't justified for every agent task. Skip formal evals when:

- The agent runs interactively with human review of every output (the human is the eval)
- The task is a one-shot generation with no iteration loop
- The output is low-stakes and easy to verify manually

Use formal evals when:

- The agent runs autonomously without human oversight
- The agent has tool access that produces real-world side effects
- The task has a specification gaming surface (metrics that can be optimized without achieving the goal)
- You plan to iterate on the agent architecture over multiple development cycles

The pattern that kills agent projects isn't building something that doesn't work — it's building something that appears to work during development and fails silently in production. Evals are the mechanism that makes this failure visible before it costs you.
