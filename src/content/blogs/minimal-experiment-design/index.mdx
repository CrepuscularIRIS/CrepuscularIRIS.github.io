Most ML experiments fail not because the hypothesis was wrong, but because the experiment wasn't designed to distinguish between plausible explanations for the result.

You run a new architecture on CIFAR-10. It scores 2% better than your baseline. Is that because:
- The architectural change actually works?
- You tuned the hyperparameters better for the new variant?
- The baseline was underfit?
- The evaluation has random variance that accounts for 2%?
- All of the above?

Without a designed experiment, you can't separate these. And without separating them, the paper you'd write from this result wouldn't survive scrutiny — because any reviewer who thinks carefully about these alternatives will ask for controls you didn't run.

This post is about how to design experiments that actually answer the question you're asking, before you commit to running them. The framework applies to any architectural hypothesis, but the examples are drawn from residual connection variants — a space where the question "does this improve on standard residuals?" is surprisingly easy to answer badly.

## The Three Hypotheses Behind Every Architectural Comparison

When you propose an architectural change, you're implicitly asserting at least three things:

1. **H1 (Core hypothesis)**: The structural change enables something the baseline cannot do in principle.
2. **H2 (Training hypothesis)**: The new architecture is at least as trainable as the baseline — it converges, doesn't destabilize, and the hyperparameter sensitivity is manageable.
3. **H3 (Cost hypothesis)**: The improvement, if real, is worth the compute, memory, and implementation complexity it adds.

A minimal experiment design has one key condition per hypothesis. An experiment that addresses H1 without H2 and H3 isn't ready for a paper. An experiment that addresses H2 and H3 without H1 is benchmarking, not hypothesis testing.

## Designing for H1: The Core Mechanism

The cleanest way to test a core hypothesis is to construct a task where the proposed mechanism is necessary and the baseline cannot succeed.

For residual connection variants, the core claim is typically something like: *"Dynamic layer aggregation enables the network to route information more efficiently than fixed-weight summation."* The mechanism is selective attention to prior layer outputs based on content.

A task designed to require this mechanism:

```python
class SelectiveDepthDependencyTask:
    """
    A synthetic task that requires selective access to features at specific depths.
    Standard residuals (fixed aggregation) should struggle because they can't
    selectively amplify the relevant depth's contribution.
    """
    def __init__(self, n_layers: int = 12, embed_dim: int = 256):
        self.n_layers = n_layers
        # Signal is injected at a random depth during training
        # The model must learn to retrieve it regardless of when it was injected
        self.signal_depth_sampler = torch.distributions.Categorical(
            probs=torch.ones(n_layers) / n_layers
        )
    
    def generate_batch(self, batch_size: int = 64):
        """
        Input: Gaussian noise + a "signal" injected at one random layer.
        Target: Recover the signal content, regardless of injection depth.
        """
        x = torch.randn(batch_size, self.embed_dim)
        signal_depth = self.signal_depth_sampler.sample((batch_size,))
        signal_content = torch.randn(batch_size, self.embed_dim)  # what to retrieve
        
        return {
            "input": x,
            "signal_depth": signal_depth,
            "signal_content": signal_content,
        }
```

The insight: if the mechanism works, the model should be able to retrieve signal content regardless of which layer it was injected into. A standard residual can only efficiently access the signal if it happens to be in the immediately preceding layers (because distant information gets diluted through the chain of additions). An attention-based residual can query for the signal directly.

This is a **constructed task**, not a real benchmark. That's intentional. Real benchmarks measure many things simultaneously; a constructed task measures one. If your proposed mechanism doesn't work on a task specifically designed to require it, the paper is not ready.

## Designing for H2: Training Dynamics

Architecture papers often underreport training instability. A method that achieves SOTA in the hands of the authors — who spent months tuning it — may be brittle in the hands of practitioners with different hardware, different learning rates, or slightly different data preprocessing.

The minimum H2 experiment is a learning rate sweep:

```python
def h2_training_stability_sweep(
    model_factory,
    learning_rates: list[float],
    seeds: list[int],
    dataset: str = "cifar10",
    epochs: int = 50,
) -> dict:
    """
    For each learning rate and seed combination, train the model and record:
    - Final validation loss
    - Loss at training failure (NaN/Inf) if it occurs, or None
    - Gradient norm statistics at each epoch
    """
    results = {}
    
    for lr in learning_rates:
        for seed in seeds:
            torch.manual_seed(seed)
            model = model_factory()
            optimizer = torch.optim.AdamW(model.parameters(), lr=lr)
            
            train_losses, grad_norms = [], []
            failed_at_epoch = None
            
            for epoch in range(epochs):
                loss, grad_norm = train_epoch(model, optimizer, dataset)
                
                if torch.isnan(torch.tensor(loss)) or torch.isinf(torch.tensor(loss)):
                    failed_at_epoch = epoch
                    break
                    
                train_losses.append(loss)
                grad_norms.append(grad_norm)
            
            results[(lr, seed)] = {
                "final_loss": train_losses[-1] if train_losses else None,
                "failed_at_epoch": failed_at_epoch,
                "grad_norm_trajectory": grad_norms,
            }
    
    return results
```

What you want to see from an H2 sweep: the proposed architecture is stable across at least the same range of learning rates as the baseline. An architecture that requires lr=1e-5 when the baseline works fine at lr=1e-3 is not a practical contribution — it's a research toy.

The gradient norm trajectory is more informative than the final loss. A model that trains at 1/10th the learning rate because its gradient norms are 10× larger has a hidden hyperparameter dependency that will bite production users.

## Designing for H3: Compute/Accuracy Pareto

No architecture contributes to the field if it's 1% better but 3× slower. The H3 experiment must characterize the tradeoff.

The right framing is a **Pareto frontier**: for a given compute budget (measured in FLOPs, wall time, or GPU-hours), what is the best accuracy achievable by each architecture variant?

```python
from dataclasses import dataclass
from typing import Callable

@dataclass(frozen=True)
class ExperimentPoint:
    model_variant: str
    flops_per_forward: float  # measured, not estimated
    params: int
    val_accuracy: float
    training_hours: float     # on your specific hardware, stated explicitly

def compute_pareto_frontier(
    experiment_points: list[ExperimentPoint],
) -> list[ExperimentPoint]:
    """
    Returns the subset of points where no other point is better on
    both accuracy and flops simultaneously.
    """
    pareto = []
    for p in experiment_points:
        dominated = any(
            q.val_accuracy >= p.val_accuracy and 
            q.flops_per_forward <= p.flops_per_forward and
            q != p
            for q in experiment_points
        )
        if not dominated:
            pareto.append(p)
    return sorted(pareto, key=lambda x: x.flops_per_forward)
```

A contribution is on the Pareto frontier if, at some compute budget, it achieves higher accuracy than all alternatives. If the proposed architecture is Pareto-dominated everywhere — slower *and* worse than some combination of baseline variants — it's not a contribution, regardless of what happens on any single benchmark.

## The Four-Cell Design Matrix

A minimal complete experiment has four cells:

| | Synthetic Task (mechanism test) | Real Benchmark |
|---|---|---|
| **Baseline** | Accuracy on mechanism-specific task | CIFAR-10 / LM perplexity / downstream task |
| **Proposed** | Same task with proposed architecture | Same benchmarks |

The synthetic task tests H1 (mechanism). The real benchmark tests generalization. The baseline numbers establish that both are measured fairly.

What this matrix catches that single-benchmark evaluations miss: a proposed architecture that shows large gains on real benchmarks but no improvement on the synthetic mechanism test is probably benefiting from an implicit inductive bias in the benchmark, not from the stated mechanism. The paper claim and the empirical support are misaligned.

## Practical Rules for Controlling Variance

Architectural comparison experiments are routinely run with insufficient controls for variance. The minimum:

**Match the parameter count.** If the proposed architecture adds parameters (e.g., attention projection layers in attention residuals), the baseline should have the same parameter budget used differently (e.g., deeper or wider). Parameter differences explain performance differences more often than the mechanism does.

**Use identical data pipelines.** Any difference in data augmentation, normalization, or loading between variants introduces a confound. The safest approach is to implement both variants in a single training script that shares all data pipeline code.

**Report variance over seeds.** A single run with one random seed is not a result. Report mean ± std over at least 3 seeds. For a 1-2% difference to be meaningful at 95% confidence, you typically need 5+ seeds on most benchmarks.

**Measure gradient norms during the comparison run.** Not in a separate experiment — during the actual comparison run. Gradient norms reveal whether the architectures are training in comparable regimes or whether one is getting a soft learning rate advantage.

## What a Complete Write-Up Shows

The reporting standard for an architectural paper should include:

1. The synthetic task result (H1: mechanism confirmed or refuted)
2. The learning rate sweep result (H2: stability characterized, range stated)
3. The Pareto frontier plot (H3: efficiency tradeoff quantified)
4. A confound section: explicitly state what the experiment does *not* control for, and why you believe those confounds don't explain the result

Section 4 is the most valuable and the most commonly omitted. An honest confound section is stronger than no confound section — it shows the authors understand their experiment's limits, which is a prerequisite for reviewers trusting their conclusions.

The goal is not to run more experiments. It's to run the minimum experiments that make the hypothesis and the evidence clearly legible to a skeptical reader. In most architectural papers, that means fewer but better-designed experiments than are typically submitted.
