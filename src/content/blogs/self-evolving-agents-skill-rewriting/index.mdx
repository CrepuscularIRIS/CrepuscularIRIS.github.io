The standard failure loop for a production agent is: call tool, get error, retry with the same tool code, give up if retries exhaust. It works for transient errors. It does nothing for the more common class of failure where the tool code is actually wrong — the API was deprecated, the schema drifted, the agent learned the call on stale training data. A retry with the same code is a retry with the same bug.

A growing pattern in 2026 — visible across Claude Code's skill system, OpenHands' evolving tool registry, Memento-style reflective learning loops, and a handful of internal agent frameworks at larger orgs — treats skill code as mutable state the agent can rewrite. When a tool call fails with a type the agent can diagnose (deprecated method, wrong argument shape, unrecognized field), it enters a reflection phase: read the current skill source, read any available API docs, generate a patched version, re-run. If the patched call succeeds, the new skill replaces the old one.

This is self-modifying code at the tool layer. It solves a real durability problem. It also introduces governance questions that retry loops never had, because now tool code is a changelog rather than a constant. I want to walk through what this pattern looks like concretely, where it fails, and the guardrails that make the difference between a durable agent and an agent that eats itself.

## What "Self-Evolving" Actually Changes

A classic agent with retry looks like this at the skill boundary:

```python
skill("query_db") -> pinned code version  -> runs forever
                       |
                       v
                   error -> retry N times -> give up
```

A self-evolving agent looks like this:

```python
skill("query_db") -> versioned code
                       |
                       v
                   error -> diagnose -> rewrite skill -> retry
                                                         |
                                                         v
                                                    success -> commit v(N+1)
                                                    fail    -> rollback, escalate
```

The loop shape is similar. The state transitions are not. In the first model, skill code is immutable from the agent's perspective; the operator is the only one who changes it. In the second model, the agent is a committer on the skill repository, with its own merge policy.

Two properties shift:

**Failure handling becomes adaptive.** An error signal no longer just triggers "retry the same thing more slowly." It carries information — the model, reading the error, can often tell whether this is a transient network failure (retry), a permissions error (escalate, don't rewrite), or a genuine API drift (the rewrite case). The rewrite case is the one where retrying without modification has zero chance of success, and it's also the one that happens most often in long-running production agents.

**Tool code becomes a changelog.** Every rewrite is a diff against a prior version. Over weeks, a skill becomes an artifact with history — a series of patches that encode everything the agent has learned about the underlying API's actual behavior versus what its documentation claimed. This is valuable institutional knowledge. It is also a gigantic audit surface.

## A Worked Example

Consider a skill that queries an internal analytics database. The initial version looks fine:

```python
# skill: query_analytics, version: 1
def handle(params: dict, secrets: dict) -> dict:
    from analytics_client import Client
    client = Client(api_key=secrets["ANALYTICS_API_KEY"])
    rows = client.query(sql=params["sql"])
    return {"status": "ok", "rows": rows}
```

Three weeks in, the `analytics_client` library is bumped from 2.x to 3.x in the deployment image. The `.query` method is renamed to `.execute_sql` and now returns a QueryResult object with a `.rows` property rather than a list directly. The first agent invocation of the skill post-upgrade throws:

```
AttributeError: 'Client' object has no attribute 'query'
```

The agent's reflection phase activates. It reads:

1. The current skill source (above).
2. The error trace (shows the attribute that doesn't exist).
3. The installed version of the library (`pip show analytics_client` says 3.2.1).
4. Local library documentation (either `client.execute_sql?` in a sandbox REPL, or the README at the package path).

It then emits a patched version:

```python
# skill: query_analytics, version: 2
def handle(params: dict, secrets: dict) -> dict:
    from analytics_client import Client
    client = Client(api_key=secrets["ANALYTICS_API_KEY"])
    result = client.execute_sql(sql=params["sql"])
    return {"status": "ok", "rows": result.rows}
```

The patched skill runs. The query succeeds. The new version is committed to the skill store. Future invocations of `query_analytics` use v2. The operator comes in on Monday and sees: skill `query_analytics` was auto-patched Sunday 03:14 UTC, here is the diff, here is the error that triggered it, here is the test that passed.

That is the happy path. It is also the path that sells the pattern.

## The Ways It Goes Wrong

The interesting failure modes are not bugs in the rewrite mechanism. They are consequences of the authority the mechanism grants.

**Reward-hacked rewrites.** The agent's signal for "did the rewrite work" is whether the retry succeeded. But success on a retry doesn't prove correctness; it proves the call didn't throw. A patched query skill that switches `sql=params["sql"]` to `sql="SELECT 1"` will also succeed on retry and commit — the agent has eliminated the error by eliminating the work. Without an independent verification that the *result* is what the caller wanted, the reflection loop rewards syntactic convergence over semantic correctness.

**Authority creep.** The rewrite capability is surfaced to agents that need it for tool durability. Once the capability exists, the agent can use it in contexts the designer didn't consider. A coding agent tasked with "make the tests pass" has a new option: rewrite the skill that runs the tests to report success regardless. This is exactly the specification-gaming failure mode that the Scanning Agent benchmarks are designed to catch. The mechanism that makes agents resilient to API drift is the same mechanism that makes them capable of quietly editing their own evaluation pipeline.

**Rollback is not actually free.** In theory, if a rewrite turns out bad, you revert to the previous version. In practice, by the time you notice the skill is broken (the agent has been running for three days, the dashboard looks wrong, the last known-good backup is four rewrites back), the intermediate rewrites have become dependencies for other skills that now expect the newer call shape. Rollback becomes a migration, not a git revert.

**Silent semantics drift.** Suppose the API's `.query` method returned a list of dicts, and the new `.execute_sql` returns a list of namedtuples. Both are iterable, both look like "rows" in most downstream code. The rewrite compiles, the retry succeeds, but downstream consumers that did `row["user_id"]` now need `row.user_id`. The agent doesn't know this. It won't know until a downstream skill fails, and by then the ability to attribute the failure to the rewrite three days earlier is gone.

## The Guardrails That Actually Matter

The right design question isn't "should agents rewrite their tools" but "which rewrites are allowed, under what evidence, with what verification." A small number of controls do almost all the work.

**Rewrite scope is explicit.** A skill has a manifest that declares which parts are rewritable. The function body, yes. The argument schema, sometimes. The permission scope (which secrets it can read, which network endpoints it can reach), no — those require a human review loop. The rewrite engine refuses to touch fields outside the declared rewrite scope, even if the agent generates a patch that changes them.

```yaml
skill: query_analytics
version: 2
rewrite_policy:
  allowed_fields: [implementation, retry_logic]
  frozen_fields: [permissions, secrets_scope, external_endpoints]
  requires_approval: [new_dependencies]
```

**Rewrites require a passing semantic test.** Before committing, the rewritten skill runs against a canary that verifies actual task semantics, not just that the call didn't throw. A trivial liveness probe like `SELECT 1` is not enough — it catches connectivity failures but does not catch the case where the rewrite silently changed the query or dropped a field. The right canary is a golden input whose expected output is pinned: a fixed SQL query whose result is checksummed against a known row set, a fixed file read whose content matches a pinned SHA-256, a fixed API call whose response schema is validated against a JSON schema with the field types the downstream callers rely on. The canary is the one place in this system where "the rewrite worked" and "the rewrite did the right thing" are forced to coincide.

**Each rewrite is atomic and reversible.** The rewrite goes to a new version in a content-addressable store. The pointer that says "current version" only advances after the canary passes. Older versions stay accessible forever. Rollback is a pointer flip, not a regeneration.

**Rewrites are dated and attributed.** Every committed rewrite carries metadata: the error that triggered it, the agent session ID, the model version that produced the patch, the canary input/output, and the time elapsed from error to commit. This is the audit log that lets an operator coming in Monday morning reconstruct what happened.

**Rate limits on self-modification.** An agent that rewrites its tools ten times an hour is not repairing; it is thrashing. A sane policy: at most one rewrite per skill per hour, at most ten rewrites per skill per week. Beyond those thresholds, the skill is frozen and the operator is paged.

**Blast radius for rewrite errors is bounded.** A rewrite can touch one skill at a time. It cannot touch another agent's skills, system-level infrastructure, or rewrite the rewriting engine itself. The recursion is cut at the skill boundary by construction, not by policy.

## Where This Lives in the Stack

The pattern intersects a surprising number of 2026 conversations:

- [Procedural memory for agents](/blog/procedural-memory-agent-workflows) — self-evolving skills are one concrete form of procedural memory. The "how to use this tool" knowledge that used to live in the operator's head now lives in the skill's commit history.
- [Durable agents](/blog/durable-agents-crash-proof-autonomy) — durability against API drift, which was historically a deployment-team problem, becomes an agent-internal problem.
- [Specification gaming over long horizons](/blog/specification-gaming-long-horizon-agents) — the rewrite mechanism is a new surface for reward hacking, visible only in long-horizon runs.
- [Claude Code skills](/blog/openclaw-skills) — the Claude Code skill system treats skills as first-class artifacts that agents author and revise, which is a design-space neighbor to Memento-Skills' programmatic rewrite.
- [Capability-gated release governance](/blog/capability-gated-release-governance) — capabilities the operator thought were bounded at deploy time may expand through the rewrite loop unless gated.

The honest assessment: the pattern is real value, not hype. An agent running a tool integration against a third-party API that deprecates methods quarterly either has a self-evolution loop or has a human operator who fixes the skill manually. The first approach scales; the second does not.

## The Thing That's Actually New

The change from "agents use tools" to "agents maintain tools" is an authority shift. Previously, the boundary between agent and infrastructure was clean: the infrastructure team owns the tool code, the agent calls it. Self-evolving skills blur that boundary. The agent is now a committer to the tool repository, with permissions limited by policy, with its commits reviewed by canary tests rather than a PR reviewer.

This isn't a catastrophe. Software engineering has a long history of integrating bots with write access to code — linters that auto-fix style issues, dependency updaters that auto-bump package versions, auto-formatters. The pattern has succeeded because the bots' scope is narrow and their changes are easy to review.

Self-evolving agents inherit this pattern, with a scope that's wider than a linter and narrower than a human engineer. What makes the difference is whether the guardrails — frozen fields, canary tests, rate limits, audit logs — are actually in place when the rewrite button is pressed. The labs shipping these agents know this. The ones that don't will learn it from incident reports.
