A recurring pattern in AI engineering: a capability that works at small scale gets harder and harder to manage as you add more of it. More parameters improve language models until they don't. More data helps until it introduces noise. More agents — the current frontier — are presumed to improve task performance by parallelizing intelligence.

A March 2026 analysis formalizes why this presumption fails mathematically — and the argument has direct consequences for anyone architecting multi-agent pipelines.

## The Core Claim

The MIT result can be stated plainly: without introducing new exogenous information into the system, a team of $N$ agents is typically dominated by a single well-designed agent. "Dominated" has a precise meaning here — the expected utility of the collective output is bounded above by the expected utility of the best individual agent in the group, absent a mechanism that genuinely aggregates independent information.

This is uncomfortable to hear if you've been building multi-agent pipelines. But the mathematics follow directly from a few first principles.

## Why Independent Agents Can't Aggregate Intelligence

Consider what happens when you send the same problem to five agents and then aggregate their outputs. The aggregation can only create value if the agents are providing genuinely independent signal about the answer. Independence requires that the agents' reasoning paths have low mutual information — they're effectively running different random experiments on the same question.

In practice, they're not. Language models share training distributions. Given the same problem statement, models from the same family — even with temperature-added diversity — tend to sample from similar regions of their output distribution. The result is correlated errors, not independent estimates.

For agents with per-agent accuracy $p > 0.5$ and pairwise error correlation $\rho$, majority voting accuracy without correlation ($\rho = 0$) is:

$$P(\text{majority correct}) = P\left(\text{Binomial}(N, p) > N/2\right)$$

With positive correlation, the effective number of independent votes $N_{\text{eff}}$ shrinks. A first-order approximation under exchangeable pairwise correlations is:

$$N_{\text{eff}} \approx \frac{N}{1 + (N-1)\rho}$$

When $\rho$ approaches 1 (fully correlated agents), $N_{\text{eff}} \approx 1$ regardless of $N$. Adding more agents doesn't help. You have one vote, repeated $N$ times.

```python
import numpy as np
from scipy.stats import binom, norm

def majority_vote_accuracy(n_agents: int, per_agent_accuracy: float,
                            correlation: float) -> float:
    """
    Estimate majority vote accuracy under pairwise error correlation.
    
    rho=0: exact binomial.
    rho>0: N_eff approximation (N / (1 + (N-1)*rho)) fed into a normal
           approximation. Underestimates at small N; treat as indicative.
    """
    p = per_agent_accuracy
    if n_agents == 1:
        return p  # Trivially exact; single agent accuracy is p

    if correlation == 0.0:
        # Exact: P(strictly more than half correct)
        return float(binom.sf(n_agents // 2, n_agents, p))

    # Correlated: effective-N normal approximation
    n_eff = n_agents / (1 + (n_agents - 1) * correlation)
    mean = n_eff * p
    std  = np.sqrt(n_eff * p * (1 - p))
    return float(1 - norm.cdf(n_eff / 2, loc=mean, scale=std))


print(f"{'N':>4} | {'rho=0 (exact)':>14} | {'rho=0.3 (approx)':>17} | {'rho=0.7 (approx)':>17}")
print("-" * 60)
for n in [1, 3, 5, 10, 20]:
    print(f"{n:>4} | {majority_vote_accuracy(n,0.7,0.0):>14.3f} "
          f"| {majority_vote_accuracy(n,0.7,0.3):>17.3f} "
          f"| {majority_vote_accuracy(n,0.7,0.7):>17.3f}")
# Run this to see the actual values.
# Key pattern: at rho=0.7, accuracy barely improves past N=3 because
# N_eff converges to ~1.4 as N grows — the plateau is the point.
```

At high correlation ($\rho = 0.7$), $N_{\text{eff}}$ saturates near 1.4 as $N \to \infty$ — adding agents provides negligible benefit. At $\rho = 0$, scaling from $N=1$ to $N=20$ raises accuracy from 0.70 to roughly 0.95 (exact binomial). The correlated regime is nearly flat by comparison. LLMs from the same family on the same prompt have empirically high inter-agent correlations — the independent case rarely applies in practice.

## The False Consensus Amplification Problem

The independence failure compounds into something worse: correlated agents can collectively arrive at wrong answers with higher confidence than any individual agent would.

This is the false consensus failure mode. When five agents all make the same reasoning error (which is likely if they're correlated), the aggregation — whether by majority vote, LLM synthesis, or averaging — interprets the agreement as high-confidence signal. The multi-agent system doesn't just fail; it fails confidently.

```python
from dataclasses import dataclass
from typing import Any
import anthropic

client = anthropic.Anthropic()

@dataclass 
class AgentVote:
    reasoning: str
    answer: str
    confidence: float

def run_agent_panel(question: str, n_agents: int = 5) -> dict:
    """
    Run multiple agents on the same question.
    Illustrates how correlated agents create false consensus.
    """
    votes = []
    
    # Same prompt to all agents — high correlation expected
    for i in range(n_agents):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=512,
            temperature=0.8,  # Some diversity, but model still correlates
            messages=[{
                "role": "user",
                "content": f"{question}\n\nGive your answer and a confidence score 0-1."
            }]
        )
        # Parse response (simplified)
        votes.append(response.content[0].text)
    
    # Aggregate: a naive aggregation treats agreement as confidence
    # This is where false consensus amplifies errors
    unique_answers = {}
    for vote in votes:
        # Simplified: extract answer and count agreement
        # Real implementation would parse structured output
        unique_answers[vote[:50]] = unique_answers.get(vote[:50], 0) + 1
    
    # The "winner" by majority vote — possibly wrong with high apparent confidence
    majority_answer = max(unique_answers, key=unique_answers.get)
    apparent_confidence = unique_answers[majority_answer] / n_agents
    
    return {
        "majority_answer": majority_answer,
        "apparent_confidence": apparent_confidence,
        "n_agreeing": unique_answers[majority_answer],
        "total_votes": n_agents,
        # Warning: apparent_confidence is inflated if agents are correlated
    }
```

The aggregation layer has no way to distinguish "five agents agree because they independently computed the same answer" from "five agents agree because they're samples from the same flawed distribution." For this reason, raw agreement counts are misleading metrics for multi-agent system quality.

## When Multi-Agent Systems Actually Work

The MIT analysis isn't an argument that multi-agent systems are useless. It's an argument that they need a specific property to generate value: **exogenous information injection**.

Agents that bring different context to the problem — different tool access, different knowledge bases, different specializations — can generate genuine independence. This is the architectural principle that distinguishes effective multi-agent systems from naive parallelism.

**Specialization as independence source**: An agent with read access to a codebase and an agent with read access to API documentation have different information. Their reasoning will be genuinely independent on questions that span both. This is what LangGraph's state-machine model exploits — agents don't just run in parallel; they operate on different slices of shared state, contributing information the others don't have.

**External verification as independence source**: A code execution agent that runs code and returns concrete outputs provides a hard signal uncorrelated with the generator's language distribution. This is why the generator-verifier pattern (AlphaProof, o-series, best-of-N with unit tests) works better than ensembling unconstrained generators. The verifier is in a different information class.

```python
from dataclasses import dataclass
from typing import Callable
import subprocess

@dataclass(frozen=True)
class SpecializedAgent:
    """An agent with a distinct information source — the source of genuine independence."""
    name: str
    tools: list[str]          # What external systems can this agent access?
    context_scope: str        # What slice of context does it have?
    verification_fn: Callable  # Hard verifier, if available

def design_independent_pipeline(task: str) -> list[SpecializedAgent]:
    """
    Good multi-agent design: agents with non-overlapping information sources.
    
    BAD: 5 agents running the same prompt with temperature diversity
    GOOD: agents with distinct tools/context/verification mechanisms
    """
    return [
        SpecializedAgent(
            name="code_generator",
            tools=["write_file", "read_codebase"],
            context_scope="full repository context",
            verification_fn=None  # No self-verification
        ),
        SpecializedAgent(
            name="test_runner",
            tools=["execute_code", "read_test_results"],
            context_scope="execution environment only",
            # Hard verifier: test pass/fail is ground truth, not correlated with generator
            verification_fn=lambda output: subprocess.run(
                ["pytest", output["test_file"]], capture_output=True
            ).returncode == 0
        ),
        SpecializedAgent(
            name="doc_checker",
            tools=["read_api_docs", "read_spec"],
            context_scope="API documentation and specifications only",
            verification_fn=None
        ),
    ]
    # These agents have genuinely independent information: 
    # generator knows code, runner knows execution, checker knows spec
    # Their agreement on a solution actually means something
```

**Role asymmetry as independence source**: The OpenClaw five-agent architecture (Lacia/Methode/Satonus/Snowdrop/Kouka) doesn't run five agents on the same problem — it runs five agents on five different *phases* of the problem. Satonus reviews what Methode built, using different success criteria. Snowdrop researches what Kouka needs to decide. The asymmetry is the point.

## The Scaling Mode That Actually Works

Multi-agent systems scale effectively along *specialization depth*, not *headcount*. The question to ask isn't "how many agents should I deploy?" but "what information sources are currently outside my single-agent's access, and which agent design would provide genuine independence?"

Practical implication: a single well-designed agent with access to code execution, documentation search, and a verification function usually outperforms five undifferentiated agents on a reasoning task. The five-agent system is only better when the five agents genuinely operate on non-overlapping information — different databases, different tools, different verification mechanisms.

The false confidence failure makes it worse than a single agent in the correlated error case: it fails with high apparent certainty, which is harder to detect and correct than a single agent's uncertainty.

The practical design principle: every agent in a multi-agent system should be able to answer "what do I know that the other agents don't?" If the answer is "nothing — I'm just another sample from the same distribution," the agent is not adding value. It's adding correlated noise.
