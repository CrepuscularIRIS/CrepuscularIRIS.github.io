当自主智能体链路在生产环境中失效时，直觉反应是增加日志记录。每一步都会发出一个 JSON 事件，每个事件都存入某个存储，链路出问题后有人查询日志，试图重建事件经过。这能撑一周。到第二周，日志量已经难以承受。到第三周，有意义的失败已经淹没在有意义的成功之中——一条"看起来正常"的轨迹，因为每一步都返回了 200，实际上在第七步静默失败了：下游智能体收到了垃圾数据，但根本不知道应该投诉。

更深层的问题在于：日志回答了错误的问题。日志告诉你*系统做了什么*。你真正需要知道的是*系统是否做了正确的事*，而"正确的事"是链路的属性，而非任何单个步骤的属性。为此，你需要验证门控——在链路执行的特定节点上必须满足的离散、可测试属性，以及使判决可事后审查的证据制品。

本文描述了一种基于门控的验证模式，我在过去一个月里将它运行在一条多智能体链路的生产环境中。该链路驱动自主博客维护、代码审查和研究综合。该模式与语言无关，其机制可迁移到任何长时间运行的智能体系统。具体来说，它把"智能体跑了"这种模糊表述变成了结构化的陈述："在以下时间点，以下属性成立，以下制品可以证明。"

## 门控的形态

验证门控由三部分组成：**名称**、**谓词**和**证据束**。名称是你写在验证报告里的东西。谓词是一个布尔函数，它消费链路的状态并返回判决。证据束是谓词在生成判决时查阅的所有文件、追踪或哈希。

```python
from dataclasses import dataclass, field
from enum import Enum
from pathlib import Path

class GateVerdict(Enum):
    PASS = "pass"
    FAIL = "fail"
    PENDING = "pending"

@dataclass(frozen=True)
class Evidence:
    description: str
    artifact_path: Path | None = None
    sha256: str | None = None

@dataclass(frozen=True)
class Gate:
    name: str
    description: str
    required: bool = True

    def evaluate(self, state: dict) -> tuple[GateVerdict, tuple[Evidence, ...]]:
        raise NotImplementedError

@dataclass(frozen=True)
class GateReport:
    gate: Gate
    verdict: GateVerdict
    evidence: tuple[Evidence, ...]
    notes: str = ""
```

关键的约束是：证据束必须是*具体的*。"调度器跑了"不是证据。"调度器的 cron 日志 `/var/log/cron.log` 在 16:13Z 和 16:43Z UTC 各包含一条记录，退出状态均为 0"是证据。区别在于前者只是断言了一个主张，而后者允许他人独立验证。

这听起来很迂腐。确实很迂腐。但这也是我在验证系统中施加的最有用的单一约束。每次我写出一个满足于主张而非制品的门控，主张最终都会在生产环境中变成错误的——调度器"跑了"意味着 Python 进程启动了，但实际任务因为缺少某个环境变量而提前退出了，而门控检查的日志是前一次运行的日志。真实的制品不会像摘要那样撒谎。

## 我实际验证的链路

该模式运行的具体链路：一个桥接进程监听 StepFun 用户事件，将其路由到本地控制平面，控制平面再将任务分派到五个智能体 profile 之一，每个 profile 要么直接处理任务，要么为较重的工作启动一个 Claude Code 会话。结果在合并回工作区仓库之前，先经过一个 Codex 风格的代码审计器和一个 Gemini 风格的语义检查器的双重审查。一个心跳 cron 每 30 分钟驱动整个循环运行一次。

该链路有九个我关心的属性：

| 门控 | 属性 |
|------|----------|
| G1 调度器 | 心跳 cron 最近已执行且结果确定 |
| G2 智能体在场 | 所有配置的智能体 profile 均响应了上次健康检查 |
| G3 分派 | 提交的任务到达了目标智能体 profile |
| G4 双重审查 | 两个审查阶段均针对上次完成的任务运行 |
| G5 端到端 | 一个来自桥接的合成事件到达了最终输出 |
| G6 工作区 | 工作区仓库接受了已提交制品 |
| G7 构建 | 工作区的下游构建通过 |
| G8 制品 | 管道发出了其业务制品（这里是博客文章） |
| G9 稳定性 | 链路连续两次运行完成，无静默失败 |

每一条都可以归约为对特定文件、日志或哈希的谓词。G1 读取 `/var/log/cron.log` 并用 grep 匹配预期条目。G5 在最终输出存储中查找合成事件的关联 ID 并验证往返延迟。G9 读取最近两次管道摘要并确认两者中均未出现错误哨兵值。这些都不需要人做判断。

```python
@dataclass(frozen=True)
class SchedulerRanGate(Gate):
    log_path: Path = Path("/var/log/cron.log")
    expected_entries: int = 2
    pattern: str = "pipeline-heartbeat"

    def evaluate(self, state: dict) -> tuple[GateVerdict, tuple[Evidence, ...]]:
        if not self.log_path.exists():
            return (GateVerdict.FAIL, (
                Evidence(description=f"{self.log_path} does not exist"),
            ))
        text = self.log_path.read_text(errors="replace")
        count = sum(1 for line in text.splitlines() if self.pattern in line)
        evidence = (Evidence(
            description=f"{count} matches of {self.pattern!r} in {self.log_path}",
            artifact_path=self.log_path,
        ),)
        verdict = (
            GateVerdict.PASS if count >= self.expected_entries
            else GateVerdict.FAIL
        )
        return (verdict, evidence)
```

无聊的代码。无聊是好事。如果一个验证门控需要巧妙的逻辑来判断自己是否通过，那这个门控迟早会报告假阳性，彻底摧毁你对整个系统的信任。

## 为什么门控优于端到端测试

逐门控验证的直观替代方案是端到端测试：用合成输入运行完整链路，检查输出是否符合预期。这是一种合理的手段，我仍然在运行（G5 本质上就是这样的）。但它本身是不够的，原因有三。

**端到端测试会压缩失败模式。** 一个运行链路并检查最终输出的测试告诉你*有东西出问题了*，但不告诉你*哪里*出问题了。在一条有九个步骤的链路上，调试会话的搜索空间比实际需要的大了九倍。逐门控验证会告诉你 G3 通过了但 G4 失败了，立刻缩小搜索范围。

**端到端测试会隐藏静默成功。** 一条链路可以通过错误的路径产生正确的输出。测试通过了，但系统依赖的是你没有意图也没有授权的行为。逐门控验证能够捕获这种情况，因为中间状态必须符合预期形态，而不仅仅是最终输出。

**端到端测试与非确定性交互很差。** 多智能体链路有合理的非确定性：LLM 采样、重试调度、并行工具调用中的竞态条件。一个断言精确最终输出的端到端测试是过度指定的。一组断言结构属性的门控——"分派日志显示关联 ID 和智能体 profile 之间存在匹配"，而不是"智能体 profile 选择了选项 B"——对允许范围内的变化具有鲁棒性，而对不允许范围内的变化不具有鲁棒性。

综合模式——对结构属性进行逐门控验证加对行为正确性进行端到端测试——是我在生产环境中运行多智能体系统一年后收敛到的结果。每个门控运行耗时秒级；完整套件耗时不到一分钟。端到端测试耗时更长（因为它实际执行链路，需要分钟级），但运行频率较低。

## 证据哈希与"证明"位

将门控从诊断工具转化为验证工具的最后一个模式，是对证据进行内容寻址。每个查阅制品的门控记录评估时该制品内容的 SHA-256 哈希。这就是区分"我检查了日志"和"我能证明检查时日志的内容"的机制。

```python
import hashlib

def hash_artifact(path: Path) -> str:
    h = hashlib.sha256()
    with path.open("rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            h.update(chunk)
    return h.hexdigest()
```

哈希被记录在门控报告中，且可复现：任何人在拥有相同制品的情况下重新运行验证，会得到相同的哈希，或者哈希不同——差异就是有意义的发现。这使得验证报告成为一种可签名、可归档、可事后审计的东西。我将报告存储在与制品哈希本身相邻的只追加日志中，这样一年后我可以确认链路在特定日期表现符合预期，而不需要信任任何摘要。

这比听起来更重要。多智能体链路是那种一个月后相关问题变成"管道是否产生了制品 X，哪个版本的每个智能体产生了其哪部分"的系统。一份列出证据哈希的验证报告给你一个确定性的答案。一份写着"看起来正常"的验证报告给你一场诉讼。

## 门控不是什么

该模式不做的三件事，以及为什么它不是适合的工具。

**门控不能证明语义正确性。** G8 验证博客文章被发出了。它不验证博客文章是否准确、论证是否充分、是否切题。那是 G4 中的双重审查负责的——而且即使是 G4 也只验证审查运行了，而非审查是正确的。语义正确性仍需要对照真实情况进行评估，那是另一门学科。

**门控不是端到端属性测试。** 端到端测试断言：给定输入 I，系统产生输出 O。门控断言：在链路中的点 P，属性 Q 成立。这是不同的形状。一套门控全部绿色的系统，如果门控定义不足，仍然可能端到端产出垃圾。

**门控不是韧性测试。** 一个完美健康的系统上的通过门控，不能告诉你当其中一个智能体变慢、网络分区、或工具返回格式错误的 JSON 时链路会怎样。为此你需要故障注入，那是另一个工作流。门控验证的是快乐路径和结构化失败路径；混沌工程验证的是非结构化失败路径。

## 最小化设置

如果你正在运行多智能体链路且没有类似这样的机制，最小可行设置比看起来要少。一天的工作给你四样东西：

1. 你的链路在其输入和输出之间必须满足的五到十个属性的列表。
2. 每个属性一个谓词，可归约为读取一个具体制品——日志、文件、指标存储中的指标。
3. 一个小型运行器，评估所有谓词并发出 `GateReport`——每个门控一行，包含判决、描述和证据制品路径。
4. 一个检查：门控报告本身作为每个管道运行的制品被提交。

回报是即时的。第一次你的链路表现异常——上线一周内必然会发生——门控报告在几秒内告诉你哪一步负责、它产生了什么制品、以及失败时刻该制品的哈希。四十五分钟的根因分析和三分钟的根因分析之间的差异，通常在于 incident 前是否已部署这套基础设施。

## 为什么这比听起来更有价值

更宏观的论点：**没有验证的自主不是自主，是委托，性质不同。** 委托把工作交给下级并信任下级主动暴露问题。自主把工作交给下级并*根据规范验证结果*。区别在于验证步骤，而在多智能体链路中，验证必须分布在链路内部，而非集中在端点。

那些能够连续数月自主运行的系统——提交到生产环境的编码智能体、决定下一篇读什么的调研智能体、花钱的运营智能体——其运营者能够指着一条门控链说："管道工作的前提是这九件事为真，这里是证明每一条的制品，这里是使证明可复现的哈希。"说不出这一点的系统不是自主的。它们只是未被监督的，而区别很重要。

增加门控很便宜。不增加门控只是暂时很便宜，直到第一次 incident。
