## 引言：为什么需要多个专职 Agent？

单一通用 AI Agent 在面对长周期任务时，面临两个根本性困境：

1. **上下文污染**：实现、审查、研究三类任务混在同一会话中，导致注意力漂移和错误决策
2. **无法止损**：没有外部仲裁者，Agent 容易陷入无限重试或过度工程化陷阱

这不是理论问题。在实际运行中，一个同时负责编码和审查的 Agent 会倾向于"自我合理化"——它不会真正审视自己写的代码，因为它的上下文中充满了自己做出每个决策的理由。审查变成了确认偏差的练习。

OpenClaw 的回答是**职责隔离的 5-Agent 体系**，每个 Agent 有明确的输入/输出契约，通过 30 分钟心跳周期和原子化 Mailbox 通信解耦协作。

---

## 五个 Agent 的角色模型

| Agent | 角色 | 核心职责 | 不做什么 |
|-------|------|----------|----------|
| **Lacia** | 编排/调度 | 任务分解、优先级排序、资源分配 | 不做实现、不做研究 |
| **Methode** | 构建/执行 | 代码实现、测试、构建验证 | 不做架构决策 |
| **Satonus** | 审查/门控 | 代码审查、质量评估、PASS/FAIL 判定 | 不做实现修复 |
| **Snowdrop** | 研究/证据 | 文献调研、数据收集、方案对比 | 不做最终决策 |
| **Kouka** | 交付/封装 | 打包交付物、止损判断、seen 记录更新 | 不做研究、不做实现 |

这种分工不是任意的。核心设计原则是 **"写代码的人不审查代码，做决策的人不收集证据"**。Satonus 永远不修复它发现的问题——修复者不能是审查者，防止自我合理化。Snowdrop 收集方案对比但不做最终选择——证据提供者不应是决策者，避免确认偏差。

### 为什么是五个而不是三个或十个

Agent 数量的选择基于任务分解的最小完备集。软件工程的核心环节是：规划、实现、验证、研究、交付。少于五个意味着某个 Agent 承担多重职责（回到上下文污染），多于五个则增加通信开销且收益递减。

信息论视角：5 个 Agent 之间的通信通道数量是 $\binom{5}{2} = 10$，但 OpenClaw 的 Mailbox 协议通过星型拓扑（所有通信经过 Lacia 路由）将实际通道数降至 4。这是关键的复杂度控制——去中心化通信在 Agent 数量 >3 时迅速变得不可控。

## 心跳调度机制

系统以 **30 分钟为心跳周期**运作。30 分钟不是随意选择——它大约等于一个 Claude Code 会话在密集工作模式下的有效上下文窗口。超过这个时间，上下文压缩开始丢失关键细节；短于这个时间，Agent 没有足够时间完成有意义的工作单元。

每次心跳的执行流程：

```
1. Agent 读取自己的 Mailbox（~/inbox/<agent>.jsonl）
2. 按优先级排序待处理消息
3. 执行最高优先级任务
4. 执行 Cron 任务（如 Blog-Maintenance-Kouka）
5. 若无工作 → 发送 idle_report 给 Lacia
6. Lacia 聚合 idle 信号，决定是否向用户升级
```

**关键设计**：Agent 不主动轮询彼此的状态，只通过 Mailbox 接收消息。这避免了忙等待，也使每个 Agent 的行为完全由其输入决定——给定相同的 Mailbox 内容，Agent 的行为是确定性的。这对调试至关重要：你可以回放任何 Agent 的 Mailbox 历史来复现它的决策路径。

### 心跳同步与错位

五个 Agent 并非同时启动心跳。实际部署中，它们的心跳周期存在有意的**错位（stagger）**：

```
T=0min   Lacia 心跳 → 分发任务
T=5min   Methode 心跳 → 读取任务，开始实现
T=10min  Snowdrop 心跳 → 读取研究请求
T=15min  Satonus 心跳 → 读取审查请求
T=20min  Kouka 心跳 → 读取交付请求
T=25min  空闲缓冲
T=30min  Lacia 心跳 → 新一轮
```

这种 5 分钟错位确保 Lacia 分发的任务在 Methode 启动前已经写入 Mailbox，避免 "读到空信箱" 的竞态条件。代价是一轮完整的 "分发→实现→审查→交付" 流程需要 ~30 分钟而非理论上的 ~5 分钟。这是正确的取舍——可靠性比吞吐量重要。

## Mailbox 通信协议

Mailbox 是系统的通信脊梁，采用 JSONL 格式。每条消息是一个独立的 JSON 对象，追加写入对应 Agent 的收件箱文件。

```jsonl
{"id":"msg-001","from":"lacia","to":"methode","type":"task_assignment","priority":1,"subject":"实现控制平面日志模块","body":{"spec":"...","acceptance_criteria":["日志写入 JSONL","含时间戳和 Agent ID"],"deadline":"2026-04-10T12:00:00Z"},"timestamp":"2026-04-10T10:00:00Z"}
{"id":"msg-002","from":"satonus","to":"lacia","type":"review_verdict","priority":1,"subject":"控制平面日志 PASS","body":{"verdict":"PASS","findings":[],"coverage":"87%"},"timestamp":"2026-04-10T11:30:00Z"}
{"id":"msg-003","from":"kouka","to":"lacia","type":"idle_report","priority":3,"subject":"idle tick","body":{"last_action":"blog-maintenance","next_cron":"2026-04-12T10:00:00Z"},"timestamp":"2026-04-10T10:20:00Z"}
```

### 消息类型契约

| type | 发送方 | 接收方 | 含义 | 优先级 |
|------|--------|--------|------|--------|
| `task_assignment` | Lacia | Methode/Snowdrop | 分配任务 | 1 (高) |
| `review_request` | Lacia/Methode | Satonus | 请求审查 | 1 (高) |
| `review_verdict` | Satonus | Lacia | PASS/FAIL 结论 | 1 (高) |
| `delivery_note` | Kouka | Lacia | 交付完成确认 | 2 (中) |
| `idle_report` | 任意 | Lacia | 空闲心跳报告 | 3 (低) |
| `research_dispatch` | Kouka/Lacia | Snowdrop | 研究任务 | 2 (中) |
| `escalation` | 任意 | Lacia | 异常上报 | 0 (紧急) |

### 为什么是 JSONL 而非数据库

选择 JSONL 文件而非 SQLite 或 Redis 的原因：

1. **Git 友好**：每条消息是一行，diff 清晰可读，可以用 `git log` 追踪通信历史
2. **无依赖**：不需要运行数据库进程，减少故障点
3. **原子追加**：POSIX `O_APPEND` 保证单条消息的写入原子性，多个 Agent 可以安全地向同一文件追加（虽然实际上星型拓扑避免了写冲突）
4. **人类可读**：调试时直接 `cat` 文件即可查看完整通信记录

代价是查询效率——在 Mailbox 消息超过 10,000 条时，逐行扫描会变慢。当前解决方案是每月归档旧消息到 `~/inbox/archive/YYYY-MM/`。

## Cron 驱动的定期任务

除了响应 Mailbox 消息，Agent 还通过 Cron 触发定期任务。以 **Blog-Maintenance-Kouka** 为例：

```
Schedule: 0 10 * * 2,5  (每周二/五 10:00 Asia/Shanghai)
```

执行流程：

```python
# Blog-Maintenance-Kouka 伪代码
async def blog_maintenance_cycle():
    # 1. 收集本周交付物
    deliveries = await read_delivery_log(since=last_tuesday())
    methode_prs = [d for d in deliveries if d.agent == "methode"]
    satonus_passes = [d for d in deliveries if d.type == "review_verdict" and d.verdict == "PASS"]

    # 2. 研究需要时，派发 Snowdrop
    if needs_research(topic_queue):
        await mailbox.send(
            to="snowdrop",
            type="research_dispatch",
            body={"topics": topic_queue[:3], "deadline": "2h"}
        )

    # 3. 审计现有文章
    existing_posts = await audit_blog_posts()
    rewrite_candidates = [p for p in existing_posts if p.quality_score < 0.6]

    # 4. 写新文章
    for topic in selected_topics[:2]:
        draft = await write_post(topic, research=snowdrop_results)
        await save_draft(f"~/blog/drafts/{today()}-{topic.slug}.mdx", draft)

    # 5. 构建验证
    build_ok = await run_build("pnpm build")
    if not build_ok:
        await mailbox.send(to="lacia", type="escalation", body="Blog build failed")
        return

    # 6. 追加交付记录
    await append_to_queue("Blog maintenance completed", posts=selected_topics)
```

这种 Cron + Mailbox 混合模式使 OpenClaw 能够在**事件驱动**（响应任务）和**时间驱动**（定期维护）两个维度上工作。Cron 保证了即使没有外部任务分配，系统也会定期自我维护。

## 止损机制

止损是 OpenClaw 最重要的设计决策之一。在无人值守的 AI Agent 系统中，最危险的失败模式不是崩溃——崩溃是安全的，因为它立即停止。最危险的是**无限循环的善意努力**：Agent 反复重试一个注定失败的任务，消耗 tokens 和时间却没有产出。

Kouka 不只是打包者，更是**止损决策者**。止损规则是硬编码的，不依赖 LLM 的判断：

**触发条件（满足任一即触发）：**

| 条件 | 阈值 | 理由 |
|------|------|------|
| 任务挂起时间 | >24h 无进展 | 超出单任务合理周期 |
| 连续无状态变化 | 2 轮心跳 | Agent 可能在空转 |
| Token 消耗异常 | >50k tokens 无 commit | 投入产出比失衡 |
| 构建连续失败 | 3 次 | 可能是系统性问题而非单次错误 |

**止损动作：**

```
1. 标记任务为 wontfix 或 deferred
2. 记录止损理由（含上下文快照）
3. 通知 Lacia 重排优先级
4. 如果涉及未完成的 Git 分支，创建 WIP commit 保存现场
```

关键设计：止损判断使用**硬编码规则而非 LLM 推理**。如果让 Agent 自己决定"是否应该停止"，它几乎永远会选择继续——因为 LLM 倾向于乐观地评估自己的进展。止损必须是外部的、机械的、不可协商的。

## 与单 Agent 系统的对比

| 特性 | 单 Agent | OpenClaw 5-Agent |
|------|----------|------------------|
| 上下文干净度 | 低（混合职责） | 高（职责隔离） |
| 错误隔离 | 无 | Satonus 门控 |
| 止损能力 | 依赖 prompt | 结构性保证（硬编码规则） |
| 可观察性 | 低（黑盒） | 高（Mailbox 审计日志） |
| 扩展性 | 线性复杂度增长 | 按需增加 Agent 类型 |
| 部署复杂度 | 低（单进程） | 中（需要心跳调度和 Cron） |
| 单任务延迟 | 低（无通信开销） | 高（~30min 完整周期） |

OpenClaw 的 5-Agent 架构不适合所有场景。对于简单的单步任务（"修复这个 typo"），单 Agent 更高效。5-Agent 架构的价值在 **长周期、多阶段、需要质量门控** 的任务中体现——比如每周博客维护、多日功能开发、或需要研究-实现-审查完整流程的项目。

## 当前限制与未来方向

**已知限制：**

- **延迟高**：一轮完整的 "分发→实现→审查→交付" 流程需要 ~30 分钟。对于紧急修复，这不可接受
- **Mailbox 无序保证**：在极端情况下（多个 Agent 同时写入同一个 JSONL 文件），消息可能交错。当前通过星型拓扑缓解，但非根本解决
- **Agent 重启时状态丢失**：心跳之间的中间状态存在于 Agent 的上下文窗口中。如果 Agent 会话意外终止，这些状态丢失。当前通过频繁的 checkpoint commit 缓解

**进行中：**

- 控制平面决策日志——记录 Lacia 的每次调度决策及其理由，用于事后审计
- Agent Mailbox 并发安全测试——模拟 5 Agent 高频写入场景

**计划中：**

- **自适应心跳频率**：根据任务紧急程度动态调整心跳间隔（紧急任务 5min，常规任务 30min）
- **Satonus 审查指标量化**：从二元的 PASS/FAIL 升级为置信度分数，允许 "有条件通过" 的中间状态
- **上下文压缩策略**：在心跳边界自动压缩上下文，保留关键决策点，丢弃中间推理过程
